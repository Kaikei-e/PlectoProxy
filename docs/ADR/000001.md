# WASM Component Model / WIT を拡張モデルに採用し、相補的な二つの構成要素でゲートウェイを構成する

## Status
accepted

## Date
2026-06-20

## Affected Components
- fast path (native Rust) — 接続受付・TLS 終端・HTTP/1.1/2/3・ルーティング・LB・upstream 管理
- extension plane — 各リクエストの判断を担う WASM フィルタ実行基盤（新設）

## Context
ゲートウェイは「カスタムロジックをどこに置くか」に必ずぶつかる。従来の三択——設定/DSL（安全だが表現力に天井）、
本体組み込み再コンパイル（最速だが untrusted を動かせず言語固定・全体を巻き込む）、別プロセス呼び出し
（安全・多言語だが毎リクエスト往復で遅い）——は「プロセス内の速さ・サンドボックスの安全・言語の自由・無停止差し替え」を
同時に満たせない。この四つを同時に満たせる事実上唯一の技術が WASM である。「データプレーンのフィルタは WASM」は
Envoy/proxy-wasm が約10年かけて切り拓き実証した結論であり、本設計はその洞察に多くを負っている。proxy-wasm は
初期の ABI v0.2.1 を対象としており、Component Model への移行は現時点でロードマップ化されていない。River（Pingora ベース）の
WASM 拡張は post-1.0 のスコープにある。現代の Component Model / WIT に立脚し安全に多言語プラグインを差せるセルフホスト型
L7 ゲートウェイは 2026 年中盤時点でまだ少なく、Plecto はその領域を担う。

## Decision
拡張モデルとして **WebAssembly Component Model + WIT** を採用し、Plecto を **相補的な二つの構成要素**で構成する:
**fast path**（性能とポータビリティが価値になる接続・TLS・HTTP・ルーティング・LB・upstream を素の Rust）と、
**extension plane**（安全な隔離と言語自由が価値になる各リクエストの判断を WASM フィルタ）。両者の契約は WIT で型付けする。

検討した代替案と却下理由:
- **legacy proxy-wasm ABI 互換**: 初期 ABI のシム層を抱えると、現代の Component Model を主軸に据える Plecto の設計意図がぼやける。却下。
- **設定 / 独自 DSL のみ**: 任意ロジック・込み入った状態を表現できない。却下。
- **ネイティブプラグイン / 再コンパイル**: untrusted を動かせず、バグが全体を巻き込み、言語が Rust 固定。却下。
- **別プロセス（ext_proc 等）**: 毎リクエストのネットワーク往復が遅い。却下。

本決定はファウンディング設計の Tenet 1 / §1 に対応する。

## Consequences

### Pros
- プロセス内の速さ・サンドボックスの安全・言語の自由・無停止差し替えを同時に得られる。
- 速い経路を素の Rust に保てる（Cloudflare の Pingora 実測: 旧 NGINX 比 CPU 約 70%減・メモリ約 67%減・TTFB p95 80ms 減 が Rust 速い経路の妥当性を裏づける）。
- Envoy と機能数で競わず「現代的で安全な拡張モデル × セルフホスト × データ主権」の一点で価値を出せる。

### Cons / Tradeoffs
- WASM 境界のデータコピーと Component Model のホストコール・オーバーヘッド（後述 ADR で扱う）を負う。
- Component Model / WASI は仕様変動が速く、確定事実と projected を区別し続ける運用負担がある。
- **再検討条件**: 上流 `wasi:http` 等に composable filter の標準が整い Plecto の独自層が薄くできるなら、その方向へ縮小する。
