# plecto:filter ワールドを独自定義し wasi:http の request/response 型を再利用する

## Status
accepted

## Date
2026-06-20

## Affected Components
- filter contract (WIT) — `plecto:filter` ワールド（decision / hooks / host-API の語彙）
- fast path / host runtime — フィルタチェーンの一段としてフィルタを呼ぶ

## Context
フィルタとホストの契約をどの WIT ワールドで表すかを決める必要がある。`wasi:http/middleware` は
「コンポーネント＝完全な HTTP ハンドラ」モデルで、Plecto が求める「ホスト管理のチェーン内の一段」とは粒度が違う。
wasi-http 自身も、HTTP intermediary チェーンの完全実現には Preview 3 timeframe の追加機能が必要と明記している。
一方で polyglot エコシステムとの型互換は維持したい。

## Decision
**独自の `plecto:filter` ワールドを定義**し、その内部で **`wasi:http` の request/response 型を再利用**する。
これにより、decision / short-circuit・init / per-request フック・host-API という Plecto 固有の語彙を自分の粒度で設計しつつ、
`wasi:http` 型の再利用で多言語コンポーネント（Rust/Go/JS/Python）との互換を保つ。契約は contract-first（先に `.wit`、
それから各言語バインディング生成）で進め、ホストと全コンポーネントは同一 WIT バージョンをターゲットにする。

検討した代替案と却下理由:
- **`wasi:http/middleware` 準拠**: 粒度が「完全ハンドラ」で、ホスト管理のチェーン一段に合わない。却下。
- **完全独自型（wasi:http 非再利用）**: polyglot 互換とツールチェーン資産を捨てることになる。却下。

本決定はファウンディング設計の Fork 2 に対応する。

## Consequences

### Pros
- decision / hooks / host-API を Plecto の語彙で設計でき、チェーン一段の粒度を正確に表せる。
- `wasi:http` 型再利用で多言語フィルタ作者のエコシステム互換を保てる。

### Cons / Tradeoffs
- 独自ワールドの維持・バージョニング（semver）の責任を負う。
- **再検討条件**: Preview 3 timeframe で `wasi:http` に composable filter の標準ワールドが追加され要件を満たすなら、
  独自ワールドをその上の薄い拡張へ縮小する。
