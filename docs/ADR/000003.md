# 非同期ファースト契約を採り、stream<u8> ボディと wasm32-wasip2→P3 移行を据える

## Status
accepted

## Date
2026-06-20

## Affected Components
- filter contract (WIT) — ボディは `stream<u8>`、フックは非同期前提
- build toolchain — 当面 `wasm32-wasip2` でビルドし P3 WIT をターゲット

## Context
契約の同期/非同期の基盤を決める。WASI 0.3.0（2026-06-11 リリース）で native async（`async func` / `stream<T>` /
`future<T>`）が Component Model の Canonical ABI に入り、ホストが単一イベントループを全コンポーネントで共有するようになった。
0.2 時代の「コンポーネントごとに独立イベントループが必要で stream/async が合成できない」制約は解消された。ただし
`wasm32-wasip3` はまだ rustc 正式ターゲットになく、ゲスト toolchain 対応は進行中。stream splicing（中間バッファなし直結）は
WASI 0.3 本体に間に合わず 0.3.x の早期フォローアップに先送りされた。

## Decision
契約の基盤を **非同期ファースト**にし、ボディを **`stream<u8>`** で表す。toolchain は移行戦略として
**当面 `wasm32-wasip2` でビルドしつつ P3 WIT をターゲット**し、ゲスト toolchain が `wasm32-wasip3` を正式サポートしたら追従する。

検討した代替案と却下理由:
- **同期ファースト**: 「ボディを流しながら変換する」がチェーン合成で自然に表せず、async が標準化された今は後退。却下（ただし下記再検討条件あり）。
- **`wasm32-wasip3` を即採用**: rustc 正式ターゲット未満でビルド再現性・DX が不十分。時期尚早として却下。

本決定はファウンディング設計の Fork 1 に対応する。確定事実（WASI 0.3 の async ABI）と projected（P3 ターゲット化・stream splicing）を区別して扱う。

## Consequences

### Pros
- フィルタチェーンの合成が型として自然になり、ボディ・ストリーミング変換を表現できる。
- 単一イベントループ共有で、async フィルタ同士が合成可能。

### Cons / Tradeoffs
- stream splicing 先送りのため、当面ボディ変換はホット経路の中間コピーを許容する。
- **再検討条件**: `wasm32-wasip3` の rustc 正式ターゲット化が遅れ async バインディングの DX が実用に達しないなら、
  ヘッダ系フィルタを当面 **同期契約**に留める二段構えを採る。
