# フィルタは worker ごとプール再利用・ステートレスとし、状態はホスト KV(redb) に置く

## Status
accepted

## Date
2026-06-20

## Affected Components
- host runtime — worker thread ごとに `InstancePre` を保持し pooling allocator で再利用。untrusted は per-request 生成にオプトイン
- host-API — KV / counter / token-bucket を redb 永続バックエンドで提供

## Context
フィルタインスタンスのライフサイクルと状態の置き場を決める。prior-art は三分される: Envoy/proxy-wasm（worker thread ごと
永続 VM + root/per-request context 分離）、Spin（リクエスト毎に新インスタンス）、Redpanda（shard ごと 1 インスタンス）。
Wasmtime の `InstancePre`（型チェック・import 解決を事前実施）+ pooling allocator は生成を劇的に速くする
（公式報告で SpiderMonkey.wasm のインスタンス化が約 2ms→約 5µs、約 400倍）。一方、フィルタ・ローカルに状態を持たせると
プール再利用・無停止差し替えと衝突し、状態漏洩リスクも増す。

## Decision
- **インスタンス**: 信頼できる自家製フィルタは **worker thread ごとに `InstancePre` で事前生成 → pooling allocator で再利用**。
  untrusted なサードパーティ・フィルタは **per-request 新規生成**にオプトイン（[[000006]] のゼロ化と併用）。
- **状態**: フィルタは **ステートレス**。レート制限・セッション・キャッシュ等の状態は **ホスト提供の KV / カウンタ host-API**
  に置き、永続バックエンドは **redb（組込みトランザクショナル KV）**。

検討した代替案と却下理由:
- **全フィルタを Spin 型の per-request 新規生成**: 強い分離は得るが全フィルタに生成コストが乗る。trusted には過剰。却下（untrusted のみ採用）。
- **フィルタ・ローカル永続状態（Durable Objects 的）**: プール再利用・無停止差し替えと衝突し状態漏洩リスク。非目標として却下。

本決定はファウンディング設計の Fork 3・Fork 4 に対応する。

## Consequences

### Pros
- trusted フィルタのインスタンス化が極めて速く、スケールする。
- ステートレス化で無停止差し替え・プール再利用・分散スケールが自然になる。
- redb のトランザクショナル性質が host KV の整合に噛み合う。

### Cons / Tradeoffs
- 信頼レベルで生成戦略を分岐するロジックをホストが持つ必要がある。
- ホスト KV 越えのホストコール（async で約 3.5x）がホット・カウンタで効く場合がある。
- **再検討条件**: 許容できないほどホットなカウンタが出たら、ホスト側ネイティブな lock-free カウンタ（count-min sketch 等）を
  fast-path として用意し WIT 越えを回避する（[[000005]] と連動）。
