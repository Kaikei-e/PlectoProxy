# フィルタは worker ごとプール再利用・ステートレスとし、状態はホスト KV(redb) に置く

## Status
accepted

## Date
2026-06-20

## Affected Components
- host runtime — worker thread ごとに `InstancePre` を保持し pooling allocator で再利用。untrusted は per-request 生成にオプトイン
- host-API — KV / counter / token-bucket を redb 永続バックエンドで提供

## Context
フィルタインスタンスのライフサイクルと状態の置き場を決める。prior-art は三分される: Envoy/proxy-wasm（worker thread ごと
永続 VM + root/per-request context 分離）、Spin（リクエスト毎に新インスタンス）、Redpanda（shard ごと 1 インスタンス）。
Wasmtime の `InstancePre`（型チェック・import 解決を事前実施）+ pooling allocator は生成を劇的に速くする
（公式報告で SpiderMonkey.wasm のインスタンス化が約 2ms→約 5µs、約 400倍）。一方、フィルタ・ローカルに状態を持たせると
プール再利用・無停止差し替えと衝突し、状態漏洩リスクも増す。

## Decision
- **インスタンス**: 信頼できる自家製フィルタは **worker thread ごとに `InstancePre` で事前生成 → pooling allocator で再利用**。
  untrusted なサードパーティ・フィルタは **per-request 新規生成**にオプトイン（[[000006]] のゼロ化と併用）。
- **状態**: フィルタは **ステートレス**。レート制限・セッション・キャッシュ等の状態は **ホスト提供の KV / カウンタ host-API**
  に置き、永続バックエンドは **redb（組込みトランザクショナル KV）**。

検討した代替案と却下理由:
- **全フィルタを Spin 型の per-request 新規生成**: 強い分離は得るが全フィルタに生成コストが乗る。trusted には過剰。却下（untrusted のみ採用）。
- **フィルタ・ローカル永続状態（Durable Objects 的）**: プール再利用・無停止差し替えと衝突し状態漏洩リスク。非目標として却下。

本決定はファウンディング設計の Fork 3・Fork 4 に対応する。

## Consequences

### Pros
- trusted フィルタのインスタンス化が極めて速く、スケールする。
- ステートレス化で無停止差し替え・プール再利用・分散スケールが自然になる。
- redb のトランザクショナル性質が host KV の整合に噛み合う。

### Cons / Tradeoffs
- 信頼レベルで生成戦略を分岐するロジックをホストが持つ必要がある。
- ホスト KV 越えのホストコール（async で約 3.5x）がホット・カウンタで効く場合がある。
- **再検討条件**: 許容できないほどホットなカウンタが出たら、ホスト側ネイティブな lock-free カウンタ（count-min sketch 等）を
  fast-path として用意し WIT 越えを回避する（[[000005]] と連動）。

## 実装ノート（確定事項・2026-06-20）

本決定を `plecto-host` に実装した際に固まった具体判断。ADR 000011 の再検討条件
（「ADR 000004 実装時に本定義を pin する」）への応答でもある。

1. **trust 機構は `LoadOptions { isolation: Isolation::{Trusted, Untrusted} }`。**
   ロード時に明示の enum で選ぶ。本番ランタイムの調査（Fastly の reusable / per-request sandbox、
   Spin の per-request 既定、Envoy の `vm_id`/`singleton`）はいずれも分離を「trust スコア」でなく
   **lifecycle モード**で表しており、それに倣い名前は `TrustLevel` でなく `Isolation`。default は
   `Untrusted`（fail-closed）。「**誰が** trusted か」の判定基準は機構と分け、ADR 000006 / OCI 署名へ
   繰り延べたまま（本 ADR は**様式の機構**だけを確定）。

2. **trusted=pooling / untrusted=on-demand を 2 エンジンで持つ。**
   wasmtime では allocation strategy が **`Engine` 単位・不変**（1 Engine で pooling と on-demand を
   混在不可）。よって trusted 用 Pooling エンジンと untrusted 用 OnDemand エンジンを分ける。untrusted の
   OnDemand は毎リクエスト fresh memory ＝**構造的にゼロ化**（slot 再利用が無いので CVE-2022-39393 の
   表面が出ない）。`memory_init_cow` は両エンジンで**有効維持**（CVE の正しい姿勢、ADR 000006）。

3. **trusted は「単一インスタンス直列再利用」に落とす（per-worker sharding は server へ繰延）。**
   本文の「worker thread ごとに `InstancePre` 保持」は per-worker-thread sharding を指すが、fast-path
   server（接続受付・スレッド）がまだ無い。v0.1 は **単一インスタンスを `Mutex` で直列再利用**し、
   `init` をロード時に一度だけ走らせる。これで Tenet 4 の**効果**（init 派生物の常駐）が出る。pooling
   allocator の slot 再利用利得は dealloc が起きて初めて顕在化するため、単一永続インスタンスの現状では
   まだ inert——per-worker sharding 到来時に効く。pool 容量は v0.1 控えめ（64）。

4. **counter は WIT を跨ぐ / token-bucket はホストネイティブ（ADR 000005 整合）。**
   `host-counter` は `increment(key, delta: s64) -> s64`（wasi:keyvalue/atomics と同形）で 1 回の安い
   コール。`host-ratelimit` は **refill とバケット状態を Rust+redb のホスト側**に置き、WIT は薄い
   `try-acquire(key, cost, spec) -> acquire` だけ——超ホット経路のカウントは WASM 境界を跨がない。
   これは本 ADR の「再検討条件（ホットなカウンタは host-native fast-path へ）」を token-bucket では
   **最初から**満たした形。バケット spec は宣言的マニフェスト（ADR 000007）未整備の v0.1 では
   コール内に渡す暫定で、マニフェスト到来時に host 側へ移し `try-acquire(key, cost)` に痩せる。

5. **状態バックエンドは `KvBackend` trait（ADR 000011 の seam）。**
   host-API 実装と lifecycle は具体ストアを名指さず、in-memory（既定）↔ redb の差し替えが局所。
   sync・**非 poison ロック**（parking_lot）。counter / token-bucket の atomicity は redb の single-writer
   write transaction（メモリ実装は lock 内 RMW）で担保。wasmtime 46 で host call が async 化したら、
   redb 実装内だけ blocking pool 越しへ移し、呼び出し側は据え置く。

6. **KV 名前空間に primitive tag を追加。**
   キーは `{filter_id}\u{1f}{tag}\u{1f}{key}`（tag = `k`/`c`/`r`）。ADR 000011 のフィルタ identity 名前
   空間化を primitive 次元へ広げ、同一フィルタの kv `"x"` と counter `"x"` の衝突を断つ。

## 実装ノート（追補・2026-06-20、状態レイヤ durability レビュー反映）

外部レビュー（2026-06-20）が、状態レイヤが最ホット経路で redb の durability/並行性プロファイルと衝突する点を
指摘した。検証の上、以下を確定した。

7. **再検討条件は発火済み——ただし想定と違う理由で。**
   本 ADR の再検討条件は「ホットなカウンタが出たら host-native fast-path へ」。発火の理由は当初想定の
   「WIT 越えのコスト」では**ない**（token-bucket は最初から WIT を跨いでいない、note #4）。実際の律速は
   **redb の既定 `Durability::Immediate`＝毎コミット fsync** と **single-writer 直列化**だった。ADR 群は redb
   single-writer を *atomicity*（正しさ）としてのみ記し *throughput* として書いていなかった——その隙を埋める。

8. **ephemeral primitive はメソッド粒度で `Durability::None` に落とす（fsync を外す）。**
   `KvBackend` は既に primitive ごとにメソッドが分かれている（`set`/`get`/`delete`=durable KV、
   `increment`=counter、`try_acquire`=ratelimit）ので、**trait を割らずに** redb 側で `increment` /
   `try_acquire` の write txn を `Durability::None` にし、durable KV は `Immediate` のまま残した。レート制限/
   カウンタはクラッシュで直近の数増分を失っても無害（ephemeral）で、atomicity は単一 write txn の RMW で保たれる。
   加えて `host-counter.get`（`increment(key,0)`）は **read txn に分岐**し single-writer 列にも commit にも乗らない。
   （redb 4.1.0 の `Durability` は `None`/`Immediate` の 2 値。`Eventual` は当該版に無い。）

9. **single-writer 直列化と sharding は繰延（明記）。** `Durability::None` は fsync 律速を外すが、全フィルタの
   状態書き込みが**単一ライタに直列化**される点は残る。MemoryBackend の単一グローバル `Mutex<HashMap>` も同形。
   解は key-hash シャード（複数 redb DB／シャード化 in-memory ＋スナップショット）だが、fast-path server が並行
   負荷を載せて初めて効くので、その到来時に **(a) per-worker インスタンス sharding・(b) epoch deadline・
   (c) シャード化状態 backend** の三点セットで入れる。`KvBackend` の seam が綺麗なので差し替えは局所に収まる。

10. **bucket decode 破損を fail-closed に統一。** 16 bytes でない格納値は従来 None 経由で「満タン start」＝
    fail-OPEN だった。`bucket_input` で「キー不在（None）＝初見・満タン」と「破損＝空バケット・deny・refill で
    自己回復」を分け、明示の fail-closed 方針に揃えた（tag 名前空間化ゆえ通常運用では踏まないが一貫性のため）。
