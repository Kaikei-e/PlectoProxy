# header-only と body-transform をフィルタ契約で分離し、ホット経路はネイティブに落とす

## Status
accepted

## Date
2026-06-20

## Affected Components
- filter contract (WIT) — 「ボディ非接触」を表現し、ボディ・ストリームをバイパス可能にする
- fast path — body-untouching フィルタのゼロコピー・バイパス、レート制限のホスト・ネイティブ実装

## Context
WASM 税の負わせ方を決める。proxy-wasm 仕様自身が、ペイロード変換はサンドボックス内外の大量データコピーで性能が落ち、
ネイティブ比 2x 未満の劣化を見込むと認めている。さらに Component Model async のホストコールは同期経路でも約 3.5x の
オーバーヘッドを加える（Bytecode Alliance の計測、Cranelift 最適化で 1.0 後に解消見込み）。stream splicing は 0.3.x 先送り。
したがって全リクエストでボディを WASM に通すのは非効率。

## Decision
**ヘッダのみフィルタとボディ変換フィルタを契約レベルで分離**し、判断の基準を
「ユーザー固有のロジック・ポリシー・WAF・認証・書換は WASM／TLS終端・ルーティング・LB・接続プール・グローバル・カウンタはネイティブ」とする。
具体的には: ヘッダのみ操作は小さなコピーで済ませる／ボディに触れないフィルタはボディ・ストリームを**ゼロコピーでバイパス**させる／
レート制限のような超ホット経路は **ホスト・ネイティブ**に落とす。WASM 税はリクエスト判断ロジックにのみ課す。

検討した代替案と却下理由:
- **全フィルタで一律にボディを WASM へ**: コピーが支配的で性能が出ない。却下。
- **全部ネイティブ**: 拡張モデルの存在意義（[[000001]]）を失う。却下。

本決定はファウンディング設計の Fork 6 に対応する。

## Consequences

### Pros
- ボディ非接触フィルタが多い実運用で WASM 税をほぼゼロに抑えられる。
- グローバル・カウンタをネイティブに置くことで超ホット経路の性能を確保。

### Cons / Tradeoffs
- 契約に「ボディ非接触」を表す仕組みと、ホスト側のバイパス判定・実装が必要。
- ネイティブ/WASM の責務境界の判断を設計者が都度行う必要がある。
- **再検討条件**: WASI 0.3.x で stream splicing と caller-supplied buffers（ゼロコピー経路）が入り、
  Component Model 1.0 で同期呼び出しの約 3.5x が解消されたら、ボディ変換フィルタの WASM 化の閾値を下げる。
