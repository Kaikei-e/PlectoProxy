# header-only と body-transform をフィルタ契約で分離し、ホット経路はネイティブに落とす

## Status
accepted

## Date
2026-06-20

## Affected Components
- filter contract (WIT) — 「ボディ非接触」を表現し、ボディ・ストリームをバイパス可能にする
- fast path — body-untouching フィルタのゼロコピー・バイパス、レート制限のホスト・ネイティブ実装

## Context
WASM 税の負わせ方を決める。proxy-wasm 仕様自身が、ペイロード変換はサンドボックス内外の大量データコピーで性能が落ち、
ネイティブ比 2x 未満の劣化を見込むと認めている。さらに Component Model async のホストコールは同期経路でも約 3.5x の
オーバーヘッドを加える（Bytecode Alliance の計測、Cranelift 最適化で 1.0 後に解消見込み）。stream splicing は 0.3.x 先送り。
したがって全リクエストでボディを WASM に通すのは非効率。

## Decision
**ヘッダのみフィルタとボディ変換フィルタを契約レベルで分離**し、判断の基準を
「ユーザー固有のロジック・ポリシー・WAF・認証・書換は WASM／TLS終端・ルーティング・LB・接続プール・グローバル・カウンタはネイティブ」とする。
具体的には: ヘッダのみ操作は小さなコピーで済ませる／ボディに触れないフィルタはボディ・ストリームを**ゼロコピーでバイパス**させる／
レート制限のような超ホット経路は **ホスト・ネイティブ**に落とす。WASM 税はリクエスト判断ロジックにのみ課す。

検討した代替案と却下理由:
- **全フィルタで一律にボディを WASM へ**: コピーが支配的で性能が出ない。却下。
- **全部ネイティブ**: 拡張モデルの存在意義（[[000001]]）を失う。却下。

本決定はファウンディング設計の Fork 6 に対応する。

## Consequences

### Pros
- ボディ非接触フィルタが多い実運用で WASM 税をほぼゼロに抑えられる。
- グローバル・カウンタをネイティブに置くことで超ホット経路の性能を確保。

### Cons / Tradeoffs
- 契約に「ボディ非接触」を表す仕組みと、ホスト側のバイパス判定・実装が必要。
- ネイティブ/WASM の責務境界の判断を設計者が都度行う必要がある。
- **再検討条件**: WASI 0.3.x で stream splicing と caller-supplied buffers（ゼロコピー経路）が入り、
  Component Model 1.0 で同期呼び出しの約 3.5x が解消されたら、ボディ変換フィルタの WASM 化の閾値を下げる。

## 実装ノート（確定事項・2026-06-20）

本決定の実装可能な機構を `plecto-host` に突き合わせた際に固まった具体判断。**done** と **deferred** を区別する。

1. **機構1（ヘッダ小コピー）・機構3（レート制限ホストネイティブ）は [[000004]] 実装で充足済み。**
   現契約はヘッダのみを copy する（[[000010]] で header-only に確定）。`host-ratelimit` の token bucket は
   refill とカウントを Rust + redb/memory の**ホスト側**に置き（`crates/host/src/backend.rs` の `apply_bucket`）、
   WIT は薄い `try-acquire` だけ——超ホット経路のカウントは WASM 境界を跨がない。ADR 000004 実装ノート #4 が
   token-bucket で「最初から」満たした形で、これが本 ADR の機構3そのもの。

2. **機構3を E2E conformance として固定（反証可能）。**
   `untrusted`（リクエストごとに線形メモリが fresh、init 再実行で `init-calls` が 1→2→3 と climb）でも
   capacity-2 のバケットが drain し続け、3 本目が 429 に落ちる——カウントが WASM 線形メモリでなくホスト側に
   あることの反証可能な証明。`crates/host/tests/e2e.rs::rate_limit_counting_is_host_native_across_fresh_untrusted_instances`。

3. **機構2（ボディ非接触の契約表現＋ゼロコピー bypass）は named-deferred。**
   現契約（[[000010]]）は header-only でボディ型を持たないため、bypass する対象（ボディ・ストリーム）が
   そもそも存在しない。よって契約レベルの header/body 分離と bypass は「ブロック中の忘れ物」ではなく、
   ボディ増分とともに着手する**予定済みの設計要素**として扱う。ゲートは二段（[[000010]] の再検討トリガと同一）:
   (1) `stream<u8>` ボディは wasmtime 46（Component Model async を default 化）以降、
   (2) ゼロコピーの実体（stream splicing / caller-supplied buffers）は WASI 0.3.x の後続。
   それまで body-transform フィルタは未対応で、全フィルタが事実上 header-only。

4. **契約レベルの分類に CONTEXT で名前を与えた（Body disposition）。**
   header-only / body-transform を分ける契約レベルの性質を `Body disposition` として `CONTEXT.md` に追加。
   機構2 が着手される際、host のゼロコピー bypass 判定はこの分類を読む。様式の機構（host 側 enum か WIT
   自己申告か）は body 増分時に決める（本 ADR では分類の**語彙**のみ確定）。
