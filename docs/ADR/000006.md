# deny-by-default capability・epoch 計量・OCI 署名・pooling ゼロ化のセキュリティモデルを定める

## Status
accepted

## Date
2026-06-20

## Affected Components
- host runtime — `Linker` での host-API 付与、`Store` 計量、component ロード時の検証
- filter contract — フィルタが import できる能力集合の上限（最小権限）

## Context
Plecto は untrusted な WASM をプロセス内で動かすため、サンドボックスの強度がそのまま製品の安全性になる。
Component Model はコンポーネントが import/export した関数経由でしか相互作用できず互いのメモリにアクセスできない
（サンドボックスが強制する規律）。計量は epoch interruption と fuel の二択で、Wasmtime 公式は epoch のほうが軽量で
最大 2-3x 速いと報告（fuel の利点は決定性）。マルチテナント分離には CVE-2022-39393（RUSTSEC-2022-0075、
CVSS 8.6 HIGH）の教訓——pooling allocator + memory-init-cow 有効時に linear memory slot 再利用で前インスタンスの
初期ヒープが見えうる（Wasmtime 2.0.2 / 1.0.2 で修正）——がある。

## Decision
- **deny-by-default の import 設計**: フィルタが触れてよい host capability を「request/response の許可範囲の読書き・
  ホスト KV/counter・metrics/trace・log・clock/random」程度に絞る。任意 outbound HTTP・FS・socket はデフォルトで貸さない。
  能力ごとに別 interface に切り、付与を明示的にする。
- **計量**: データプレーンには **epoch interruption** を採用（fuel より軽量、低オーバーヘッドのデッドラインが要件）。
  加えて `Store` のメモリ上限で誤割当てを封じる。
- **provenance**: フィルタは OCI artifact として配布し、**cosign 署名と SBOM をロード時に検証**してから instantiate する（fail-closed）。
- **pooling ゼロ化**: pooling allocator を使うなら instance deallocation 時の linear memory ゼロ化と memory-init-cow の挙動を
  明示検証し、untrusted テナント・フィルタは per-request 新規生成（[[000004]]）と組み合わせる。

検討した代替案と却下理由:
- **計量に fuel を常用**: 計装が重く実行が遅い。決定性が要る検証用途に限定。データプレーン常用は却下。
- **署名検証なしロード**: 無署名アップデートは integrity 障害（OWASP A08）。却下。

本決定はファウンディング設計の Fork 7・Tenet 2 に対応する。

## Consequences

### Pros
- 「プロセス内で untrusted を動かしても安全」を、規約でなくサンドボックスで強制できる。
- epoch で低オーバーヘッドにフィルタ暴走を抑止、メモリ上限で誤割当てを封じる。
- 署名検証でサプライチェーン（A03/A08）を担保。

### Cons / Tradeoffs
- 能力を増やすたびに Linker 配線と最小権限の判断が要る（意図的な摩擦）。
- epoch も fuel も「ホスト呼び出し内でブロックした WASM」は起こせない。
- **再検討条件**: ホスト呼び出しでブロックする外部 I/O 待ちが増えるなら、ホスト側の非同期 I/O とタイムアウト機構を別途設計する。

## 実装ノート（確定事項・2026-06-20）

本決定を `plecto-host` に実装した際に固まった具体判断。**done** と **named-deferred** を区別する。
本 ADR は4本柱（deny-by-default / 計量 / provenance / pooling ゼロ化）を立てたが、実装時に各柱が
他 ADR の実装でどこまで充足済みかが判明したので、その整理も含めて記録する。

### 既充足（他 ADR 実装の副作用）

1. **deny-by-default の import 設計は [[000010]] / [[000004]] 実装で充足済み。**
   `Linker` は plecto host-API（log / clock / kv / counter / ratelimit）の 5 interface のみ add し
   `wasmtime_wasi` を一切リンクしない。能力ごと別 interface は既に成立しており、本 ADR では再確認に留めた。

2. **pooling ゼロ化は untrusted=OnDemand engine で構造的に充足、`memory_init_cow` は両エンジン有効維持
   （[[000004]] note #2）。** 本 ADR ではこれを**反証可能なテスト**にした（下記 #7）。

### Phase A — データプレーン計量（done）

3. **epoch interruption を [[000004]] note #9 の三点セット（sharding + epoch + sharded state）から
   切り離して単独実装した。** epoch は engine epoch を刻むスレッド + per-Store deadline + trap→fail-closed
   だけで成立し、fast-path server も sharding も要らない。むしろ untrusted を動かす現状でこそ要る（runaway な
   フィルタが worker をハングさせる）。`Config::epoch_interruption(true)` を両エンジンに、`Host` が 1ms 粒度の
   ticker スレッドを 1 本持ち両 engine の epoch を進める。`init`（既定 5s）と per-request hook（既定 100ms）で
   deadline を分離し（`LoadOptions`、Tenet 4 の init/per-request 分離と整合）、重い init を誤って trap しない。
   これに伴い [[000004]] note #9 を「epoch のみ先行」と更新した。

4. **Store メモリ上限を `StoreLimits` で両エンジンの全インスタンスに付けた。** pool 予約（[[000004]]）とは別に
   per-Store の linear-memory cap（既定 64 MiB）を強制する。untrusted の OnDemand engine は従来 per-Store の
   上限を持たなかったので、ここで封じた。

5. **fail-closed を typed `RunError { Deadline, Trap, Instantiate }` で表現し、`decision` と型を分けた。**
   trap / deadline は `decision`（フィルタの**意図的**出力）ではなくフィルタの**失敗**なので別型にし、呼び出し側
   （将来の fast-path）が必ず fail-closed（5xx 合成・upstream 非到達）する契約にした。`fail_closed_response()`
   が 504（deadline）/ 502（trap）を合成する。型を分けたことで trap 率を健全性シグナルとして観測できる。
   CLAUDE.md の「フィルタ trap / deadline 超過で素通り（fail-open）させない」を型で担保した形。

6. **trusted 永続インスタンスは trap 後に破棄→再生成（自己回復）。** trap 後の linear memory は不定なので
   再利用せず、次リクエストで再 instantiate + 再 init する。trusted slot を `Mutex<Option<Instance>>` にして
   `None`＝要再生成を表す。untrusted は元々 per-request fresh なので影響なし。

7. **zeroization を反証可能にした。** conformance fixture（filter-hello）に guest 線形メモリの可変状態
   （`local-state`、host KV ではない関数ローカル `static`）を露出させ、untrusted では毎リクエスト fresh
   （`local-state` が 1 のまま）、trusted では持ち越し（1→2→3）を E2E で固定した。CVE-2022-39393 の表面が
   OnDemand で構造的に不在であることの反証可能な証明であり、[[000011]] の「trusted の statelessness は
   強制でなく信頼」という挙動差とも対になる。

### Phase B — provenance（done）

8. **keyed cosign（ローカル公開鍵・オフライン）を採り、keyless（Fulcio/Rekor）は将来オプトインへ繰延。**
   self-host / single-node-first / 運用の単純さ（Tenet）に整合し、ロード時の network 依存と外部 trust root を
   持たない。実装は `sigstore` crate（sigstore-rs）0.14 を `default-features = false` で使い、offline な
   `crypto` モジュール（keyed verify）だけに限定する。これで Fulcio / Rekor / OCI registry client / reqwest が
   依存ツリーに入らないことを実機ビルドで確認した（`tokio(rt)` と `aws-lc-rs` は常時同梱だが network スタックは
   不在）。**lock-in**: provenance 検証の実装を sigstore-rs に固定する（API は pre-1.0 で流動的）。

9. **署名対象は component bytes 直接（cosign `sign-blob` 相当）で、OCI manifest digest ではない。**
   registry も image-layout も介さない「bytes in-hand」境界に整合する。ECDSA P-256 / ASN.1 DER。

10. **署名はロードの門であり、Isolation を駆動しない。** 有効署名（信頼鍵による component + SBOM 検証）が
    無ければそもそもロード不可（fail-closed）。Trusted / Untrusted は `LoadOptions` の明示選択を維持する。
    第三者の鍵による有効署名も「信頼コードである」ことは意味しないので、署名者 identity → Isolation の自動
    マッピングは宣言的マニフェスト（[[000007]]）へ繰延する。[[000004]] note #1 が署名へ繰延した「誰が trusted か」は、
    本 ADR で「ロード可否」までを確定し、「lifecycle 様式の自動決定」は引き続き繰延という形で半分だけ解いた。

11. **trust anchors は `Host` 構築時の `TrustPolicy`（操作者の trust root 一元管理）。** `load` は artifact +
    署名のみを受け取り、Host が持つ信頼鍵で検証する。`TrustPolicy::empty()` は何もロードしない＝deny-by-default。
    production API に「unsigned を許す」抜け道は作らない（テストはエフェメラル鍵で実署名する `test_support`）。

12. **署名付き SBOM の存在を必須化した（内容ポリシーは繰延）。** `SignedArtifact` は component とその署名に
    加え SBOM とその署名を必須で持ち、同じ keyed trust で両方を検証する。SBOM の不在・空・署名不一致は
    fail-closed。CVE / license などの内容ポリシーは後続。

13. **境界: registry fetch と base64 `.sig` decode は [[000007]] / wkg の領分。** `SignedArtifact` は raw DER
    署名を受け取り、OCI reference → bytes の取得や cosign の base64 デコードを host に埋め込まない。ADR6
    （検証）と [[000007]]（配布）を疎結合に保つ。

14. **WIT 契約は不変。** epoch / メモリ上限 / 署名はすべて host 側で、`plecto:filter` ワールドを一切変えない。
    deny-by-default の契約面を揺らさずに計量と provenance を足せた。

### named-deferred（予定済みの繰延）

- **OCI registry pull / wkg fetch**（[[000007]]）— host は bytes + 署名を受け取るだけ。
- **署名者 identity → Isolation の自動マッピング**（manifest, [[000007]]）。
- **SBOM 内容ポリシー**（CVE / license scanning）。
- **keyless（Fulcio / Rekor）オプトイン**。
- **per-worker sharding と pooling slot 再利用時の実ゼロ化検証**（fast-path server 到来時、[[000004]] note #9 の残り）。

## レビュー対応（f000003・2026-06-20）

外部レビュー `docs/feedback/f000003.md` が本セキュリティ実装を再評価し、前回指摘の解消と ADR 006 の正しい実装を
反証可能なテスト付きで確認した上で、前向きの hardening を提示した。実コードに突き合わせて各指摘の実在を検証し、
以下を反映した。

**done（対応済み）:**

15. **SBOM ↔ component の束縛（指摘 #1・最重要）。** ゲートは従来「component 署名 ∧ SBOM 署名 ∧ SBOM 非空」を
    *独立*に検証しており、SBOM が*この component を指す*ことは見ていなかった——正規署名済みの component A と
    正規署名済みの無関係な SBOM B を**ミスマッチ**させて通せた（偽造は不能だが断片の組換えは可能）。v0.1 は SBOM
    opaque で無害だが、内容消費（CVE / license スキャン）開始の瞬間「脆弱な A を無関係でクリーンな B でスキャン
    通過」が成立する潜在ギャップ。SBOM を **in-toto 風 statement** とし、`subject[].digest.sha256 ==
    sha256(component_bytes)` を instantiate 前に検証（fail-closed）。`load_rejects_sbom_for_other_component` が反証。

16. **untrusted の `table.grow` 上限（指摘 #2）。** `StoreLimits` は `memory_size`(linear memory) のみで
    `table.grow`（巨大 funcref テーブル）は上限の外だった。`StoreLimitsBuilder::table_elements`（既定 100_000）を
    全 Store に付け、epoch が最終的にトラップする前の単発巨大 grow を bounding（安価な defense-in-depth）。

17. **trusted 決定論トラップの circuit-breaker（指摘 #5）。** 決定論的にトラップする trusted フィルタは従来
    毎リクエスト instantiate + init をやり直していた（re-init ストーム、重い init で増幅）。`TrustedSlot` に連続
    トラップ数と cooldown を持たせ、連続 `TRUSTED_TRAP_BREAKER_THRESHOLD`(=3) 回で一定時間(500ms) rebuild を止めて
    即 fail-closed（`RunError::Unavailable` = 503）。単発トラップの自己回復（rebuild + retry）は維持。
    `trusted_trap_storm_opens_circuit_breaker` が反証。untrusted は元々 per-request fresh なので対象外。

**deferred（記録のみ・指摘の通り並行サービング / async と同時期、または既知）:**

- **single-writer のシャード化（指摘 #3）** — [[000004]] の再検討条件（hot counter は host-native fast-path /
  key-hash シャード）のまま。`Durability::None` で fsync 律速は外れた。並行サービングが載るまで不要。
- **epoch × host コール内ブロック（指摘 #4）** — 本 ADR が自認済み（Cons の再検討条件）。host コールが async 化
  （wasmtime 46）したら host I/O 側に別タイムアウトを設計する。
- **ticker のプロセス単位化（指摘 #6）** — per-worker sharding 到来時に「プロセスに 1 ticker」を保つ。`sleep(1ms)`
  の実効粒度が OS 依存（Windows 既定 ~15ms）な点は安全境界として許容（コメントに明記済み）。
- **cosign の on-the-wire 形式（指摘 #7）** — base64 `.sig` デコード・OCI referrers/attachment 取得は [[000007]] の
  wkg/loader 層。本実装は bundled raw-DER を前提（[[000007]] 実装ノート #5/#13）。
