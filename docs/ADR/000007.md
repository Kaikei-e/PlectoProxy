# OCI artifact + wkg で配布し、宣言的マニフェストの content hash で無停止リロードを整合させる

## Status
accepted

## Date
2026-06-20

## Affected Components
- distribution — フィルタを OCI artifact として配布、`wkg` で fetch/publish、digest 固定
- host runtime / control — 宣言的マニフェスト、無停止リロード手順

## Context
フィルタと設定の配布・バージョニング・ホットリロードの方式を決める。CNCF TAG Runtime が Wasm OCI Artifact レイアウト
（config mediaType `application/vnd.wasm.config.v0+json`、layer mediaType `application/wasm`）を定義し、Bytecode Alliance の
`wkg`（wasm-pkg-tools）が fetch/publish を担い、`wkg.lock` で content digest を固定でき、cosign 署名や SBOM など既存の
OCI 投資を流用できる。Envoy は「拡張を control plane から runtime にリロードできる」agility を proxy-wasm の主要価値に挙げる。
Plecto は重い control plane を持たない（[[000008]]）。

## Decision
- 配布は **OCI artifact + `wkg`**。フィルタ参照は OCI ダイジェストで固定し、`wkg.lock` 相当のロックファイルで再現性を担保。
- フィルタと設定のバージョンは **単一の宣言的マニフェスト**（ルート・チェーン定義・フィルタ OCI 参照＋ダイジェスト）で
  **content hash により整合をロック**する。
- 無停止差し替えは **「新インスタンスを並行生成 → アトミックに切替 → 旧インスタンスを drain」**。

検討した代替案と却下理由:
- **独自配布フォーマット**: OCI の署名/SBOM/レジストリ等の既存投資を捨てる。却下。
- **ダイジェスト非固定（タグ運用）**: 再現性が壊れ、サプライチェーン整合を担保できない。却下。

本決定はファウンディング設計の Fork 8 に対応する。

## Consequences

### Pros
- 既存の OCI エコシステム（レジストリ・cosign・SBOM）をそのまま流用できる。
- content hash 固定で「何がロードされているか」が再現可能・監査可能。
- 公式・未改変のホストでカスタム拡張をロードでき、無停止で差し替えられる。

### Cons / Tradeoffs
- マニフェストと OCI ダイジェストの整合運用（更新時のロック更新）が要る。
- **再検討条件**: フィルタ数が数百規模になり静的マニフェスト管理が破綻したら、動的配布レイヤを段階導入する（[[000008]]）。
