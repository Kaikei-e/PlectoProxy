# 単一ノード・ファーストとし、分散はオプトイン、設定は静的宣言的 + 無停止 reload とする

## Status
accepted

## Date
2026-06-20

## Affected Components
- control plane — 静的な宣言的設定 + 無停止 reload を第一に
- distribution (opt-in) — `foca`（SWIM gossip）でメンバーシップ・配布、`openraft`（Raft）で設定/ルート複製

## Context
デプロイ単位と設定供給の方式を決める。wasmCloud の分散ファースト（NATS lattice 必須）は、NATS への運用依存と
未認証 lattice の攻撃面という重い信頼インフラを伴う。Plecto のデータ主権・セルフホスト志向には、単一ノードで完結し
分散はオプトインという River 型（設定駆動の単一バイナリ）が合う。Envoy の xDS は強力だが、実装者が外部の gRPC/REST 設定
プロバイダ API を提供する必要があり、ソロ運用には過大。Envoy 公式も hot restart は稀な変更にのみ必要であるべきとしている。

## Decision
- **単一ノード・ファースト**。状態は各ノードローカル（redb、[[000004]]）に置き、状態共有を分散の前提にしない。
- **分散はオプトイン・レイヤ**: `foca`（SWIM gossip）でノード・メンバーシップとフィルタ/設定配布、`openraft`（Raft）で
  設定・ルートの強整合複製。分散は「設定の合意」に限定する。
- **設定は静的・宣言的 + 無停止 reload を第一**とし、xDS 的な動的 push は採用しない。

検討した代替案と却下理由:
- **分散ファースト（wasmCloud 型 / NATS lattice 必須）**: 重い信頼インフラと攻撃面。セルフホスト志向に反する。却下。
- **xDS 動的 push**: 外部設定プロバイダ API の実装が要り、ソロ運用に過大。却下。
- **全ノード横断のグローバル状態共有**: 分散の前提にすると単純さを失う。却下。

本決定はファウンディング設計の Fork 5・Fork 10 に対応する。

## Consequences

### Pros
- 単一バイナリ・単一ノードでセルフホストが完結し、運用が単純。
- 分散が必要になっても「設定の合意」に限定でき、状態共有の複雑さを持ち込まない。

### Cons / Tradeoffs
- 全ノード横断の厳密なグローバル・レート制限は標準では提供しない。
- 高頻度な動的ルート/フィルタ更新は静的マニフェストでは扱いにくい。
- **再検討条件**: 厳密なグローバル・レート制限が必須要件化したら専用の分散カウンタを追加（デフォルトには入れない）。
  動的更新が高頻度で必要になったら openraft 複製の上に薄い push API を足す。
