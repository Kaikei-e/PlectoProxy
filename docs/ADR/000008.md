# 単一ノード・ファーストとし、分散はオプトイン、設定は静的宣言的 + 無停止 reload とする

## Status
accepted

## Date
2026-06-20

## Affected Components
- control plane — 静的な宣言的設定 + 無停止 reload を第一に
- distribution (opt-in) — `foca`（SWIM gossip）でメンバーシップ・配布、`openraft`（Raft）で設定/ルート複製

## Context
デプロイ単位と設定供給の方式を決める。wasmCloud の分散ファースト（NATS lattice 必須）は、NATS への運用依存と
未認証 lattice の攻撃面という重い信頼インフラを伴う。Plecto のデータ主権・セルフホスト志向には、単一ノードで完結し
分散はオプトインという River 型（設定駆動の単一バイナリ）が合う。Envoy の xDS は強力だが、実装者が外部の gRPC/REST 設定
プロバイダ API を提供する必要があり、ソロ運用には過大。Envoy 公式も hot restart は稀な変更にのみ必要であるべきとしている。

## Decision
- **単一ノード・ファースト**。状態は各ノードローカル（redb、[[000004]]）に置き、状態共有を分散の前提にしない。
- **分散はオプトイン・レイヤ**: `foca`（SWIM gossip）でノード・メンバーシップとフィルタ/設定配布、`openraft`（Raft）で
  設定・ルートの強整合複製。分散は「設定の合意」に限定する。
- **設定は静的・宣言的 + 無停止 reload を第一**とし、xDS 的な動的 push は採用しない。

検討した代替案と却下理由:
- **分散ファースト（wasmCloud 型 / NATS lattice 必須）**: 重い信頼インフラと攻撃面。セルフホスト志向に反する。却下。
- **xDS 動的 push**: 外部設定プロバイダ API の実装が要り、ソロ運用に過大。却下。
- **全ノード横断のグローバル状態共有**: 分散の前提にすると単純さを失う。却下。

本決定はファウンディング設計の Fork 5・Fork 10 に対応する。

## Consequences

### Pros
- 単一バイナリ・単一ノードでセルフホストが完結し、運用が単純。
- 分散が必要になっても「設定の合意」に限定でき、状態共有の複雑さを持ち込まない。

### Cons / Tradeoffs
- 全ノード横断の厳密なグローバル・レート制限は標準では提供しない。
- 高頻度な動的ルート/フィルタ更新は静的マニフェストでは扱いにくい。
- **再検討条件**: 厳密なグローバル・レート制限が必須要件化したら専用の分散カウンタを追加（デフォルトには入れない）。
  動的更新が高頻度で必要になったら openraft 複製の上に薄い push API を足す。

## 実装ノート（確定事項・2026-06-20）

本決定の単一ノード制御面を `plecto-control`（[[000007]] で起こしたクレート）上で具体化した際に固まった判断。
ADR 8 は (1) 単一ノード・ファースト (2) 静的宣言的設定 + 無停止 reload (3) 分散 opt-in の三本を立てたが、
(1) は redb ノードローカル（[[000004]]）、control plane の骨格は [[000007]] で実装済みのため、今回は
**reload を運用契機まで届かせる**ことと、**分散 opt-in の境界を明示する**ことに絞った。done と named-deferred を区別する。

### done

1. **semantic な manifest content-hash = config version。** `Manifest::content_hash()` が
   `sha256(serde_json の正規形)` を返す。生 TOML ではなく正規化後にハッシュするので、コメント・空白・キー順・
   明示デフォルト（`isolation = "untrusted"` を書く／略す）は同値に畳まれ、意味のある変更だけがハッシュを変える。
   これが reload の冪等判定単位であり監査単位であり、将来の分散合意（openraft）が合意する「設定バージョン」と同一物。
   manifest は map を持たず順序付き `Vec` だけなので serde_json 出力は決定的。

2. **path-aware な reload と冪等スキップ。** `Control::from_manifest_path` / `load_at` がマニフェストの
   ディスク path を保持し、`reload_from_disk() -> ReloadOutcome { Unchanged, Reloaded { hash } }` が
   再読 → content-hash 比較 → 変化時のみ rebuild + アトミック swap を行う。同一 config version なら no-op
   （rebuild も drain もしない）。build 失敗時は旧 set 据え置き（all-or-nothing・fail-closed）。path を持たない
   構築（`load` / `from_manifest`）では `NoManifestPath` を返す。

3. **reload トリガは `ReloadSource` seam。** トリガは内容を持たず「マニフェストを今読み直せ」だけを意味する
   （[[000007]] が named-deferred にした file-watch／SIGHUP トリガの具体化）。`serve_reloads(&Control, &mut dyn ReloadSource)`
   が契機ごとに `reload_from_disk` を呼ぶブロッキングループ。実トリガは **SIGHUP**（`SignalReloadSource`・unix・
   signal-hook）で、operator が「マニフェストを編集した、拾え」と明示 push する慣習に対応する。これは ADR 8 の
   「静的マニフェスト・xDS 的動的 push は不採用」に整合する——push されるのは設定そのものではなく『読み直せ』の合図だけ。
   trait なので fake で決定的にテストでき、process-global なシグナルを送らずにループを検証できる。

4. **reload は trust root を変えない（セキュリティ境界）。** `reload_from_disk` は filter set + chain だけを差し替え、
   trust policy は構築時固定（[[000006]] / [[000007]] note#9）。マニフェストの `[trust] keys` を書換えて SIGHUP を
   送っても trust は変わらず、新たに宣言した鍵だけで署名された filter は **reload で fail-closed に落ちる**（黙って
   信頼しない）。trust root の変更はプロセス再起動を要する。「マニフェスト編集権限だけでは信頼を昇格できない」を保つ。

5. **分散（foca/openraft）は依存ごと入れない。** opt-in の境界は config version（content-hash）に集約する、と
   文書で固定した。将来分散を有効化するときは、各ノードが openraft で合意するのはこの content-hash であり、
   `reload_from_disk` の「ディスクから読む」を「合意した manifest を適用する」に差し替える形で `ConfigSource` 的な
   seam を入れる。今は消費者が一つしかないため trait seam を先に立てず（投機的抽象を避ける・bp-rust）、境界の所在
   だけを明示する。

### named-deferred

- **foca メンバーシップ + openraft 設定/ルート複製**（本文の再検討条件——厳密なグローバル・レート制限が必須要件化／
  動的更新が高頻度化——に達したら）。
- **file-watch トリガ**（notify）。SIGHUP を第一とし、自動検知は運用要望が出たら `ReloadSource` の別 adapter を足す。
- **drain（in-flight 追跡）**（fast-path server が並行負荷を載せてから。現状は ArcSwap の Arc drop が論理 drain、[[000007]]）。
- **route + named chain**（fast-path server 到来時、[[000007]]）。
- **config version のロックファイル外部化**（現状は pin 済み digest + manifest hash がロックそのもの、[[000007]]）。
