# 可観測性は wasi-otel を契約に据え、トレース文脈はホストが span state を管理して伝播する

## Status
accepted

## Date
2026-06-20

## Affected Components
- host runtime — span state の track/maintain、メトリクス集約、in-process OTel 処理
- filter contract — wasi-otel ゲスト契約（成熟に追従する二段構え）

## Context
フィルタ境界をまたぐ可観測性の方式を決める。汎用すぎた旧 WASI Observe は、上流 OpenTelemetry API に密結合した
**wasi-otel** として再出発した。host が span state を track/maintain し、component と core module で isomorphic という性質が、
フィルタ境界をまたぐ文脈伝播に効く。in-process でホストが OTel を処理すればネットワークホップを除去でき native 並みのコストになる。
ただし wasi-otel は WASI プロセスの Phase 0 で、言語サポートは Rust 先行・JS/Go 進行中・Python 未着手と未成熟。

## Decision
可観測性の契約として **wasi-otel** を据え、トレース文脈は **ホストが span state を管理してフィルタ境界をまたいで伝播**する。
メトリクスは **ホスト集約**。短命フィルタは telemetry をホストへバッファするだけにする。
ただし wasi-otel の未成熟を踏まえ、**当面はホスト側で OTel コレクタへ流す実装を主**とし、wasi-otel ゲスト契約は安定化に追従する
二段構えを採る。

検討した代替案と却下理由:
- **各フィルタが独自に OTel を送る**: 文脈伝播が分断され、ネットワークホップが増える。却下。
- **wasi-otel ゲスト契約を即・全面採用**: 言語サポートが未成熟（Python 未着手等）で実用に達しない。時期尚早として却下。

本決定はファウンディング設計の Fork 9 に対応する。当初 status は `proposed`（wasi-otel 成熟待ち）だったが、**二段構えの
前段——ホスト側集約 stage——を実装した**ため `accepted` に進めた。後段の wasi-otel ゲスト契約は成熟するまで named-deferred
のまま（再検討条件を参照）。

## Consequences

### Pros
- ホストが span state を持つことでチェーンをまたぐ文脈伝播が成立する。
- in-process 処理でネットワークホップを除去でき native 並みのコスト。

### Cons / Tradeoffs
- wasi-otel は Phase 0 で言語サポートが未成熟、契約の安定化に追従し続ける必要がある。
- **再検討条件**: wasi-otel のゲスト言語サポートが安定したら、ホスト側集約主体からゲスト契約主体へ寄せる。

## 実装ノート（確定事項・2026-06-21）

二段構えのうち **host 側集約 stage** を実装した。Web 調査（opentelemetry crate を no-tokio で使えるか／wasi-otel 成熟度）
の上で採用範囲を絞った。done と named-deferred を区別する。

### done

1. **OTel データモデルを API crate だけ借りる（no-tokio）。** `opentelemetry`（`default-features=false, features=["trace"]`）で
   `TraceId`/`SpanId`/`SpanKind`/`Status`/`KeyValue`/`Event` を借り、tokio は引かない（実測: 追加依存は futures-core/sink・
   pin-project-lite・thiserror のみ）。Web 調査で SDK の `SpanExporter` が async（tokio 誘導）と確認したため、SDK exporter は
   実装せず、自前の **sync** `TelemetrySink` を境界に置いた。OTLP / SDK 連携はそこで初めて tokio が入る named-deferred。

2. **host が span state を管理し filter 境界を跨いで伝播。** `RequestTrace`（trace-id + request span-id）を chain driver が起こし、
   各 filter 実行はその子 span になる。filter は自分の trace context を持たない——ただ走り、host が timing と outcome を記録する。
   W3C `traceparent` の in/out で（将来の fast-path server が）inbound trace を継続できる。

3. **filter 実行ごとに 1 span を emit。** `LoadedFilter::on_request/on_response` が `&RequestTrace` を取り、実行を計時し、
   outcome（continue / modified / short-circuit / trap / deadline / unavailable / instantiate-error）と filter の host-log 行を
   span event として `FilterSpan` に詰め、host 保持の `TelemetrySink` に出す。**trap / deadline も fault span として記録**する
   （観測の死角を作らない）。戻り値 `(decision, logs)` は低レベル直接アクセスとして従来どおり維持し、波及は `&RequestTrace`
   引数追加のみに留めた。

4. **これまで捨てていた filter ログを sink に載せた。** chain dispatcher は filter の logs を破棄していた（[[000007]]）。host が
   span event として sink に流すことで telemetry の消失を止めた。

5. **メトリクスは host 集約（RED 系）。** `MetricsSink` が span ストリームから total / errors / short-circuits / latency を
   in-process 集約する（「メトリクスはホスト集約」を具体化）。filter 向け metrics API は足さない——host が outcome と timing を
   知る立場なので不要で、かつ wasi-otel と将来競合しない。sink は `NoopSink`(既定) / `InMemorySink`(test) / `MetricsSink` /
   `FanOutSink`(合成)。

6. **trace context は `ConfigSnapshot` に乗る（f000004 #2 と統合）。** request transaction の snapshot が `RequestTrace` を保持し、
   `on_request` / `on_response` が同一 trace で走る——request 半と response 半の span が 1 trace を共有する。

7. **WIT 契約は不変。** filter 向け wasi-otel 契約は据え置き。host 側 stage は host-API（`plecto:filter` ワールド）を一切変えない。

### named-deferred

- **wasi-otel ゲスト契約**（WASI Phase 0・Rust 先行 / JS・Go 進行 / Python 未着手。成熟したら host 集約主体から寄せる——再検討条件）。
- **OTLP ネットワーク export**（`opentelemetry_sdk` の async `SpanExporter` + `opentelemetry-otlp`。`TelemetrySink` が継ぎ目で、ここで tokio）。
- **OTel metrics 型へのマッピング**（現状は自前の RED カウンタ。exporter 追加時に `opentelemetry` metrics へ）。
- **span のバッチング / サンプリング**（現状は per-call 同期 emit + 常時サンプル）。
