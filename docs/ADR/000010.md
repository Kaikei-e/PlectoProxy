# 初回フィルタ契約を sync・自前 http 型・wasm32-unknown-unknown で実装し、async/wasi:http 再利用を wasmtime 46 へ繰り延べる

## Status
accepted

## Date
2026-06-20

## Affected Components
- filter contract (WIT) — `plecto:filter@0.1.0` を **sync・header-only・自前 http 型**（`header` / `http-request` / `http-response` / `*-edit` / `decision`）で確定
- host runtime — wasmtime 45 を sync 経路で埋め込み、`Linker` は deny-by-default、`InstancePre` で事前型チェック
- build toolchain — 無 WASI な header-only フィルタは `wasm32-unknown-unknown` でビルドし `wit-component` の `ComponentEncoder` で component 化

## Context
ADR1/ADR2 を実装するにあたり、契約の async/sync 基盤（ADR3）・型再利用（ADR2）・build target（ADR3）を
**実機の toolchain 現況**と突き合わせる必要があった。確定事実として: (a) WASI 0.3.0 は 2026-06-11 に
ratified され `wasi:http` のボディは `stream<u8>`（native async）になった。(b) しかし crates.io の wasmtime
stable は **45.0.2 が最新で 46 は未リリース**。(c) wasmtime 45 の Component Model async は **experimental**
（`component-model-async` フラグ、公式が "very incomplete" と明記）で、default 有効化は wasmtime 46。
(d) wit-bindgen（0.58）生成の `wasi:http@0.3` component が wasmtime のリンクで `wasi:http/types` の resource
mismatch を起こす **未解決バグ**（wit-bindgen#1554 / wasmtime#12714, OPEN）。

つまり「完全 async-first（`stream<u8>`）× `wasi:http@0.3` 型再利用 × wasmtime 45」を同時に満たすと、
experimental フラグと既知バグ込みのフラジャイルな実装になる。一方 ADR3（Fork 1）は自身に再検討条件
——「`wasm32-wasip3` 正式化が遅れ async バインディングの DX が実用に達しないなら、ヘッダ系を当面
sync 契約に留める二段構え」——を持ち、現況（wasm32-wasip3 は Tier 3、wasmtime 45 の async は "very
incomplete"）はその条件が満たされた状態である。本 ADR は確定事実と projected（wasmtime 46 / stream
splicing / wasi:http 再利用）を区別して扱う。

加えて build target を検証した。セキュリティ境界の実体は **Linker**——host が plecto host-API しか add せず
`wasmtime_wasi` を一切リンクしない——であり、これは**ターゲット非依存**で成立する。`wasm32-wasip2` ゲストが
std のために declare する `wasi:io/*`・`wasi:cli/*` import は、host が満たさなければ未解決 import として
instantiate に失敗するだけで、勝手に権限が漏れるわけではない（fs/socket を deny したまま境界は保てる）。
ただし wasip2 では std を満たす最小 `WasiCtx` を能動的に scope（または stub）する必要があり、設定表面が増えて
over-grant を踏みやすい。一方 `wasm32-unknown-unknown` + `wit-component` の component は **plecto 能力
（host-log / types）だけを import** し（実測 25KB / WASI ゼロ、wasip2 は 68KB / 多数の WASI import）、追加の
`WasiCtx` 配線なしに最小・最小サイズで**今日フラグ無しで green に通る**。なおこの「WASI import ゼロ」の純粋性は
header-only かつ host-async-call 無しのフィルタ固有の性質で、body / `stream<u8>` が入れば消える。

## Decision
初回増分（`plecto:filter@0.1.0`）を次の三点で実装する。いずれも **wasmtime 46 到来までの暫定**で、
contract-first（先に `.wit`、それからバインディング）を保つ。

1. **sync・header-only 契約**（ADR3 の二段構え fallback を発動）。`decision` variant（`continue` /
   `modified` / `short-circuit`, Tenet 3）と init / on-request / on-response フック（Tenet 4）は確定させ、
   ボディ（`stream<u8>`）と async func は繰り延べる。
2. **自前の最小 http 型**（`record` ベース）で実装し、`wasi:http@0.3` 型の再利用（ADR2/Fork 2）は繰り延べる。
   これで #12714 を回避し、wasi:http の host resource 配線も不要になる。
3. **build target は `wasm32-unknown-unknown` + `wit-component`**（CLAUDE.md/ADR3 の `wasm32-wasip2` から
   **v0.1 限定で逸脱**）。根拠は「今日フラグ無しで green に通る最小・最小サイズの足場」であること。境界自体は
   Linker が担うのでターゲット非依存だが、unknown-unknown は WASI import ゼロで追加の `WasiCtx` scope が要らず
   便宜が良い。これは v0.1 の便宜であり、body 増分で wasip2/wasip3 へ戻る前提（下記再検討条件）。

host-API は ADR6 の最小スライス（log / clock / kv）を能力ごと別 interface で付与。kv backend は初回 in-memory
で、redb は ADR4 へ繰り延べる。instance は per-request 新規生成（pooling / epoch / OCI 署名は ADR4/6/7 へ）。

検討した代替案と却下理由:
- **完全 async + wasi:http 再利用を wasmtime 45 experimental で強行**: 既知リンクバグ #12714 と "very
  incomplete" な async ABI でフラジャイル。初回 green を最優先するため却下（wasmtime 46 で本採用予定）。
- **`wasm32-wasip2` を維持**: 境界は Linker で保てる（fs/socket は deny のまま）が、std を満たす最小
  `WasiCtx` を能動的に scope する設定表面が増え、over-grant を踏みやすい。v0.1 では最小足場を優先し不採用
  （body 増分で本採用）。
- **wasmtime 46 を待つ**: 未リリースでビルド不能。待たずに足場を作る。

本決定は **ADR3（Fork 1）の再検討条件を発動**し、**ADR2（Fork 2）の型再利用を一時繰り延べ**、
**ADR6 / Tenet 2 の deny-by-default を build target 選択で強化**するものである。

## Consequences

### Pros
- stable な wasmtime 45 上で、実験フラグ無し・既知バグ無しに end-to-end が green（E2E / WIT-conformance /
  unit が揃う）。ADR1（相補的な二つの構成要素を WIT で駆動）と ADR2（独自 `plecto:filter` ワールド）を実証できた。
- deny-by-default を純粋に実証: フィルタは貸与された plecto 能力だけを import し、host Linker も plecto
  host-API のみ。WASI / network / filesystem / socket は一切到達不能。
- contract-first の WIT が確定し、`decision` / init / per-request / host-API スライスという Plecto の語彙が
  以後の async 増分の足場になる。

### Cons / Tradeoffs
- ボディを扱えない（header-only）。body-transform フィルタは未対応（ADR5 の body 経路は次増分）。
- `wasi:http` 型を再利用しないため、当面 polyglot 互換の一部（wasi:http 型共有）を得られず、自前型の維持
  責任を負う。
- build target が二系統に分岐する（無 WASI = `wasm32-unknown-unknown` / WASI・body = `wasm32-wasip2`）。
- pooling 未実装ゆえ instance = per-request となり、init を per-request 毎に呼ぶ＝Tenet 4 の「効果」は
  まだ出ていない（契約の「形」のみ実証。事実上 untrusted プロファイルを全員に当てた状態）。stateless の
  精密定義と init/pooling/zeroization の結節点は [[000011]]、正しい init-once 意味論は ADR4 で解消する。
- **再検討条件（2段トリガ）**: trigger を host 側と guest 側に分ける。
  - **(1) host が P3 を走らせられる** = **wasmtime 46（Component Model async + WASI 0.3 を default 化）リリース**（目前）。
  - **(2) P3 ゲストを実用 DX で書ける** = **`wasm32-wasip3` の Tier 2 化 / wit-bindgen async の成熟**（より後）。
  async + `stream<u8>` ボディ・`wasi:http@0.3` 型再利用・body 系フィルタの着手は **(2)** に紐づける（46 が
  出た瞬間に body を始めると guest 側が未整備で再び詰まりうる。#1554 はまさにこの guest 生成側の荒れ）。
  stream splicing / zero-copy 経路は WASI 0.3.x の後続なので、ゼロコピー bypass の本実装はさらにその先。
  移行時の再確認チェックリスト: #12714 / #1554 の解消。
- **wasip2 移行時の新セキュリティ表面**: wasip2 へ戻ると std を満たす**最小 `WasiCtx` の scope 設計**が
  security-critical な新表面として登場する（fs/socket を deny し、必要な io/cli のみ最小付与）。named task として残す。
