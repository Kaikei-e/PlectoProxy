# 「ステートレス」を可変業務状態に限定し、init 派生物の常駐と trusted/untrusted 分岐を必然化する

## Status
accepted

## Date
2026-06-20

## Affected Components
- filter contract — 「ステートレス」(Fork 4) を *可変業務状態* に限定し、*不変 init 派生物* のインスタンス常駐を許す
- host runtime — instance ライフサイクル（trusted=pooled / untrusted=per-request）と、その分岐が必然である根拠
- host-API — `host-kv` をフィルタ identity で名前空間化（host 側の偽造不能 prefix）

## Context
ADR 000010 の v0.1 host は per-request に `instantiate → init → on_request` を回す。結果、Tenet 4
（重い init をホット経路から追い出す）の「**効果**」は出ておらず（毎回 init）、契約の「形」だけを実証して
いる。事実上、untrusted プロファイル（毎回 fresh）を全フィルタに当てた状態である。

ADR 000004（pooling / stateless）を実装する前に、**3つの設計判断が1つの結節点として絡む**ことを明文化
しておく必要がある。さもないと ADR4 実装時に矛盾に突き当たる:

- **Tenet 4 が成立するには**、init の産物（compiled regex / parsed config）が per-request 呼び出しまで
  **インスタンスに残らねばならない**。
- だが **Fork 4 は「フィルタはステートレス」**。
- さらに **ADR 000006 の zeroization**（CVE-2022-39393 / RUSTSEC-2022-0075）は、インスタンス再利用時に
  linear memory を**ゼロ化せよ**と言う——ゼロ化は init 産物も消すので**再 init を強制**する。

ゆえに「pooling で init-once を効かせる（trusted）」と「再利用間で linear memory をゼロ化する（untrusted）」は
**同一インスタンス上で両立しない**。

レビュー（`docs/feedback/f000001.md`）でこの結節点と、`host-kv` が全フィルタ共有のフラット単一キースペース
である点（チェーン化した瞬間に capability 隔離の穴）が指摘された。後者は本 ADR と同時にコードで塞いだ。

## Decision

**(a) 「ステートレス」を精密化する。**
ステートレスとは「**リクエスト間に持ち越す*可変*状態を持たない**」ことである。
- *可変の業務状態*（session / rate-limit counter / cache）は **host KV**（[[000004]]）に置く。
- *不変の init 派生物*（compiled regex / parsed config / built schema）は**インスタンス常駐で良い**。
これは「ステートレス」に反しない（持ち越す可変状態ではない）。

**(b) trust でインスタンス戦略を分ける——これは perf 最適化ではなく上記矛盾ゆえの必然。**
- **trusted（自家製）= pooled + init-once**。同一信頼ドメインなので request 間で linear memory をゼロ化しない。
  init 派生物が常駐し、Tenet 4 の効果が出る。
- **untrusted（第三者）= per-request fresh + zeroize**。再利用間ゼロ化で状態漏洩を断つ。その対価として
  **init が毎回走り、Tenet 4 の利得を失う**——これは untrusted 隔離の不可避なトレードであり、意識的な
  ポジションとして受け入れる。

**(c) host KV をフィルタ identity で名前空間化する（実装済み）。**
`Host::load(filter_id, …)` で host が偽造不能な prefix（`"{filter_id}\u{1f}"`）を全 KV キーに付ける。
フィルタは prefix を観測も改変もできず、他フィルタの名前空間へ到達できない。共有フラット keyspace は
チェーン（＝本来の目的）で capability 隔離の穴になるため、フィルタが1個のうちに塞ぐ。

検討した代替案と却下理由:
- **全フィルタを常時 pooled + init-once**: untrusted の状態漏洩リスク（CVE-2022-39393）。却下（trusted のみ）。
- **全フィルタを常時 per-request + zeroize**: trusted にも毎回 init コストが乗り Tenet 4 を捨てる。却下（untrusted のみ）。
- **`host-kv` をフラット共有のまま**: チェーンで隔離が崩れる。却下。

本決定は **Fork 4（ステートレス）/ Tenet 4（init 分離）/ ADR 000006（zeroization）** を結び、ADR 000004 の
trust 分岐の根拠を確定する。

## Consequences

### Pros
- 「どの状態がどこへ行くか」が決まり、ADR 000004 実装の判断が一意になる。
- trust 分岐の根拠が「perf」でなく「矛盾ゆえの必然」と記録され、将来「なぜ分けるのか」を再質問されない。
- KV 名前空間化により、チェーン化しても capability 隔離（deny-by-default, ADR 000006）が保たれる。

### Cons / Tradeoffs
- untrusted は init-once の利得を得られない。重い init を持つ untrusted フィルタは perf 不利——trusted 化の
  検討か、init を軽く保つフィルタ設計が要る。
- **trusted は「強制」でなく「信頼」**: ステートレス（Fork 4）が host に**構造的に強制されるのは untrusted
  だけ**（fresh memory による）。trusted は zeroize しないので、ステートレス違反のフィルタ（guest メモリに
  可変状態を貯める）はリクエスト間に黙って持ち越す。同一信頼ドメインゆえセキュリティ侵害ではないが、Tenet 4
  が効くことの裏返しの挙動差として明示する（host 実装の `Isolation::Trusted` 注記と対）。
- host が trust レベルでロード経路を分岐するロジックを持つ（ADR 000004）。
- **async-aware KV の布石**: wasmtime 46 で host call が async 化したら、`.await` を跨ぐ `std::sync::Mutex` は
  古典的フットガン。redb のトランザクションも blocking。ADR 000004 の redb 統合は最初から async-aware
  （blocking pool 越し / 非 poison ロック）にする。v0.1 は sync ゆえ `std::sync::Mutex` + poison 回復で可（実装済み）。
- **再検討条件**: ADR 000004（pooling / instance lifecycle）の実装時に本定義を pin する。trusted/untrusted の
  判定基準（誰が trusted か）は配布・署名（[[000006]] / OCI 署名）と連動して別途定める。
