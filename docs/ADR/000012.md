# trusted フィルタを固定容量インスタンスプールで checkout 再利用し、枯渇は有界待ち後 fail-closed とする

## Status
accepted

## Date
2026-06-21

## Affected Components
- host runtime — trusted の `Mutex<TrustedSlot>`（単一インスタンス直列再利用）を `TrustedPool`（固定容量・checkout 方式）へ置換
- host runtime — circuit breaker を pool-wide 化（決定的 trap は pool 単位で 1 回 open）、instance discard/rebuild は個別、`max_requests_per_instance` で周期 recycle
- host-API（`LoadOptions`）— `trusted_pool_size` / `checkout_timeout_ms` / `max_requests_per_instance` を追加（既定値つき、`Untrusted` では無視）

## Context

ADR 000004 / 000011 は trusted=pooled・init-once を確定したが、v0.1 実装は「単一インスタンスを単一 `Mutex` で直列再利用」＝ concurrency=1 の placeholder だった（ADR 000004 実装ノート #3）。この形では pooling allocator が 1 インスタンスしか作らず**実質 dead weight**（VA 予約コストのみ）で、Tenet 4 の効果（init 派生物の常駐）は出るが並行性は無い。foundation plan §6.3 はこれを M1 の残作業「per-worker instance 分散」とし、**最大の前方リスク**と位置づけた。

§6.3 は二つの★判断を含む：(1) trusted を worker 数規模の pre-instantiated プールへ、(2) tokio/quinn fast path への sync↔async ブリッジ（blocking pool / fiber）。後者は「M2 のデータパス設計と同時に ADR 化」と明記されている。**並行対応のプール自体は fast-path server 非依存で建てられ・N スレッドからテストできる**ため、本 ADR は (1) を M1 で realize し、(2) は M2 へ繰り延べる――§6.3 の★を M1/M2 境界で割る。

**枯渇ポリシーは Web 先行事例調査で決めた（確定事実）。**

- **wasmtime の pooling allocator は Engine 生成時に容量固定・VA 前予約**で、スロット枯渇時の instantiate は内蔵待ち行列なしに即 `PoolConcurrencyLimitError` を返す。実行時の上限拡張は不可。公式 Fast Instantiation ドキュメントは「embedder 側でセマフォ的バックプレッシャを被せ、最大同時数を超える instantiate を避けよ」と明示する。→ 「lazy grow で上限を超える」は物理的に不可能で、論理プールサイズは必ず pooling cap 以下に固定し、枯渇時の待機/timeout は host 側で自前に被せるしかない。
- **成熟プール（bb8 / r2d2 / HikariCP）の総意は「固定上限＋有界 timeout 待ち＋超過で fail-fast」**。HikariCP は動的サイジングを明示拒否し「小さい固定プールを待機で満杯にする」を中核思想とする。
- **production WASM ホストは instance-per-request（Spin / Fastly 既定）か worker ごと永続 VM（Envoy）**で、中央 checkout プールの枯渇待ち問題を構造的に回避している。Envoy は VM 非共有ゆえ checkout 競合が原理的に発生しない。
- **再利用するなら max-requests / max-lifetime の周期 recycle（Fastly reusable-sandbox）＋失敗時即破棄（Envoy `FAIL_RELOAD`）**が確立パターン。wasmtime のゼロ化は instance（Store）破棄で初めて発動するため、同一 instance を回す checkout プールでは guest-heap は clean されない。

加えて §6.6 の footgun：trusted はゼロ化しないため、誤って linear memory に可変状態を溜めるフィルタはプールだと「**どのインスタンスに当たるか**」で挙動が変わり（非決定性）、同一インスタンス上で req A→B にデータが見える。同一信頼ドメインゆえ侵害ではないが、正しさの足元を崩す。

本決定は Fork 3（インスタンス・ライフサイクル / 並行性）の placeholder を解消し、Tenet 4（init 分離）を「instance ごと init-once」で維持する。

## Decision

**採用：固定容量・lazy fill（eager 1）の checkout プール。**

- **eager 1 ＋ lazy fill**：load 時に 1 インスタンスだけ事前生成して `init` を走らせる（壊れた init を load 時に検出、かつ単一スレッドの caller は以後それを再利用するので init-once が観測上維持される）。残りは並行需要が来たとき cap まで遅延生成する。これにより**直列アクセスでは 1 インスタンスしか作られず、既存の init-once / メモリ持ち越しテストは無改変で通る**。並行時のみプールが追加インスタンスを生成し、ここで pooling allocator が初めて非 inert になる。
- **checkout**：idle を再利用 → `live < cap` なら lazy build → 飽和（全 checkout 済み）なら host 側 `Condvar` で `checkout_timeout` まで有界待ち → timeout 超過で `RunError::Unavailable`（fail-closed 503）。wasmtime に待ち行列が無いので、この有界待ちが公式推奨の host-side バックプレッシャに当たる（§6.5 の二層レイテンシ保証――epoch はゲスト計算の上限、host-timeout がブロック経路の穴を塞ぐ――を checkout にも適用）。
- **容量と既定値**：論理 cap ≤ engine pooling cap（`TRUSTED_POOL_SLOTS = 256`、全 trusted フィルタで共有する VA-reservation 予算）。既定プールサイズは `available_parallelism` を `[1, 8]` にクランプ（fast-path server 不在での worker-scale 近似）。複数フィルタが pooling 予算を食い合う配分は、宣言的マニフェストのレジストリ（ADR 000007）が server 到来時に担う。
- **circuit breaker は pool-wide**：trap は「filter ＋ 入力」の性質なので、決定的に trap するフィルタは pool 単位で 1 回だけ cooldown を開く。**instance の discard/rebuild は個別**（trap が壊すのは当該インスタンスの linear memory のみ）。これで決定的 trap フィルタの再 init storm を N 倍にしない。
- **周期 recycle**：インスタンスが `max_requests_per_instance` を超えたら破棄・再生成（init 再走）。§6.6 footgun (a) の状態蓄積 bound。加えて (b) の「mutable static 禁止」は guidance として `Isolation::Trusted` の doc と host `CONTEXT.md` に明文化した。

**検討した代替案と却下理由：**

- **(A) lazy grow ＋ 満杯時は無制限ブロック**：wasmtime の pooling は上限超過 grow が物理的に不可能。無制限ブロックは fail-closed（ロードシェディング）方針とも相性が悪い。却下。
- **(B) 固定事前生成 ＋ 満杯時に即 reject**：瞬間的バーストを吸収できず tail latency が荒れる。成熟プールの総意は有界待ち。却下（C を採用）。
- **(C) 固定 cap ＋ 有界 semaphore 待ち ＋ timeout 超過で fail-closed**：wasmtime の固定容量制約・connection pool の timeout 思想・WASM ホストの知見すべてに整合。**採用**。なお C は当初直感 (A) の「block」を Plecto の fail-closed tenet に合わせた洗練版に当たる。
- **per-thread（thread-local）インスタンス（Envoy 型）**：checkout 競合は無いが thread 数＝server 概念に縛られ M1 では建てられず、VM 非共有はステートレス原則と緊張する。M2 で per-worker 小プールとしてハイブリッド検討する余地として残す。
- **per-instance breaker**：決定的 trap で N インスタンスが各々閾値に達するまで再 init storm が N 倍。却下。

本決定は ADR 000004 実装ノート #3（trusted=単一インスタンス直列再利用）を**更新**し、その三点セット（per-worker sharding / epoch deadline / シャード化状態 backend）のうち epoch（ADR 000006 で先行実装）に続いて per-worker 並行の足場を置く。設計原則（安全 × ポータビリティ × セルフホスト性 × 運用の単純さ）と矛盾しない：飽和も trap も recycle も fail-closed 側に倒れ、変更は host crate 内に局所化される。

## Consequences

### Pros
- concurrency=1 placeholder を解消。pooling allocator が並行需要で初めて非 inert になり、ADR 000004 の本来の利得（高速インスタンス化のスケール）に到達する足場ができた。
- 単一スレッドでは init-once が維持され、既存の e2e / conformance（37 件）が無改変で green。追加 init コストは並行時にのみ発生する。
- 枯渇が fail-closed（`Unavailable` 503）でロードシェディングし、無制限ハングを断つ。wasmtime 公式・connection pool・production WASM ホストの先行事例に整合。
- §6.6 の trusted state footgun を周期 recycle で bound（Fastly の max-requests に倣う）。
- `KvBackend` seam や host-API 契約（WIT world）は不変。変更は host 内に閉じ、control crate も無影響。

### Cons / Tradeoffs
- **sync↔async ブリッジは未着手**。現状の checkout 有界待ちは sync `Condvar`。`!Send` な wasmtime `Store` は `.await` を跨げないため、M2 の tokio fast path に載せる際は専用 blocking スレッドプール（将来は fiber）へ移す必要があり、これは別 ADR（§6.3 ★(2)）。
- 既定プールサイズ `available_parallelism` は server 不在の近似で、複数 trusted フィルタが engine pooling 予算（256 slots）を食い合いうる。マニフェストのレジストリ（ADR 000007）が per-filter cap を配分するまでは控えめ default で凌ぐ。
- **trusted のステートレスは依然「信頼」であって「強制」ではない**。recycle は蓄積を bound するだけで、req 間の持ち越し自体は断たない。構造的に強制されるのは untrusted（fresh-per-request memory）のみ（ADR 000011）。
- **再検討条件**：
  - (a) M2 fast-path server 到来 → checkout を blocking-pool / fiber へ移し、per-worker 小プールへの sharding と §6.4 状態 backend の sharding を**同時に**再設計する（ADR 000004 実装ノート #9 の三点セット）。
  - (b) untrusted engine 選択の再論（§6.6 / ADR 000006）：on-demand vs pooling+COW fresh-per-request をベンチで決める。pooling 化するなら本プール機構を untrusted へ拡張できる。
  - (c) `checkout_timeout_ms`（暫定 250ms）は M2 で実 SLO に紐付けてチューニングする。
  - (d) 複数フィルタで pooling slot 予算が逼迫したら、マニフェストのレジストリが per-filter cap を配分する。
