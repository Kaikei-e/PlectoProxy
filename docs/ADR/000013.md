# tokio/hyper の fast path で Plecto を実プロキシにし、sync チェーンを spawn_blocking で橋渡しする

## Status
accepted

## Date
2026-06-21

## Affected Components
- server runtime（新 crate `plecto-server`）— tokio + hyper 1.x の HTTP/1.1 listener、`spawn_blocking` による sync↔async ブリッジ、hyper-util pooling client での upstream 転送
- control plane — manifest を routes/upstreams へ進化、`ConfigSnapshot` に route 選択（`find_route`）と route 毎チェーン dispatch（`dispatch_request` / `dispatch_response`）を追加、`chain::dispatch` を chain 引数化
- manifest — `[[upstream]]`（name/address）と `[[route]]`（host?／path_prefix／inline filters／upstream／strip_prefix?）を additive 追加（既定 `[chain]` は据え置き）

## Context

foundation plan §4.1 の M2（データパス）は「最大の穴」だった。M0/M1 で「安全な拡張プレーン ＋ 供給網 ＋ 制御面」は実物として深く出来ていた一方、**接続を受ける fast path のコードが存在せず**、Plecto は *sandbox ＋ filter runtime ＋ control plane というライブラリ*であって、まだ「プロキシ」ではなかった。これを解くのが M2 であり、§6.3 はその前提として並行性モデルと sync↔async ブリッジを「**M2 のデータパス設計と同時に ADR 化**」せよとした。M1（[[000012]]）で trusted インスタンスプールは既に着地しており、本 ADR はそのプールを**実データパスに結線**する。

control plane は既に chain dispatch・無停止 reload・trace context（[[000007]] / [[000008]] / [[000009]]）を持ち、`ConfigSnapshot.on_request → ChainOutcome{Respond | Forward}` という seam を提供していた。manifest は「routes は fast-path server 到来時まで繰延」と明記しており、今がその到来である。

スコープは grill で確定した：**HTTP/1.1 平文・routing 込み・1 route に 1 upstream**。TLS（rustls）・HTTP/2（h2）・HTTP/3（quinn）・インスタンス間 LB は後続スライスへ繰延する（plan §9 のスタックに沿う）。

## Decision

**(1) 新クレート `plecto-server`（tokio + hyper 1.x の HTTP/1.1 fast path）。** CONTEXT-MAP の予告どおり fast path を独立クレートに切る。データ面（listener / upstream）と制御面（reload / manifest）の責務を分ける。

**(2) sync↔async ブリッジ ＝ `spawn_blocking` ＋ M1 中央プール。** フィルタ実行は sync で、wasmtime の `Store` は `!Send` ゆえ `.await` を跨げない。チェーン駆動（route の chain を request→pool→response で回す）を `tokio::task::spawn_blocking` に載せ、**M1 の trusted インスタンスプールがインスタンス再利用・飽和を担う**。route 照合は純粋な設定参照なので async スレッドに残し、blocking 化するのはチェーン駆動だけにする。`ConfigSnapshot` を `Clone`（Arc ＋ trace-id の安いクローン）にし、request 半と response 半を別々の `spawn_blocking` に載せつつ同一トランザクション（同一 pinned config ＋ trace）を保つ。

**(3) host-native routing。** manifest を additive に進化させ、`[[upstream]]`（name → `host:port`）と `[[route]]`（match＝host?＋path_prefix／inline `filters`＝route 毎チェーン／`upstream`／`strip_prefix`?）を加える。route 選択は host（port 無視・大文字小文字無視）一致かつ path prefix の `/` 境界最長一致で、一致なしは 404。Control が `find_route` と route 毎の chain dispatch を提供する。`chain::dispatch_*` を「chain スライス引数」化し、既定 `[chain]` の単一チェーンと route 毎チェーンを同一コードで回す。

**(4) host-native prefix strip。** route の `strip_prefix` を fast path が**転送直前**に適用し、チェーンは元 path を見たまま upstream は書き換え後を受ける。これでリバースプロキシの基本（prefix strip）を **WIT 契約変更なしに**得る。

**(5) opaque body（header-only 契約、ADR 010 / §6.7）。** リクエスト/レスポンスのボディはフィルタを介さずストリーム透過する。レスポンス側はチェーンが status/headers のみ編集でき、fail-closed のときだけ合成ボディを使う（`edited.body` が非空かで判別）。upstream へは hyper-util の pooling client（接続再利用が無償）で 1 route 1 アドレスに転送し、hop-by-hop ヘッダは双方向で除去、trace（W3C `traceparent`）を伝播する。

**検討した代替案と却下理由:**

- **sync↔async：(a) async ワーカ上で `block_in_place`** — ランタイムワーカをブロックし並行性・公平性が落ちる。却下。**(b) 専用 per-thread インスタンスプール（Envoy 型）** — M1 の中央プールと二重になり、ステートレス原則とも緊張する。却下（M1 プール再利用を採る）。**(c) fiber（M3 `async_support`）** — 上流が experimental（wasmtime 45）で時期尚早。繰延。なお §6.3 の「専用 blocking プール（per-thread instance）」という表現は、M1 が中央 checkout プールに着地した事実に合わせて**精緻化**する（per-thread 専有ではなく中央プールへの checkout）。
- **HTTP スタック：axum/tower（高レベル）** — routing/ミドルウェアは楽だが、双方向ストリーム ＋ upstream client の低レベル制御には過剰で層が増える。**自前 HTTP/1.1** — keep-alive / chunked / エッジケースの車輪の再発明。→ hyper 1.x（plan §9 指名、プロキシ用途の Rust 標準）。
- **path 書き換え：(a) 書き換えなし** — prefix strip ができず実用に欠く。**(b) WIT `set-path` を今追加（§6.7）** — @0.1.0→@0.2.0 の破壊的変更で conformance / versioning が付随しスライス 1 が重い。→ host-native rewrite（契約不変）。
- **named chain の表現** — 別テーブルの named chain（route が名前で参照）も検討したが、`[chain]`（単数テーブル）と `[[chain]]`（配列）の TOML キー衝突を避けるため、route が inline `filters` で自前チェーンを持つ形にした。route 毎に異なるチェーンを持てる（grill の要件）を満たしつつ additive。共有 named chain は後続。

本決定は Fork 3（並行性モデル）の sync↔async ブリッジを realize し（M1 のプール化に続く §6.3 の二本目の★）、Fork 6（「Rust か WASM か」）を実データパスで体現する（routing / TLS / upstream / prefix-strip は native、判断ロジックは WASM）。設計原則（安全 × ポータビリティ × セルフホスト性 × 運用の単純さ）と整合する：404 / 502 / チェーン fail-closed、宣言的静的 manifest、単一ノード。

## Consequences

### Pros
- **Plecto が初めて「接続を受けて応答する実プロキシ」になった。** E2E（実 HTTP/1.1 → routing → trusted チェーン → prefix strip → upstream 転送 → response チェーン → ストリーム応答）が green。
- sync↔async ブリッジが M1 プールを実負荷に結線。`!Send` Store は `spawn_blocking` 境界で完結し `.await` を跨がない。
- routing と prefix strip が host-native で、`plecto:filter` 契約は不変のまま。
- ボディはゼロコピー透過でフィルタ税ゼロ（header-only、§6.7）。
- manifest 進化は additive で既存 control テスト（31 件）無改変。`chain::dispatch` の chain 引数化で単一チェーンと route 毎チェーンを共通化。

### Cons / Tradeoffs
- **HTTP/1.1 平文のみ**：TLS（rustls）・HTTP/2（h2）・HTTP/3（quinn）は後続スライス。現状は本番 TLS 終端ができない。
- **1 route 1 upstream・LB なし・health check なし**：インスタンス間 LB と受動的健全性は後続。
- **`spawn_blocking` の限界**：tokio の blocking プールに載るが、真の並行上限は M1 プール cap。blocking プールサイズと pool cap の整合は将来チューニング。fiber 化（M3）まではスレッド切替コストが残る。
- **hop-by-hop は最小実装**：RFC 9110 の hop-by-hop を固定リストで除去するのみ。`Connection` で列挙されたヘッダの動的除去や trailer 透過は未対応。
- **SSRF 面**：upstream アドレスは manifest 由来（operator 信頼）で安全だが、将来フィルタ駆動の動的 upstream を入れるなら SSRF 対策が要る（security-auditor 範囲）。
- **再検討条件**：
  - (a) TLS 終端が要件化 → 次スライスで rustls 終端。
  - (b) HTTP/2,3 → wasmtime 46（M3 トリガ）とは独立に h2 / quinn を追加できる。
  - (c) 複数インスタンス LB / health check が要件化 → upstream を複数アドレス ＋ LB 戦略へ。
  - (d) M3 `async_support` 成熟 → `spawn_blocking` を fiber 実行へ移行（§6.3）。
  - (e) フィルタが path / method / authority を書き換える要求 → §6.7 の WIT `set-path` 拡張（@0.2.0、versioning 込み）。
