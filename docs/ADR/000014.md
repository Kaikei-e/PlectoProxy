# fast path の TLS 終端を rustls で行い、証明書は宣言的 manifest の静的ファイルで管理する

## Status
accepted

## Date
2026-06-21

## Affected Components
- server runtime（`plecto-server`）— 受理した TCP 接続を rustls で TLS 終端してから既存の hyper HTTP/1.1（将来は h2）へ渡す。SNI で証明書を選ぶ resolver、ALPN を広告
- manifest — `[[tls]]`（`host?` / `cert_path` / `key_path`、PEM、manifest-relative）で証明書を静的宣言。reload で差し替え

## Context

M2 slice 1（[[000013]]）は HTTP/1.1 **平文のみ**で、本番相当の TLS 終端ができない。次スライスとして TLS 終端を入れる。plan §9 が rustls を指名している（pure-Rust・openssl 非依存で、既存スタックの no-openssl 方針と整合）。証明書の管理方式は self-host / データ主権志向と「宣言的・静的設定 ＋ 無停止 reload」tenet（[[000008]]）に整合させる必要がある。TLS は ALPN を通じて HTTP/2（[[000015]]）の前提でもある（h2c は採らない）。

## Decision

- **rustls + tokio-rustls で listener を TLS 終端する。** 受理時に TLS ハンドシェイクを行い、得た平文ストリームを slice 1 と同じ hyper serve（HTTP/1.1、後に ALPN で h2）に繋ぐ。リクエスト処理（route → chain → forward）は不変。
- **証明書は manifest の静的宣言。** `[[tls]]` に `host?`（SNI、無指定はデフォルト証明書）/ `cert_path` / `key_path`（PEM、manifest-relative）。SNI で host→証明書を選ぶ rustls の cert resolver を構築。証明書は [[000006]] の cosign trust roots とは別物（こちらは TLS サーバ証明書）。
- **reload で証明書を差し替える。** 証明書パスは manifest content hash に乗るので、パス変更は通常の reload で反映。同一パスのファイル**内容**変更の自動検知は当面非対応（明示 reload か mtime/hash 監視は後続）。
- **ALPN は当面 `http/1.1` を広告。** h2 の追加は [[000015]]。

検討した代替案と却下理由:
- **ACME / Let's Encrypt 自動取得** — 運用は楽だがネットワーク依存・更新状態を抱え、オフライン/データ主権志向と緊張する。却下（後続の opt-in レイヤ）。
- **native-tls / openssl** — pure-Rust の rustls を採る（plan §9、既存の no-openssl 方針）。却下。

本決定は plan §9（rustls）を具体化し、設計原則（安全・セルフホスト・運用単純さ・宣言的設定）に整合する。

## 実装ノート（確定事項・2026-06-21）

1. **crypto provider は `ring`。** `rustls` / `tokio-rustls` を `default-features=false, features=["ring",…]` で取り、aws-lc-rs の cmake/C ツールチェーンを回避（pure-Rust 寄り・ビルド軽量）。control は sync `rustls`、server は `tokio-rustls`、両者とも ring ゆえ cargo が単一 `rustls` にまとめ、`Control::tls_config()` の `Arc<ServerConfig>` をそのまま `TlsAcceptor::from` に渡せる。
2. **ServerConfig は control の `build_active` で構築。** `ActiveConfig.tls: Option<Arc<ServerConfig>>` に載せ ArcSwap に乗せる。bad cert（読込/解析/未対応鍵/重複）は `ControlError::TlsCert` で fail-closed・reload all-or-nothing。cert を manifest-relative に解決するため `Control` に `base_dir` フィールドを追加（in-memory `load` は base=`"."`、テストは絶対パス）。
3. **SNI 選択（`ResolvesServerCert`）。** host→cert の map ＋ host-less default。SNI 未一致は default に fallback、default が無ければ `None`＝ハンドシェイク拒否（fail-closed）。重複 host / 複数 default は fail-closed。
4. **server は per-accept に `Control::tls_config()` を読む。** `Some` なら `tokio_rustls::TlsAcceptor` で終端してから hyper、`None` なら平文。reload は新規接続のみ反映、ハンドシェイク失敗は接続 drop（fail-closed、何も転送しない）。
5. **listen は CLI param 据え置き**、TLS on/off は `[[tls]]` の有無で判定（単一 listener）。ALPN は `http/1.1` のみ。
6. **content_hash は cert パスを束ねる**（内容は束ねない）。パス変更で reload、同一パスの内容変更は明示 reload まで未反映（再検討条件 (c)）。

## Consequences

### Pros
- 本番相当の TLS 終端が入り、ALPN で HTTP/2 への道が開く。
- pure-Rust・openssl 非依存・オフライン。宣言的 manifest と一貫。
- SNI マルチ証明書に対応（1 listener で複数 host）。

### Cons / Tradeoffs
- 静的証明書は operator が手動ローテ（reload）。自動更新なし。
- 同一パスの証明書**内容**変更の自動検知は未対応（パス変更 or 明示 reload を要する）。
- **再検討条件**：(a) ACME を望む operator 向けに opt-in 自動取得；(b) mTLS（client 証明書検証）が要件化；(c) 証明書ファイル内容ハッシュでの reload；(d) **upstream TLS**（https upstream への rustls client 由来）は LB スライス（[[000017]]）または専用スライスで。
