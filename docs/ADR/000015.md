# HTTP/2 を hyper の h2 で TLS+ALPN 上に終端し、h2c は採らない

## Status
accepted

## Date
2026-06-21

## Affected Components
- server runtime（`plecto-server`）— TLS の ALPN 交渉で `h2` / `http/1.1` を分岐。`h2` なら hyper の HTTP/2 で serve。リクエスト処理（route → chain → forward）は slice 1 と共通

## Context

TLS スライス（[[000014]]）が入ると ALPN が使え、HTTP/2 を終端できる。plan の M2 は「HTTP/1.1→2→3」を掲げる。HTTP/2 はクライアントとの多重化・ヘッダ圧縮の価値があり、現代的 L7 ゲートウェイの基本線。h2 はほぼ常に TLS+ALPN 上で使われ、h2c（平文 HTTP/2）は野では稀。

## Decision

- **HTTP/2 は TLS+ALPN 上でのみ終端する（h2c は採らない）。** TLS ハンドシェイクの ALPN で `h2` が選ばれたら hyper の HTTP/2 で `serve_connection`、`http/1.1` なら従来どおり。`h2` を ALPN 広告に追加する（[[000014]] の ALPN 設定を拡張）。
- **リクエスト処理は共通。** HTTP/2 の各ストリームが 1 トランザクション（1 snapshot）に対応し、route → chain（spawn_blocking↔M1 プール）→ forward の流れは slice 1 と同一。多重化で**同一接続から並行するストリーム**が増えるため、bridge と M1 プールの飽和処理（[[000012]]）が初めて実負荷で効く。
- **upstream は当面 HTTP/1.1 のまま。** クライアント側 h2 終端を先行し、upstream への h2（ALPN/prior-knowledge）は後続。

検討した代替案と却下理由:
- **h2c（平文 HTTP/2）対応** — Upgrade / prior-knowledge の経路が増え、TLS なしの h2 は実利用が稀。却下（TLS+ALPN のみ）。
- **HTTP/2 を slice 1 に前倒し** — ALPN 前提なので TLS（[[000014]]）が先。順序固定。

本決定は plan の HTTP/1.1→2 段を具体化し、M1 プールの並行性が実負荷で活きる最初の局面になる。

## 実装ノート（確定事項・2026-06-23）

grill（`grill-with-docs`）で 4 つの分岐を詰め、TDD（E2E → unit）で着地した。

1. **ALPN は `["h2", "http/1.1"]`（h2 優先）。** control の `build_server_config`（[[000014]]）で `alpn_protocols`
   を 2 値に拡張。両対応クライアントは h2 を得る。server は per-connection に negotiated ALPN を読み
   （`TlsStream::get_ref().1.alpn_protocol()`）、`h2` なら hyper `http2::Builder`、それ以外（`http/1.1` /
   ALPN 未交渉 `None`）は `http1::Builder` で serve。**平文 listener は http1 のみ**（h2c / prior-knowledge
   非対応）。リクエスト処理（route → chain → forward）は protocol 不問で共通の `serve_conn` に集約。
2. **`max_concurrent_streams = 100`（固定定数・明示上限）。** hyper の既定は version 依存で API 安定保証が
   なく、1 本の h2 接続が**固定 cap の M1 プール（[[000012]]）の checkout を並行 chain dispatch で独占**しうる。
   これを止める防御線として、かつ stream-flood DoS への defense-in-depth として、RFC 9113 推奨下限の 100 を
   定数で pin する（h2 クレートは Rapid Reset／CVE-2023-44487 を別途緩和済み）。manifest 設定化・per-route
   tuning は後続（再検討条件 (d)）。
3. **scheme 補正をスコープに追加（grill 判断）。** TLS 終端接続は chain に `scheme = "https"`、平文は `"http"`
   を見せる。これは**接続レベルの真実**（fast path が何を終端したか）であり、クライアントが詐称しうる URI の
   scheme は採らない。`to_http_request` に scheme を threading。あわせて **authority は h2 の `:authority`
   （hyper では URI authority に載る）を優先**し、無ければ Host header に fallback——h2 は Host header を持たない
   ため、host-routing を効かせるのに必要。これは元の ADR スコープ（ALPN 分岐 + h2 serve に限局）への明示的な
   追加で、[[000014]] から持ち越していた「TLS でも scheme=http 固定」の wart を解消する。
4. **依存。** hyper に `http2` feature を追加。**upstream client は HTTP/1.1 のまま**（hyper-util の
   pooling client、ADR 決定どおり upstream h2 は後続）。

## Consequences

### Pros
- 多重化・ヘッダ圧縮でクライアント側の効率が上がる。
- ハンドラ共通ゆえ実装増分は ALPN 分岐 ＋ h2 serve に限局。
- 同一接続の並行ストリームで M1 プールの並行設計（checkout・飽和 fail-closed）が初めて実証される。

### Cons / Tradeoffs
- TLS（[[000014]]）に依存（順序固定）。
- ストリーム多重化で per-connection の並行 chain dispatch が増え、blocking プール／M1 プール cap のチューニングが load-bearing になる。`max_concurrent_streams` を 100 で固定し 1 接続の M1 プール独占を防ぐが、適正値はワークロード依存（固定定数ゆえ operator は今は調整できない）。
- **再検討条件**：(a) upstream への HTTP/2；(b) フロー制御・優先度の調整が要件化したとき；(c) server push は非採用（deprecated）；(d) `max_concurrent_streams` の manifest 設定化 / per-route 化（現在は固定定数）；(e) フィルタ駆動の scheme/authority 書換が要件化したとき（現在は host-native で接続レベルの scheme を付与、契約は不変）。
