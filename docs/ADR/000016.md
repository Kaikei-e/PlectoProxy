# HTTP/3 を quinn(QUIC)+h3 の独立 UDP listener で終端し、Alt-Svc で広告する

## Status
accepted

## Date
2026-06-21

## Affected Components
- server runtime（`plecto-server`）— quinn(QUIC) + h3 crate で UDP listener を追加。HTTP/1.1/2 応答に `Alt-Svc: h3=...` を付与してアップグレードを誘導。QUIC の TLS は rustls

## Context

plan の M2 は HTTP/3 までを射程に置き、§9 が quinn を指名する。HTTP/3 は QUIC（UDP 上）で HoL ブロッキングを避け、接続移行・0-RTT を持つ。HTTP/3 は **wasmtime / Component Model とは独立**（純粋なデータパス側）なので、M3 の async/wasi トリガとは無関係に着手できる。クライアントは通常 HTTP/1.1 か 2 で接続し、`Alt-Svc` を見て HTTP/3 へ切り替える。

## Decision

- **quinn(QUIC) + h3 crate で独立 UDP listener を立てる。** QUIC の TLS は rustls（[[000014]] と同じ証明書 resolver を共有）。h3 でリクエストを受け、route → chain（spawn_blocking↔M1 プール）→ forward の共通ハンドラに繋ぐ（リクエスト処理は HTTP バージョン非依存）。
- **Alt-Svc で広告する。** HTTP/1.1/2 応答に `Alt-Svc: h3=":<port>"` を付け、対応クライアントを HTTP/3 へ誘導。
- **upstream は HTTP/1.1（または h2）のまま。** downstream の HTTP/3 終端を先行。
- **QUIC datagram / 非信頼転送は使わない**（HTTP セマンティクスのみ）。

検討した代替案と却下理由:
- **hyper の h3 サポートを待つ** — 成熟途上。quinn+h3 が現状の実路（plan §9）。却下（quinn を採る）。
- **HTTP/3 を TLS/HTTP2 より先行** — QUIC は TLS(rustls) 前提で、証明書 resolver を [[000014]] と共有するため TLS スライス後が素直。順序維持。

本決定は plan §9（quinn）を具体化し、wasmtime 非依存ゆえ M3 トリガと切り離して進められる。

## 実装ノート（確定事項・2026-06-23）

grill（`grill-with-docs`）で 4 分岐を詰め、API（quinn 0.11 / h3 0.0.8 / h3-quinn 0.0.10）と RFC
（7838 Alt-Svc / 8470 Early Data / 9114 HTTP/3 / 9000 QUIC）を一次情報で確認し、TDD（E2E → unit）で着地した。

1. **共通ハンドラを transport 非依存にする（body-generic）。** `route_and_proxy` を `proxy_core(parts,
   body: ReqBody)` に分離し、`ReqBody`/upstream client の body 型を `BoxBody<Bytes>` 化。HTTP/1.1・HTTP/2 は
   hyper `Incoming` を、HTTP/3 は h3 recv stream を box して**同一コア**に流す。h3 の request body は
   `RequestStream::poll_recv_data` で `http_body::Body` に直接アダプト（`H3ReqBody`）、response body は
   `BodyExt::frame()` で h3 send stream にポンプ。ゼロコピー透過（ADR000013）を h3 でも維持。
2. **独立 QUIC/UDP listener・TCP と同一ポート・TLS 時のみ。** `serve` が `quic_tls_config()` Some のとき、TCP
   listener の `local_addr` のポートで UDP に `quinn::Endpoint::server` を張り、`serve_h3` を spawn。`[[tls]]`
   無し → h3 無効。新しい config knob は足さない（運用単純さ）。
3. **QUIC 用 TLS config は control が構築。** `build_server_configs` が ADR000014 の `SniResolver` を一度作って
   TCP config（ALPN `[h2, http/1.1]`、TLS1.2+1.3）と QUIC config（ALPN `[h3]`、**TLS1.3 限定**）を返し、
   `ActiveConfig` の `tls` / `quic_tls` に載せて同一 ArcSwap で reload。bad cert は build を fail-closed に倒す
   既存不変条件を維持。`Control::quic_tls_config()` で公開。
4. **0-RTT 無効 + Alt-Svc。** QUIC config の `max_early_data_size` を既定 0 のままにして early data を受けない。
   これは単なる用心でなく **RFC 8470 の要求**——ゲートウェイは origin が 425 (Too Early) を返すと保証できない
   限り early-data リクエストを転送して**はならない**ため、任意 upstream を相手にする Plecto では受けないのが唯一
   安全。TCP（h1/h2）応答に `Alt-Svc: h3=":<port>"; ma=86400`（RFC 7838、host 省略=同一ホスト）を注入し h3 へ誘導。
   h3 応答には付けない。
5. **per-connection bidi ストリーム上限 = 100（ADR000015 の h2 cap と対称）。** h3 の各リクエスト = 1 bidi
   stream = 1 chain dispatch なので、`TransportConfig::max_concurrent_bidi_streams(100)` で 1 接続の M1 プール
   独占を抑止。uni stream（h3 control/QPACK）は既定維持。**QUIC の 3x anti-amplification（RFC 9000 §8/§21）は
   quinn が既定で適用**し、自前 `Endpoint::retry()` は使わない（CVE-2024-45311 回避）ので、UDP 反射増幅に既定で
   悪用されない。
6. **RFC 9114 の禁止ヘッダ。** connection-specific フィールド（connection/keep-alive/proxy-connection/
   transfer-encoding/te/upgrade）は h3 で malformed。既存 `HOP_BY_HOP` strip が双方向で完全一致除去するため、
   upstream(HTTP/1.1) 応答を h3 で返す経路は契約変更なしで compliant。h3→HTTP/1.1 の Host は hyper-util client が
   upstream_address から生成（§4.3.1 充足）。
7. **依存。** quinn 0.11（`rustls-ring`、aws-lc-rs/cmake 回避で ADR000014 と一貫）・h3 0.0.8・h3-quinn 0.0.10。
   **upstream は HTTP/1.1 のまま**（client h3 終端を先行）。

## Consequences

### Pros
- HoL ブロッキング回避・接続移行で実プロキシの体験が向上。
- 証明書 resolver を TLS スライスと共有でき、ハンドラも共通で増分が局所。
- wasmtime / WASI の成熟（M3）に依存せず着手できる。

### Cons / Tradeoffs
- UDP listener・QUIC スタック（quinn）という別経路が増え、運用面（UDP ファイアウォール・MTU・輻輳制御）の考慮が要る。
- **0-RTT は無効**（RFC 8470）。性能上の 0-RTT 利得は捨てた。安全メソッド限定の有効化は後続（再検討条件 (a)）。
- **h3 の証明書 hot-reload は未対応。** TCP は per-accept で reload 後の証明書を反映するが、QUIC Endpoint は起動時に
  一度 config を渡すだけで、reload 後の新証明書は**再起動まで h3 に反映されない**（TCP と非対称）。`Endpoint::set_server_config`
  での反映は後続（再検討条件 (d)）。
- **h3 crate は pre-1.0**（0.0.x）で API が不安定。バージョン更新時に破壊的変更を被りうる。
- **再検討条件**：(a) 0-RTT の有効化方針（安全メソッド限定 + 425 Too Early 連携）；(b) upstream への HTTP/3；
  (c) コネクション移行・輻輳制御のチューニング；(d) reload 後の証明書を `Endpoint::set_server_config` で h3 に反映；
  (e) `max_concurrent_bidi_streams` の manifest 設定化（現在は固定 100、ADR000015 と対称）。
