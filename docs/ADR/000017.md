# upstream を複数インスタンス化し、round-robin LB ＋ active/passive health check で分散する

## Status
accepted

## Date
2026-06-24

## Affected Components
- server runtime（`plecto-server`）— 1 upstream を複数アドレスへ拡張。healthy な instance 集合を round-robin で選び、background の **active health supervisor** が各 instance を定期 probe して unhealthy を eject・healthy で復帰。実リクエストの転送が接続失敗したら、その instance を **passive** に demote する
- control plane（`plecto-control`）— **upstream registry**（health 状態の持ち主）を atomically-swap される `ActiveConfig` の**外**に置く。reload は registry を `(name, address)` 差分で reconcile し、不変 instance の health を保つ。route は build 時に upstream 名を `Arc<UpstreamGroup>` へ解決して持つ
- manifest — `[[upstream]]` に複数 `addresses` と **必須**の `[upstream.health]`（probe `path` 必須、`interval_ms` / `timeout_ms` / `healthy_threshold` / `unhealthy_threshold` は既定値あり）

## Context

M2 slice 1（[[000013]]）は 1 route 1 upstream（1 アドレス）で、インスタンス間 LB が無い。可用性・スケールには複数 upstream インスタンスへの分散と、落ちたインスタンスを避ける health 判定が要る。plan §6.10 は「単一ノード・ファースト、分散はオプトイン」。LB は単一ノード内の upstream 分散であり、ノード間分散（foca/openraft）とは別レイヤ。

設計を詰める中で二つの論点が立った。(1) health 状態は**可変ランタイム状態**であり、`reload` が `ActiveConfig` を丸ごと差し替える（ArcSwap）モデルとどう両立させるか。(2) instance を起動時に healthy とみなす（optimistic）か unhealthy から始める（pessimistic）か——pessimistic は落ちを絶対に踏まない代わりに cold-start で全 instance unhealthy の窓ができる。後者は Envoy / Kubernetes など確立した実装の挙動を調査して裏取りした（[公式ドキュメント参照](https://www.envoyproxy.io/docs/envoy/latest/intro/arch_overview/upstream/health_checking)）。

## Decision

- **upstream を複数アドレスに拡張する。** `[[upstream]] name=... addresses=["h1:p","h2:p", …]`（最低 1、build 時に非空を検証）。単数 `address` は廃し、常に `addresses`。route は従来どおり upstream を名前で指す。
- **`[upstream.health]` を全 upstream に必須化する。** probe `path` のみ必須、`interval_ms`（既定 2000）/ `timeout_ms`（既定 1000）/ `healthy_threshold`（既定 2）/ `unhealthy_threshold`（既定 3）は既定値。単一インスタンスの upstream にも health を要求する（pessimistic 起動の前提）。
- **分配は round-robin。** healthy な instance 集合を `AtomicUsize` 巡回で選ぶ。選択は**転送時**に行い（`UpstreamGroup::pick`）、registry の最新 health を見る。
- **active health check（先回り検知）。** server 内の single supervisor task が各 instance を `health.path` へ `interval_ms` ごとに GET し、timeout 内 **2xx のみ成功**。連続成功/失敗が閾値に達したら healthy/unhealthy を切り替え、unhealthy は分配集合から **eject**、復帰で restore。supervisor は毎 tick registry を読み直すので reload の追加/削除に自動追従し、per-instance のタスク生成は不要。
- **pessimistic 起動 ＋ cold-start fast path。** instance は unhealthy で始まる。ただし**一度も healthy になっていない instance は最初の 1 回の成功 probe で昇格**（以後の再入には full `healthy_threshold`）。新規 instance は supervisor の初回 tick で即 probe する。これにより pessimistic の cold-start 窓を `healthy_threshold × interval` から概ね 1 probe RTT に圧縮する（Envoy の startup 単一 probe 挙動・Kubernetes の startup/readiness 分離に倣う）。
- **passive ejection も併用する。** 実リクエストの転送が**接続失敗**（connect 不能）したら、その instance を active と**同じ単一状態機**に反映して demote する。active が先回り、passive が取りこぼしを拾う。引き金になったリクエスト自体は救済しない（**retry しない**——その応答は従来どおり 502）。upstream が返した 5xx や mid-response の transport fault は health 信号にしない（接続失敗のみ）。
- **health 状態は registry に持続させる。** registry を `ActiveConfig` の外（`Control` 所有）に置き、`reload` は `(name, address)` で reconcile——不変 instance は health を維持、新規は pessimistic、消失は drop、health policy 変更時のみ再 probe。route の `CompiledRoute`/`RouteInfo` は reconcile 後の `Arc<UpstreamGroup>` を指し直す。registry の reconcile は build_active の**最後の可変ステップ**に置き、フィルタ load / route 参照検証 / TLS 構築の後に実行するので、失敗した reload が registry を `active` と desync させない（all-or-nothing を保つ）。
- **全 instance unhealthy は fail-closed。** ある upstream の healthy が 0 なら upstream に流さず **503**（`x-plecto-fault: no-healthy-upstream`）を返す。404 no-route / 502 upstream-error とは別の fault。
- **状態は単一ノードローカル。** ノード間の health 共有は §6.10 のとおり opt-in の後続。

検討した代替案と却下理由:
- **passive ejection のみ**（active probe なし）— 単純だが「次の実リクエストが踏むまで落ちを検知できない」。active を主、passive を補完として**両採用**した。
- **optimistic 起動**（起動時 healthy）— cold-start 窓が無い反面、dead instance を最初の数リクエストが踏む。落ちを絶対に踏ませない pessimistic を採り、cold-start fast path で窓を最小化した。
- **bind 前の eager 同期 probe**（healthy が 1 本立つまで listener を開かない）— 調査した本番プロキシで採用例が無く、upstream の起動可用性に boot が依存し fragile。却下。
- **panic threshold**（healthy 比率が閾値割れで全 host へ流す。Envoy の backstop）— mass-unhealthy の誤検知に強いが、**fail-closed テネットに反する**ので採らない。
- **least-conn / EWMA 分配** — 偏りに強いが per-instance の接続数/応答時間の追跡が要る。round-robin を先行し、偏りが問題化したら導入（後続）。
- **request-level retry**（接続失敗時に別 instance へ再試行）— 可用性は上がるが body 既送・冪等性の問題とスコープ拡大。passive eject のみとし retry は後続。
- **分散（ノード間）health/LB を今入れる** — §6.10 の単一ノード・ファーストに反する。opt-in 後続。

本決定は plan §6.10（単一ノード・ファースト、分散は opt-in）と整合し、fail-closed（全 unhealthy で upstream に流さない）を保つ。fail-closed テネットを upstream 層へ具体化し、M2 routing（[[000013]]）の単一 upstream 前提を更新する。

## Consequences

### Pros
- 複数 upstream インスタンスへ分散でき、可用性・スケールが上がる。
- active health で落ちたインスタンスを**実リクエストが踏む前に**回避でき、passive が probe 間隔の取りこぼしを拾う。
- health 状態が reload を跨いで生存するので、route 追加などの reload が稼働中 instance の health をリセットしない。
- cold-start fast path により pessimistic 起動の 503 窓を 1 probe RTT 程度に抑える。
- 単一ノードに閉じ、運用が単純。hot path（`pick`）は lock-free な `AtomicBool` 読みのみ。

### Cons / Tradeoffs
- 全 upstream（単一インスタンスでも）に `[upstream.health]` と health エンドポイントが必須になり、起動直後に最初の probe が通るまでの cold-start 窓（503）を持つ。
- active probe は upstream ごとに background probe トラフィック ＋ 設定（path/interval/閾値）を増やす。
- round-robin は接続数/負荷の偏りに弱い（重いリクエストが偏ると不均衡）。
- passive は**接続失敗のみ**を見る（実リクエストの full timeout は未導入）。mid-response の hang は active probe 任せ。
- health policy を変えた reload は当該 upstream の instance を pessimistic から再 probe する（health 履歴を引き継がない）。
- **再検討条件**：(a) 偏りが問題化したら least-conn/EWMA；(b) 接続失敗時の **request-level retry**（冪等メソッド限定など）；(c) passive に full request timeout を加える；(d) **upstream TLS**（https upstream への rustls client、[[000014]] と連動）；(e) ノード横断の health/LB は分散レイヤ（foca/openraft）の opt-in。
