# クライアント IP を edge モデルで伝播する（受信 X-Forwarded-* を剥がし peer から付け直す）

## Status
accepted

## Date
2026-06-25

## Affected Components
- server runtime（`plecto-server`）— 接続の peer アドレスを fast path（`serve` → `serve_conn` → `handle` → `proxy_core`、および h3 経路 `serve_h3` → `handle_h3_request`）へ通す。`proxy_core` は `to_http_request` 直後・チェーン実行の前に、受信 `Forwarded` / `X-Forwarded-For` / `X-Forwarded-Proto` / `X-Forwarded-Host` を**全て剥がし**、`X-Forwarded-For`（peer IP）と `X-Forwarded-Proto`（接続 scheme）を**新規に付与**する

## Context

M2 fast path（[[000013]]）は `listener.accept()` の peer を捨て（`_peer`）、`X-Forwarded-*` を一切触っていなかった。結果、untrusted クライアントが `X-Forwarded-For: 9.9.9.9` のように送ると、それがそのまま (1) フィルタ（IP ベースの rate limit / 認証を構想）に渡り、(2) upstream のアクセスログ・IP 制御に渡る——**送信元 IP のなりすまし**が通る状態だった（review f000005 P2#3）。

Plecto のテネット（deny-by-default・fail-closed・「判断は信頼できる入力に対して下す」）からは、クライアントが自由に書ける値をそのまま信頼してはならない。プロキシが送信元 IP を扱うときの確立した姿勢は二つ:

- **edge モデル**（このプロキシが信頼境界の外周）: 受信 `X-Forwarded-*` は全て不信、剥がして自分が観測した peer で**上書き**する。
- **trusted-proxy モデル**（前段に信頼できる LB がいる）: 信頼ホップ分の受信 `X-Forwarded-*` を尊重し、自分の peer を**追記**する。

どちらを既定にするかで挙動が真逆になるため、明示的に決める。

## Decision

- **edge モデルを既定に採る。** `proxy_core` でチェーン実行の前に、受信 `Forwarded` / `X-Forwarded-For` / `X-Forwarded-Proto` / `X-Forwarded-Host` を**剥がし**、`X-Forwarded-For = peer.ip()`、`X-Forwarded-Proto = scheme`（接続が TLS 終端なら `https`、平文なら `http`——[[000015]] の接続真実）を**新規付与**する。peer は TCP の `accept()`、h3 は QUIC `remote_address()` から取る（全 HTTP バージョンで同一経路 `proxy_core`）。
- **チェーンの前に適用する。** フィルタは Plecto が確定した `X-Forwarded-For` を見るので、IP ベースの rate limit / 認証フィルタが信頼できる値で判断できる。剥がし＋付け直しの後にチェーンが走り、編集後のヘッダが upstream へ転送される。
- **付与は For ＋ Proto のみ、剥がしは family 全体。** `X-Forwarded-Host` / `Forwarded` は**剥がす**（なりすましを残さない）が新規には設定しない（upstream は Host/`:authority` から元ホストを得られる）。最小限の確定値だけを足す。
- **trusted-proxy（追記）モードは後続の manifest knob。** 前段に信頼 LB がいる構成では受信 `X-Forwarded-For` を尊重して追記したいが、これは「どのホップまで信頼するか」の設定設計を要する。安全側の既定（上書き）を先に置き、設定可能化を後続スライスに回す。

検討した代替案と却下理由:
- **受信 `X-Forwarded-*` をそのまま信頼/追記する** — 設定なしでこれを既定にすると、edge 配置でなりすましが通る（fail-open）。安全側の edge 上書きを既定とした。
- **`X-Forwarded-Host` も確定付与する** — 元ホストは upstream が Host から得られ、付与は URL 生成で upstream が XFH を信頼する別リスクを生む。family の剥がしに留め、付与は For ＋ Proto に絞った。
- **RFC 7239 `Forwarded` を正準として発行** — 表現は新しいが採用が薄く、まず広く解釈される `X-Forwarded-*` を確定発行。`Forwarded` 発行は再検討条件。
- **PROXY protocol で L4 から peer を受ける** — 前段 LB 構成の話で、trusted-proxy モードと同じく後続。

本決定は deny-by-default / fail-closed テネットを「送信元メタデータ」へ具体化し、[[000013]] の fast path に IP 伝播を足す。設計原則（安全 × ポータビリティ × セルフホスト性 × 運用の単純さ）と矛盾しない（最小・安全側の既定、設定は後続）。

## Consequences

### Pros
- untrusted クライアントが送信元 IP / scheme をなりすませない（edge 既定）。IP ベースのフィルタ判断と upstream のログ・IP 制御が信頼できる値に乗る。
- 剥がし＋付け直しをチェーンの**前**に置くので、フィルタは確定値だけを観測する。
- HTTP/1.1・2・3 すべてで同一の `proxy_core` 経路に乗る（h3 は QUIC `remote_address()`）。

### Cons / Tradeoffs
- Plecto を**信頼 LB の背後**に置くと、edge 上書きが前段 LB の付けた本物の `X-Forwarded-For` を消す（前段の値を捨てて peer＝LB の IP で上書きする）。この構成は後続の trusted-proxy モードで対応する。
- `X-Forwarded-Host` / RFC 7239 `Forwarded` を発行しないので、それらに依存する upstream は別途設定が要る。
- **再検討条件**：(a) **trusted-proxy（追記）モード** — 信頼ホップ数を manifest 設定にし、受信 `X-Forwarded-For` を尊重して追記する；(b) RFC 7239 `Forwarded` と `X-Forwarded-Host` の発行；(c) **PROXY protocol** 受信（L4 で peer を受ける前段 LB 構成）。
