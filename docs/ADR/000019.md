# upstream への end-to-end タイムアウトを入れ、超過を fail-closed 504 にする

## Status
accepted

## Date
2026-06-25

## Affected Components
- server runtime（`plecto-server`）— `proxy_core` の upstream 転送を `tokio::time::timeout(route.upstream.request_timeout(), client.request(...))` で包む。超過時は **504**（`x-plecto-fault: upstream-timeout`）を返し fail-closed。`Duration::ZERO` は無効化（オプトアウト）
- control plane（`plecto-control`）— `Upstream` に `request_timeout_ms`、`UpstreamGroup` に `request_timeout: Duration` ＋ getter。`reconcile` が manifest 値から毎回再構築する（`health` には含めないので、timeout だけの変更は instance の health 状態を保ったままグループを差し替える）
- manifest — `[[upstream]]` に任意の `request_timeout_ms`（既定 30000、`0` で無効）

## Context

M2 fast path（[[000013]] / [[000017]]）は upstream への転送 `state.client.request(...)` に**タイムアウトを掛けていなかった**。health プローブには `timeout_ms` があるのに、本流の転送には無い——遅い / ハングした upstream に対してリクエストが無期限に張り付き、そのリクエストが占有する M1 プール枠・接続も解放されない（review f000005 P2#4）。

[[000017]] は LB / health を入れた際、再検討条件として明示的に **(c) passive に full request timeout を加える**・**(d) upstream TLS** を残していた。本 ADR は (c) を回収する。

論点は二つ。(1) 何を区切るか——応答ヘッダ到達までか、ボディ完了までか。ボディ完了まで区切ると SSE / 長寿命ストリームを切ってしまう。(2) タイムアウトを health 信号にするか——単発の遅延で instance を eject すると、閾値次第で flapping する。

## Decision

- **per-upstream の `request_timeout_ms` を manifest に入れる。** 既定 **30000**（30s）。**`0` は無効化**（long-poll / streaming upstream のためのオプトアウト）。`UpstreamGroup` が `Duration` として保持し、route は `Arc<UpstreamGroup>` 経由で読む。
- **応答ヘッダ到達までを区切る。** `tokio::time::timeout` で `client.request` を包む——これは応答**ヘッダ**が返るまでで解決し、その後のボディは無制限にストリームする。よって SSE / streaming 応答は（ヘッダさえ早く返れば）影響を受けない。
- **超過は fail-closed 504。** `x-plecto-fault: upstream-timeout` を付けて 504 を返す（404 no-route / 502 upstream-error / 503 no-healthy-upstream と別 fault）。素通り（fail-open）させない。
- **タイムアウトは health 信号にしない。** 単発のタイムアウトで passive eject はしない——[[000017]] の passive ejection が「**接続失敗のみ**」を見るのと一貫させ、生死判定は active プローバに委ねる（誤検知 flapping を避ける）。
- **timeout は `health` と独立。** `reconcile` は `health` の一致で instance を再利用するが、`request_timeout` はそこに含めない。timeout だけ変えた reload はグループを作り直すが instance の health を保つ。

検討した代替案と却下理由:
- **グローバル固定タイムアウト** — 単純だが backend ごとに SLA が違う。per-upstream を採り、per-route override は後続（必要になったら）。
- **ボディ完了までを区切る** — SSE / streaming / 大きなダウンロードを誤って切る。ヘッダ到達までに限定し、ボディ idle timeout は後続の再検討条件に。
- **タイムアウトで passive eject** — 可用性には効くが単発遅延での flapping リスク。[[000017]] の「接続失敗のみ passive」方針と揃え、active プローバに委ねた。
- **タイムアウト時に別 instance へ retry** — 可用性は上がるが body 既送・冪等性の問題でスコープが膨らむ（[[000017]] の retry 再検討条件と同じ）。passive と同様、retry は後続。
- **`0 = 無効` を許さない（常に強制）** — 安全側だが long-poll / streaming upstream を運用できなくする。明示オプトアウト（operator の判断）として `0` を許した。既定（省略）は 30s で安全側。

本決定は fail-closed テネットを upstream 転送の時間軸へ具体化し、[[000017]] の再検討条件 (c) を回収する。設計原則（安全 × 運用の単純さ）と矛盾しない。

## Consequences

### Pros
- 遅い / ハングした upstream に対してリクエストが無期限に張り付かず、504 で fail-closed に解放される。占有していたプール枠・接続も解放される。
- ヘッダ到達までの制限なので streaming / SSE 応答は影響を受けない。
- per-upstream 設定で backend ごとの SLA に合わせられ、`0` で streaming 専用 upstream をオプトアウトできる。
- [[000017]] が残した再検討条件 (c) を回収。

### Cons / Tradeoffs
- **ヘッダ到達まで**しか区切らない。ヘッダは早く返るが**ボディの途中で hang** する upstream は、このタイムアウトでは捕まらない（[[000017]] の「mid-response の hang は active probe 任せ」という穴がそのまま残る）。
- 既定 30s は backend によっては長すぎ / 短すぎる（per-upstream で調整は可能）。
- `0 = 無効` は明示すれば fail-open のオプトアウトになる（operator の責任）。
- **再検討条件**：(a) **ボディ / idle timeout**（ヘッダ後のストリーム停滞を区切る）；(b) タイムアウト時の **別 instance への retry**（冪等メソッド限定など、[[000017]] の retry 条件と連動）；(c) 反復タイムアウトでの passive ejection（flapping を抑える閾値付き）；(d) per-route override。
