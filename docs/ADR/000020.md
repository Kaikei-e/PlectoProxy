# plecto:filter を wasi:http（proxy / middleware）へ収斂させる方向を確定する

## Status
accepted

## Date
2026-06-25

## Affected Components
- wit world（`wit/world.wit`）— 現在の独自 http 型は v0.1 の便宜（[[000010]]：header-only・sync・安定ツールチェーンで動く）。M3 で wasi:http の型へ収斂させ、body / `stream<u8>` は wasi:http の型を再利用し、ワールドは `wasi:http/proxy` および開発中の `wasi:http/middleware`（HTTP ハンドラを import かつ export する＝プロキシ・ミドルウェア形）との相互運用を目標にする
- extension plane — 型を wasi:http に寄せても、ホストの `Linker` は引き続き **deny-by-default**：フィルタが必要とする最小の host-API だけを貸す（型語彙と capability 境界は別レイヤ）

## Context

`wit/world.wit` のコメント（[[000010]]）は一貫して「`stream<u8>` body・async・**`wasi:http` 型再利用は後の increment**」としてきた。理由は、当時の安定ツールチェーン（wasmtime 45）でフラグ無しに動くワールドを優先したため。その「後」を縛る外部状況が動いた:

- **wasmtime 46（2026-06-22）** が WASI 0.3 ＋ Component Model async を既定で載せ、`wasi:http` の成熟が進んだ。HTTP ハンドラを **import かつ export** する `wasi:http/middleware` ワールドが開発中で、これはまさにプロキシ・ミドルウェア（Plecto のフィルタ）の用途。
- review f000005 が「`plecto:filter` を独自に保つか `wasi:http` に収斂させるかを **ADR 1 本で明示せよ**」と指摘。後から WIT を直すコストが大きいため、方向を先に固める価値がある。

これは設計 tenets 内の緊張を含む Fork:

- **ポータビリティ / エコシステム相互運用**（Plecto の中核原則の一つ）は**収斂**を推す——`wasi:http` 向けに書かれたフィルタが Plecto で動き、Plecto 向けフィルタが他ランタイムでも動く。独自型のメンテと契約ドリフトを避けられる。
- **deny-by-default の最小 capability・運用の単純さ**は、小さく自前の契約を推す——`wasi:http` は header-only の最小契約より大きい。

## Decision

- **収斂を採る。** `plecto:filter` の独自 http 型を、M3 で **`wasi:http` の型へ収斂**させる。request / response / body / `stream<u8>` は `wasi:http` の型を再利用し、ワールドは `wasi:http/proxy` ＋ `wasi:http/middleware` との相互運用を目標にする。`wasi:http` 向けフィルタが Plecto で（その逆も）動くことを狙う。
- **収斂は M3 で漸進的に行う。** [[000003]] の 2 段トリガに沿う——Stage 1（ホストが wasmtime 46 で P3 を走らせる）→ Stage 2（`wasm32-wasip3` の Tier 化 / wit-bindgen async 成熟でゲストが実用 DX で書ける）。型の置換は body 契約と同じタイミングで入れ、本固定は streaming が枯れるまで待つ（[[000010]] の Stage 2 判断は維持）。
- **deny-by-default は型と独立に保つ。** 型を `wasi:http` から借りても、ホストの `Linker` は最小 capability だけを貸す姿勢（[[000006]]）を変えない。`wasi:http` 型の採用＝広い capability の付与ではない。
- **typed decision（Tenet 3）の写像を M3 で設計する。** Plecto の `decision`（`continue` / `modified` / `short-circuit`）と `wasi:http/middleware` の handler 形（import＋export）の対応づけは、収斂時の具体設計事項として M3 に持ち込む。

検討した代替案と却下理由:
- **独自 `plecto:filter` を恒久維持** — 契約を完全制御でき最小に保てるが、エコシステム分断・`wasi:http` 型の再発明・契約ドリフトのコストが累積する。却下。
- **独自維持＋型だけ漸進再利用（収斂はコミットしない）** — 折衷だが「いずれ寄せるかもしれない」という未決を残し、後悔の芽（contract ドリフト）を断てない。今回**収斂をコミット**する判断を採った。
- **いま即収斂（M3 を待たず main で）** — 46 はリリース直後でゲストツール（`wasm32-wasip3` は Tier 3）が未成熟。本固定は時期尚早。漸進・M3・別ブランチで進める（[[000003]] / README の Stage 2 判断と整合）。

本決定は **ポータビリティ tenet** を WIT 契約レイヤへ具体化し、[[000010]]（独自型・header-only は v0.1 の便宜）の「`wasi:http` 型再利用は後で」を「**収斂する**」という確定方向に更新する。deny-by-default（[[000006]]）は型語彙と独立に維持する。

## Consequences

### Pros
- エコシステム相互運用：`wasi:http` 向けフィルタが Plecto で動き、Plecto フィルタが他ランタイムでも動く余地。polyglot SDK（[[000010]] / M6）が馴染みある契約に乗る。
- `wasi:http` の成熟（型・canonical ABI の `stream`/`future`）を借りられ、自前型のメンテが不要。body の zero-copy 目標（`component-model-bytes` の `Bytes` 直接 lift/lower）に繋ぎやすい。
- 契約ドリフト（独自型と業界標準の乖離）を将来にわたって断てる。

### Cons / Tradeoffs
- `wasi:http` は header-only の最小契約より**大きい**。deny-by-default の最小 capability を、型を借りつつ別レイヤで守り続ける設計負担が要る。
- typed `decision` ↔ `wasi:http/middleware` handler の写像に設計が要る（収斂時の具体作業）。
- `wasi:http/middleware` は**開発中**でタイムライン risk があり、`wasm32-wasip3` は Tier 3。収斂は M3 Stage 2 のゲート（ゲストツール成熟）に従属する。
- **再検討条件**：(a) `wasi:http/middleware` が停滞、または deny-by-default に切り出せない（最小 capability に分解できない）と判明したら、「独自ワールド＋`wasi:http` 型再利用」へフォールバック；(b) 収斂可否は **M3 Stage 2 ゲート**で最終確認する（型置換を本固定する直前）。
