# wasmtime 46 を機に M3 Stage 1（ホスト async 化）を別ブランチで先行し、body 契約と header バイト等価を Stage 2 へ凍結する

## Status
accepted

## Date
2026-06-25

## Affected Components
- host runtime（`plecto-host`）— `bindgen!(async: true)` 化、`host_*::Host` 実装と `Linker` 配線を 46 の `*WithStore` トレイト形状変更（store 型パラメータがメソッドではなくトレイト自体に付く）へ追従、`run_pooled` を async / fiber 前提で再設計（epoch deadline 発火・trusted プールの recycle / circuit breaker の fiber 上挙動）、redb は async host call 着地と同時に blocking プール裏へ
- build / deps — `wasmtime = "46.0.0"` 完全固定、移行は **`main` ではなく別ブランチ**
- filter contract（WIT）— `stream<u8>` body 契約は Stage 2 まで凍結。header 値バイト等価（P3#6）は M3 の契約進化で解消する deferred 項目
- server（`plecto-server`）— body 契約着地時に `spawn_blocking` ブリッジを async host 呼び出しへ置換

## Context

[[000010]] は v0.1 を sync・header-only・自前 http 型で実装し、async / `stream<u8>` body / `wasi:http` 型再利用を **2 段トリガ**で繰り延べた——(1) host が P3 を走らせられる＝**wasmtime 46 リリース**、(2) P3 ゲストを実用 DX で書ける＝`wasm32-wasip3` の Tier 化 / wit-bindgen async 成熟。これらの ADR（000010 / [[000003]] / [[000005]]）は **46 リリース前（2026-06-20、当時「目前」）**の記述で、「46 が実際に出た後にどう着手するか」までは決めていなかった。

**確定事実（wasmtime 46.0.0、2026-06-22 リリース）**——よって trigger (1) は満たされた:
- WASI 0.3.0 が既定サポート、`component-model-async` wasm feature が既定有効（フラグ無しの安定ツールチェーンで P3 ホストが組める）。
- MSRV は Rust 1.94 で、Plecto の CI ピン（1.96）が充足。edition 2024 も問題なし（**ツールチェーン変更不要**）。
- `component-model-bytes`（旧 `component-model-async-bytes`）で `Bytes` / `BytesMut` の直接 lift/lower がサポート。`plecto-server` の body パイプラインは既に `bytes::Bytes`（`BoxBody<Bytes>` / `H3ReqBody`）で構成されており、将来の body zero-copy 経路が噛み合う**追い風**。
- bindgen の `*WithStore` トレイトが store 型パラメータをトレイト自体に持つ形へ変更。Plecto は `bindgen!` で `Filter` / `FilterPre` を生成し各 `host_*::Host` を手実装しているので、ここが async 化の最初の移行タッチポイント。
- fuel は bulk-data-transfer 命令でサイズ比例消費に変更されたが、Plecto は [[000006]] で **epoch interruption** を選んでおり無関係（epoch over fuel の判断は据え置き）。

**projected（未確定・計画）**:
- `wasm32-wasip3` は依然 **Tier 3**（CI 未テスト）で、Tier 2 化は 2026 Rust プロジェクトゴールだが未達。実務上の抜け道は `wasm32-wasip2` をビルドターゲットにしつつ P3 の WIT を wit-bindgen async で対象にする方法（wasmCloud / nats-wasip3 で実証）。
- コンポーネント間 stream コピーは 46 でもまだ修正が入る段階（streaming は枯れていない）。

したがって設計上の緊張は「46 が出た瞬間に body 契約まで一気に進めると、guest 側（wasip3 Tier 3 / wit-bindgen async）が未整備で再び詰まる」——これは 000010 / 000003 が既に警告していたとおり。host 移行と body 契約を分離するのが低リスク。

## Decision

- **Stage 1（ホスト async 化）をいま着手する。`main` ではなく別ブランチで。** `wasmtime = "46.0.0"` を**完全固定**（`"46"` で floating させない——async / bindgen の API がまだ動いている時期のため）。これは [[000010]] の再検討条件 trigger (1) の発動である。
- **着手順を固定する。** (a) `bindgen!` を async 化 → (b) `host_*::Host` 実装と `Linker` 配線を `*WithStore` 形状変更へ追従 → (c) `tests/conformance.rs` が green に戻るか確認。ここで **「async だが header-only」の中間地点**に到達し、`spawn_blocking` 撤去のリスクを body 契約から切り離して検証できる。
- **`run_pooled` を async 再設計する。** checkout → `set_epoch_deadline` → call → checkin を `async_support(true)` + Component Model async + fiber 上で組み直す。wasm 実行が `Future::poll` の中で起きる緊張は既知で、epoch deadline 発火・trusted プールの recycle / circuit breaker（[[000012]]）の fiber 上挙動を設計し直す。redb の blocking プール化（[[000004]] のシーム）は async host call 着地と同時に確定する。
- **production の `stream<u8>` body 契約は凍結する（Stage 2 まで）。** streaming（長寿命ストリーム・バックプレッシャ・接続ライフサイクル）が枯れるまで本固定しない。並行して `component-model-bytes` 前提の body フィルタ **spike を `wasm32-wasip2` + wit-bindgen async で 1 本**回し、契約の書き味だけ先に確認する（本固定はしない）。
- **header 値のバイト等価素通し（P3#6）を deferred として明記する。** 現在 `headers_to_vec` は WIT が `string` 契約ゆえ全ヘッダ値を `from_utf8_lossy` で通すので、フィルタが触らないヘッダでも非 UTF-8 / 特殊バイトが U+FFFD に化けて転送されうる。理想は「フィルタ非接触ヘッダはバイト等価で素通し」——これは body の opaque pass-through（[[000005]] 機構2）と同じ**型レベル分離**の発想で、M3 の契約進化（[[000020]] の `wasi:http` 型採用、ないし header に byte 保持表現を足す）でまとめて解消する。v0.1 は `string` の lossy 変換を許容する（header-only ゆえ実害は限定的）。

検討した代替案と却下理由:
- **`main` 上で直接移行する** — 46 はリリース 2 日後で async / bindgen API がまだ動く時期。`main` を不安定化させる。別ブランチ + pin を採る。
- **46 で body 契約まで一気に進める** — guest 側（wasip3 Tier 3 / wit-bindgen async / wit-bindgen#1554 系の荒れ）が未整備で詰まる（000010 / 000003 の警告どおり）。host 移行を先に固め body は Stage 2 凍結。
- **`wasmtime = "46"` で floating** — minor で API が動く時期にビルドが壊れうる。完全固定。
- **epoch を fuel へ切り替える** — 46 で fuel が bulk-data 比例消費になったが、[[000006]] で epoch を選択済み。据え置き（移行の本丸は epoch × async × tokio/fiber の協調）。
- **P3#6 を v0.1 で個別修正（header に byte 表現を足す）** — body 型導入前に header だけ byte 化すると契約が二度揺れる。型レベル分離（M3）でまとめて解消する方が一貫。

本決定は **[[000003]]（Fork 1）が据えた async-first 方向を前進**させ、**[[000010]] の 2 段トリガ (1) を発動**するもの。データプレーン no-panic / fail-closed テネットは移行後も保つ（async 化で `RunError` の deadline / trap 分類と fail-closed 写像を崩さない）。

## 実装ノート（Stage 1 中間地点の着地・2026-06-26）

別ブランチ `feat/m3-stage1-host-async` で「async だが header-only・conformance green」の中間地点まで着地した。確定事項と、本文の Stage 1 配分からの精緻化:

1. **`wasmtime = "46.0.0"` 完全固定（Cargo.lock で実バージョン pin）。** 取得・ビルドとも問題なし。**sync コードは無改変のまま 46 で green**（workspace check ＋ host 56 テスト）——46 の `*WithStore` 等の変更は Plecto が手実装する sync `host_*::Host` を壊さなかった。
2. **bindgen async は exports-only。** 46 で top-level `async` キーは廃止され `imports` / `exports` ブロックへ移行。`exports: { default: async }` だけを採り、guest の `init` / `on-request` / `on-response` を `call_async` で fiber 上実行する。**host-API imports は sync のまま**——log / clock / kv / counter / ratelimit は trivial・非ブロッキングで async 化の必要がなく、impl もテストも無改変・設定表面も最小に保てる（ADR の「bindgen を async 化」を**最小 capability 方向へ精緻化**）。`Config::async_support` は 46 で deprecated / no-op なので呼ばない。
3. **公開境界は `block_on`（`pollster`）で sync 維持。** `instantiate_async` / `call_async` を leaf で `block_on` 包むだけにし、`instantiate_initialized` / `run_pooled` / dispatch のシグネチャを不変に保った。結果 **`plecto-control` / `plecto-server` は無改変**（`spawn_blocking` ブリッジはそのまま）。guest は epoch trap モード・非ブロッキング imports ゆえ fiber 上で suspend せず完走するので、no-reactor executor で十分。
4. **`run_pooled` の async / fiber 再設計は Stage 2 へ繰延（本文 (b) からの差分）。** block_on bridge では epoch × async × fiber の本格協調が不要なため、真の fiber 並行（deadline 発火・breaker / recycle の fiber 上挙動）は **server の `spawn_blocking` 撤去（Stage 2）と同時**に行うのが整合的。redb の blocking プール化も「async host call 着地」が Stage 2 ゆえ同時に繰延。
5. **検証:** `cargo fmt --all -- --check` / `cargo clippy --workspace --all-targets -- -D warnings` / host 56 テスト（conformance ＋ unit）/ `plecto-control` ＋ `plecto-server` 全テスト、いずれも green（wasmtime 46.0.0）。

## Consequences

### Pros
- 000010 / 000003 が据えた async-first 方向を、trigger (1) 充足で実際に前進させられる。
- host 移行（async bindgen + linker + `run_pooled`）を **「header-only async」の中間地点**で検証でき、body 契約のリスクと分離できる。`spawn_blocking` 撤去を body と切り離して確かめられる。
- `Bytes` 直接 lift/lower（#13366）と server の既存 `bytes::Bytes` パイプラインが噛み合い、将来の body zero-copy が繋がる。
- `main` は安定のまま（別ブランチ・pin）。MSRV はクリアでツールチェーン変更不要。

### Cons / Tradeoffs
- 移行ブランチの維持コスト（46 の async / bindgen API がまだ動くので追従が要る）。
- `run_pooled` の async × epoch × fiber 協調は既知の緊張（wasm 実行が `Future::poll` 内で起きる）で、deadline 発火・breaker・recycle の設計やり直しが要る。
- stream はまだ枯れていない（コンポーネント間 stream コピーが 46 でも修正中）ので body 契約は凍結のまま——**body-transform フィルタは引き続き未対応**。
- **P3#6（header バイト等価素通し）は M3 の契約進化まで未解消**（v0.1 は `from_utf8_lossy` のまま、フィルタ非接触ヘッダでも U+FFFD 化の可能性が残る）。
- **再検討条件**：(a) **Stage 2 ゲート** = `wasm32-wasip3` Tier 2 化 / wit-bindgen async 成熟 →（i）`stream<u8>` body 契約の本固定、（ii）P3#6 の型レベル解消；(b) 移行時チェックリスト = wit-bindgen#1554 / wasmtime#12714 / コンポーネント間 stream コピー修正の解消；(c) redb の blocking プール化は async host call 着地と同時；(d) [[000020]] の `wasi:http` 型採用を本 Stage の async 化と body 契約のどちらに乗せるかは移行中に確定。
