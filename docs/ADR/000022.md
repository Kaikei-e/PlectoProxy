# クライアント IP の剥がし family を de-facto ヘッダへ広げ、X-Real-IP も確定発行する

## Status
accepted

## Date
2026-06-25

## Affected Components
- server runtime（`plecto-server`）— `set_forwarded`（[[000018]]）の `FORWARDED_HEADERS` を de-facto クライアント IP ヘッダへ拡張（`X-Real-IP` / `True-Client-IP` / `CF-Connecting-IP` / `Fastly-Client-IP` / `X-Client-IP` / `X-Cluster-Client-IP` を追加）し、`X-Real-IP` を peer から**新規発行**。IPv4-mapped IPv6 peer（`::ffff:a.b.c.d`）を dotted IPv4 へ**正規化**して `X-Forwarded-For` / `X-Real-IP` 共通の値にする

## Context

[[000018]] は edge モデルを採り、「受信 `X-Forwarded-*` / `Forwarded` を剥がし、peer ＋ scheme から付け直す」と決めた。決定文は「**剥がしは family 全体**」と書いたが、実際に `FORWARDED_HEADERS` が列挙していたのは `forwarded` / `x-forwarded-for` / `x-forwarded-proto` / `x-forwarded-host` の 4 つに留まっていた。

しかし「クライアント IP」を表す de-facto ヘッダはこの 4 つの外にも広く使われている。代表的なのは nginx 慣習の `X-Real-IP`、および CDN 由来の `True-Client-IP`（Akamai / Cloudflare Enterprise）・`CF-Connecting-IP`（Cloudflare）・`Fastly-Client-IP`（Fastly）・`X-Client-IP` / `X-Cluster-Client-IP`。多くのバックエンドやフィルタは `X-Forwarded-For` ではなくこれらを「送信元 IP」として読む。

このとき [[000018]] の実装には穴が残る:

- **`X-Forwarded-For` だけ上書きしても、なりすました `X-Real-IP: 9.9.9.9` は素通りする。** `X-Real-IP` を読むバックエンドは、Plecto が確定した XFF ではなくクライアントの詐称値を信頼してしまう。edge モデルの目的（送信元 IP を詐称させない）が、ヘッダ 1 つの読み替えで破られる。これは [[000018]] と同じ review f000005 P2#3 のなりすまし面が family の取りこぼしとして残っていたもの。
- **dual-stack listener（`[::]`）で IPv4 クライアントが繋ぐと peer は IPv4-mapped IPv6（`::ffff:a.b.c.d`）になる。** `peer.to_string()` はそのまま `::ffff:a.b.c.d` を `X-Forwarded-For` に書くため、バックエンドやフィルタが `a.b.c.d`（dotted IPv4）で IP allowlist / rate limit すると一致しない。

Plecto のテネット（deny-by-default・fail-closed・「判断は信頼できる入力に対して下す」）からは、クライアントが自由に書ける**いずれの**送信元 IP ヘッダもそのまま信頼してはならない。本 ADR は [[000018]] の edge 既定をその趣旨どおりに完成させ、あわせて [[000018]] が「付与は For ＋ Proto のみ」とした最小発行の方針を一点だけ更新する。

## Decision

[[000018]] の `set_forwarded` を次のように広げる。チェーン実行の**前**・全 HTTP バージョン共通の `proxy_core` 経路で適用する点は [[000018]] のまま不変。

- **剥がし family を de-facto クライアント IP ヘッダへ拡張する。** `FORWARDED_HEADERS` に `x-real-ip` / `true-client-ip` / `cf-connecting-ip` / `fastly-client-ip` / `x-client-ip` / `x-cluster-client-ip` を追加し、case-insensitive で全て剥がす。これは [[000018]] の「剥がしは family 全体」の忠実な補完であり、列挙不足を埋めるもの。
- **`X-Real-IP` を peer から確定発行する（[[000018]] の「付与は For ＋ Proto のみ」を更新）。** `X-Real-IP` を読むバックエンドが `X-Forwarded-For` を解さなくても Plecto の確定 peer を得られるようにする。`X-Real-IP` を**剥がすだけ**だと、それを読むバックエンドは空値を見て、構成によっては別経路へ fallback して信頼判断が揺れる。発行して authoritative にする方が運用が単純（運用の単純さテネット）。発行は `X-Forwarded-For` / `X-Real-IP` / `X-Forwarded-Proto` の 3 つに留め、CDN 系ヘッダは剥がすが発行しない（最小限の確定値）。
- **IPv4-mapped IPv6 peer を dotted IPv4 へ正規化する。** `Ipv6Addr::to_ipv4_mapped()` で `::ffff:a.b.c.d` を `a.b.c.d` に畳み、`X-Forwarded-For` と `X-Real-IP` は同一の正規化値を使う。これにより dual-stack デプロイで XFF が予測可能になり、広く使われるプロキシ実装が出す値と揃う。本物の IPv6 peer はそのまま（bare address、XFF の慣習どおり bracket を付けない）。

検討した代替案と却下理由:

- **[[000018]] の 4 ヘッダを厳守し、X-Real-IP / CDN family は触らない** — `X-Real-IP` を読むバックエンドで edge のなりすましが残り、fail-open 相当になる。安全側に family を広げた。
- **拡張ヘッダを剥がすが `X-Real-IP` は発行しない（剥がしのみ）** — [[000018]] の minimalism は保てるが、`X-Real-IP` 前提のバックエンドが空値を見て fallback で揺れうる。発行して確定値を渡す方が運用が単純なので、発行範囲の更新を本 follow-up ADR で明示した。
- **client-IP variant を網羅列挙する**（`X-Original-Forwarded-For` / `X-ProxyUser-Ip` / `X-Azure-ClientIP` / `X-Appengine-User-Ip` 等も全て） — 列挙の tail は際限がなく、広く使われる主要ヘッダに絞るのが現実的。tail は再検討条件に回す。
- **正規化しない（`::ffff:` のまま発行）** — dual-stack で IPv4 allowlist が外れ、運用上の驚きになる。予測可能な dotted IPv4 を採った。
- **IPv6 を RFC 7239 `Forwarded` で bracket＋quote 発行する** — `Forwarded` 発行は [[000018]] で再検討条件に回し済み。XFF は bare address 慣習なので正規化に留めた。

本決定は [[000018]] の決定（family の範囲・発行ヘッダ）を更新し、deny-by-default / fail-closed テネットを「送信元メタデータ」へさらに具体化する。設計原則（安全 × ポータビリティ × セルフホスト性 × 運用の単純さ）と整合する：安全側の既定（family 全体を剥がす）を保ちつつ、バックエンド互換のための確定発行は最小（`X-Real-IP` 1 つ）に絞った。

## Consequences

### Pros
- `X-Real-IP` / CDN 系ヘッダを読むバックエンド・フィルタも、なりすました送信元 IP を信頼しない（edge 既定の穴を塞いだ）。
- `X-Real-IP` 前提のバックエンドが、`X-Forwarded-For` を解さずとも Plecto の確定 peer を得る。
- dual-stack listener で `X-Forwarded-For` / `X-Real-IP` が予測可能な dotted IPv4 になり、IP allowlist / rate limit が安定する。
- 全 HTTP バージョンが同一 `proxy_core` 経路に乗り（[[000018]] と同じ）、`plecto:filter` の WIT 契約は不変（host-native のヘッダ操作のみ）。

### Cons / Tradeoffs
- 発行ヘッダが 1 つ増え（`X-Real-IP`）、[[000018]] の「付与は For ＋ Proto のみ」という minimalism を一点緩めた。`X-Real-IP` を解さないバックエンドには無害だが、ヘッダは増える。
- Plecto を**信頼できる前段 CDN / LB の背後**に置くと、edge 既定は `CF-Connecting-IP` 等の CDN ヘッダも剥がすため、前段が付けた本物の送信元 IP が消える。この構成は [[000018]] の trusted-proxy（追記）モードで family ごと扱う。
- **再検討条件**：(a) **trusted-proxy（追記）モード** — 信頼ホップ構成で受信 `X-Real-IP` / `CF-Connecting-IP` 等を尊重するか、family ごと設定可能化する（[[000018]] の再検討条件 (a) と統合）；(b) 列挙の **tail**（`X-Original-Forwarded-For` 等）の追加が要件化したとき；(c) RFC 7239 `Forwarded` 発行（[[000018]] から継続）。
