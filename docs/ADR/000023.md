# タイムアウト・接続失敗時に別の healthy instance へ bounded retry する（冪等・bodyless 限定）

## Status
accepted

## Date
2026-06-25

## Affected Components
- server runtime（`plecto-server`）— `proxy_core` の forward を retry ループに変える。timeout（応答ヘッダ未達、ADR 000019）と connect 失敗（ADR 000017）を retryable failure とし、`pick_excluding` で**別の**healthy instance へ `max_retries` 回まで再送する。timeout retry は**冪等メソッド限定**、connect retry は**任意メソッド**（upstream が未受信）。いずれも **bodyless 限定**。`Failure` / `is_idempotent` / `may_retry` / `empty_req` を追加
- control plane（`plecto-control`）— `Upstream` に `max_retries`、`UpstreamGroup` に `max_retries: u64` ＋ getter ＋ `pick_excluding`。`reconcile` が manifest 値から再構築するが `health` には含めないので、retry だけの変更は instance の health を保ったままグループを差し替える
- manifest — `[[upstream]]` に任意の `max_retries`（既定 1、`0` で無効）

## Context

M2 fast path（[[000013]] / [[000017]] / [[000019]]）は upstream 転送の失敗を「その場で fail-closed」するだけだった——接続失敗は passive eject ＋ 502、タイムアウトは 504。**別 instance への再送（retry）は無い**。可用性の観点では、安全に再試行できる失敗は別の healthy instance へ回したい。

これは過去 ADR が明示的に残した再検討条件である。[[000017]] は再検討条件 (b)「接続失敗時の **request-level retry**（冪等メソッド限定など）」を、[[000019]] は再検討条件 (b)「タイムアウト時の **別 instance への retry**」を残した。両者は同じ retry 機構で回収できる。

retry は安全性の論点が多い。(1) **冪等性**——タイムアウトは「upstream が受信・処理済みかもしれない」状態なので、非冪等メソッド（POST/PATCH）を再送すると二重実行になる。対して接続失敗は upstream が **1 byte も受信していない**ので、任意メソッドで再送して安全。(2) **body 再送**——クライアントからの body は opaque にストリーム透過する（[[000013]] / [[000010]]、ゼロコピー・フィルタ税ゼロ）ため、instance 1 に流したら instance 2 へは再生できない。buffer すれば再送できるが、それは opaque body 原則に反しメモリ・DoS 面を生む。(3) **retry storm / レイテンシ**——無制限な retry は増幅を生み、各試行が独立 timeout を持つとレイテンシが伸びる。(4) **flapping**——タイムアウトを health 信号にすると単発遅延で eject が揺れる（[[000019]] が意図的に避けた点）。

## Decision

[[000017]] (b) と [[000019]] (b) を**同時に**回収し、安全に再試行できるケースに限定して別の healthy instance へ bounded retry する。

- **per-upstream `max_retries` を manifest に入れる。** 既定 **1**、**`0` で無効**。`UpstreamGroup` が `u64` として保持し、route は `Arc<UpstreamGroup>` 経由で読む。安全ケース限定なので既定 on（1）とした（nginx の `proxy_next_upstream` が `error timeout` を既定 on にするのと同じ姿勢）。
- **retryable failure は timeout と connect 失敗の二つ。** timeout retry は**冪等メソッド限定**（RFC 9110 §9.2.2: GET/HEAD/PUT/DELETE/OPTIONS/TRACE）。connect 失敗は upstream が未受信なので**任意メソッド**で安全に retry する。非 connect の transport fault（mid-response 等）は retry せず、health 信号にもしない（[[000017]] 踏襲）。
- **bodyless 限定（buffer しない）。** opaque streamed body（[[000013]]）は再生不可なので、**body の無いリクエストだけ** retry する。判定は hyper の framing 正確な `body.size_hint().exact() == Some(0)`（false-positive を出さない＝body 付きを誤って空再送しない）。body 付きは従来どおり単発で 504/502。
- **別の healthy instance へ。** `pick_excluding(&failed)` が round-robin で**直前に失敗した instance 以外**の healthy を返す（Arc 同一性で除外）。別 instance が無ければ（単一インスタンス等）retry せず、その失敗の fault（504 / connect は 502）を即返す——単一インスタンスで同じ instance を再試行してレイテンシを二重化しない。
- **timeout は health 信号にしない。** [[000019]] を踏襲し、本 ADR でも (c)（反復タイムアウトでの passive eject）は採らない。connect 失敗は従来どおり passive eject も行う（[[000017]]）。各試行は独立に `request_timeout` を持つので、最悪レイテンシは概ね (retry 数 + 1) × `request_timeout`。
- **timeout は `health` と独立。** `reconcile` は `health` の一致で instance を再利用するが `max_retries` は含めない（[[000019]] の `request_timeout` と同じ）。retry だけ変えた reload は instance の health を保つ。

検討した代替案と却下理由:
- **body を buffer して body 付き冪等リクエストも retry する** — 可用性は上がるが、[[000013]] のゼロコピー opaque body 原則に反し、buffer がメモリ消費・DoS 面を生む。bodyless 限定に留め、bounded buffering は再検討条件へ。
- **タイムアウトでも非冪等メソッドを retry する** — 二重実行リスク。timeout は冪等限定、connect（未受信）のみ任意メソッド、と安全プロファイルで分けた。
- **`max_retries` 既定 0（opt-in）** — 最も保守的だが、retry を安全ケース（冪等 or connect・bodyless・別 instance）に限定済みなので、既定 1 で可用性を取りつつ安全側を保てる。
- **固定 1 回・設定不可** — 最小実装だが backend ごとの調整ができない。[[000017]] / [[000019]] の per-upstream 設計に揃えて knob 化した。
- **タイムアウトで passive eject も同時に入れる（[[000019]] (c)）** — flapping リスクがあり閾値設計を要する。本スライスは retry のみとし、eject は別条件に残した。
- **全 tried instance を除外して retry する** — `pick_excluding` は**直前のみ**除外する（既定 `max_retries = 1` では完全；`> 1` では 3 回目以降が既試行に戻りうる）。全 tried 追跡は単純さとのトレードオフで見送り、再検討条件へ。

本決定は fail-closed テネット（安全に再試行できないものは素通りさせず従来の fault）と「安全 × 運用の単純さ」を保ちつつ、[[000017]] (b) と [[000019]] (b) を回収する。`plecto:filter` の WIT 契約は不変（host-native のみ）。

## Consequences

### Pros
- タイムアウト / 接続失敗が、安全なケースで別の healthy instance により救済され、可用性が上がる。
- 安全ケース限定（timeout は冪等・connect は任意メソッド／いずれも bodyless／別 instance）なので、二重実行と body 再送問題を回避する。
- [[000017]] (b) と [[000019]] (b) を一度に回収。`max_retries` は per-upstream で backend ごとに調整でき、`0` で無効化できる。
- `health` と独立なので retry だけの reload が instance health をリセットしない。全 HTTP バージョンが同一 `proxy_core` 経路に乗り、契約は不変。

### Cons / Tradeoffs
- **body 付きリクエストは retry されない**（PUT with body 等は従来どおり 504/502）。`size_hint` が body 不明・非空なら bodied 判定なので、GET-with-body のような病的ケースも安全側に倒れて retry されない。
- 既定 `max_retries = 1` は、タイムアウト時の最悪レイテンシを最大およそ 2 × `request_timeout` にしうる（各試行が独立 timeout）。
- `max_retries > 1` では retry が**直前の instance のみ**除外（全 tried を追跡しない）ので、3 回目以降が既試行 instance に戻りうる。
- タイムアウトは依然 health 信号ではない（[[000019]] (c) 未採用）。反復タイムアウトの instance は active prober 任せ。
- **再検討条件**：(a) **ボディ / idle timeout**（ヘッダ後のストリーム停滞、[[000019]] (a)）；(b) 反復タイムアウトでの **passive ejection**（flapping 閾値付き、[[000019]] (c)）；(c) **per-route override**（[[000019]] (d)）；(d) body 付き retry のための **bounded buffering**；(e) **全 tried instance 除外** / per-request の retry 時間上限（budget）。
