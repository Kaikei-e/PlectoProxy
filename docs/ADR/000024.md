# round-robin を healthy 集合上で回し、reload で RR カーソルを引き継ぐ

## Status
accepted

## Date
2026-06-26

## Affected Components
- control plane（`plecto-control`）— `UpstreamGroup::pick` / `pick_excluding` を `pick_inner(exclude)` に集約。グローバルカーソルから前方スキャンして最初の healthy を返す旧実装を、**healthy（retry 時は `exclude` を除く）集合の要素数 `eligible` で `rr % eligible` を割って index する**実装に置換。`reconcile` は新 `UpstreamGroup` の `rr` を、同名の旧グループの `rr` から移送する（health policy / instance 集合の変化に依らず移送。カーソルは単なる回転カウンタ）

## Context

[[000017]] は「healthy な instance 集合を round-robin で分配する」と決め、`pick` を実装した。だがその実装は**グローバルカーソルから前方スキャンして最初の healthy を返す**形だった。3 台 `[a, b, c]` で中央の b を eject すると、カーソル `start` の剰余ごとに `a / c(b→c) / c` を返し、**死んだ instance の「枠」がその次の instance に吸われる**。結果、縮退時の分配が `a:c = 1:2` に偏る。

- **定量根拠（本 ADR の RED テストで再現）**：`[a, b(down), c]` で 600 回 `pick` すると `a=200, c=400`（厳密に 1:2）。これは core-pinned ベンチで観測された「縮退時 ~1:2」と一致する。
- 既存テスト `round_robin_distributes_over_healthy_only` は **pick の集合（healthy のみが返るか）しか検証しておらず**、分配の偏りは素通ししていた。全 healthy 時は前方スキャンでも厳密に均等なので、偏りは縮退時だけ顕在化する。
- 影響：部分障害下で各 dead instance の隣が約 2 倍の負荷を負う。N 台中 1 台落ちで 1 台が過負荷になりうる、availability・perf 双方の実害。

あわせて軽微な関連バグ：`reconcile` が新グループを `rr: AtomicUsize::new(0)` で作るため、**reload 直後の最初の pick が必ず instances[0] に偏る**（頻繁な reload で 0 番へ僅かに偏る）。カーソルは health 状態とは独立な回転カウンタなので、[[000017]] が health 状態を reload 跨ぎで生存させたのと同じ精神で、カーソルも生存させるべきだった。

本 ADR は [[000017]] の「round-robin over healthy」を**実装レベルで正す**もので、fail-closed テネット（全 unhealthy で 503）と「hot path は lock-free」「データプレーン no-panic」（`CLAUDE.md`）を崩さないことを条件とする。

## Decision

- **eligible 集合上で回す。** `pick` / `pick_excluding` を `pick_inner(exclude: Option<&Arc<UpstreamInstance>>)` に集約。eligible = `is_healthy()` かつ（retry 時は）`!Arc::ptr_eq(inst, exclude)`。**1 パス目で eligible の数 `n` を数え、`rr.fetch_add(1) % n` で対象 index を決め、2 パス目でその index の eligible を返す。**死んだ／除外された枠が隣に吸われないので、縮退時も eligible 集合上で均等になる（全 healthy 時の挙動は不変）。
- **アロケーションフリーを保つ。** 2 パスとも `instances` を走査するだけで、ヒープ確保も追加 lock も無い（[[000017]] の hot-path lock-free 性質を維持）。`SmallVec` 等のスナップショットは採らない（依存追加で `deny.toml` を触る）。
- **パス間 race は安全側に倒す。** 1 パス目と 2 パス目の間に instance の health が flip しても、2 パス目は「最後に見た eligible」を保持して返す（`target` に届かず走り抜けた場合のフォールバック）。**eligible が 1 つでもあれば spurious な `None`（不要な 503）を返さない。**flip で本当に 0 になった場合のみ `None`＝fail-closed。
- **reload でカーソルを移送する。** `reconcile` は同名の旧グループの `rr` を読み、新グループの初期値にする。instance 集合や health policy が変わっても移送する（カーソルは整数の回転カウンタで、新しい eligible 数での剰余として意味を保つ）。乱数 seed はしない。

検討した代替案と却下理由:
- **前方スキャンのまま（現状維持）** — 死んだ枠が隣に吸われる構造的偏りが残る。却下（本 ADR の主目的）。
- **healthy を `SmallVec` にスナップショットして `healthy[rr % len]`** — 読みは 1 パスで済むが、`smallvec` 依存の追加が `deny.toml`（cargo-deny の CI ブロッキング）とサプライチェーン面を増やす。2-pass・アロケーションフリーで同じ均等性とほぼ同じ race 頑健性が得られるので、依存を足さない方を採った。
- **カーソルに乱数 seed を入れて reload リセットを散らす** — 0 番偏りは消えるが、round-robin の決定性（テスト容易性・再現性）を捨てる。移送なら決定性を保ったまま偏りも消える。却下。
- **reconcile で health policy / instance が変わったらカーソルを 0 にリセット** — カーソルは health とも instance 同一性とも独立なので、変化時にリセットする理由が無い。常に移送する方が単純で偏りも残さない。

本決定は [[000017]] の round-robin/health 設計を実装レベルで具体化・修正する。fail-closed（eligible 0 で `None`→503）と hot-path lock-free、データプレーン no-panic（`get`/`iter` のみ、添字 panic 無し）を保つ。

## Consequences

### Pros
- 縮退時の分配が eligible 集合上で均等になり、dead instance の隣が過負荷になる実害が消える（RED テストで `a:c` が `200:400`→`300:300`）。
- 全 healthy 時の round-robin 挙動は不変（既存テスト `round_robin_distributes_over_healthy_only` 緑のまま）。
- reload 直後の instances[0] 偏りが消える（カーソル移送）。health 状態と同じく「reload を生き残る」一貫性。
- アロケーションフリー・lock-free を維持。`pick`/`pick_excluding` の重複ロジックが `pick_inner` に集約され読みやすい。

### Cons / Tradeoffs
- `pick` が `instances` を 2 回走査し `is_healthy()` を 2 回読む（従来は最悪 1 パス）。典型プールは小さく、いずれもアロケーション無しの `AtomicBool` 読みなのでコストは無視可能だが、O(n) の定数は僅かに増える。
- 1 パス目と 2 パス目の間の health flip は「最後の eligible へフォールバック」で吸収する（厳密なスナップショット一貫性ではない）。これは round-robin の公平性が瞬間的に 1 つズレるだけで、安全側（spurious 503 を出さない）。
- カーソル移送は instance 集合が変わっても剰余で回るだけで、「どの instance に対応する位置か」を remap しない（カーソルは整数のまま）。意図どおりで負債ではないが、将来 least-conn/EWMA を入れるときは別設計になる。
- **再検討条件**：(a) 偏りが接続数/負荷レベルで問題化したら least-conn/EWMA へ（[[000017]] 再検討条件 (a) を継承。そのとき本 ADR の純カウンタ round-robin は置換される）；(b) 2-pass の定数コストがベンチで有意になったら、`is_healthy` のスナップショットを 1 度だけ取る形（アロケーション無しの固定長バッファ等）を再検討。
