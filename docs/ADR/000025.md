# body プレーンを wasi:http 収斂から分離し、自前 body 契約を wasm32-wasip2 で先行する

## Status
accepted

## Date
2026-06-27

## Affected Components
- filter contract（`wit/world.wit`）— body を契約に入れる。body 型は **`body` resource**（`stream<u8>` ＋ `future<trailers>` を包む）。header hook と body hook を**別 export**にし、body hook は `body-decision { continue(body) | short-circuit(http-response) }` を返す
- extension plane（`plecto-host`）— body hook 非 export のフィルタへ body を渡さず stream を**ゼロコピー bypass**。host-API は不変。example/conformance フィルタの build target を `wasm32-wasip2`（＋ wit-bindgen async）へ
- fast path（`plecto-server`）— body-transform ルートは `continue` 後に upstream へ転送（buffer-then-decide）
- roadmap — `wasi:http` 収斂を body から分離し gated な別マイルストーンへ

## Context

[[000020]] は `plecto:filter` を `wasi:http`（proxy / middleware）型へ**収斂**させる方向を確定し、その型置換を **body 契約と同じタイミング**で入れるとした。同時に再検討条件 (b) で「収斂可否は **M3 Stage 2 ゲート**で最終確認する」と置いた。[[000021]] は M3 Stage 1（host async・header-only）を着地させ、`stream<u8>` body 契約と `spawn_blocking` 撤去を Stage 2 へ凍結した。[[000005]] は header-only と body-transform を契約で分け、body 非接触フィルタへ body をゼロコピー bypass する「機構2」を named-deferred とし、その様式（host 側 enum か WIT 自己申告か）は body 増分時に決めるとした。本 ADR はこの **M3 Stage 2 ゲート**に立ち、収斂可否と body 契約の具体形を確定する。

**確定事実（2026-06-27 の web 調査、一次情報で検証）— runtime はもう障害ではない**:
- `wasi:http` 0.3.0 は **FINAL**（2026-06-11 批准、`prerelease:false`）。`proxy` world は消え `service` + `middleware`（`include service; import handler`）へ分割確定。
- **wasmtime 46（2026-06-22）は `wasi:http` 0.3 の `service` / `middleware` を既定で serve**。旧ブロッカー issue #12714 は **2026-03-30 に Alex Crichton が「spec-compliant＝バグでない」として close**——本質は WIT スナップショット日付の不一致 × 厳格リンカ（guest がビルドした WIT と wasmtime 内蔵の WASI snapshot がズレると `resource implementation is missing` で正しく弾く）。**snapshot を同期すれば現在も正常リンク・起動する。**
- コンポーネント間 `stream<T>` copy は **修正済み**（wasmtime PR #13418、46 に同梱）、guest-drop の stream leak も close。

**projected（未確定・guest 側が弱点）**:
- `wasm32-wasip3` は依然 **Tier 3**（nightly 専用）。Tier 2 化は 2026 Rust プロジェクトゴール（goal #656）だが未着地・promotion MCP も未提出。
- **wit-bindgen#1554 は OPEN だが「コードの blocker」ではない**——runtime 側（#12714）は spec-compliant として close 済みで、本 issue は wit-bindgen 側の **UX / エコシステム同期の placeholder**。WASIp3 が GA で WIT 型が固定されるまで、古い RC WIT を持ち込めば再現する（報告は wit-bindgen 0.51 / wasmtime 41 / `0.3.0-rc-2026-01-06` の旧版）。問題は date-stamped な `wasi:http/types` を import する**収斂パス固有**で、**自前 WIT はこの snapshot-sync クラスを構造的に丸ごと回避する**。収斂自体は snapshot 同期＋`-Sp3` / `component-model-async` フラグで今でも到達可能だが、experimental な WASIp3 の破壊的 RC 間 **snapshot 同期 treadmill** を負う。
- `component-model-async`（stream/future）API は 46 で既定有効だが公式に「very incomplete」。streaming のバックプレッシャは現状不安定（wasmCloud #5028）。

ここから出る含意が本決定の核心: **これらの guest 側ブロッカーが gate するのは `wasi:http` への収斂であって、body そのものではない。** `stream<u8>` は WASI ではなく Component Model の機能なので、自前 WIT に持てば `wasm32-wasip2`（Tier 2・安定）でビルドでき、`wasi:http/types` の resource 形を import しないので #1554 を構造的に踏まない（wasmCloud が実証するパターン）。よって body は今すぐ着手でき、収斂だけを待てばよい。

## Decision

- **body と `wasi:http` 収斂を分離する。** [[000020]] の再検討条件 (b) を発動し、収斂（`wasi:http` 0.3 型再利用・`middleware` world・typed `decision` の写像）を body から切り離して **gated な別マイルストーン**にする。go/no-go 信号は **WASIp3 GA / WIT 型安定（破壊的 RC snapshot treadmill の解消）＋ `component-model-async` 成熟**（「#1554 close 待ち」ではない——#12714 は spec-compliant close 済みで、収斂自体は snapshot 同期で今でも可だが experimental な treadmill を負う）。保守路線として **Preview 2（`wasi:http`@0.2.0、安定）への収斂**もフォールバック候補。
- **body は自前契約で先行する。** **v1（`list<u8>`）は WASI/async 不要の素の Component Model 型なので、既存の `wasm32-unknown-unknown` ターゲットのままでよい**（実装で確認——既存フィルタは引き続き unknown-unknown でビルドし、`http-response.body` も元々 `list<u8>`）。`wasm32-wasip2` ＋ wit-bindgen async への移行（[[000010]] 予告）は **`stream<u8>` true-streaming 増分**まで不要。
- **body 型は `body` resource**（trailers 込み）。`wasi:http` の incoming/outgoing-body の**形だけを import せず模倣**し、将来の収斂時に置換しやすくする。**ただし spike の発見により v1 は resource の中身を buffered `list<u8>` にする**——wit-bindgen 0.58 の `stream<u8>` export は低レベル（`RawStreamReader<&'static StreamVtable<u8>>`、`StreamVtable` は private module）で書き味が悪く、かつ buffer-then-decide なら契約面に streaming は不要だから。`list<u8>` は `component-model-bytes` / `Bytes` でゼロコピー・wit-bindgen で trivial。**`stream<u8>` への差し替えは true-streaming（stream-and-signal）増分で**行う（resource 形を保つので contract は揺れない）。
- **header hook と body hook を別 export に分ける**（[[000005]] 機構2 を**型レベル分離**で実装）。`on-request` は常に。body hook（`on-request-body` / `on-response-body`、request/response 対称）は body-transform フィルタだけが export する。**host は body hook を export しないフィルタには body を渡さず、body stream をゼロコピー bypass する**——disposition が契約の export 形から構造的に決まる。
- **buffer-then-decide を採る。** `on-request-body(body: body) -> body-decision { continue(body) | short-circuit(http-response) }`。フィルタは入力 body を読み（[[000006]] の memory 上限内で必要なら buffer）、変換後 body か short-circuit を返す。**host は `continue` を受けてから初めて upstream へ転送する**ので、**mid-stream short-circuit は常に upstream 接続より前に起き、クリーン**（upstream reset 不要・[[000023]] の「streamed body は再生不可ゆえ bodyless 限定 retry」の緊張を踏まない）。
- **host-API は不変に保つ。** body は resource / stream の関数引数で流し、**新規 import capability を足さない**。deny-by-default の表面（`host-log` / `host-clock` / `host-kv` / `host-counter` / `host-ratelimit`）はそのまま（[[000006]]、[[000020]]「型を借りても capability は最小」）。
- **最初の一歩は spike——実施済み（2026-06-27）。** `wasm32-wasip2` 導入 OK。wit-bindgen 0.58 が自前 WIT の `stream<u8>`（`func` / `async func`）を wasip2 向けに**コンパイル可能**と確認した（`wasi:http` 型を import しないので snapshot-treadmill を踏まない＝本 ADR の核心を実証）。**発見**: stream を返す export の生成 trait は低レベル `RawStreamReader<&'static StreamVtable<u8>>`（`StreamVtable` は wit-bindgen の private module）で書き味が悪い → **v1 を buffered `list<u8>` に切替**（上記）。残作業（有効 component の完成＋host ロード/駆動）は list<u8> 版で進め、`stream<u8>` 増分の前に backpressure を別途検証する。

検討した代替案と却下理由:
- **今すぐ収斂する**（`wasi:http` 0.3 型 ＋ `middleware` world を body と同時採用）— serve は 2 週間前に着地したばかり、guest は Tier-3/nightly、#1554 が OPEN。early-adopter リスクが高く、[[000020]] が「収斂可否を Stage 2 ゲートで最終確認」と置いた趣旨に照らし時期尚早。型再利用の狙い自体は維持し、gate 充足時に再評価する。
- **素の `stream<u8>` を body 型にする** — 最小だが trailers を表せず、EOF / lifecycle の扱いが曖昧で、将来の `wasi:http` body（resource 形）への収斂で二度手間。resource を採る。
- **stream-and-signal の真 streaming**（フィルタが出力を逐次 emit、host が live で upstream へ流す）— ゼロ追加レイテンシだが、mid-stream short-circuit で送信済み partial を抱えた upstream を RST する必要があり、不安定な backpressure（#5028）と相まって v1 には重い。真 streaming は後続増分へ。
- **disposition を manifest 宣言にする** — host が宣言を読んで bypass を決める案。だが**契約の export 形**から bypass が導ける方が、型と実装の乖離が無く堅い。型レベル分離を採る。
- **body 向け capability を新設する**（bounded-buffer / chunk allocator 等）— deny-by-default の表面を広げる。body は能力でなくデータフローなので、引数で流し capability は据え置く。

本決定は [[000003]]（async-first / `stream<u8>`）の方向を body へ具体化し、[[000005]] の Fork 6（header-only / body-transform 分離）の機構2 を実装方針まで確定する。[[000020]] の収斂タイミングを**後ろ倒し**（再検討条件 (b) 発動）、[[000021]] の Stage 2 を **body 同梱**で具体化する。データプレーン no-panic / fail-closed テネットは body 経路でも保つ（buffer 上限超過・stream エラーは fail-closed 写像）。

## Consequences

### Pros
- body フィルタ（body 変換・body 走査 WAF・transcoding）を、`wasi:http` 0.3 の guest DX 成熟（Tier-2 / #1554）を待たずに**今すぐ**出せる。gateway 最大の機能欠落（[[000005]] の body プレーン）が埋まる。
- 自前 `body` resource ＋ `wasm32-wasip2` は #1554 を構造的に踏まず、安定 Tier-2 ツールチェーンで動く。runtime 側の修正（stream copy #13418）は wasmtime 46 で享受できる。
- header / body を別 export に分けたので、body 非接触フィルタの**ゼロコピー bypass が契約から構造的に保証**される（host 側の特別扱いでなく型で決まる）。
- buffer-then-decide により mid-stream short-circuit が upstream 接続前で常にクリーン——upstream reset / 再生不可 body の難所を v1 で回避。
- host-API 表面が不変なので、[[000006]] の deny-by-default 監査面が body 追加で広がらない。
- `body` resource を `wasi:http` の形に寄せたので、将来の収斂で置換コストが小さい。

### Cons / Tradeoffs
- **buffer-then-decide は stream-through を諦める**: body-transform フィルタは body を保持してから upstream へ転送するので、大 body でレイテンシ・メモリ（[[000006]] 上限で bound）を負う。真 streaming は後続増分。
- 別 export ＋ `body-decision` variant ＋ resource で**契約面が太る**（v0.1 の header-only より複雑）。conformance スイートを body 用に増やす必要がある。
- `wasm32-wasip2` ＋ wit-bindgen async は **wit-bindgen の API がまだ churn**（リリースごとに破壊的変更）、`component-model-async` が「very incomplete」。toolchain バージョンを pin し、feature flag 裏で進める。
- response 側 short-circuit の意味（応答 body 途中での停止）は request 側ほど自明でなく、spike で詰める未決事項。
- **再検討条件**:
  - **`wasi:http` 収斂**: **WASIp3 GA / WIT 型安定（snapshot treadmill 解消）＋ `component-model-async` 成熟**を go/no-go 信号に再評価（型再利用・`middleware` world・typed `decision` の写像をここで入れる）。収斂自体は snapshot 同期で今でも到達可だが experimental treadmill を負うので待つ。保守路線として Preview 2（`wasi:http`@0.2.0）収斂も候補。
  - **真 streaming（stream-and-signal）**: `component-model-async` の backpressure / cancellation が枯れたら、buffer-then-decide に streaming オプションを足す。
  - **production 本固定**: spike が buffer-then-decide の書き味・#1554 回避・backpressure を反証可能に確認してから契約を本固定する（それまで body 契約は spike 検証中の暫定）。
