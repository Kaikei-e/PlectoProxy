# ingress path 正規化・host-state quota・inbound リソース上限で untrusted マルチテナント＋L7 脅威モデルを fail-closed に塞ぐ

## Status
accepted

## Date
2026-06-27

## Affected Components
- fast path（`plecto-server`）— ingress path 正規化、接続・body buffer 上限、inbound timeout、hop-by-hop の追補
- routing（`plecto-control` `route.rs`）— `normalize_path` 新設・`normalize_host` の trailing-dot 正規化
- extension plane（`plecto-host`）— host-state quota、untrusted init deadline の分離、host-log の cap／サニタイズ
- control plane（`plecto-control`）— manifest 値検証、OCI blob 読みの境界化、TLS cert/key 一致検証
- supply chain（CI / `deny.toml`）— action の SHA 固定、`sources` gate

## Context

[[000006]] が deny-by-default capability・epoch 計量・OCI 署名・pooling ゼロ化で**サンドボックスと provenance** を固めた。しかしその4本柱は (a) **L7 データパス**（リクエストのルーティング解釈）、(b) **host が保持する state の resource bound**、(c) **inbound のリソース上限** を範囲外にしていた。[[000013]] で fast path が実体化し、`SECURITY.md` が脅威モデルを「untrusted・マルチテナントな WASM フィルタ＋L7 データパス」と明示したことで、この3面が load-bearing になった。運用は **untrusted な第三者フィルタをマルチテナントで実運用する**前提（運用者確認）で、かつ **per-route の filter 束縛を認証/認可の隔離境界として使う**（運用者確認）。

2026-06 のセキュリティ監査（OWASP Top 10:2025 / ASVS 5.0＋Plecto 二軸：WASM sandbox / L7 proxy）が、確定事実として3つの gap を特定した:

1. **path confusion（CWE-436 / CWE-22）**: ルート選択は生 path の prefix match で、`.`/`..`/percent-encode を解決しない。`/public`（filter なし）と `/public/admin`（auth filter）がある時、`/public/%2e%2e/admin` は短い `/public` を選び、upstream が `/admin` に正規化し戻すため auth filter を迂回できる。front-end と origin の正規化差を突く既知クラス（PortSwigger "Gotta cache 'em all"）。per-route filter が認証境界である以上、ルート選択そのものがアクセス制御。
2. **host-state の無制限消費（CWE-770）**: KV/counter/ratelimit に quota がなく、`StoreLimits`（guest 線形メモリ 64MiB）は host 側 store を縛らない。untrusted filter が distinct key で `set`/`increment`/`try_acquire` を回すと host RSS / redb ファイルが無制限に肥大する。さらに untrusted は per-request fresh 生成で `init` を毎リクエスト 5s budget（per-request 100ms の 50 倍）で再実行し、`init` 内 busy-loop が worker を ~5s/req 占有できる。
3. **inbound 上限の欠如（CWE-400）**: 受理接続ごとに上限なく task を spawn し、body buffering（16MiB/req）に global 予算も時間境界もない。一次情報の裏取りで、hyper http1 は `header_read_timeout` 既定 30s を持つが **Timer 未設定では無効**、また接続 cap は存在しないことを確認した。

本決定は設計 tenets の **fail-closed**・**データプレーンで panic 禁止**・**deny-by-default** と Fork 7（security）を、[[000006]] が触れなかった L7 データパスと host-held state に具体化する。[[000006]] を supersede せず**補完**する。

## Decision

### 1. ingress で path を正規化し、ルート選択・チェーン・転送を同一正規化 path で行う

fast path の単一チョークポイント（`proxy_core`、`to_http_request` 直後）で `plecto_control::normalize_path` を適用する。`.`/`..` をレキシカルに解決し、root を脱出する `..` は **400 で拒否**、percent-encoded separator/dot（`%2e`/`%2f`/`%5c`）・制御バイト・backslash を含む path も **fail-closed で拒否**する。query（`?` 以降）は不変。**送出 path = 正規化 path** なので、origin が再正規化しても差異が生じない。これで per-route filter が信頼できる認証境界になる。

検討した代替案と却下理由:
- **decode-then-normalize（encoded を解いて正規化し転送）**: origin が `%2f` を path separator として扱うかは origin 依存で、proxy が独自に解いて送ると別経路で再び食い違う。decode 後の安全な再 encode も要り複雑。曖昧入力を**拒否**する方が fail-closed かつ単純で、Apache `AllowEncodedSlashes Off`／多くの WAF の既定挙動と一致する。却下。
- **正規化を origin に委ね proxy は素通し**: 本監査が突いた脆弱性そのもの。却下。
- **`path_under_prefix` 側でのみ `..` を見る**: 転送 path が非正規のままだと proxy↔origin の差が残る。正規化は「ルート選択にも転送にも効く単一地点」でなければ意味がない。却下。

### 2. host-held state に per-filter＋host-wide の quota を課し、untrusted の init を per-request 級に bound する

`KvQuota`（per-namespace と host-wide の entries／bytes を会計）を **capability 境界（`HostState`）** に置き、`set`/`increment`/`try_acquire` の成長を挿入前に検査して over-quota を fail-closed で drop する。KV の値・キーには単発上限も設ける。`init` deadline は **untrusted=250ms / trusted=5s** に分離する（trusted は init-once で再利用するため generous を維持、untrusted は毎リクエスト走るため per-request 級に締める）。

検討した代替案と却下理由:
- **quota を `KvBackend` 実装に持たせる**: backend が namespace 境界（`{filter_id}\u{1f}…`）を解釈する必要が生じ、[[000011]] の「backend は opaque bytes として扱う」seam を壊す。能力境界で課す方が「ホストが貸した能力の上限」という deny-by-default の自然な延長になる。却下。
- **untrusted init を per-request deadline で一括計量**: 重い init を持つ正当な untrusted filter を毎回 100ms で trap してしまう。init と per-request を分ける Tenet（[[000011]]）に沿って別 budget（250ms）にした。
- **eviction（TTL/LRU）を同時導入**: untrusted state のライフサイクル方針は [[000011]] の領分で、まず無制限増加を fail-closed で塞ぐことを優先。eviction は再検討条件に残す。

### 3. inbound のリソース上限を入れる

accept **前**に取得する host-wide の接続 `Semaphore`（TCP＋QUIC 共通。飽和時はバックログから引かずバックプレッシャ）、body buffering の同時数 `Semaphore`、http1 の `timer`（`TokioTimer`）＋ `header_read_timeout` ＋ `max_headers`、body-read の deadline（超過は 408）を追加する。hyper の既定は Timer 必須で無効なので明示的に張る。quinn の `max_idle_timeout`/stream cap は健全な既定（30s / 100）を維持。

検討した代替案と却下理由:
- **hyper/quinn の既定に委ねる**: header timeout は Timer 未設定で無効、接続 cap は存在せず、body buffering に時間境界がない（一次情報で確認）。inbound DoS が現行スコープの gap（運用者確認）である以上、既定委任は却下。

### 補完的に確定した hardening（同じ脅威モデルの一部）

- `normalize_host` が absolute-form の trailing-dot を剥がし wildcard route への取りこぼしを防ぐ。
- OCI blob 読みを symlink 拒否＋descriptor size 境界＋hex 検証で bound する（traversal は oci-spec の `Digest` 検証で既に不成立だが、path 部品の検査を局所化して暗黙依存を断つ）。
- TLS は `CertifiedKey::keys_match()` で cert/key 不一致を build 時に fail-closed（rustls は `new()` で検証しない）。`Unknown`（provider が公開鍵を出せない）は不一致でないので許容する。
- `FilterEntry::validate` が manifest の不正な metering/ratelimit 値を build で拒否する。
- `HOP_BY_HOP` に `proxy-authorization`/`proxy-authenticate` を加え、proxy-scoped credential の漏出を防ぐ。
- 供給チェーン: GitHub Actions を full commit SHA に固定、cargo-deny に `sources` gate を追加（OWASP A03）。

## Consequences

### Pros
- **per-route filter が認証境界として信頼できる**: path confusion を構造的に封じ、ルート選択＝アクセス制御が proxy↔origin で一致する。
- **untrusted マルチテナントで noisy-neighbor を抑止**: 1 テナントの filter が host メモリ/redb/CPU を枯渇させられない。
- **internet 直曝に対して bounded**: 接続・body・header すべてに上限と時間境界があり、OOM/slowloris が成立しない。
- 既存の強み（SSRF 不成立・smuggling 双方向 strip・fail-closed チェーン・cosign 二重署名）に、L7 データパスと host-state の安全を積み増した。
- 供給チェーンの可変タグ／未知 source を CI で hard-fail に。

### Cons / Tradeoffs
- **encoded separator/dot を含む正当な path も 400 になる**（稀だが要文書化）。**再検討条件**: `%2f`/`%2e` を path 値として正当に使うユースケースが現れたら、per-route の "decode-then-normalize を許す" opt-in を設計する。
- **quota は eviction を持たない**: cap 到達後、その filter の write は拒否され続ける（fail-closed write-deny）。**再検討条件**: untrusted state に TTL/LRU eviction が要るなら [[000011]] を更新する。
- **quota 会計は Host 再構築でリセットされる**: redb の既存データは再カウントされないため、実効上限は「既存 + cap」になる。起動時の redb 再走査は高コストなので許容するが、運用監視の対象。
- **ratelimit/counter/set の hot path に novelty 判定の pre-read が 1 回増える**。会計の正確性を優先した。**再検討条件**: 並行サービングで律速になれば、会計を backend の write txn に畳み込む。
- **上限値は単一定数で manifest 可変ではない**（接続 1 万・buffer 同時 64・header 15s・body 30s・KV value 256KiB・per-filter 64MiB 等）。**再検討条件**: per-route/per-filter override が要れば manifest registry（[[000007]]）で配分する。
- **CI の SHA 固定で `dtolnay/rust-toolchain` は `toolchain` input の再指定が要る**（ref 名からの自動選択が効かなくなるため）。
- epoch は host 呼び出し内でブロックした guest を中断できない（[[000006]] 自認）点は不変。現状 host 呼び出し（KV/redb）は非ブロッキングなので顕在化しない。
