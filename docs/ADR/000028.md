# upstream ごとの circuit breaker を concurrency cap として導入し、health とは別関心事に分離する

## Status
accepted

## Date
2026-06-30

## Affected Components
- server runtime（`plecto-server`）— `proxy_core` が forward に入る前（`pick` の直前）に `UpstreamGroup::try_acquire` で in-flight slot を取る。取得できなければ upstream に流さず **503**（`x-plecto-fault: circuit-open`、body `upstream overloaded`）。permit は 1 リクエスト 1 枚で retry ループ全体を跨いで保持し、応答ヘッダ到達（または失敗）で解放
- control plane（`plecto-control`）— `UpstreamGroup` に `max_requests: usize` ＋ `in_flight: AtomicUsize` を追加し、`try_acquire(self: &Arc<Self>) -> Option<RequestPermit>`（optimistic increment、cap 超過は back-out）と Drop で減算する `RequestPermit` を実装。`reconcile` が manifest の値から毎回構築する（health policy の一部ではないので breaker 変更は instance health を保つ）
- manifest — `[[upstream]]` に任意セクション `[upstream.circuit_breaker]`（`max_requests` のみ、`0` = unlimited が既定）
- observability — `plecto_circuit_open_total` counter を fast-path metrics に追加（[[000009]]）

## Context

M2 の upstream 層（[[000017]]）は active/passive health で**落ちた**インスタンスを rotation から外すが、**生きているが飽和した** upstream を保護する機構が無い。遅い/過負荷の backend に対してプロキシは無制限に並行転送を積み上げ、in-flight が膨れて自身の resource（コネクション・メモリ・スレッド）を食い潰し、連鎖障害（cascading failure）の起点になる。これは成熟した L7 proxy が Tier-0 で備える circuit breaking の不在であり、先行事例（Envoy の cluster `circuit_breakers.thresholds` = max_connections/pending/requests/retries による hard cap → fast 503）を調査して裏取りした（[公式ドキュメント参照](https://www.envoyproxy.io/docs/envoy/latest/intro/arch_overview/upstream/circuit_breaking)）。

本決定は maturity roadmap の Stage B（resilience 硬化）の一部であり、**役割駆動**の native 採用基準に従う: circuit breaking は per-request の filter decision では表現できない**横断的データプレーン関心事**（upstream という共有資源の保護）なので native に置く（Fork 6）。`crates/server/CONTEXT.md` は従来「circuit breaking は無く、no-healthy fail-closed とは別物」と明示していた——本 ADR がその欠落を埋める。

設計の核心の論点は「circuit breaker を health/outlier detection（[[000017]] が単一状態機に**意図的に畳み込んだ** passive ejection）と統合するか、別関心事として分離するか」だった。

## Decision

- **circuit breaker を per-upstream の concurrency cap として導入する。** `[upstream.circuit_breaker] max_requests = N` が、その upstream へ同時に転送中（in-flight）のリクエスト数の上限。`0` = unlimited（既定・opt-in）。`request_timeout`/`max_retries`（[[000019]]/[[000023]]）と同じく `UpstreamGroup` のフィールドで health policy の外に置き、breaker のみの変更は reconcile で group を作り直しても instance health を保つ。
- **cap 到達は fail-closed 503。** `try_acquire` が `None` を返したら pick せず `x-plecto-fault: circuit-open`（body `upstream overloaded`）で即返す。404 no-route / 503 no-healthy-upstream / 504 upstream-timeout とは別の fault 値で、運用が区別できる。
- **permit は RAII。** `try_acquire` は optimistic に `in_flight` を increment し、prior 値が cap 以上なら back-out して `None`。成功時は `RequestPermit`（`Arc<UpstreamGroup>` 保持）を返し、Drop で減算する。`proxy_core` は 1 リクエスト 1 枚を `pick` 直前に取り、retry ループ全体を跨いで保持、応答ヘッダ到達/失敗で関数を抜けるときに解放——`?` や early return を含む**全ての return path で必ず解放**される（data-plane no-panic / leak-free テネット）。unlimited（`max_requests == 0`）は `group: None` の no-op permit でゼロコスト。
- **health とは別関心事として分離する（統合しない）。** health は「この instance は up か?」を、breaker は「この upstream は飽和しているか?」を答える別の問い。**breaker が shed したリクエストは instance を demote しない**（過負荷は不健全ではない——飽和した backend を unhealthy 扱いして rotation から外すと、残りに負荷が集中して連鎖障害を悪化させる）。[[000017]] が passive ejection を health に畳み込んだのは「connect 失敗 = 落ち」だからで、本 ADR はその境界を尊重し別軸を足す。
- **観測可能にする。** shed のたび `plecto_circuit_open_total` を計上し admin `/metrics`（[[000009]]）に出す。Stage A の観測性が Stage B の効果計測を支える。
- **状態は単一ノードローカル。** `in_flight` はこのノードの当該 group のカウンタ。ノード横断の global concurrency 制御は分散レイヤ（foca/openraft）の opt-in 後続（Fork 5/10）。

検討した代替案と却下理由:
- **error-rate trip breaker（open/half-open/closed、連続エラー率で trip し cooldown 後 half-open で試行）** — Envoy の outlier detection や古典的 breaker の形。誤検知耐性は高いが、timing 状態機（trip 時刻・cooldown・half-open 探索）と閾値群を持ち込み運用が複雑化する。v1 は concurrency cap を採る——状態を持たず（瞬時のカウンタのみ）、resource 使用量を**直接**bound し、運用が単純（operational simplicity テネット）。error-rate breaker は再検討条件に置く。
- **breaker を health 状態機に統合する** — 一見コード共有できるが、shed = demote になり「飽和 backend を unhealthy にして残りへ負荷集中」という連鎖障害増幅を生む。過負荷と不健全は別軸なので**分離**した。
- **per-route cap** — route 単位の方が細かいが、飽和する共有資源は upstream（backend）であって route ではない。Envoy の cluster-level に倣い **per-upstream** にした。複数 route が同一 upstream を指すなら合算で cap がかかるのが正しい。
- **max_connections / max_pending_requests も同時に入れる**（Envoy フル）— hyper-util の pooling client では「pending」「connection」の境界が曖昧で、最も意味が明確な **max_requests（同時 active）** を v1 の単一ノブにした。残りは再検討条件。
- **応答 body 完了まで permit を保持する** — より厳密だが、body は応答ヘッダ後に stream され（プロキシ側の上限管理外）、ヘッダ到達で解放する方が「backend が処理中の窓」を素直に表す。streaming 応答を不当に長く 1 slot 占有させない。

本決定は Fork 6（横断データプレーン関心事は native）を具体化し、fail-closed・no-panic・single-node-first・operational simplicity の各テネットに整合する。[[000017]]（health）/[[000019]]（timeout）/[[000023]]（retry）を**補完**し、いずれも supersede しない。

## Consequences

### Pros
- 過負荷/低速 upstream に対しプロキシ自身の resource を bound でき、連鎖障害の起点を断つ（Tier-0 resilience の充足）。
- health と独立なので、飽和した healthy backend を誤って rotation から外さない（負荷集中の悪化を避ける）。
- hot path のコストは `max_requests == 0` で完全ゼロ（既定）、有効時も atomic 1 回 ＋ RAII の減算のみ。lock-free。
- `circuit-open` という独立 fault 値 ＋ `plecto_circuit_open_total` で、503 の原因（no-route/no-healthy/overload）を運用が切り分けられる。
- RAII permit により retry ループ・early return・`?` を跨いで slot leak が起きない。

### Cons / Tradeoffs
- v1 は **concurrency cap のみ**。error-rate に基づく trip（短時間の高エラー率で先回りして遮断）は未対応——遅延ではなくエラーで死ぬ backend は、in-flight が低いままなら cap に当たらない。
- cap 直後の reload は新 group の `in_flight` を 0 から始める（旧 group の in-flight は旧カウンタを permit 経由で減算）。reload 跨ぎで瞬間的に実効上限を僅かに超え得るが、旧リクエストの drain で自己補正する。
- per-upstream 単位なので、同一 upstream を共有する route 間で「うるさい隣人」を route 単位に隔離できない（per-route cap は未導入）。
- ノードローカル。水平展開時のノード横断 global concurrency は保証しない。
- **再検討条件**：(a) error-rate / open-half-open-closed breaker（[[000017]] の outlier detection 独立化と統合検討）；(b) `max_pending_requests` / `max_connections` / `max_retries` の追加閾値；(c) per-route override；(d) ノード横断の global concurrency（分散レイヤ opt-in）；(e) backend からの load 報告（ORCA 風）に基づく適応制御。
