# L7/API-gateway としての成熟を「役割駆動」で進め、native/WASM の配置基準を固定する

## Status
accepted

## Date
2026-06-30

## Affected Components
- fast path（`plecto-server` / `plecto-control`）— native が育つ範囲を**横断的データプレーン関心事**（timeout / circuit breaking / load balancing / observability）と**グローバルカウンタ**に限定する判断基準を据える。この基準に当たらない機能は native 化を**明示的に declined** とし、後続スライスが native へ滲み出すのを止める
- extension plane（`plecto-host` / `plecto:filter`）— user-policy 形状の機能（auth / WAF / OPA / 複雑 authz / rewrite ロジック）は WASM 拡張面に留め置く（Fork 6）。拡張モデルは WASM Component Model + WIT で、`wasi:http` への収斂（[[000020]]）を進める。proxy-wasm / http-wasm は**先行事例の capability チェックリストとして調査しただけで、採用しない**
- roadmap / governance — 成熟の駆動軸を Envoy/Kong との feature parity（埋め尽くし）から**役割駆動**へ固定。Stage A–G の見通しを projected として残し、本 ADR が Stage B–G の各 ADR を統べる

## Context

Plecto は M0（`plecto:filter` 契約 + wasmtime ホスト）・M1（filter-runtime 堅牢化 + trusted プール）・M2 slice 1–5（HTTP/1.1·2·3 fast path + routing + rustls TLS 終端 + 複数 upstream の round-robin LB / active+passive health, [[000017]]）を着地させ、直近では Stage B（resilience 硬化）の最初のスライスとして per-upstream circuit breaker（[[000028]]）を入れた。ここで上位の問いが顕在化した——**「Plecto は L7/API gateway としてどこまで、何を駆動軸に成熟するのか」** を明文化しないと、成熟の指針が暗黙に「成熟プロキシ（Envoy/Kong）が持つ機能を埋めていく」という **parity チェックリスト** に流れる。

これは設計の根に反する。プロジェクトのコア原則は **安全 × ポータビリティ × セルフホスト性 × 運用の単純さ ＞ 機能網羅性 × 強い権限 × 分散デフォルト**（`CLAUDE.md`）であり、**機能網羅性を明示的に後位に置いている**。したがって「機能の数」を最大化する parity 駆動は、tenet 順序と正面から衝突する。先行プロキシの feature 一覧は **capability の地図（prior art）** として有用だが、それを**到達目標（target）にしてはならない**——この区別を governance として固定するのが本 ADR の主目的である。

仕様変動領域なので **確定事実 / 現時点の計画（projected）** を分ける。確定事実は (1) 拡張モデルが WASM Component Model + WIT であること（[[000001]]）、(2) header-only と body-transform をフィルタ契約で分離しホット経路は native に落とすこと（[[000005]]）、(3) 単一ノード・ファースト、分散は opt-in（[[000008]]）、(4) `plecto:filter` を `wasi:http` へ収斂させる方針（[[000020]]）、(5) circuit breaker が Stage B の最初のスライスとして着地済み（[[000028]]）。一方、下記の Stage A–G の並びは**前向きの見通し（projected）**であり、各 Stage は着手時に独立した ADR で確定する。

本決定は **Fork 6（native vs WASM 配置）** と **コア原則の順序付け** を具体化するもので、新しい機能の決定ではなく、以後の機能決定が従う**配置の判断基準そのもの**を固定する。

## Decision

- **成熟は「役割駆動（role-driven）」で進める。parity を目標にしない。** Plecto の到達目標は Envoy/Kong の機能網羅ではなく、**自分の役割（安全に・ポータブルに・セルフホストで・単純に L7 を捌く）を深く満たすこと**。先行プロキシの feature 一覧は capability の地図として参照するが、チェックリストを消化することを設計の駆動力にしない。tenet 順序（機能網羅性は後位）に従属する。

- **native fast-path の採用基準を固定する（本 ADR の load-bearing なルール）。** native は次のいずれかに当たるときに**だけ**育てる:
  - (i) **per-request の filter decision では表現できない横断的データプレーン関心事**——timeout / circuit breaking / load balancing / observability のように、個々のリクエストの判断ではなく共有資源・接続・経路の制御に属するもの。
  - (ii) **グローバルカウンタ**——ノード内で集約が必要な計数（例: native rate limiting のトークン/ウィンドウ）。

- **user-policy 形状の機能は WASM 拡張面に留め、native 化を declined とする。** auth / WAF / OPA / 複雑な authz / rewrite ロジックは「このリクエストをどう判断するか」という per-request decision であり、ユーザのポリシーで形が変わる。これらは `plecto:filter` の `decision` variant（`continue` / `modified` / `short-circuit`）で表現し、**native には実装しない**（Fork 6）。判断は型で、ホット経路は軽く保つ（コア原則）。WAF のような重い検査も、native の固定実装ではなく**拡張面に置けるフィルタ**として配置する（配置先は Stage F で詰める）。

- **分散/グローバル機能は opt-in を保つ。** native rate limiting のグローバルカウンタもノード横断の global 制御も、まずは単一ノードローカルで成立させ、ノード横断は分散レイヤ（foca/openraft）の **opt-in 後続** とする（[[000008]], Fork 5/10）。single-node-first をデフォルトに据える。

- **拡張モデルは WASM Component Model + WIT、`wasi:http` 収斂で確定。proxy-wasm は採らない。** Plecto の拡張面は Component Model のフィルタを WIT 契約で差し込む形（[[000001]]）で、`plecto:filter` は `wasi:http` の proxy/middleware 型へ収斂させる（[[000020]]）。**proxy-wasm / http-wasm は先行事例の capability チェックリストとして調査しただけで、拡張 ABI として採用しない**——これらは host-call 列挙の参照であって、Plecto の契約の出発点ではない。

- **Stage 並びを projected として残す（前向きの見通し・拘束はしない）。** 役割駆動の native 成熟が辿る見込みの順序:
  - **Stage A — observability の配線**（着地済み。[[000009]] を実体化）。後段の効果計測の土台。
  - **Stage B — resilience 硬化**（circuit breaking は [[000028]] で着地。残: retry backoff / retry-on-5xx、timeout モデル、outlier detection）。
  - **Stage C — native rate limiting**（グローバルカウンタ＝採用基準 (ii)）。
  - **Stage D — routing/matching の成熟 ＋ LB アルゴリズム**（横断＝採用基準 (i)）。
  - **Stage E — filter の outbound capability**（拡張面に貸す能力の拡張、deny-by-default を保つ）。
  - **Stage F — reference フィルタ / SDK ＋ WAF の配置**（user-policy 形状＝拡張面）。
  - **Stage G — response-body ＋ `wasi:http` 収斂**（[[000005]] / [[000020]] の body 方向）。
  各 Stage の確定は着手時の個別 ADR で行い、本並びは順序保証ではない。

検討した代替案と却下理由:
- **parity 駆動（埋め尽くし）**——成熟プロキシの feature 一覧を到達目標に置き、上から消化する。地図が明確で進捗が測りやすい一方、**機能網羅性を後位に置く tenet 順序と正面衝突**し、運用の単純さ・ポータビリティ・セルフホスト性を機能数のために犠牲にしやすい。capability の参照（地図）としては使うが、**駆動軸にはしない**。
- **tier-by-tier ハイブリッド（Tier-0 は parity を native、Tier-1+ は役割駆動）**——「最低限（Tier-0）の resilience/observability は先行プロキシ並みに native で揃え、その上は役割駆動」という折衷。実務的だが、**採用基準が二段になり不純**——「どこまでが Tier-0 か」を都度線引きする曖昧さが残り、parity の論理が Tier-0 という名で native へ滲む。本 ADR は **単一の採用基準（横断 or グローバルカウンタ）** に一本化し、線引きを feature の格付けではなく**性質（per-request decision か否か）**で行う。
- **拡張 ABI に proxy-wasm を採用**——既存フィルタ資産と知見が再利用できるが、host-call 列挙ベースの古い ABI で、Component Model + WIT の型契約・`wasi:http` 収斂（[[000001]] / [[000020]]）から後退する。**参照に留め、不採用**。

本決定は **Fork 6（横断データプレーン関心事は native、user-policy は WASM）** と **コア原則の順序付け**（機能網羅性は後位）を具体化し、deny-by-default capability / 判断は型で / fail-closed / single-node-first / データプレーン no-panic の各テネットに整合する。Stage B–G の各 ADR は本 ADR の採用基準に従属する。

## Consequences

### Pros
- 成熟の駆動軸が **tenet 順序と一致**する。機能数ではなく役割の深さで進むので、運用の単純さ・ポータビリティ・セルフホスト性が機能追加で侵食されにくい。
- native/WASM の配置が **単一の判断基準（per-request decision か、横断 or グローバルカウンタか）** に固定され、後続スライスごとの「どっちに置く」議論が短くなる（[[000028]] が circuit breaking を native に置いた論法が、以後 reusable な定石になる）。
- user-policy 形状の機能を拡張面に集約することで、ホット経路（native fast path）が軽く保たれ、deny-by-default capability の境界も明快になる。
- Stage 並びを projected として明示するので、ロードマップを**拘束せずに方向だけ共有**でき、各 Stage の確定 ADR が独立して書ける。
- proxy-wasm を参照に留める判断が記録され、拡張 ABI の議論が `wasi:http` 収斂（[[000020]]）に一本化される。

### Cons / Tradeoffs
- 役割駆動は **「いつ十分か」の客観指標が parity より弱い**。feature チェックリストのような明確な完了線が無く、成熟度の対外説明は「何を declined にしたか」を都度言語化する負担を負う。
- 「per-request decision か、横断関心事か」の線引きが**境界事例で割れ得る**（例: 認証は user-policy だが、global token-bucket rate limit はグローバルカウンタで native——両者を跨ぐ機能の置き場は個別 ADR の判断が要る）。
- WAF のような重い検査を拡張面に置く方針は、**native 固定実装より per-request コストが高く**なり得る。性能が問題化したら配置（Stage F）を再検討する。
- Stage 並び（A–G）は projected であり**順序保証ではない**。優先度や前提が変われば組み替わる——ロードマップとして固定視されると誤解を生む。
- **再検討条件**：(a) ある機能で「横断/グローバルカウンタ」とも「per-request decision」とも言い切れない境界事例が頻発し、二分基準が破綻するとき；(b) 拡張面に置いた重いフィルタ（WAF 等）の per-request コストが運用上許容できず、native co-processor 化が必要になるとき；(c) `wasi:http` / Component Model の標準が proxy-wasm 系へ収斂するなど、拡張 ABI 前提が外的に変わるとき；(d) 単一ノードで成立させたグローバルカウンタ（rate limit 等）にノード横断の一貫性要求が出て、分散レイヤ opt-in（[[000008]]）を前倒しするとき。
