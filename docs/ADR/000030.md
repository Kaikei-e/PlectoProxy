# retry を硬化する——jittered exponential backoff と retriable-5xx での別 instance 再試行を加える

## Status
proposed

## Date
2026-06-30

## Affected Components
- server runtime（`plecto-server`）— `proxy_core` の retry ループ（[[000023]]）を二点で硬化する。(1) 試行間に **full-jitter exponential backoff** の sleep を入れる（現在は試行間に遅延ゼロ）。(2) timeout / connect 失敗に加えて **retryable upstream 5xx response**（gateway-error class 502/503/504 ＋ 設定可能な retriable-status set）も retry trigger にする。いずれも**冪等メソッド・bodyless 限定**（[[000023]] の body 制約を維持）。**projected（未実装）**
- control plane / manifest（`plecto-control`）— `[[upstream]]` に per-upstream の `retry_on` 条件集合・`num_retries`・backoff `base`/`cap`、必要なら retry budget（active リクエストに占める retry の割合上限）を追加。`request_timeout`/`max_retries`/`circuit_breaker` と同列で health policy の外に置く。**projected（未実装）**

## Context

[[000023]] で fast path は upstream 転送失敗を別の healthy instance へ bounded retry できるようになった。ただし現状の retry プロファイルは狭く、二つの弱点を残している。

第一に、**試行間に遅延がない**。失敗したら即座に次の instance へ再送する。upstream group 全体が過負荷・低速に傾いた瞬間、全クライアントの retry が同期して殺到し（thundering herd / retry storm）、弱った backend をさらに押し倒す。[[000028]] の circuit breaker が in-flight 上限で resource を bound しても、retry の**到着レート**そのものを均す機構は無い。

第二に、retry trigger が **timeout と connect 失敗の二つだけ**である。upstream が **5xx を返した**ケース——reload 中やヘルスチェック前で `503 Service Unavailable` を返す instance、あるいは前段の gateway が `502/504` を返すケース——は、別の healthy instance なら成功し得るのに retry されず、その 5xx がそのままクライアントへ抜ける。timeout（無応答）だけでなく「明示的な gateway-error 応答」も、安全なケースに限れば別 instance への再試行で救済できる。

これは [[000023]] が明示的に残した再検討条件の延長である（同 ADR Cons (e)「per-request の retry 時間上限（budget）」、および body/trigger 拡張の含み）。本 ADR は [[000023]] を **supersede せず拡張**し、[[000019]]（timeout）・[[000028]]（circuit breaker）を**補完**する。仕様変動領域（backoff のデフォルト値）は **projected（Envoy reference）** として確定事実と区別する。`CLAUDE.md` の resilience テネット（fail-closed・data-plane no-panic・single-node-first・運用の単純さ）に従属する。

## Decision

[[000023]] の retry ループを二軸で硬化する。安全プロファイル（冪等・bodyless・別 healthy instance・health 非連動）は [[000023]] から不変に継承し、その上に backoff と 5xx trigger を足す。

- **試行間に full-jitter exponential backoff を入れる。** 各 retry の前に `sleep(random(0, min(cap, base * 2^attempt)))` だけ待つ。full-jitter（待ち時間を `0..上限` の一様乱数にする）にすることで、複数クライアントの retry が時間軸上にばらけ、同期再送を崩す。デフォルトは **projected（Envoy reference）**: base ≈ 25ms、cap ≈ 250ms（= base の約 10 倍）。**これらは計測値ではなく Envoy のリファレンス既定を投影した出発点**であり、Plecto 自身のベンチで確定するまで暫定とする。backoff は health 信号でも circuit 状態でもない、retry ループ内だけの遅延である。
- **retryable upstream 5xx response も retry trigger にする。** timeout（[[000019]]）/ connect 失敗（[[000017]]）に加えて、upstream が **gateway-error class の 5xx（502 / 503 / 504）** を返した場合、**冪等メソッド限定**で別の healthy instance へ再試行する。さらに `retriable-status-codes` を設定で拡張できる（例: `503`, `429`）。5xx でも **bodyless 限定**は維持する——bodied retry のための bounded buffering は [[000023]] 同様に未設計のまま再検討条件に残す。
- **projected config（per-upstream）。** `retry_on`（`timeout` / `connect-failure` / `retriable-5xx` / 明示的な status set の組）、`num_retries`、backoff `base`/`cap`。retry の**増幅**を抑えるため、active リクエストに占める retry の割合を上限化する **retry budget**（例: active の 20% まで）を入れるかは再検討条件とする。いずれも manifest 上で health policy の外に置き、retry-only の変更が instance health をリセットしないようにする（[[000023]]/[[000028]] と同じ分離）。
- **fail-closed と no-panic を維持する。** 安全に再試行できないもの（非冪等メソッド、body 付き、別 healthy instance が無い、retriable でない 5xx）は素通りさせず、従来どおりその fault（5xx をそのまま、または 504/502）を返す。backoff sleep・5xx 判定・budget 計上は untrusted upstream 応答に対して panic しない。
- **retry は health を resurrect しない。** 再試行は引き続き `pick_excluding` で**別の** healthy instance に着地し、retry 機構そのものは health を変えない——retry で迂回された instance を retry が demote することも、迂回先の instance を promote することもない（[[000017]] の passive ejection 境界＝connect 失敗のみ、を尊重）。5xx を返した instance の demote は別軸（将来の outlier detection）の判断に委ね、本 ADR では行わない。

検討した代替案と却下理由:
- **即時 retry のまま（[[000023]] 現状、backoff なし）** — 実装は最小だが、過負荷 upstream に対し同期 retry storm を生み thundering herd で backend を押し倒す。backoff（特に jitter）はこの増幅を均すための業界標準であり、ここで導入する。
- **固定（非 jitter）backoff** — 遅延は入るが、全クライアントが**同じ**時間だけ待って一斉に再送するため山が時間方向にずれるだけで同期は崩れない。full-jitter で待ち時間を乱数化し、retry を時間軸に分散させる。
- **全 5xx を retry する** — 不採用。retry は **gateway-error class（502/503/504）＋ 冪等メソッド**に限る。`500 Internal Server Error` のような application バグ由来の決定的 5xx は、別 instance でも同じ入力なら同じく失敗する公算が高く、retry は capacity を浪費して 500-bug を叩き続けるだけになる。また非冪等メソッドを 5xx で再送すると、upstream が副作用を適用済みのケースで二重実行になる（timeout と同じ冪等性論点）。retriable な gateway-error class と冪等性で trigger を絞る。

本決定は [[000023]] の安全プロファイルを保ったまま retry を硬化し、`CLAUDE.md` の resilience テネット（fail-closed・no-panic）と「安全 × 運用の単純さ」に整合する。`plecto:filter` の WIT 契約は不変（host-native のみ）。

## Consequences

### Pros
- full-jitter backoff により、過負荷 upstream への同期 retry storm を時間軸に分散し、thundering herd を抑える（[[000028]] の concurrency cap が *量* を、backoff が *到着レート* を bound する補完関係）。
- retryable 5xx（502/503/504）が別の healthy instance で救済され、reload 中や一過性の gateway-error をクライアントへ漏らしにくくなる——timeout/connect だけでなく「明示的なエラー応答」も可用性の対象に入る。
- gateway-error class ＋ 冪等 ＋ bodyless に絞ることで、application 500-bug の叩き続けと非冪等な副作用の二重実行を回避する。
- backoff/`retry_on`/`num_retries`/budget は per-upstream で backend ごとに調整でき、health policy の外なので retry-only の reload が instance health を保つ。

### Cons / Tradeoffs
- backoff sleep が**最悪レイテンシを増やす**。worst ≈ (num_retries + 1) × `request_timeout` ＋ Σ backoff sleeps。これは [[000019]] の end-to-end timeout と**相互作用**するため、backoff の cap と num_retries は timeout budget の内側に収まるよう設計する必要がある（backoff budget の整合）。
- backoff のデフォルト（base ≈ 25ms / cap ≈ 250ms）は **projected（Envoy reference）であり Plecto の計測値ではない**。実負荷でのベンチ確定までは暫定で、過小だと storm 抑制が弱く過大だとレイテンシ税になる。
- 5xx を返した instance は retry 機構では demote されない（health 非連動を維持）。決定的に 5xx を吐き続ける instance は、num_retries を消費して最終的に 5xx を返すまで rotation に残り得る——outlier detection は別軸の未採用機能。
- body 付きリクエストは 5xx でも依然 retry されない（[[000023]] の bodyless 制約を継承）。
- **再検討条件**：(a) **retry budget**（active に占める retry 割合の上限）を入れて増幅を hard-cap するか；(b) bodied retry のための **bounded buffering**（[[000013]] の opaque body 原則との折衝）；(c) **per-route override** の retry/backoff；(d) backoff デフォルトの **計測に基づく再較正**（projected → measured）；(e) 5xx を返す instance の **outlier detection**（[[000017]] の health 軸への統合可否）；(f) `429` 等で upstream が返す `Retry-After` を backoff に反映するか。
