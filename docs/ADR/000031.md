# upstream timeout を per-try と overall(per-route) の二段に分離し、overall 超過を fail-closed 504 にする

## Status
proposed

## Date
2026-06-30

## Affected Components
- server runtime（`plecto-server`）— `proxy_core` の転送を二段の deadline で包む。**per-try** が単一試行（attempt）の応答ヘッダ到達を、**overall** が最初の試行＋全 retry＋backoff 待ちを含む transaction 全体を区切る。overall 超過は fail-closed **504**（`x-plecto-fault: request-timeout`、[[000019]] の per-try `upstream-timeout` とは別 fault）。**projected**（未実装）
- manifest — [[000019]] の per-upstream `request_timeout_ms` が **per-try** の意味へ降格し、別途 **overall（per-route）** のノブを足す。既定値・キー名・per-route override は **projected**（再検討対象）

## Context

[[000019]] が入れた upstream timeout は **per-upstream の単一値**で、応答**ヘッダ**到達までを区切る（body はヘッダ後に無制限ストリーム）。これは「ひとつの転送 `client.request(...)` がヘッダを返すまで」を bound するものであり、本質的に **1 回の試行（attempt）の bound** である。

その後 [[000023]] が「timeout / connect 失敗時に別 healthy instance へ bounded retry する」を入れた。retry が入ると、[[000019]] の timeout は **試行ごと**に掛かる——つまり実態として **per-try** に振る舞う。問題は、**retry 列全体を区切る上位の deadline が存在しない**ことである。`max_retries` 回ぶん per-try timeout を積み上げ得るうえ、[[000030]] の backoff を入れると試行の合間に待ち時間が更に積まれる。結果として、retry 込みの**実際の総所要時間が operator の意図した総予算（client SLA / 上流 deadline）を超え得る**。fail-closed テネットを「時間の上限」へ具体化した [[000019]] の意図が、retry の下では崩れている。

確定事実: 今日の `request_timeout_ms` は per-attempt の bound であり、retry 列全体を縛る overall deadline は無い。仕様変動領域（projected）: 新しい overall ノブの既定値・キー名、および後述の追加ノブ（connect / idle / per-route override）。

関係する Tenet: **fail-closed**（超過は素通りさせず 504）、**データプレーンで panic 禁止**、**運用の単純さ**、**single-node first**（本 ADR の deadline はノードローカルで完結し分散合意を要さない）。

## Decision

timeout を**二つの別概念**に分離する。

- **per-try timeout（単一試行の bound）。** [[000019]] の per-upstream `request_timeout_ms` の意味をここへ移す。挙動は [[000019]] のまま——`tokio::time::timeout` で 1 回の `client.request(...)` を包み、応答**ヘッダ**到達で解決し、body は無制限にストリーム。`0 = 無効`（long-poll / streaming upstream のオプトアウト）も継承。per-upstream 既定（30000ms）はそのまま per-try の既定として残る。
- **overall（per-route）request timeout（transaction 全体の bound）。** route 単位の上位 deadline。最初の試行・[[000023]] の retry・[[000030]] の backoff 待ちを**すべて含む** transaction 全体を区切る。超過したら、進行中の試行を打ち切り **fail-closed 504**（`x-plecto-fault: request-timeout`、per-try の `upstream-timeout` と区別して運用が切り分けられる別 fault 値）を返す。streaming のため `0 = 無効` のオプトアウトを overall にも残す（[[000019]] と同じ operator 責任）。
- **不変条件: `overall ≥ per-try`。** overall が per-try より短いと、retry の余地が無いどころか単一試行すら最後まで走れず無意味になる。この invariant を manifest 検証で強制し、違反は reload を拒否する。これは先行事例（Gateway API の `timeouts.backendRequest ≤ timeouts.request`）と同型の関係であり、本 ADR が**独自に持ち込む発明ではない**——一般的な慣行として projected に位置づけて引用する（具体実装・既定値は本リポジトリ独自）。
- **fail-closed・no-panic を維持。** overall 超過時も素通り（fail-open）させず 504。打ち切りは `tokio::time::timeout` の cancellation で表現し、保持中の circuit permit（[[000028]]）や M1 プール枠は RAII で確実に解放する（untrusted upstream の挙動で worker を巻き込まない）。

追加ノブ（いずれも **projected**・再検討条件）:
- **connect timeout** — TCP / TLS ハンドシェイク確立までの別 bound。今は per-try のヘッダ到達に畳み込まれている。
- **idle / stream timeout** — ヘッダ到達**後**の body ストリーム停滞を区切る（[[000019]] が「mid-response hang は捕まらない」として残した穴に対応）。
- **per-route override of the upstream default** — per-try（既定は per-upstream）を route 単位で上書き可能にする。overall は最初から per-route なので対象外。

検討した代替案と却下理由:
- **単一 timeout を維持する（現状／[[000019]] のまま）** — 単純だが、retry（[[000023]]）の下では総所要時間が **`(max_retries+1) × per-try ＋ backoff 総和`** まで膨らみ得て、transaction 全体が**無制限**に近づく。operator の総予算を表現する手段が無く、不十分（insufficient）。却下。
- **overall timeout のみにする（per-try を捨てる）** — 一見シンプルだが、**per-attempt の bound を失う**。1 回の遅い試行が overall 予算を丸ごと食い潰し、retry に回る前に予算が枯渇する——「早い試行を素早く諦めて別 instance へ振る」という retry の前提（[[000023]]）が成立しなくなる。per-try の早期カットこそが overall 予算内で複数試行を可能にするので、両方必要。却下。
- **overall を per-upstream に置く** — per-try と同じ粒度で揃うが、operator が縛りたいのは「この route のリクエストの総時間」（client SLA 由来）であって upstream resource ではない。overall は per-route が自然（per-upstream resource 保護は [[000028]] の breaker が担う）。per-route を採用。

本決定は [[000019]] を **supersede しない**——その per-upstream timeout は per-try として存続し、上に overall を足す**拡張**である。fail-closed テネットを「retry を含む transaction の総時間」へ拡張し、設計原則（安全 × 運用の単純さ）と矛盾しない。

## Consequences

### Pros
- retry（[[000023]]）と backoff（[[000030]]）を入れても、transaction の総時間が overall で**確実に bound** され、operator の総予算（client SLA）を表現できる。
- per-try を残すので「遅い単一試行を素早く諦めて別 healthy instance へ振る」retry の効きが保たれる——overall だけにすると失われる per-attempt の機敏さを維持。
- `request-timeout`（overall）と `upstream-timeout`（per-try、[[000019]]）の別 fault 値で、504 の原因を運用が切り分けられる。
- overall も `0 = 無効`で streaming / long-poll route をオプトアウトでき、[[000019]] の運用モデルと一貫。
- `overall ≥ per-try` の invariant を reload 時に強制することで、無意味な設定（overall < per-try）を早期に弾ける。

### Cons / Tradeoffs
- ノブが 1 個から 2 個（将来さらに connect / idle）へ増え、operator が理解すべき timeout 軸が増える（運用の単純さへの軽い負債——invariant とドキュメントで緩和）。
- per-try で**ヘッダ到達後**に body が hang するケースは、overall が無効（`0`）または十分長い streaming route では依然捕まらない（idle timeout を入れるまで [[000019]] と同じ穴が残る）。
- overall は本ノードの当該 transaction にローカル。ノード横断の global deadline は対象外（single-node first；分散は Fork 5/10 の opt-in 後続）。
- 既定値の選定が未確定（projected）。overall を per-try より十分大きく取らないと retry の余地が消える一方、大きすぎると総予算の意味が薄れる——既定の妥当性は実装後に再評価する。
- **再検討条件**：(a) **connect timeout** の独立化；(b) ヘッダ後の **idle / stream timeout**（[[000019]] が残した body hang の穴を回収）；(c) per-try の **per-route override**；(d) overall 既定値の妥当性（retry 既定回数・backoff 既定との整合）；(e) deadline 残量を retry 判断（[[000023]]）へ渡し、残予算が per-try を下回るなら最後の試行を打ち切る連携。
