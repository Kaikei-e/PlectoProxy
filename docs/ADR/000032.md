# outlier detection を health 状態機から独立した第三の resilience 軸として昇格する

## Status
accepted — 連続 gateway-5xx による時間ベース ejection（exponential backoff + `max_ejection_percent` cap、health とは独立軸）を `UpstreamGroup::record_outcome` / `pick` に実装。統計的検知（success-rate / failure-percentage）と ejection 後の eject_count 時間減衰は再検討条件として deferred。

## Date
2026-06-30

## Affected Components
- control plane（`plecto-control`）— `UpstreamGroup`（registry）に、health の `AtomicBool` とは**独立**した per-instance の outlier 状態（連続 5xx カウンタ・連続 gateway 失敗カウンタ・ejection deadline・連続 ejection 回数による backoff 倍率）と ejection 制御（base ejection time × 指数 backoff、`max_ejection_percent` cap）を持たせる。`reconcile` が manifest から構築し、health policy の**外**（circuit breaker と同列）に置くので outlier 設定の変更は instance health を保つ（projected）
- server runtime（`plecto-server`）— `proxy_core` が転送完了時に応答 outcome（status class = application 5xx / gateway 失敗 = connect・timeout・transport fault）を outlier detector に報告する。`pick` の eligible 集合を「health-healthy **かつ** outlier-非 eject」に絞る（projected）
- manifest — `[[upstream]]` に任意セクション `[upstream.outlier_detection]`（`consecutive_5xx` / `consecutive_gateway_failure` / `failure_percentage`（統計閾値）/ `max_ejection_percent` / `base_ejection_time`、未指定 = disabled で opt-in）（projected）
- observability — outlier ejection 系メトリクス（ejection 累計・現在 eject 中の instance 数・eject 比率）を admin `/metrics`（[[000009]]）に出す（projected）

## Context

M2 の upstream 層は今、resilience を**二軸**で持つ。(1) active health（[[000017]]）＝「この instance は reachable か?」を background prober が `health.path` への合成 probe で判定し、(3) circuit breaker（[[000028]]）＝「この upstream は飽和しているか?」を in-flight concurrency cap で守る。だが**二軸の間に穴**がある。

[[000017]] は passive ejection を**単一の per-instance health 状態機に意図的に畳み込んだ**。畳み込みの境界は「connect 失敗 = 落ち」という等式で、実リクエストの転送が**接続失敗**したときだけ instance を demote する。その結果として、**接続は張れるが live traffic 上で 5xx を吐き続ける / 遅く失敗する** upstream は eject されない——active prober の合成 probe が（たまたま）2xx を返す限り、その instance は healthy のまま rotation に残り、実トラフィックを吸い続ける。[[000017]] 自身がこれを Cons（「passive は**接続失敗のみ**を見る。mid-response の hang は active probe 任せ」）と再検討条件に挙げ、[[000028]] も「[[000017]] が passive ejection を health に畳み込んだのは connect 失敗 = 落ちだから」とその境界を明示して尊重していた。本 ADR はその**畳み込みの再検討条件を回収**し、欠けた第二軸を埋める。

これは成熟した L7 proxy が備える **outlier detection**（live request outcome から個々の host の不調を見抜き、一時的に rotation から外す）の不在である。先行事例として Envoy の outlier detection（`consecutive_5xx` / `consecutive_gateway_failure` / `success_rate`・`failure_percentage` 統計、base ejection time × 連続 ejection 回数の指数 backoff、`max_ejection_percent` による pool 保護）を参照する（[公式ドキュメント参照](https://www.envoyproxy.io/docs/envoy/latest/intro/arch_overview/upstream/outlier)）。本 ADR は `proposed` であり、ここに挙げる Envoy 由来の既定値は**参照値（projected）であって計測値ではない**。本決定は maturity roadmap の resilience 硬化に属し、upstream という共有資源に対する**横断的データプレーン関心事**なので native に置く（Fork 6）。状態は単一ノードローカルに保つ（Fork 5/10）。

## Decision

- **outlier detection を独立した第三の resilience 軸として昇格する。** それを駆動するのは connect 失敗だけでなく**観測された request outcome** である。判定信号は二系統:
  - **連続カウンタ型** — `consecutive_5xx`（application が返す連続 5xx）と `consecutive_gateway_failure`（connect 失敗・upstream timeout・transport fault・502/503/504 等の transport 級失敗の連続）。前者は「backend が動いているが壊れた応答を返す」、後者は「経路が壊れている」を捉える別信号として分ける。
  - **統計型** — 一定窓・最低リクエスト数の上で success-rate / `failure_percentage` が閾値を割った host を outlier とする。瞬間的なスパイクではなく**継続的な不調**を捉える。
- **ejection 制御を持つ。** outlier と判定した instance を rotation から**一時 eject** し、`base_ejection_time` 経過後に再投入する。同じ instance が再び outlier になるたび ejection 時間を**連続 ejection 回数で指数 backoff** する（flapping する bad host が短周期で復帰と再 eject を繰り返すのを抑える）。eject は**時間ベースで自己回復**し、永久 eject はしない。
- **`max_ejection_percent` で pool を守る。** outlier detection が eject できるのは pool の最大 N%（projected 既定 ≈ 10%）まで。**全 instance が同時に 5xx を吐く**ような事象は単一 host の不調ではなく**系全体の問題**（downstream 依存断・スキーマ不整合など）の徴候であり、そこで pool を空にすると残りに負荷が集中して自分で連鎖障害を起こす。cap は「一匹の outlier は外すが、皆が不調なら誰も外さない」を保証する——**fail-closed であって self-DoS にはしない**（[[000017]] が panic threshold を却下した精神と整合し、active health の「全 unhealthy なら 503」とは別の保護点）。
- **eligible 集合 = healthy かつ非 eject。** `pick` は「active/passive health で healthy」**かつ**「outlier で eject されていない」instance のみを round-robin 対象にする。outlier 状態は health の `AtomicBool` とは**別フィールド**（カウンタ＋deadline＋backoff 倍率）で、`reconcile` は health policy の外で再構築するので outlier 設定変更が instance health を消さない（[[000028]] が breaker を health の外に置いたのと同じ分離）。
- **三軸を明確に分離し、再畳み込みしない。** 各軸は別の問いに答える:
  1. **active health（[[000017]]）** — 「instance は reachable か?」/ 合成 probe / connect 失敗の passive 補完。
  2. **outlier detection（本 ADR）** — 「instance は live traffic 上で misbehave しているか?」/ 実 request outcome。
  3. **circuit breaker（[[000028]]）** — 「upstream は飽和しているか?」/ concurrency cap。
  軸が**別の問い**である以上、コード共有の誘惑で一つの状態機に戻さない。
- **projected な manifest 形。** `[upstream.outlier_detection]` を opt-in（未指定 = disabled、breaker と同じ既定オフ）で足す。参照既定値（**projected, Envoy 参照, 未計測**）: `consecutive_5xx ≈ 5` / `consecutive_gateway_failure ≈ 3` / `max_ejection_percent ≈ 10%` / `base_ejection_time ≈ 30s`。これらは accepted 化の際に計測で裏取りして確定する。
- **untrusted upstream 応答で panic しない。** outcome 分類（status class / gateway 失敗の判定）は data plane が untrusted な upstream バイト列を見て行う**total な分類**であり、未知 status や欠落でも panic させない（data-plane no-panic テネット）。
- **状態は単一ノードローカル。** 各ノードは自分が観測した outcome で自分の rotation を eject する。ノード横断の outlier 共有は分散レイヤ（foca/openraft）の opt-in 後続（Fork 5/10）。

検討した代替案と却下理由:
- **ejection を health に畳み込んだまま残す（現状 = [[000017]]）** — 単一状態機でコードは最小だが、**connect 失敗しか demote しない**。5xx storm を吐く host・live traffic 上で slow-error する host を取りこぼし、合成 probe が緑な限り bad host が rotation に残る。実 outcome を見る独立軸が必要なので**却下**。
- **outlier detection を circuit breaker（[[000028]]）に畳み込む** — 一見どちらも「悪い upstream を避ける」だが、breaker が答えるのは**飽和（overload）**で、outlier が答えるのは**不調（misbehaviour）**。**overload ≠ misbehaviour**: 飽和は健全な backend でも起きる正常な背圧で shed して守るべきもの、不調は個々の host を rotation から外すべきもの。畳み込むと「飽和 = eject」になり、[[000028]] が退けた「飽和 backend を unhealthy 扱いして残りに負荷集中」を再現する。別軸として**分離**する。
- **永久 eject（時間ベース再投入なし）** — bad host を確実に締め出せるが、一過性の不調から自己回復できず、運用が手で戻す必要がある。base ejection time ＋指数 backoff の**時間ベース自己回復**を採る。
- **per-route の outlier detection** — route 単位は細かいが、不調なのは共有資源である upstream（backend host）であって route ではない。[[000028]] の per-upstream 判断に倣い **per-upstream/instance** にする。

本決定は Fork 6（横断データプレーン関心事は native、user-policy は WASM）を具体化し、fail-closed（ただし `max_ejection_percent` で self-DoS を防ぐ）・no-panic（untrusted upstream 応答）・single-node-first・operational simplicity の各テネットに整合する。[[000017]]（health）/[[000028]]（breaker）/[[000030]]（retry）を**補完**し、いずれも supersede しない——[[000017]] の「passive ejection を health に畳み込む」決定はそのまま残し、本 ADR が**その外に**第二軸を足す形を取る。

## Consequences

### Pros
- live traffic 上で 5xx storm / slow-error する host を、合成 probe が緑でも rotation から外せる——[[000017]] が connect 失敗しか見なかった死角を埋める。
- 三軸（reachable / misbehaving / saturated）が**別フィールド・別問い**として分離され、後から読む人が「なぜこの 503 か」「なぜこの host が外れたか」を軸ごとに切り分けられる。
- `max_ejection_percent` cap で、系全体の不調時に pool を空にして self-DoS する事故を防ぐ（fail-closed と self-DoS 回避の両立）。
- ejection が時間ベース＋指数 backoff で自己回復し、flapping host の復帰・再 eject 振動を抑える。手動介入が要らない。
- outlier 状態が health の外にあるので、outlier 設定の reload が instance health をリセットしない（[[000028]] と同じ reconcile 分離）。
- ejection メトリクス（[[000009]]）で eject 件数・現在 eject 中・eject 比率が可視化され、`max_ejection_percent` に張り付く＝系全体の不調、という運用シグナルが得られる。

### Cons / Tradeoffs
- 第三の状態を data plane の hot path（転送完了時の outcome 報告）に足す。`pick` の eligibility 評価に outlier ビットの読みが 1 つ増える。
- 連続カウンタ＋統計窓＋ejection timer＋backoff 倍率という新たな調整パラメータ群を運用に持ち込む。誤設定で過敏に eject すると可用性を逆に削り得る（`max_ejection_percent` が backstop だが閾値調整は要る）。
- application 5xx は「backend の正当な業務エラー（例: 503 maintenance）」と「不調」の区別が本質的に曖昧で、`consecutive_5xx` は false-positive を生み得る。`consecutive_gateway_failure` と統計型で緩和するが完全ではない。
- **active health（合成 probe）と outlier（実 outcome）が同じ instance について食い違い得る**（probe 緑だが実トラフィックで eject、など）。これは設計上の意図（別軸）だが、運用者には二系統の判断を理解する負荷がかかる。
- [[000030]]（retry-on-5xx）との相互作用に注意が要る: ある instance の 5xx を別 instance への retry で**client から隠した**場合でも、その 5xx は**元の instance の outlier 信号として計上し続ける**（retry が outlier の目を塞いではならない）。retry が成功体験を増幅して bad host を温存しないよう、計上は retry の前段（per-attempt outcome）で行う。
- 状態はノードローカル。水平展開時、各ノードが独立に eject 判断するので pool 全体の eject 比率はノード横断では揃わない。
- **再検討条件**：(a) `proposed` の参照既定値（`consecutive_5xx`/`consecutive_gateway_failure`/`max_ejection_percent`/`base_ejection_time`）を計測で裏取りして accepted 化する；(b) 統計型（success-rate / `failure_percentage`）を連続カウンタ型より先に出すか後に出すかの段階分け；(c) outlier eject と active health unhealthy を `pick` でどう合成集計するか（二重カウントの扱い）；(d) [[000028]] の error-rate trip breaker 構想との境界（open/half-open/closed の timing 状態機を outlier 側に寄せるか breaker 側に置くか）；(e) ノード横断の outlier 共有（分散レイヤ opt-in）。
