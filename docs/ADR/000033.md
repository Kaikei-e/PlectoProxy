# native L7 rate limiting を fast path に導入し、filter に貸す host-ratelimit とは別の粗粒度ベースラインとする

## Status
proposed

## Date
2026-06-30

## Affected Components
- server runtime（`plecto-server`）— projected: `proxy_core` が forward に入る前に native の per-route token bucket を consult。bucket が枯渇していれば upstream を pick せず **429**（`x-plecto-fault: rate-limited`、body `rate limit exceeded`）で即返す。token-bucket の計算は host-native で完結し、WASM 境界を跨がない
- control plane / manifest（`plecto-control`）— projected: `[[route]]` に任意セクション `[route.rate_limit]`（`rate` / `burst`、`key` = `route` または `client-ip`、既定 = 無制限）。`reconcile` が manifest から limiter を構築（`request_timeout` / circuit breaker と同列、health の外）
- observability — projected: `plecto_rate_limited_total` counter を fast-path metrics に追加（[[000009]]）

## Context

現在 Plecto に存在する rate limit 機構は **filter に貸す `host-ratelimit`（[[000026]]）の 1 つだけ**である。これは host 側の token bucket を WASM フィルタが `try-acquire(key, cost)` で consult する **filter capability** であり、spec（capacity / refill）は per-filter で manifest に置かれ、フィルタが「consult するか・どのキーで・cost いくつで」を判断する。つまり rate limit は**フィルタが効いているルートにしか掛からない**。

ここに穴がある——**フィルタを差し込んでいない素のルートには、リクエストレートに対する built-in な DoS 保護が何も無い**。filterless route は upstream へ無制限にリクエストを通す。すべてのルートに最低限のレート上限を掛けたければ、現状では**そのためだけに WASM フィルタを 1 枚書いて全ルートに配線する**必要があり、native proxy が Tier-0 で備えるべき粗粒度のレート防御が欠けている。

役割駆動の native 採用基準（Fork 6）に照らすと、**リクエストレートの上限は per-request の filter decision では表現できない横断的データプレーン関心事**である。レートは「このルート / この client-IP に向かう全リクエストにまたがる共有カウンタ」であり、global counter は native に置く（[[000029]] の placement rule「global counters are native」、roadmap 参照）。これは [[000028]]（circuit breaker = upstream の in-flight cap）が「飽和の保護」を native に置いたのと同じ系譜の、**inbound 方向のレート保護**である。

本件は DoS 対策ファミリの一部でもある。[[000027]] が host-state quota と inbound リソース上限（接続数・ヘッダサイズ・body 上限など）で**量**を bound したのに対し、本 ADR は**レート（時間あたりのリクエスト数）**を bound する——両者は別軸で相補する。

> 注: 本 ADR は proposed であり未実装。設定キー名・manifest 形状・counter 名・status code marker は現時点の計画（projected）であり、実装 ADR で確定する。性能上の主張（hot path コスト等）は実測前のため数値は載せない。

## Decision

- **native の per-route（and/or per-IP）token-bucket rate limiter を fast path に導入する。** `[route.rate_limit] rate = N, burst = B` が、その route（または client-IP）へ向かうリクエストの token bucket を定義する。`proxy_core` は forward 前にこの bucket を consult し、token が無ければ upstream を pick せず即 fast-fail する。token-bucket の math は **host-native で完結**し、WASM 境界を一切跨がない（filter chain とは独立した素のデータプレーン経路）。
- **超過は 429 Too Many Requests で fast-fail。** 上限到達時は `x-plecto-fault: rate-limited`（body `rate limit exceeded`）を付けて即返す。**status は 429 を採用**し、[[000028]] の circuit breaker が返す **503**（`circuit-open` = upstream 飽和）とは**別の status / 別の fault 値**にする——「クライアントがレート超過（429）」と「upstream が過負荷（503）」を運用が切り分けられる。
- **filter に貸す host-ratelimit（[[000026]]）は残す。** native limiter は**粗粒度の native ベースライン**（route / client-IP 単位の素朴な上限）であり、`host-ratelimit` は**フィルタが決める policy 形状のレート制限**（per-API-key・per-tenant・cost-weighted など、ビジネスロジックを伴う細粒度）である。両者は**相補関係**であって置換ではない: native が「全ルートに掛かる床」を、filter が「policy に応じた細かい天井」を担う。
- **single-node-first を守る。** native limiter は**ノードローカル**な token bucket（このノードを通るリクエストのみカウント）。ノード横断の global / distributed rate limiting（外部 RLS = rate limit service）は、**保留中の foca/openraft 分散レイヤ（[[000008]]）の opt-in 後続**に置き、本 ADR では**引き込まない**。この境界を明示的に記録する。
- **設定は projected。** per-route limit（`rate` + `burst`、key = route または client-IP）を manifest の `[[route]]` 配下に宣言的に置く。spec は operator が所有し、フィルタは触れない（[[000026]] が host-ratelimit で確立した「spec は host/operator 側」の原則を native でも踏襲）。

検討した代替案と却下理由:
- **filter-only のまま（現状維持）** — 本 ADR が塞ぐ穴そのもの。filterless route に native ベースラインが無く、最低限のレート上限ですら**すべて WASM フィルタを要求**する。native proxy としての Tier-0 防御の欠落を放置することになる。却下。
- **distributed RLS を今入れる** — ノード横断の global レート制御は魅力的だが、外部 RLS / コンセンサスの依存を持ち込み **single-node-first テネット（[[000008]]）に反する**。分散は opt-in の後続レイヤに置くのが Plecto の方針。却下（再検討条件に置く）。
- **fixed-window counter** — 実装は単純だが、window 境界で burst が 2 倍通る（境界跨ぎ問題）。**token bucket は burst を `burst` パラメータで明示的に吸収**でき、平滑な refill でレートを素直に表現できる。先行事例（Envoy の local rate limit は token bucket を採用）にも倣い token bucket を採る。fixed-window は却下。

本決定は Fork 6（横断データプレーン関心事は native、user-policy は WASM）を inbound レート方向に具体化し、**fail-closed**（bucket 枯渇は素通しせず 429 で fail-closed に止める）・**data-plane no-panic**（untrusted な client 入力で worker を巻き込まない）・**single-node-first**・**operational simplicity** の各テネットに整合する。[[000027]] の inbound リソース上限を**レート軸で補完**し、[[000026]] / [[000028]] のいずれも supersede しない。

## Consequences

### Pros
- **filterless route が native の DoS ベースラインを得る**——WASM フィルタを書かずとも、route / client-IP 単位の素朴なレート上限を manifest だけで掛けられる。native proxy の Tier-0 レート防御の充足。
- **native（粗粒度）と filter（細粒度 policy）が相補**し、役割が明確に分かれる（床 vs 政策的天井）。どちらか一方に機能を寄せる無理が無い。
- token-bucket の計算が **host-native で完結**し、native limiter の判定が **WASM 境界を跨がない**（filter chain を起動しない素の経路）。hot path を軽く保つ意図（コストは実測前のため数値は projected）。
- **429（rate-limited）が 503（circuit-open）と別 status / 別 fault 値**なので、「クライアント側のレート超過」と「upstream 側の過負荷」を運用が `x-plecto-fault` と metrics で切り分けられる。
- fail-closed・no-panic を維持。inbound レートを bound して DoS 耐性を上げ、[[000027]] の量的上限を時間軸で補完する。

### Cons / Tradeoffs
- **ノードローカル**——水平展開時のノード横断 global レートは保証しない。N ノードに分散すると実効レートは概ね N 倍になり得る（distributed RLS は opt-in の後続、[[000008]]）。
- **rate limit 機構が 2 つ**になる（native の per-route / per-IP と、filter の host-ratelimit）。operator は「粗粒度の床は native、policy 的な細粒度は filter」という使い分けを理解する必要がある（ドキュメント負債）。
- 設定面（manifest）の表面積が増える（`[route.rate_limit]`）。projected な形状であり実装 ADR で確定するまで変動し得る。
- v1 の key は route / client-IP のみを想定。より多次元のキー（header 値・path prefix・組み合わせ）は policy 寄りで、filter（host-ratelimit）側か後続の native 拡張に委ねる。
- **再検討条件**:
  - (a) ノード横断の **global / distributed rate limiting** が要件化したら、外部 RLS もしくは分散レイヤ（foca/openraft, [[000008]]）の opt-in として設計する。
  - (b) route / client-IP 以外の**キー次元**（per-header・per-path-prefix 等）が native ベースラインに必要になったら拡張を検討（policy 寄りなら filter 側に寄せる）。
  - (c) 429 応答に `Retry-After` を載せるか、bucket の残量を可視化するか（運用フィードバック）。
