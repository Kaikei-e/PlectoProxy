# host + path-prefix routing に header/method/query match と weighted traffic split を足してルーティングを成熟させる

## Status
accepted

## Date
2026-07-01

## Affected Components

> 設計は grill（`/grill-with-docs`）で、match semantics / precedence / weighted backendRefs は **Gateway-API v1.5.0 仕様**に、weighted-pick アルゴリズムは **配分法（apportionment / error-diffusion）の学術技法**（特定プロキシ実装ではなく）に照らして確定した。projected だった precedence・config 形・weighted-pick アルゴリズム・unhealthy 時の扱いはすべて確定値に解決した（後述）。

- control/route（`plecto-control`）— route 選択の compiled 形 `CompiledRoute` に、host＋path-prefix に加えて header(exact)・HTTP method・query-parameter の match dimension を持たせ、転送先を単一 upstream から **weighted backends（複数 upstream group ＋ 整数 weight）** へ拡張する。`select` は最長 prefix 単一規則から **specificity 順の多軸照合**へ拡張する
- server runtime（`plecto-server`）— `proxy_core` が method/headers/query を `find_route` に渡し、route 一致後に backend を weight で選び、選んだ group 内は既存の healthy-set LB（[[000024]]）で instance を選ぶ。match 評価は今日と同じく async スレッド上の純粋な config 参照に留める
- manifest — `[[route]]` の match を `[route.match]`（`host` / `path_prefix` / `method` / `headers` / `query`）にネストし、`upstream`（単数 shorthand）と weighted `[[route.backends]]` を排他に持てるよう拡張する

## Context

[[000013]] が fast path に host-native routing を入れた時点の match 軸は **host ＋ path-prefix の二つだけ**（host は port 無視・大文字小文字無視、path は `/` 境界の最長 prefix 一致、不一致は 404）。これは「実プロキシになる」最初の一歩としては十分だったが、成熟した L7 gateway が当然備える表現力には届かない。具体的に今日表現できないもの:

- **method による分岐** — 同一 path で `GET` と `POST` を別の backend / 別チェーンへ。
- **header による分岐** — API バージョンヘッダ、tenant ヘッダ、`Accept` などで宛先を変える。
- **query による分岐** — feature flag やバージョン指定の query で分岐。
- **canary / weighted traffic split** — 一つの宛先に対して新旧バージョンを **重み付きで同時に流す**（例: v2 を 5%、v1 を 95% から徐々にシフト）。今は別 route に割るしかなく、同一 match 条件のトラフィックを比率で割れない。

これらは gateway の table-stakes であり、本 ADR は maturity roadmap（[[000029]] が定める role-driven な native 採用基準）に沿って **routing を native 軸で成熟させる**位置づけにある。

成熟度の物差しとして Kubernetes **Gateway-API の conformance level（Core / Extended）** に投影しておく（**projected**: これは実際の conformance 認証ではなく、機能選定の位置づけのための対応づけ）:

- path-prefix / header(exact) match … **Core** 相当
- method / query match … **Extended** 相当
- weighted `backendRefs`（per-route の重み付き分配）… **Core** 相当

つまり今日の Plecto は Core の一部（path）に留まり、canary という Core 機能すら欠く。本 ADR は Core を満たし Extended の有用な一部（method/query）を取り込む線を引く。

設計原則上の制約（[[000013]] / [[000027]] から継承）:

- **routing は native**（Fork 6、`CLAUDE.md`）。match は per-request の filter decision ではなく横断的な fast-path 関心事なので Rust 側に置く。
- match 評価は [[000013]] と同じく **allocation-free / 純粋な config 参照**で、blocking チェーン経路には載せず async スレッドに残す。
- **untrusted な header/method/query 値で panic しない**（data-plane no-panic テネット）。header value は非 UTF-8 を含み得るのでバイト単位の exact 比較、malformed query は error ではなく no-match として扱う。
- **path 正規化は fail-closed のまま不変**（[[000027]]）。match は既に正規化済みの path に対して行い、query は正規化が touch しない（[[000027]] は query 不変）。新しい match 軸は新たな正規化面を増やさない（exact byte / token 比較のみ）。

## Decision

**(1) match dimension を増やす（exact 起点）。** route の match を `[route.match]` テーブルに集約し、host＋path-prefix から次へ拡張する（Gateway-API v1.5.0 の HTTPRoute match semantics に倣う）:

- **header match（exact）** — header 名は HTTP 規則どおり case-insensitive、値は exact（**バイト一致**。非 UTF-8 ヘッダ値でも no-panic）。複数指定は AND。
- **method match** — request line の method token を exact 一致（既に hyper がパース済み・upper-case token）。
- **query match** — 指定 query parameter の exact 一致。**名は case-sensitive**（header と非対称、Gateway-API に一致）、malformed query は no-match、重複キーは**最初の値**で照合。

指定された軸はすべて満たす必要があり（AND）、指定のない軸は wildcard。regex match は **意図的に繰延**する（後述の代替案参照）。まず exact ＋ Extended 集合（method/query）から始める。

**(1b) precedence を specificity 順に厳密化する（最大の open question を解決）。** 軸が増えると「どの route が勝つか」の決定的順序が要る。Gateway-API v1.5.0 が規範化する precedence（より specific な match が勝つ・同点は continue）に倣い、当たった route 群を次のキーの辞書式で比較し最上位を選ぶ:

1. **host 指定 > wildcard**（Gateway-API は hostname を rule precedence より先に解決するので host を最上位 specificity に置く）
2. **最長 path prefix**（文字数）
3. **method 指定あり**
4. **header 一致数が多い**
5. **query 一致数が多い**
6. 同点は **manifest 出現順**（最古 index）で安定 tie-break（Gateway-API の creationTimestamp / `{namespace}/{name}` に相当する決定的順序を Plecto は manifest 順で与える）

一致なしは 404（[[000013]] のまま）。これにより、より specific な route が必ず勝つ曖昧さ無しの順序が確定する。

**(2) weighted traffic split / canary を入れる。** `[[route.backends]]` を `{ upstream, weight }` のリストにし、route 一致後に **weight で backend group を選ぶ**。これは canary の正準プリミティブ（一つの match 条件のトラフィックを比率で分配）であり、別 route に割る方式では表現できない。weight は `u32`・既定 `1`・上限 `1_000_000`（Σ overflow 回避、Gateway-API の Maximum に一致）、`0` は drain（その backend へ流さない）。**単一 `upstream`（shorthand）は 1 要素の weighted backends（weight 1）に正規化**し、runtime は単一経路を持つ（Envoy の `cluster` vs `weighted_clusters` と同じ shorthand）。`upstream` と `backends` は排他。

**(2b) weighted-pick は apportionment（最大剰余 / error-diffusion）を precompute table で実装する（決定的・lock-free）。** 選択アルゴリズムは **Webster / Sainte-Laguë 配分法（maximum-deficit 規則）**を採る: 各ステップで全 backend にその weight を「deficit（公平分配からの不足）」として積み、deficit 最大の backend を選んで total を引く。これは整数 **Bresenham error diffusion**（1965）の N 要素一般化であり、配分法の中で唯一 *unbiased* な divisor 法（Balinski & Young, *Fair Representation*, 1982）で、理想比率からの最大偏差を `< 1` に抑える（Tijdeman, chairman-assignment problem, 1980）。決定的・均等インタリーブ・バースト無しで、weighted-random（非決定・テストしづらい・`rand` 依存増）より優れる。この規則はプロキシ界隈で "smooth weighted round-robin" とも呼ばれるが、**本 ADR は特定プロキシのコードではなく上記の汎用配分技法を実装する**。

素朴な実装は per-pick の deficit RMW で lock が要り hot-path lock-free テネット（[[000024]]）と緊張するが、この規則は **周期 `L = Σ(gcd 縮約 weight)` で周期的**（L pick 後に各 backend がちょうど縮約 weight 回選ばれ deficit が零状態に戻る）なので、**1 周期がそのまま無限スケジュール**になる。よって compile 時に 1 周期分を backend-index の table に展開し、runtime は `cursor.fetch_add(1) % L` で引く（[[000024]] の純カウンタ RR と同型の lock-free pick）。Gateway-API は分配アルゴリズムを規定せず統計比率（epsilon 許容）のみ要求するので、決定的 apportionment は準拠する（周期境界では epsilon 零）。縮約後 table 長は **65536 を上限**にし、超える病的な coprime weight は build で fail-closed 拒否する。

**(3) 既存の upstream group ＋ LB の上に重ねる（二層）＋ renormalize over healthy。** weighted pick は「どの upstream group へ送るか」だけを決め、選んだ group 内の instance 選択は既存の **healthy-set round-robin（[[000024]]）** がそのまま担う。retry（[[000023]]）・circuit breaker（[[000028]]）・health（[[000017]]）も選ばれた group の既存挙動を踏襲する。**weighted-chosen backend が unhealthy（その group に eligible instance ゼロ）のときは、その backend を分配集合から外し残る healthy backend で再正規化する（renormalize over healthy）。** これは [[000024]] の pick_inner が unhealthy instance を eligible 集合から外すのと同型で、table を前方 skip して次の eligible backend を返す。全 backend が ineligible（または全 weight 0）なら eligible 集合が空 → 503（fail-closed、[[000017]]/[[000024]] の no-healthy と同じ倒し方）。

**(4) native・allocation-free・fail-closed を維持する。** match 評価は async スレッド上の純粋な config 参照（[[000013]] の並行性モデルを保つ）。`CompiledRoute` は manifest reconcile 時に一度コンパイルし、per-request では割り当てを増やさない。untrusted 値は exact byte/token 比較で no-panic。path 正規化（[[000027]]）は不変で、match は正規化後 path に対して行う。

**(4b) chain / strip_prefix / rate limit は per-route 粒度に保つ。** weighted backends を持つ route でも filter chain・strip_prefix・rate limit は route 単位で全 backend に共通に掛かる。canary は「どの upstream か」だけを変え policy（認証・rate limit・path 書換）は canary 版間で同一にする。backend ごとに異なる policy が要るなら別 route に割る。これで backend は純粋な `{upstream, weight}` に保てる。

**(5) confirmed config（確定形）。** per-route の形は次:

```toml
[[route]]
  # 転送先: 単一 shorthand（[[route.backends]] と排他）…
  upstream = "svc"
  # …または weighted backends（canary）:
  # [[route.backends]]
  #   upstream = "svc-v2"
  #   weight   = 5
  # [[route.backends]]
  #   upstream = "svc-v1"
  #   weight   = 95

  filters      = ["auth"]        # route 単位の chain（全 backend 共通）
  strip_prefix = "/v2"           # route 単位

  [route.match]                  # match 軸はここにネスト
  host        = "api.example.com"   # 任意
  path_prefix = "/v2"               # 必須
  method      = "POST"              # 任意（Extended）
  headers     = { "x-api-version" = "2" }   # 任意・exact・AND（Core）
  query       = { "beta" = "on" }   # 任意・exact（Extended）

  # [route.rate_limit] は従来どおり route 直下（ADR 000033）
```

match 軸を `[route.match]` にネストするのは「リバプロ／API Gateway としてあるべき宣言形」を優先した判断で、host/path-prefix を route 直下に持っていた [[000013]] のフラット形からの**移行を伴う**（既存 manifest / テストは書き換える。additive ではない）。

**(5b) 不正な config は build で fail-closed 拒否する（[[000027]]/[[000028]] の build-time validation に倣う）。** `upstream` と `backends` の両指定・どちらも無し、空 `backends`、全 weight 0、`weight` の上限超過、縮約後 apportionment table 長の上限（65536）超過、未知 upstream 参照を build 時に拒否する。reload は all-or-nothing（[[000013]]）のまま。

**検討した代替案と却下理由:**

- **host＋path-prefix のまま据え置く** — 最も単純だが、method 分岐も canary も表現できない。両者は gateway の table-stakes であり、「成熟した L7 gateway」を名乗る上で欠かせない。却下。
- **最初から regex を全面採用する（path/header を regex match）** — 表現力は最大だが、(a) 各実装で意味論が割れる implementation-specific な機能で portability を損ない、(b) untrusted 入力に対する **ReDoS** 攻撃面を data-plane に持ち込む（no-panic / fail-closed テネットと緊張）。よって regex は繰延し、**exact ＋ Extended 集合（method/query）から始める**。regex は再検討条件（ReDoS-bounded エンジン前提）に置く。
- **traffic split を「別 route を並べる」だけで表現する** — 既存機能の範囲で済むが、同一 match 条件のトラフィックを **比率で分配**できない（route は match 条件で選ぶもので、一致した route の中で確率分配はできない）。weighted split は一つの route 内で完結する canary の正準プリミティブなので、route 内の `backends` weight として入れる。却下。
- **match 軸を all-or-nothing で Gateway-API フル準拠まで一気に入れる** — Extended のうち header-regex・header-presence・`not` match・gRPC method 等まで一括導入する案。スコープが過大で、ReDoS 面や precedence の複雑性を一度に抱える。Core ＋ method/query という有用な最小集合に絞り、残りは再検討条件へ。

本決定は Fork 6（横断的データプレーン関心事は native、user-policy は WASM）を具体化し、native routing を Core 充足まで成熟させる。fail-closed・no-panic・operational-simplicity・single-node-first の各テネットと整合する（match は正規化後 path に対する no-panic な exact 比較、weighted split は単一ノードローカル）。[[000013]]（routing 基盤）を **extend** し、supersede しない。

## Consequences

### Pros

- method / header / query による分岐と canary（weighted split）を表現でき、routing が Gateway-API の Core を満たし Extended の有用部分（method/query）を取り込む。「成熟した gateway」の table-stakes が揃う。
- weighted split が **既存の upstream group ＋ LB の上にきれいに重なる**（weight が group を選び、group 内は [[000024]] の healthy-set RR）。retry / circuit breaker / health の既存挙動を再利用し、新しい失敗モードを増やさない。
- match は今日と同じく **allocation-free / async スレッド上の純粋 config 参照**で、blocking チェーン経路に載らない（[[000013]] の並行性モデルを保つ）。per-request の追加割り当てなし。
- match は **正規化後 path** に対して行い、新軸は exact byte/token 比較のみなので、[[000027]] の fail-closed 正規化に新たな攻撃面を足さない。
- manifest は additive。既存の host＋path route ＋単一 `upstream` はそのまま動く。

### Cons / Tradeoffs

- **match precedence / specificity が非自明になる。** 軸が増えると「どの route が勝つか」の決定的な順序が要る（[[000013]] の最長 path-prefix 単純規則では足りない）。本 ADR (1b) で **Gateway-API v1.5.0 の precedence に倣い host > path-prefix長 > method有 > header一致数 > query一致数 > manifest順 の辞書式に確定**した。ambiguous routing は無いが、host を最上位に置く（path より優先）のは Plecto 固有の選択で、Gateway-API の hostname 先行解決に対応づく。
- **regex を繰延した分、表現力に穴。** 一部の実 header/path match は exact では書けない。ReDoS を避けるための意図的な妥協で、再検討条件に置く。
- **match 軸（特に header/method/query）は untrusted。** クライアントが自由に付けられる属性なので、これらを**権限・レート区分を分岐させる唯一の根拠にしない**。authz と隔離は [[000027]] が定めた「route に束ねた filter」が担い、match はあくまで宛先選択である（攻撃者が `x-tier: premium` を名乗るだけで privileged backend に到達する、といった設計を避ける）。重複ヘッダは**先頭オカレンスで判定**し（query と同じく）、転送先 origin が別オカレンスを読むことによる解釈差（CWE-436）を避ける。
- **weighted split はノードローカル。** weight は各ノードで独立適用されるため、ノード横断の global な比率保証はしない（分散レイヤの opt-in 後続、Fork 5/10）。低トラフィック時は per-node の weighted 選択が目標比率から短期的にずれ得る（量が溜まるまで。決定的 apportionment なので per-node では均等インタリーブ）。
- **renormalize over healthy は近似。** weighted-chosen backend が unhealthy のとき、本 ADR (3) で **残る healthy backend へ再正規化（table を前方 skip）**すると確定した。skip-forward は厳密な eligible-proportional ではなく smooth 順で次へ回す近似（[[000024]] の eligible-set 近似と同型）だが、典型 canary 比では実用上十分。代償として、canary backend がダウンするとそのトラフィックは 503 ではなく他 backend（多くは安定版）へ流れ、canary 障害が end-user には surface しない（health/outlier 状態・メトリクスで観測する）。
- **apportionment table のメモリ。** 縮約後 table は backend index 列で、典型 canary 比（{5,95}→len 20 等）では数十バイト。病的な coprime 大 weight だけ table が膨らむため上限 65536 で build 拒否する（fail-closed、運用者に比例的な小さい weight を促すエラー）。
- **config / compiled route の表面積が増える。** match 句が増えれば per-request の照合コストは dimension 数に比例（allocation-free のまま O(dimensions)）、compiled route table も大きくなる。manifest の不正な match/weight（空 backends・全 weight 0・両指定・上限超過等）は (5b) で **build 時に fail-closed で拒否**する。
- **reload で apportionment cursor を引き継がない。** [[000024]] は RR カーソルを reload 跨ぎで移送したが、weighted route は安定した name を持たない（match キーで選ぶ）ため新 cursor を 0 にリセットする。apportionment schedule の位置 0 は最大 weight backend（＝意図した majority）なので [[000024]] が直した instances[0] 偏りのような実害は無い。頻繁 reload で比率が問題化したら移送を再検討（下記）。
- **確定済み（本 ADR で解決）:**
  - ~~(c) weighted 選択アルゴリズム~~ → **apportionment（error-diffusion / Webster 配分法）を precompute table で確定**（決定的・lock-free、(2b)）。[[000035]] の per-instance LB とは層が別（route→group の split weight と group 内の instance LB weight は別物）。
  - ~~(e) weighted-chosen backend が unhealthy のときの方針~~ → **renormalize over healthy で確定**（(3)）。
- **再検討条件（残）:**
  - (a) regex match（path/header）を ReDoS-bounded エンジン前提で追加。
  - (b) header presence/absence・`not` match・value の case-insensitive 等、Extended の残り。
  - (d) canary の sticky session（consistent hashing で同一 client を同一 backend へ固定、[[000035]] の ring hash と協調）。
  - (f) ノード横断の global weighted split（分散レイヤ opt-in）。
  - (g) gRPC / method-specific routing 拡張。
  - (h) reload 跨ぎの apportionment cursor 移送（頻繁 reload で per-node 比率が問題化したら [[000024]] のカーソル移送に倣う）。
  - (i) apportionment table 上限（65536）の manifest override / 大 weight 自動縮約の高度化。
