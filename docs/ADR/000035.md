# load-balancing アルゴリズムを round-robin 以外へ拡張する（least-request P2C ＋ consistent hashing）

## Status
proposed

## Date
2026-06-30

## Affected Components
- control/upstream（`plecto-control`）— `UpstreamGroup::pick` に per-upstream の algorithm 選択を追加。weighted least-request は per-instance の active-request 数を見て選ぶため、[[000028]] が持つ per-group の `in_flight: AtomicUsize` を **per-instance 粒度へ一般化**して信号に再利用する（projected）
- manifest — `[[upstream]]` に `lb_algorithm`（既定 `round_robin`）と、consistent hashing 用の hash-key 設定（projected: request の header / cookie / source-IP のいずれかを投影）を追加（projected）

## Context

M2 の upstream 層は現在、**healthy 集合上の round-robin** が唯一の LB アルゴリズムである。[[000017]] が「healthy な instance 集合を round-robin で分配」と決め、[[000024]] が縮退時の偏り（dead 枠が隣に吸われる ~1:2）と reload カーソルのリセットを **実装レベルで正した**。この RR は均等回転で公平だが、二つの構造的限界を持つ:

- **リクエストのコストが不均一だと負荷が偏る。** RR は「次の枠」を機械的に回すだけで、ある instance に重い／長命なリクエスト（例: streaming 応答、遅い backend、大きな body）が積もっていても次の pick は無頓着。in-flight の偏りを均せない（[[000024]] の Cons でも「round-robin は接続数/負荷の偏りに弱い」と明記済み）。
- **session affinity / cache locality を表現できない。** 同じ論理キー（ユーザ / セッション / cache key）が毎回別の instance に飛ぶため、backend 側キャッシュのヒット率や sticky session を要求するワークロードに向かない。

重要なのは、[[000017]] が **再検討条件 (a) に「least-conn/EWMA 分配」を明示**し、[[000024]] がそれを継承していたことだ。本 ADR はその (a) を回収する——ただし naive least-conn ではなく、より運用の単純な代替を選ぶ（後述）。

役割駆動の native/WASM 配置基準（[[000029]]）に照らすと、LB は per-request の filter decision では表現できない **横断データプレーン関心事**（upstream という共有資源の選択）であり、native の fast path に置く（Fork 6）。hot path は [[000024]] が確立した **lock-free / アロケーションフリー**を崩さないことを前提にする。

**確定事実 / 現時点の計画（projected）の区別:** 現状の healthy-set round-robin は確定実装。本 ADR が足す P2C least-request と consistent hashing は projected（設計提案）であり、まだ実装・計測していない。先行事例として挙げる Envoy（`least_request` の P2C、ring hash、maglev）・HAProxy（`balance random` の P2C、consistent hashing）・"power of two choices"（Mitzenmacher 系の理論）は **projected prior art であって Plecto 上での計測値ではない**。版や挙動の細部は未検証で、ここでは設計方向の裏付けとしてのみ参照する。

## Decision

- **per-upstream で LB アルゴリズムを選択可能にする。** `[[upstream]] lb_algorithm = "round_robin" | "least_request" | "ring_hash" | …`（projected）。**round-robin は default のまま**。既存 manifest は無変更で従来挙動（[[000024]] の eligible 集合上 RR）を保つ。

- **(1) weighted least-request via power-of-two-choices（P2C）。** healthy 集合から **2 instance を一様乱択**し、active-request 数が少ない方へ流す（重み付きは 2 択の比較に upstream weight を織り込む）。全 instance を走査せず **O(1)** で「ほぼ最小」を選ぶため、variable-cost / long-lived なワークロードで RR より in-flight を均す。
  - **信号は per-instance の active-request count。** [[000028]] の circuit breaker が既に持つ in-flight カウンタ（per-group `AtomicUsize` ＋ RAII の `RequestPermit` で increment/decrement）を **per-instance 粒度へ一般化**して再利用する。新しい計測機構を起こさず、forward 中の active 数という同じ事実を instance 単位で読むだけにする。hot path は atomic 2 read ＋ 乱数で lock-free を保つ。

- **(2) consistent hashing（ring hash / ketama、必要に応じて maglev）。** request 属性（projected config: header / cookie / source-IP のいずれか）を hash key にして、ring（または maglev table）上で instance を選ぶ。**host の add/remove が乱す key を概ね ~1/N に限定**し、全 key の再分配を避ける。session affinity（sticky session）や cache locality（backend 側キャッシュのヒット率最大化）を要求するワークロード向け。ring hash と maglev のどちらを採るかは lookup 均一性とテーブルメモリのトレードオフで決める（後述・再検討条件）。

- **default は round-robin に保つ。** P2C / consistent hashing は **opt-in**。`balance random`（P2C）を default 寄りに置く方向（HAProxy 系、projected; 版を含む細部は未検証）もあるが、(i) 既存挙動の **最小の驚き（least surprise）**、(ii) [[000024]] の純カウンタ RR が持つ **決定性（再現性・テスト容易性）** を優先し、RR を default に据える。default の切替は再検討条件に残す。

検討した代替案と却下理由:

- **round-robin only（現状維持）** — 不均一コストに弱く（[[000024]] Cons 既述）、affinity をまったく表現できない。本 ADR の主目的が拡張なので却下する。ただし **default としては残す**（上記）。
- **pure least-connections（全 instance を full scan して最小接続を選ぶ）** — 厳密に最小を選べるが **O(n)** で、かつ瞬間的に最小の 1 台へ pick が殺到する **herd effect（thundering herd）** を生む。P2C は 2 択で「ほぼ最小」を **O(1)** で選び、herd を避けつつ実用上ほぼ同等の均し効果が得られる（projected prior art: power of two choices）。よって naive least-conn ではなく **P2C** を least-request の実装に採る。
- **EWMA / latency-based（応答時間の指数移動平均で重みづけ）** — より賢いが、per-instance の latency 状態と減衰の timing（半減期等）を持ち込み、状態と運用が増える。まず active-request 数だけを見る P2C を先行し、EWMA は **P2C の後**に検討する（再検討条件）。
- **random+P2C を default 化（`balance random` 方向、projected）vs RR default を維持** — P2C を default にすると未設定でも均し効果を得られるが、既存挙動を変え（最小の驚きに反する）、決定性を捨てる。**RR default を推奨**し、P2C を default にする選択は option として再検討条件に残す。

本決定は [[000017]] / [[000024]] の round-robin/health 設計を **supersede せず拡張**する（RR は default として残り、algorithm は同じ `pick` に slot in する）。[[000017]] 再検討条件 (a)「least-conn/EWMA」を P2C の形で回収し、Fork 6（横断データプレーン関心事は native）を具体化する。fail-closed（eligible 0 で 503）・hot-path lock-free・データプレーン no-panic・single-node-first・operational simplicity の各テネットを崩さないことを条件とする。

## Consequences

### Pros

- **不均一コスト / long-lived リクエストで P2C が RR より負荷を均す**（次の枠ではなく active-request の少ない方を選ぶため）。
- **consistent hashing が session affinity / cache locality を表現でき**、host 増減時の再分配を ~1/N に抑える（全 key 再シャッフルを避ける）。
- **[[000028]] の in-flight counter を一般化して再利用する**ので、least-request の信号を新規に作らない。circuit breaker（飽和保護）と least-request（選択）が同じ「active 数」という事実を共有する。
- **per-upstream 選択なので RR default は不変**で、既存デプロイ・既存テストに影響しない。hot path は lock-free を維持（P2C は atomic 2 read ＋ 乱数、ring hash は table lookup）。

### Cons / Tradeoffs

- **P2C は per-instance の active-request count を要求する**——[[000028]] の per-group counter を instance 粒度へ広げ、RAII の increment/decrement 粒度を増やす（per-group の circuit cap と per-instance の選択信号を両立させる設計が要る）。
- **consistent hashing は hash-key 設定と ring 構築のコストを持ち込む。** reload で instance 集合が変わるたび ring（または maglev table）を作り直す必要があり、[[000024]] の「カーソル移送」のように軽くは済まない（ring の再構築は別設計）。
- **projected ゆえ未計測。** Envoy / HAProxy / power-of-two-choices は prior art であって Plecto 上のベンチではない。実装後に **P2C vs RR の均し効果、ring lookup のホット経路コスト、maglev table のメモリ**を計測して裏取りする必要がある（憶測を確定にしない）。
- **default を RR に保つ**ので、P2C / affinity の利点を得るには明示設定が要る（最小の驚きと引き換えのコスト）。
- **単一ノードローカル。** active-request count も ring も当該ノードのもの。ノード横断で一貫した hashing / 負荷観測は分散レイヤ（foca/openraft）の opt-in 後続（Fork 5/10）。
- **再検討条件**：(a) EWMA / latency-based を **P2C の次**に導入（[[000017]] 再検討条件 (a) の残りを回収）；(b) default を random+P2C へ切替（運用上 RR の驚き／偏りが問題化したら、[[000024]] の決定性を失う代償と引き換えに）；(c) **maglev** を ring hash に加える（lookup 均一性 vs table メモリで選ぶ）；(d) hash key の複数属性合成 / fallback（指定 header 欠落時の RR フォールバック等）；(e) ノード横断の consistent hashing / global least-request（分散レイヤ opt-in）。
