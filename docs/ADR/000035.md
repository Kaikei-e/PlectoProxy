# load-balancing アルゴリズムを round-robin 以外へ拡張する（weighted least-request P2C ＋ weighted maglev consistent hashing）

## Status
accepted

## Date
2026-07-01

## Affected Components

> 設計は grill（`/grill-with-docs`）で一次ソースに照らして確定した。P2C の保証は **Azar–Broder–Karlin–Upfal（balanced allocations, STOC'94/SICOMP'99）** と **Mitzenmacher（two choices, OOPSLA-系/TPDS'01）**、consistent hashing は **Karger ら（STOC'97）** と **Maglev（Eisenbud ら, NSDI'16）**、weighted maglev は論文が**意図的に省く**ため一般技法（smooth weighted-frequency。Envoy PR #2982 / Cilium / Katran-V2 が de-facto 標準だが**特定実装の写しではなく**配分技法として実装）、乱数は **SplitMix64（Steele–Lea–Flood, OOPSLA'14, public domain）＋ Lemire 非偏 bounded（TOMACS'19）**、hash は **MurmurHash3_x64_128（Appleby, public domain）** に拠る。projected だった algorithm 選択・weighting・hash-key・config 形・table 構築はすべて確定値に解決した（後述）。

- control/upstream（`plecto-control`）— `UpstreamGroup::pick` / `pick_excluding` を per-upstream の `lb_algorithm` で dispatch する（既定 `round_robin` は [[000024]] の eligible-set RR をそのまま）。`UpstreamInstance` に per-instance の `in_flight: AtomicUsize` ＋ `weight: u32` を追加。[[000028]] の per-group `in_flight` は overload 保護（circuit breaker）の信号として残し、**per-instance の `in_flight` を別フィールドとして足して** weighted least-request の選択信号にする（二つは別関心事）。maglev の lookup table は reconcile 時に instance 集合から precompute して group に持たせる
- manifest — `[[upstream]]` に `lb_algorithm`（`round_robin` ｜ `least_request` ｜ `maglev`、既定 `round_robin`）、per-instance weight（`addresses` の各要素を bare string ＝ weight 1 か `{address, weight}` の untagged 形へ拡張＝既存 manifest 無変更）、`[upstream.hash]`（`key = "header" | "source_ip"`、header 名、`table_size` 既定 65537）を追加
- server runtime（`plecto-server`）— `proxy_core` が `group.hash_key_source()` を読み、header の値 bytes（借用）か peer-IP の octets を `HashInput` として `pick` に渡す（allocation-free）。per-instance load は `Pick`（RAII の `InstanceLoad` を内包）で attempt 単位に計数し、retry で持ち替えると古い instance の `in_flight` が減って新しい方が増える

## Context

M2 の upstream 層は現在、**healthy 集合上の round-robin** が唯一の LB アルゴリズムである。[[000017]] が「healthy な instance 集合を round-robin で分配」と決め、[[000024]] が縮退時の偏り（dead 枠が隣に吸われる ~1:2）と reload カーソルのリセットを **実装レベルで正した**。この RR は均等回転で公平だが、二つの構造的限界を持つ:

- **リクエストのコストが不均一だと負荷が偏る。** RR は「次の枠」を機械的に回すだけで、ある instance に重い／長命なリクエスト（例: streaming 応答、遅い backend、大きな body）が積もっていても次の pick は無頓着。in-flight の偏りを均せない（[[000024]] の Cons でも「round-robin は接続数/負荷の偏りに弱い」と明記済み）。
- **session affinity / cache locality を表現できない。** 同じ論理キー（ユーザ / セッション / cache key）が毎回別の instance に飛ぶため、backend 側キャッシュのヒット率や sticky session を要求するワークロードに向かない。

[[000017]] は **再検討条件 (a) に「least-conn/EWMA 分配」を明示**し、[[000024]] がそれを継承していた。本 ADR はその (a) を回収する——ただし naive least-conn ではなく、より運用の単純な代替を選ぶ（後述）。役割駆動の native/WASM 配置基準（[[000029]]）に照らすと、LB は per-request の filter decision では表現できない **横断データプレーン関心事**（upstream という共有資源の選択）であり、native の fast path に置く（Fork 6）。hot path は [[000024]] が確立した **lock-free / アロケーションフリー**を崩さないことを前提にする。

**確定事実 / projected の区別:** 現状の healthy-set round-robin は確定実装。本 ADR が足す weighted least-request（P2C）と weighted maglev は確定設計（本 ADR で実装着地）だが、**Plecto 上での負荷計測はまだ無い**。先行事例として参照する Envoy・HAProxy・Maglev 論文・"power of two choices" の理論は **prior art であって Plecto 上の計測値ではない**。版や挙動の細部は実装後にベンチで裏取りする（憶測を確定にしない）。

## Decision

- **per-upstream で LB アルゴリズムを選択可能にする。** `[[upstream]] lb_algorithm = "round_robin" | "least_request" | "maglev"`。**round-robin は default のまま**。既存 manifest は無変更で従来挙動（[[000024]] の eligible 集合上 RR）を保つ。algorithm は同じ `UpstreamGroup::pick` / `pick_excluding` に dispatch を足す形で slot in し、hot-path 呼び出し側（`proxy_core`）のメソッド名は不変。

- **per-instance weight を manifest に足す。** `addresses` の各要素を **bare string（`"h:p"` ＝ weight 1）か inline table（`{ address = "h:p", weight = N }`）** の untagged 形に拡張する。既存の string 配列はそのまま weight 1 として読めるので **後方互換**。weight は `1..=MAX`（既定 1）、`0` は build で拒否（least-request の比は weight で割るため、drain は address を消すことで表す）。weight は weighted least-request と weighted maglev の双方に効く（一貫させる）。

### (1) weighted least-request via power-of-two-choices（P2C）

- healthy（かつ非 outlier-ejected、retry 時は非 exclude）の **eligible 集合から 2 instance を一様乱択**し、**inverse active-request weighting** で「`(in_flight + 1) / weight` の小さい方」へ流す（整数の交差積 `(aᵢ+1)·wⱼ < (aⱼ+1)·wᵢ` で比較、float なし）。`+1` は idle（active 0）でも weight が効くようにするため（Envoy の `active_request_bias = 1` 相当の一般形）。全 instance を走査せず **O(1) 近傍**で「ほぼ最小」を選ぶ。eligible が 1 つならそれを返す。
- **信号は per-instance の active-request count。** [[000028]] が既に持つ per-group の in-flight（circuit breaker の cap 用）を **per-instance 粒度へ一般化**する——ただし *統合はしない*。per-group の `in_flight` は overload 保護（飽和なら 503）として残し、新たに `UpstreamInstance::in_flight` を足して least-request の選択信号にする。選択時に勝者を increment し、**`Pick` が内包する RAII の `InstanceLoad` guard が attempt 終了（成功・retry・transport error の全 return path）で decrement** する。retry で別 instance に移ると古い方が減って新しい方が増える。hot path は atomic 2 read ＋ 乱数 ＋ atomic 1 increment で lock-free を保つ。**round-robin / maglev のときは per-instance 計数を一切行わない**（既定 RR の hot path コストは不変）。
- **乱数は自前の thread-local SplitMix64。** `rand` / `getrandom` を直接依存に足さず（`deny.toml` を触らない・[[000024]] が `smallvec` を避けた精神）、worker ごとに 1 回 seed する SplitMix64（OOPSLA'14、public domain。近接 seed の decorrelator として作者自身が推奨）を thread-local に持つ。区間内一様は **Lemire の nearly-divisionless**（除算をほぼ使わず厳密に非偏）で取り、2 つの distinct index は `j = bounded(n-1); if j >= i { j += 1 }` で得る。retry jitter（[[000030]]）の自前 `jitter` と同じ「自前・非暗号」路線だが、per-call の wall-clock syscall を排して hot path を軽くする。

- **P2C の根拠（[[000024]] からの補正）。** ナイーブな全走査 least-connection は厳密最小を選べるが **O(n)** で、かつ瞬間的最小の 1 台へ pick が殺到する **herd（thundering herd）** を生む。ただし「herd 回避」の効きは文脈に依る——P2C の優位が最大化するのは *分散/stale-state*（多数の独立バランサが古い負荷観測を共有する）であり、**単一ノードで正確な in-process atomic を持つ Plecto は本来「集中型1台」側**で、そこでは厳密 least-conn の精度が高い（HAProxy 自身のベンチでも単一正確バランサでは strict least-conn が P2C を僅かに上回る）。それでも P2C を採るのは: (i) **O(1)**（小プールでも RR と同型の軽さ）、(ii) Plecto に残る **read→compare→increment の TOCTOU 窓**——選択は in_flight を読んで比較し、その後 increment する——で並行 pick が同じ瞬間最小に重なり得るが、P2C の乱択がその衝突確率を下げる（単一ノードでも成立する正直な herd 論）、(iii) [[000024]] の純カウンタ RR と同型の lock-free 設計に綺麗に重なる。よって本 ADR は ADR35 proposed 段の「分散 herd」根拠を **単一ノードの concurrency-window herd ＋ O(1)** へ補正して P2C を採る。EWMA / latency-based はより賢いが状態と運用を増やすので P2C の **後**に検討する（再検討条件）。

### (2) consistent hashing via weighted maglev

- request 属性（`[upstream.hash] key`：**header の値**または **source-IP**）を hash key にして、**maglev の lookup table**（サイズ M、素数）上で instance を選ぶ。**ring hash ではなく maglev を採る**（proposed 段では ring hash 寄り＝maglev を再検討条件 (c) としていたが、grill で覆した。後述の代替案参照）。session affinity（sticky session）や cache locality（backend 側キャッシュのヒット率最大化）を要求するワークロード向け。
  **（2026-07-06 追補）**hash key は **chain 前（inbound 原本）のヘッダ値／接続 peer から導出する**——filter がアフィニティ対象ヘッダを書き換えても LB キーには反映されない。これは意図した意味論として固定する（アフィニティは「クライアントが何を送ったか」で決まり、filter の書換と独立。retry ループを跨ぐ借用の実装都合とも整合）。`source-ip` は [[000057]] 後、前段 LB 配下でも real client IP で意味を持つ。

- **table 構築は Maglev 論文の permutation（offset/skip）＋ weighted な populate。** instance 名から 2 つの**独立**な hash で `offset = h1(name) mod M`、`skip = h2(name) mod (M−1) + 1`（範囲 `[1, M−1]`、M 素数で `(offset + j·skip) mod M` が完全置換）。weighting は論文が意図的に省くため **smooth weighted-frequency 方式**（weight `w` の backend は `max_w / w` 反復ごとに「turn」を得る；反復1で全 positive-weight backend が必ず ≥1 エントリを取るので zero-target 問題が構造的に消える）を一般技法として実装する。全 weight 等しいときは論文の素の round-robin populate に一致する（unweighted は weighted の特殊形）。
- **2 つの独立 hash は 1 つの 128-bit から取る。** MurmurHash3_x64_128（Appleby, public domain）で instance 名を hash し、**low 64 = offset 用 h1 / high 64 = skip 用 h2** に分割する。「FNV を seed 替えで 2 回回す」は**乗法構造を共有して相関**し、Maglev の double-hashing（`offset + j·skip`、Kirsch–Mitzenmacher が h1⊥h2 の独立性を要求）の前提を壊すので採らない。request key → slot も同 MurmurHash3 の low 64 を `mod M` する（FNV は SMHasher 上 uniformity が弱く、slot 分布は均一性が効くため）。MurmurHash3 / SplitMix64 とも public domain で、**一般アルゴリズムを出典付きで自前実装**する（特定 OSS のコード写しはしない）。
- **table は instance 集合上で固定し、health flip では作り直さない。** table は reconcile（instance の add/remove）でのみ再構築し、health/outlier の bit 変化では不変——affinity を health flip で崩さない。pick は `slot = hash(key) mod M`、`idx = table[slot]` を引き、その instance が eligible なら返す。**primary が ineligible（unhealthy/ejected/excluded）または key 欠落のときは eligible-set RR へ fallback**（[[000017]] の no-healthy と同じ倒し方。Envoy も hash policy 不成立で random へ落ちる）。これで lookup は通常 O(1)、down 時のみ O(n) fallback で、65537 規模の table を hot path で線形走査しない。
- **table_size は configurable、既定 65537（素数）。** build で **素数・`M ≥ instance 数`・上限**を検証（fail-closed）。memory は `M × 2 byte`（backend index は `u16`；instance 数は `< 65536` を検証）＝既定で ~128KB/group（単一ノード小規模なら無視可能）。Plecto の小プール（2–10 台）では M ≫ N なので uniformity は常に `⌊M/N⌋..⌈M/N⌉`（差 ≤ 1）。`[upstream.hash]` は **`lb_algorithm = "maglev"` のときのみ**許す（他 algorithm との併記は build 拒否）。maglev なのに hash 未設定も build 拒否（key が無いと常に RR fallback で無意味）。

### 共通

- **default は round-robin に保つ。** least-request / maglev は **opt-in**。既存挙動の **最小の驚き**と [[000024]] の純カウンタ RR が持つ **決定性** を優先する。default 切替は再検討条件に残す。
- **native・lock-free・fail-closed を維持する。** pick は引き続き `instances` を走査する純粋関数で、ヒープ確保も追加 lock も無い（P2C は atomic 中心、maglev は table lookup）。eligible 0 は `None`＝503 fail-closed。untrusted な header 値 / IP は exact byte / octet を hash するだけで no-panic。

**検討した代替案と却下理由:**

- **round-robin only（現状維持）** — 不均一コストに弱く（[[000024]] Cons 既述）、affinity をまったく表現できない。本 ADR の主目的が拡張なので却下する。ただし **default としては残す**。
- **pure least-connections（全 instance を full scan して最小を選ぶ）** — 厳密最小だが **O(n)**＋瞬間最小への herd。P2C は 2 択で「ほぼ最小」を O(1) で選び、TOCTOU 窓の衝突も乱択で薄める。単一ノードでは精度差は小さいが、O(1) と lock-free 同型性を採って P2C を選ぶ（上記「P2C の根拠」）。
- **unweighted P2C / unweighted maglev（per-instance weight を入れない）** — manifest 無変更で済むが、heterogeneous backend（容量差）と canary 容量配分を表現できない。`addresses` の untagged 拡張で後方互換に weight を入れられるので、weighted を採る。weight 0 drain は least-request の除算と衝突するため v1 では不採用（再検討条件）。
- **ring hash / ketama を consistent hashing の v1 にする（proposed 段の lean）** — 最小撹乱（~1/N）・低メモリ（N·R）・reload 時 incremental 更新が安い利点があるが、(i) lookup が O(log V)、(ii) virtual-node 数 R で分布の σ∝1/√R が決まり小プールでは imbalance が残る。**maglev は near-perfect uniformity（差 ≤ 1）＋ O(1) lookup** を固定 table（既定 128KB、単一ノードで無視可能）で得る。撹乱は ring より大きい（~2×、ただし weighted maglev の撹乱を厳密定量した一次ソースは存在せず、緩和は「M を大きく」のみ——正直なギャップ）が、Plecto は reload が稀で health flip では table を作り直さないので churn の実害は小さい。よって **maglev を採り、ring hash を再検討条件**へ回す。
- **EWMA / latency-based（応答時間の指数移動平均で重みづけ）** — より賢いが per-instance の latency 状態と減衰 timing を持ち込む。まず active-request 数を見る P2C を先行し、EWMA は **P2C の後**に検討する（再検討条件）。
- **FNV-1a を hash に使う / seed 替えで 2 hash を作る** — FNV は tiny で決定的だが SMHasher 上 uniformity が弱く、seed 替えは独立にならない（相関）。slot 分布と offset/skip 独立性が効くので MurmurHash3_x64_128 を採り、128-bit 分割で h1⊥h2 を 1 pass で得る。
- **`rand` crate を直接依存にする** — 実績はあるが direct dep 追加＝`deny.toml` レビュー＋supply-chain 面拡大。SplitMix64 / Lemire / MurmurHash3 はいずれも public domain の単純アルゴリズムで、出典付き自前実装の方が依存を増やさず項目を統制できる（[[000024]] が `smallvec` を避けた判断と同型）。

本決定は [[000017]] / [[000024]] の round-robin/health 設計を **supersede せず拡張**する（RR は default として残り、algorithm は同じ `pick` に slot in する）。[[000017]] 再検討条件 (a)「least-conn/EWMA」を weighted P2C の形で回収し、Fork 6（横断データプレーン関心事は native）を具体化する。fail-closed・hot-path lock-free・データプレーン no-panic・single-node-first・operational simplicity の各テネットを崩さない。

## Consequences

### Pros

- **不均一コスト / long-lived リクエストで weighted least-request が RR より負荷を均す**（次の枠ではなく `(active+1)/weight` の小さい方を選ぶ）。weight で heterogeneous backend の容量差を反映できる。
- **maglev consistent hashing が session affinity / cache locality を表現でき**、near-perfect な負荷均一性（差 ≤ 1）と O(1) lookup を固定 table で得る。table を instance 集合上で固定するので health flip で affinity が崩れない。
- **[[000028]] の in-flight を per-instance に一般化して least-request の信号に再利用する**ので、新規の計測機構を起こさない。overload 保護（per-group cap）と選択信号（per-instance count）は同じ「active 数」という事実を別粒度で読む。
- **per-upstream 選択なので RR default は不変**で、既存デプロイ・既存テストの挙動に影響しない（manifest は per-instance weight が additive、それ以外は無変更）。hot path は lock-free を維持し、least-request 以外では per-instance 計数を行わない。
- **依存を増やさない。** 乱数・hash とも public domain の一般アルゴリズムを出典付きで自前実装し、`deny.toml` / supply-chain 面を広げない。

### Cons / Tradeoffs

- **per-instance の active-request count を要求する**——[[000028]] の per-group counter とは別フィールドを足し、RAII の increment/decrement 粒度を attempt 単位に増やす（per-group の circuit cap と per-instance の選択信号を両立させる）。least-request 有効時のみだが hot path に atomic が増える。
- **maglev は reconcile 毎に table を再構築する**（既定 65537 で ~1–2ms、cold path）。instance 集合不変でも v1 は作り直す（health flip では作り直さないが、reload では作る）。table 再利用は最適化として後続（再検討条件）。
- **weighted maglev の撹乱は文献で未定量。** 一次ソースは運用的な「table_size を上げよ」のみ。緩和は M を大きく取ること。Plecto は reload が稀なので実害は小さいが、確定値ではない。
- **P2C は単一ノードでは精度劣位になり得る。** 正確な集中カウンタ下では strict least-conn の方が均しは強い（HAProxy ベンチ）。O(1) と lock-free 同型性を採った意図的トレードオフで、均しが問題化したら strict least-conn / EWMA を再検討する。
- **hash key は untrusted。** header 値はクライアントが自由に付けられ、同一 key を大量送出すれば 1 instance に集まる（affinity の定義上そうなる）。さらに unkeyed hash は意図的衝突（HashDoS）で別 key を同 slot に集め得る。affinity は性能最適化であって権限境界ではない（[[000034]] の untrusted-match と同じ注意）。source-IP は **peer socket address の octets**（spoofable な XFF は使わない）。keyed hash（cluster 共有 secret の SipHash）は再検討条件。
- **default を RR に保つ**ので、利点を得るには明示設定が要る（最小の驚きと引き換え）。
- **単一ノードローカル。** active-request count も maglev table も当該ノードのもの。ノード横断で一貫した hashing / 負荷観測は分散レイヤ（foca/openraft）の opt-in 後続（Fork 5/10）。
- **再検討条件**：(a) EWMA / latency-based を **P2C の次**に（[[000017]] 再検討条件 (a) の残り）；(b) default を least-request / P2C へ切替（運用上 RR の偏りが問題化したら、[[000024]] の決定性を失う代償と引き換えに）；(c) **ring hash / ketama** を maglev に加える（最小撹乱・低メモリが要るワークロード向け）；(d) hash key の複数属性合成 / cookie 対応；(e) **keyed hash（SipHash＋cluster secret）** で HashDoS 硬化；(f) per-instance **weight 0 = drain** を least-request 側でも安全に扱う；(g) maglev table の reconcile-跨ぎ再利用（不変集合で作り直さない）；(h) ノード横断の consistent hashing / global least-request（分散レイヤ opt-in）；(i) [[000034]] の canary sticky session（route→group split と group 内 affinity の協調）。
