# filter に outbound HTTP capability（wasi:http/outgoing-handler）を deny-by-default ＋ per-filter allowlist ＋ SSRF ガードで貸与する方向を定める

## Status
accepted（方向 proposed → 実装確定。詳細サンドボックス設計は下記「実装ノート」で確定し、`security-auditor` レビュー済み）

## Date
2026-06-30

## Affected Components
- host runtime（`plecto-host`）— `wasi:http/outgoing-handler` を deny-by-default `Linker`（[[000006]]）に**明示的に**足し、貸す瞬間に per-filter allowlist・SSRF ガード・epoch deadline・レスポンスサイズ上限・同時実行上限を強制する（projected）
- WIT contract（`wit/world.wit`）— filter が outbound HTTP を要求するための capability import を追加する（収斂方針に沿い `wasi:http` の標準形を採る、projected）
- manifest（[[000007]] 系）— filter ごとに許可する送信先（host/port の allowlist）を宣言する欄を設け、host がその宣言を強制する（projected）

## Context

今日の `plecto:filter` ワールドが import する host-API は `host-log` / `host-clock` / `host-kv` / `host-counter` / `host-ratelimit` の 5 つだけで、**outbound を貸す capability はひとつも存在しない**。つまり filter は文字どおりネットワーク呼び出しを 1 本も発行できない。これは [[000006]] の deny-by-default を厳密に守った結果であり、それ自体は正しい初期姿勢だった。

しかしこの欠落は、成熟した gateway が当然備える filter の**まるごと 1 クラス**を不可能にしている:

- **ext_authz 風の外部認可** — リクエストの可否を外部の authorization サービスへ問い合わせて決める（Envoy の ext_authz / ext_proc が prior art。ここでは一般的な能力比較として参照するだけで、その ABI を採るわけではない）。
- **JWT 検証のための JWKS 取得** — 署名鍵セットを発行者のエンドポイントから取りに行く。
- **OAuth token introspection** — bearer token を IdP の introspection エンドポイントで検証する。
- **OPA 風のポリシー問い合わせ** — 認可判断を外部 policy エンジンへ委譲する。

これらはいずれも「per-request の判断のために外部へ 1 本だけ HTTP を投げ、その応答で `decision` を返す」という同じ形をしている。[[000029]] が役割駆動で確定したとおり、auth / WAF / 複雑な authz といった **user-policy 形状の機能は extension plane（WASM 側）に置く**のが配置基準である。ところが outbound capability が無いために、まさにその extension plane に置くべき機能群が**契約レベルで実装不能**になっている。これは現状の拡張契約における**単一で最大の空白**と言ってよい。

本決定はファウンディング設計の **Fork 7（security / capability 境界）** と Tenet「deny-by-default capability」「fail-closed」に直結する。outbound は最も濫用されやすい能力であり、SSRF（[[000027]] が upstream 構築面で既に塞いだのと同じクラスの脅威）を extension plane 側に新しく開く。したがって本 ADR は**方向（direction）を記録する** proposed であり、詳細なサンドボックス設計は `security-auditor` スキルでのレビューを経て後続で確定する。本文中の具体（allowlist 強制点・SSRF 判定範囲・各上限値）はすべて現時点の計画（projected）であり、確定事実ではない。

## Decision

**outbound HTTP capability を、`wasi:http/outgoing-handler` という標準形で、deny-by-default の同じゲートを通して filter に貸与する方向を採る。** ext_authz / JWKS / introspection / OPA クラスを extension plane で実装可能にする。具体は projected:

- **同じ deny-by-default `Linker` を通す。** outbound は [[000006]] の `Linker` ベース能力付与の延長として、**明示的に・filter 単位で**貸す。既存の host-API と同様「ホストが明示的に貸した能力以外、何も触れない」を保つ。`wasi:http` 型を借りても capability 境界は型語彙と独立（[[000020]] の原則）——標準型を採用することは広い権限の付与を意味しない。デフォルトは従来どおり**貸さない**。
- **per-filter allowlist を必須にする。** outbound を要求する filter は、許可される送信先（host/port）の allowlist を宣言する。host はその宣言を**強制**し、allowlist 外への接続を拒否する。wholesale な「どこへでも出られる」貸与はしない。
- **SSRF ガード（load-bearing・`security-auditor` レビュー必須）。** 内部 / link-local / loopback / cloud-metadata 宛て（クラウドの metadata エンドポイント。実アドレスは本書に記さず一般語で表す）のアドレスを**ブロックし、deny-by-default**で判定する。判定は名前解決後のアドレスにも効かせ、DNS rebinding 的な迂回を想定する。**upstream 構築の経路を流用しない**——fast path の upstream connector とは別の、allowlist と SSRF 判定を内蔵した専用経路にする（[[000027]] が「正規化はルート選択にも転送にも効く単一地点でなければ意味がない」と整理したのと同じ精神で、outbound は別境界として独立に設計する）。
- **fail-closed と資源境界。** filter の outbound 呼び出しが失敗 / deadline 超過しても**素通り（fail-open）させない**——タイムアウトした外部 authz 問い合わせは「許可」に倒さない。他の host capability と同じく concurrency / size / time を bound する: epoch deadline、レスポンスサイズ上限、同時 outbound 数の上限。

`wasi:http/outgoing-handler` を採るのは、[[000020]] でコミット済みの **wasi:http 収斂の自然な乗り物**だからである。outbound は `wasi:http` の標準的な形であり、独自の `host-http` import を発明する代わりにこれを採ることは、収斂を**前進させる**（別契約の発明を増やさない）。proxy-wasm の `proxy_http_call` は「outbound を貸す能力」の prior art として**比較参照するだけ**で、Plecto は proxy-wasm の ABI を採らない（Plecto は Component Model + wasi:http、[[000029]] で不採用を確定済み）。

検討した代替案と却下理由:

- **outbound を永久に貸さない（現状維持）。** capability 面では最も安全だが、ext_authz / JWKS / introspection / OPA という gateway の中核 filter クラスをまるごと断念することになる。成熟した gateway としては制約が強すぎる。却下。
- **outbound を host 側 ext_authz として native に実装する（filter には貸さない）。** SSRF 面は楽になるが、extension plane の強み（polyglot な filter で任意言語のポリシーを差し込める＝拡張プレーンの存在意義）を失う。[[000029]] の役割駆動配置（user-policy 形状は WASM 側）にも反する。却下。
- **`wasi:http` を allowlist 無しで wholesale に貸す。** 実装は最小だが、任意の内部サービス・metadata への到達を許す**容認できない SSRF 面**を開く。deny-by-default の真逆。却下。
- **独自の `host-http` import を `wasi:http` の代わりに作る。** 契約を完全制御できるが、[[000020]] の収斂方針から**逸脱**し、標準と再び乖離する contract drift を招く。却下。

本決定は **deny-by-default capability**（型語彙と独立に維持）と **fail-closed** の Tenet、Fork 7 を、[[000006]] が範囲外にしていた **outbound** 面へ具体化する。[[000006]] / [[000027]] を supersede せず**補完**する。これは Fork 7 における**新規の capability ＋ サンドボックス面**であり、実装前に `security-auditor` スキルで設計する前提を明記する。本 ADR は方向の確定に留め、詳細なサンドボックス設計（allowlist の表現・SSRF 判定アルゴリズム・各上限値・WIT import の正確な形）は後続で確定する。

## Consequences

### Pros
- ext_authz / JWKS 取得 / OAuth token introspection / OPA ポリシー問い合わせという gateway 中核 filter クラスが、ようやく extension plane で**実装可能**になる（拡張契約の最大の空白を埋める）。
- 標準 `wasi:http/outgoing-handler` を採ることで [[000020]] の収斂を前進させ、`wasi:http` 向けに書かれた既存 filter / SDK が outbound を馴染んだ形で使える。独自 host-http の再発明と contract drift を避けられる。
- outbound という最も濫用されやすい能力を、**貸す瞬間に** allowlist ＋ SSRF ガード ＋ fail-closed ＋ 資源境界で囲う形を最初から設計に織り込める（後付けの hardening にしない）。
- [[000029]] の役割駆動配置（user-policy は WASM 側）を能力面から裏付ける——native へ ext_authz を吸い込まずに済む。

### Cons / Tradeoffs
- **新規の攻撃面（SSRF）を extension plane に開く。** [[000027]] が upstream 構築面で塞いだのと同じクラスの脅威を、別境界として独立に塞ぎ直す設計負担が要る。`security-auditor` レビューを実装前提とする。
- **deny-by-default の意図的な摩擦が増える。** outbound を貸すたびに allowlist 宣言・強制・SSRF 判定の配線が要る（[[000006]] が認める「能力を増やすたびの最小権限判断」の延長）。
- **epoch は host 呼び出し内でブロックした guest を中断できない**（[[000006]] / [[000027]] が自認）。outbound はまさに host 内で I/O 待ちする呼び出しなので、この制約が顕在化する。outbound 専用の deadline を host I/O 側に設計する必要があり、これは [[000006]] が記録した「ホスト呼び出しでブロックする外部 I/O が増えたら別タイムアウト機構を設計する」再検討条件の発動に当たる（wasmtime 46 の async host call と同時期）。
- allowlist は eviction / 動的更新の方針を別途要する（manifest 配信 [[000007]] の領分）。
- **再検討条件**: (a) `security-auditor` レビューで SSRF ガードを deny-by-default かつ upstream 経路と独立に切り出せないと判明したら、本方向（filter への貸与）を取り下げ host 側 ext_authz へフォールバックする；(b) `wasi:http/outgoing-handler` の async / WIT 形が M3 の wasi:http 収斂タイミングに乗らないなら、収斂ゲート（[[000020]] の再検討条件 (b)）に従属させて後ろ倒す；(c) outbound のブロッキング待ちが worker を律速するなら host I/O 側の非同期タイムアウト機構を先に設計する。

## 実装ノート（確定事項・2026-07-01）

本 ADR が「方向」に留めた具体を、grill セッション（4 つの Fork を確定）＋ 実装で固めた。`security-auditor`
レビュー（下記）を通した上で **feature-gated end-to-end** で着地。本文の projected はすべて下記で確定に置換する。

### 範囲とシーケンス（Fork A → feature-gated end-to-end）

1. **`outbound-http` feature（off-by-default）の裏で `wasi:http/outgoing-handler` を実配線した。** default の
   shipped world は sync・WASI-free・header-only（[[000010]]）のまま——ADR20 の M3 収斂 gate まで default 契約は
   触らない。これで本 ADR の再検討条件 (b) を「WIT 形は feature 裏に隔離し、default 契約化は収斂 gate に従属」
   という形で満たす。`streaming-body`（[[000025]] 系）と同じ gating パターン。
2. **outbound を使う guest は multi-thread な tokio runtime で駆動する。** 既存の header-only path は host-API
   import が決してブロックしないので no-reactor の `pollster` で足りたが、outbound は実ソケット I/O でブロック
   するため reactor が要る。これは [[000006]] が記録した「host 呼び出しでブロックする外部 I/O が増えたら
   host 側の非同期 timeout 機構を設計する」再検討条件、および本 ADR の再検討条件 (c) の発動——**host I/O 側の
   `tokio::time::timeout` で実装して解消**した（epoch は host 呼び出し内の guest を中断できないため）。

### SSRF ガード（Fork B → private は per-filter opt-in）／ IP ピン留め（load-bearing）

3. **一般アルゴリズムは OWASP SSRF Prevention に従い、自前の octet 判定で実装した**（特定 OSS 実装の写しではない）。
   2-tier: **常時 deny の floor**（loopback 127/8・link-local 169.254/16＝cloud-metadata 含む・unspecified 0/8・
   multicast・broadcast・240/4・100.64/10 CGNAT・IETF/benchmark/documentation・IPv4-mapped/compat IPv6 は
   埋め込み v4 に還元して再判定）は allowlist にも private opt-in にも**優先して**効く。private（RFC1918 / ULA
   fc00::/7）は operator が per-filter に `allow_private`（CIDR で絞る）を宣言したときのみ許可。
4. **判定は名前解決後の全アドレスに効かせ、1 つでも block なら request 全体を拒否する**（rebinding な A レコード
   集合対策・OWASP の最小より厳しめ）。**IP ピン留め接続**——host 自身が DNS 解決し、検査を通った具体 IP に
   直接接続する（ホスト名で再接続しない）ことで検査と接続の間の TOCTOU（DNS rebinding）を塞ぐ。TLS の SNI /
   証明書検証は**元のホスト名**で行う。[[000027]] の upstream connector は流用せず、専用経路に allowlist ＋ SSRF
   判定を内蔵した。

### allowlist（Fork C → exact scheme+host+port）／ 資源境界（Fork D → outbound 専用 timeout 分離）

5. **allowlist は `[filter.outbound]` の exact `(scheme, host, port)` タプル**（wildcard/suffix なし・host は
   case-insensitive）。operator 所有で、untrusted filter は供給も上書きもできない（`host-ratelimit` の
   「operator が限度を持つ」モデル [[000026]] と同形）。デフォルトは貸さない・列挙外は deny。
6. **資源境界**: connect timeout・**whole-call の host-side timeout**・response body 上限（`Limited`, CWE-770）・
   per-filter 同時 outbound 上限（shared semaphore）。**guest 供給の `wasi:http` RequestOptions timeout は無視**し
   host policy 値を使う（guest は境界を緩められない）。値は manifest で宣言し host max に clamp。redirect は
   追わない（低レベル hyper conn を使うので 3xx はそのまま guest へ返る＝redirect SSRF 経路を持たない）。
7. **fail-closed**: allowlist deny / SSRF block / DNS 失敗 / connect 失敗 / timeout / protocol はすべて guest に
   `wasi:http` の `error-code` として届く（沈黙成功にしない）。例 `filter-extauthz` は任意の outbound エラーで
   403 に short-circuit する。判断（decision）の fail-closed は guest の責務（標準 `wasi:http` 契約上 host は
   decision を強制できない）——契約と SDK ガイダンスで担保する。

### capability 境界（deny-by-default 維持）

8. **貸すのは `wasi:http/outgoing-handler` ＋ 最小 WASI base（io / clocks / random / cli-stdio）だけ。**
   **filesystem も raw sockets も貸さない**——outbound はすべて gated な `send_request` フックを通り、guest は
   任意 TCP/UDP を開けない。`wasi:http` 型を借りても capability 境界は型語彙と独立（[[000020]]）を維持。

### security-auditor レビュー

9. **F-001（Medium・修正済み）**: 全 outbound を駆動する tokio runtime が current-thread だと複数スレッドからの
   並行 `block_on` で contend する（可用性）。**2-worker の multi-thread runtime に変更**して解消（I/O bound な
   ので 2 で足り、スレッド数も bound する）。
10. **F-002（Info・by-design）** guest がエラーを握り潰して continue する余地は標準契約上 host 強制不能。
    **F-003（out of scope）** 将来の fast-path server 統合（async 文脈からの呼び出し）は [[000021]] §4 の
    spawn_blocking 撤去と同時期に設計する（本 increment は sync host API ＋ テストで完結）。

### テスト

11. 反証可能に固定: SSRF 分類器の全レンジ（metadata / v4-mapped / CGNAT / private opt-in 等）＋ allowlist
    exact match の unit、connector の実サーバ成功 ＋ body cap ＋ 並行上限、そして**実 wasip2 guest
    （`filter-extauthz`）越しの E2E**——非 allowlist は `HttpRequestDenied`、allowlist 済みだが loopback に解決
    する名前は（実サーバが listen していても）`DestinationIpProhibited` で fail-closed。
