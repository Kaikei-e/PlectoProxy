# WAF を native では declined とし、拡張プレーン（WASM filter）/host-side engine に配置する（参照エンジンは deferred）

## Status
accepted — placement-only (native WAF declined; reference engine selection remains deferred per Decision)

## Date
2026-06-30

## Affected Components
- extension plane（`plecto-host` / `plecto:filter`）— WAF（rule マッチング・anomaly scoring・CRS 相当の検査）は WASM filter または host-side engine（capability 経由）として載せる。native fast path には WAF を持たせない
- reference engine — 出荷可能な参照 WAF filter は research / deferred。採用エンジンは未確定で、本 ADR は配置の決定であってエンジンの選定ではない

## Context

成熟した L7 / API-gateway にとって WAF（Web Application Firewall）は Tier-0 級の保護機能であり、Plecto でも「どこに置くか」を先に確定しておく必要がある。配置の判断軸は roadmap ADR [[000029]] が固定した**役割駆動**の基準に従う: native が育つのは「横断的データプレーン関心事 or グローバルカウンタ」に限り、それ以外は native 化を明示的に declined とする（Fork 6）。

WAF は本質的に **user-policy 形状**の機能である。ルールセット・除外・anomaly scoring の閾値・false-positive チューニングはすべて運用者のポリシーであり、リクエストごとの判断としてフィルタの `decision` 契約で表現できる。これは [[000005]] が配置分割の基準を「ユーザー固有のロジック・ポリシー・WAF・認証・書換は WASM」と明記した通りで、本 ADR はその分類を WAF について具体化する。すなわち **native WAF は declined**、WAF は拡張プレーン側の関心事である。

ここで配置の議論とは別に、**実装手段に関する確定事実**を正直に記録しておく必要がある。

- production-grade な **pure-Rust の SecLang / OWASP-CRS エンジンは存在しない**。広く使われる埋め込み可能 WAF の旗艦である **OWASP Coraza は Go 実装**で、Go ライブラリとして埋め込むか、proxy-wasm filter（`coraza-proxy-wasm`）として出荷される。後者は **Envoy の proxy-wasm ABI** を対象にしており、Plecto が採る **Component Model + `wasi:http`** とは別の拡張モデルである（[[000020]] / [[000029]] で proxy-wasm 不採用を確定済み）。
- pure-Rust を志向する取り組み（例: `zentinel-modsec` 系）は early-stage / single-vendor の段階で、CNCF / OWASP grade の依存として据えられる成熟度には達していない。

したがって「native に CRS エンジンを埋め込む」も「proxy-wasm の Coraza をそのまま載せる」も、現時点では成立しない。本 ADR は**配置（WAF = 拡張プレーン、not native）を決定**し、参照エンジンの実装手段は open question として明示的に残す。

## Decision

- **native WAF を declined とする。** WAF は user-policy 形状の機能なので、native fast path には組み込まず、**拡張プレーン（WASM filter）または host-side engine（capability 経由）**で提供する。これは Fork 6 / [[000029]] の役割駆動配置を WAF について具体化したもので、加えて native に埋め込めるだけの pure-Rust CRS エンジンが存在しないという事実とも整合する。
- **参照 WAF filter は research / deferred とする。** 本 ADR ではエンジンを選定しない。現実的な実装手段を、いずれもコミットせずに選択肢として記録するに留める:
  - (a) WAF エンジンを **WASM guest filter として componentize** し、`plecto:filter`（Component Model）の契約で載せる。
  - (b) WAF エンジンを **host-side engine として呼び出し**、フィルタには capability として最小貸与する。
  - (c) 既存エンジンを **bridge** する（例えば Go 実装を host 側で動かし、判断だけを契約越しに渡す）。

  どれも相応のコストを伴い、free な選択肢は無い。本 ADR が確定するのは **placement（WAF は拡張プレーン、not native）と、未解決の実装課題の所在**であって、特定エンジンの採用ではない。
- **依存関係を明示する。** リクエストボディを検査する WAF は **body 契約（[[000025]]）に依存する**——header-only 契約では body を見られないため、body-inspecting WAF は body hook（buffer-then-decide）の上に乗る。また reputation lookup（IP / payload の外部照会）等を要する場合は、フィルタの **outbound capability（[[000036]]）が必要**になり得る。いずれも deny-by-default 境界の内側で、ホストが明示的に貸した能力としてのみ成立する。

検討した代替案と却下理由:

- **native fast path に WAF を組み込む** — 却下。WAF は policy 形状なので役割駆動配置（Fork 6）により拡張プレーンが正しい置き場であり、かつ native に埋め込めるだけの成熟した pure-Rust CRS エンジンがそもそも存在しない（上記 Context）。配置原則とエンジンの現実の両面から native は成立しない。
- **`coraza-proxy-wasm` をそのまま載せる** — 却下。これは **proxy-wasm ABI** 向けの成果物であり、Plecto は proxy-wasm を意図的に採用しない（Component Model + `wasi:http` 収斂、[[000020]] / [[000029]]）。ABI が異なるため「そのまま載せる」は技術的に成立せず、また拡張モデルの一貫性を崩す。
- **部分的な CRS を自前で fabricate / hand-roll する** — 却下。WAF は security-critical であり、網羅性・誤検知耐性・CRS の継続メンテに追従できない自前実装は、**保護しているという誤った安心感（false sense of protection）**を生むだけで、保護が無いより危険になり得る。security-critical な機能を中途半端な自作で埋めない。

本決定は Fork 6（横断データプレーン関心事は native、user-policy は WASM）を具体化し、deny-by-default / fail-closed / operational simplicity の各テネットと整合する。[[000005]] の配置分割と [[000029]] の役割駆動原則を **補完**し、いずれも supersede しない。

## Consequences

### Pros
- native fast path を WAF ロジックで肥大させない。ルールセットの更新・チューニングは hot-swap 可能な拡張プレーン側に閉じ、rule 更新のたびに native を rebuild しなくて済む。
- エンジンの現実を正直に記録できる。成熟していない依存（early-stage の pure-Rust 実装や ABI 違いの proxy-wasm 成果物）へ**早すぎるコミット**をせずに済む。
- WAF 専用の native サブシステムを発明せず、既存の境界（sandbox / deny-by-default、body 契約 [[000025]]、outbound capability [[000036]]）を再利用する設計に落ちる。

### Cons / Tradeoffs
- 本 ADR の時点で **出荷可能な参照 WAF が無い**。CRS 相当の保護を求める運用者に対する turnkey な選択肢は、エンジン課題が解けるまで提供されない。
- body を検査する WAF は **WASM 税**（[[000005]]）を負う。サンドボックス内外のコピーと buffer-then-decide（[[000025]]）による遅延・メモリのコストが、header-only フィルタより重い。
- Go 実装の Coraza を Component Model / host-side に取り込む経路（(a)/(b)/(c)）はいずれも非自明な統合作業で、bridge には保守コストが付く。
- **再検討条件**: (a) production-grade な pure-Rust の SecLang/OWASP-CRS エンジンが成熟した場合（native ではなく**拡張プレーン側のエンジンとして**採用を再検討）；(b) WASI 0.3.x の stream splicing 等で body 検査のコストが下がり、body-inspecting WAF の閾値が変わった場合（[[000005]] / [[000025]] の再検討トリガと同期）；(c) Component-Model-native な WAF エンジンが登場した場合に手段 (a) を具体化。配置（拡張プレーン）は据え置き、変わるのはエンジン選定のみ。
