# header-only フィルタの request body を buffer せずゼロコピー素通しし、buffer 経路の残余 RSS を glibc arena 上限で抑える

## Status
accepted

## Date
2026-07-01

## Affected Components
- **wit (plecto:filter)** — 単一 `world filter` を、header-only の `filter`（init / on-request / on-response）と body-reading の `filter-body`（+ on-request-body）へ分割。`on-request-body` は必須 export ではなくなった。
- **host runtime (plecto-host)** — load 時に `Component::get_export_index("on-request-body")` で export 有無を検出し `LoadedFilter::reads_body()` として公開。body-hook は optional export なので bindgen 経由でなく index 取得 + typed-func で呼ぶ。
- **control (plecto-control)** — `CompiledRoute` / `RouteInfo` に `reads_body` を持たせ、route の各フィルタの `reads_body()` を OR で集約。
- **fast path (plecto-server)** — proxy の body buffer を `route.reads_body && !bodyless` で gate。`plecto` bin 起動時（worker spawn 前）に `mallopt(M_ARENA_MAX, 4)`。
- **example filters** — auth / header-stamp 系（apikey / quickstart / template / extauthz）を header-only（world `filter`）へ、body を読む hello / noop を `filter-body` へ。

## Context

「ボディ税」調査（[docs/servey body-tax report](../servey/body-tax-memory-2026-07-01.md)）で、buffer-then-decide の request body 経路（[[000025]]）が 1MB body × 50 並列で **~355MB RSS** を載せることを、proxy 単体・時系列計測・交絡統制つきで分解した。結果は当初の 2 仮説の順位を覆した:

- **(A) host↔guest コピー往復** — payload×concurrency 比例。`list<u8>` は Component Model canonical ABI 上、guest linear memory に lower / lift される本質的コピー。live working set ~100MB。
- **(B) pooling allocator の固定常駐費** — 反証。`PoolingAllocatorMetrics::unused_memory_bytes_resident()` が全セル 0（`linear_memory_keep_resident` 未設定＝既定 0）。pool の恒久分 ≤~16MB。
- **(C) glibc マルチアリーナ断片化**（当初 2 仮説に無し） — 最大要因。1MB×50 の RSS の **~63%**。glibc は 64bit で既定 `8×コア`（本調査ホストで最大 192 arena）まで per-thread arena を作り、bursty な大 alloc の free がアリーナに居座り OS へ返らない。`MALLOC_ARENA_MAX=1` で peak 373→139MB（−63%）・スループット悪化なし。

**根治**は、body を読まないフィルタに対して body を guest memory へ materialize しないこと＝(A) の原因除去であり、同時に大 transient alloc が消えて (C) も縮む。現状は全フィルタが `on-request-body` を export し（多くは pass-through）、route にフィルタが有れば body を必ず buffer していた。これは [[000005]] mechanism 2 が予告した「export-presence-based zero-copy bypass」の未実装状態そのもので、`wit/world.wit` と filter-apikey のコメントに follow-up として明記されていた。

関係する Tenet / 原則: **「判断は型で」**（buffer 要否を契約＝export 有無に従属させる）、**fail-closed**（body を読むフィルタを取りこぼさない）、**ゼロコピー**（body-untouching は stream をバイパス）、**single-node / 運用の単純さ**（arena 上限は移植可能な既定）。

## Decision

二段構えで対処する。**主**は契約由来の zero-copy bypass（根治）、**従**は buffer が残る経路への防御的 allocator 上限。

### 1. export presence による body buffer の要否判定（zero-copy bypass）

`plecto:filter` world を二つに分割する:

- `world filter` — `init` / `on-request` / `on-response` のみ（header-only）。`on-request-body` を **export しない**。
- `world filter-body` — 上記に `on-request-body: func(list<u8>) -> request-body-decision` を加えたもの。

host は load 時に `Component::get_export_index(None, "on-request-body")` で export 有無を検出し、`LoadedFilter::reads_body()` として返す。control は route の各フィルタの `reads_body()` を OR して `RouteInfo.reads_body` に畳み込み、fast path はそれが `true` の時だけ body を buffer して `on-request-body` を回す。`false`（header-only フィルタのみ）の route は body を upstream へストリーム素通しする。判定源が **コンポーネントの export（＝契約）** なので sound（fail-closed）: フィルタは契約に宣言せずに body を読めない。

body-hook は optional export で bindgen の必須世界に無いため、host は取得済み `ComponentExportIndex` から `Instance::get_typed_func::<(&[u8],), (RequestBodyDecision,)>` で借用のまま呼ぶ（host 側の追加コピー無し）。

**検討した代替案と却下理由:**

- **manifest 宣言 `reads_body`（operator 申告、既定 true）** — 実装は最小だが、body を読むフィルタ（WAF / body 検証）を誤って `reads_body=false` と申告すると hook が黙って skip される＝**fail-open**。Plecto の fail-closed tenet と衝突するため却下。export presence は host が真実を知るので不整合が起きない。
- **WIT の optional export キーワード** — Component Model 仕様上まだ未実装（将来機能）。`@unstable` gate は encode 時の話で per-component の実行時 optionality にならない。よって「必須世界から外して host が名前で lookup」が現状唯一の手段。
- **`include filter` で filter-body を DRY に** — WIT は include した world の型 `use` を新しい export の scope へ伝播しないため（`request-body-decision does not exist` になる）、共有形を filter-body に明示的に重複記述した。
- **streaming `stream<u8>` へ即移行** — buffer 経路そのものを bounded chunk 化する本命だが、`streaming.rs`（feature `streaming-body`）は experimental で wasm32-wasip3 Tier 2 待ち。今回の対象外（[[000020]] の M3 収斂で扱う）。

これは [[000005]] mechanism 2 の実装であり、Fork「フィルタ判断は型で（decision variant）」を body 経路へ拡張、Tenet「fail-closed」「ゼロコピー」を具体化する。

### 2. glibc arena 上限（buffer が残る経路への防御的補完）

`plecto` bin 起動時、tokio runtime の worker spawn 前に `mallopt(M_ARENA_MAX, 4)` を呼ぶ（`#[tokio::main]` をやめ手動 runtime）。`M_ARENA_MAX` は **新規** arena 生成のみを絞り既存を回収しないので、worker 生成前に効かせる必要がある。既定 **4**、`PLECTO_MALLOC_ARENA_MAX` で上書き（`0` で glibc 既定）、glibc 限定 cfg（musl / macOS では no-op）。

**N の決め方:** 実測では `MALLOC_ARENA_MAX=1` が RSS 最小（buffer 経路 326→137MB）だが、それは**特定ホストへの overfit**であり、Plecto はセルフホストで多様な多コア機に載る OSS。高RPS 小 alloc 経路の lock 競合を避けつつ断片化を大幅に抑える移植可能値として、多スレッドサービスの業界既定 **4**（Elasticsearch / Cloud Foundry 系）を採り、機ごとに env で調整可能にした。

**検討した代替案と却下理由:**

- **`MALLOC_ARENA_MAX` 環境変数** — glibc は allocator 初期化時（≒ 最初の malloc、main 到達前）に一度だけ読むため、`std::env::set_var` では手遅れ。かつ edition 2024 で unsafe。プロセス外の launch wrapper に依存させたくないので、in-process の `mallopt` を採用。
- **jemalloc / mimalloc へ差し替え** — jemalloc 既定は `muzzy_decay` の MADV_FREE 保持で RSS がむしろ増える（実測 377MB）。チューニング（`MALLOC_CONF`）前提の大きな依存追加は、供給網最小化の方針に対し過剰。glibc 上限は追加依存が libc のみで最小。

## Consequences

### Pros
- **header-only フィルタは body 税ゼロ。** filter-quickstart 単体の route に 1MB×50 の POST を流すと peak RSS **83MB**（`/baseline` 91MB とほぼ同じ、buffer する `/body` は 352MB）＝ **−76%**、かつスループット 5720 rps（body 1585 の 3.6×）。auth / rate-limit / header 書換のような大多数のフィルタが恩恵を受ける。
- **判定が sound（fail-closed）。** operator 申告でなく契約 export に従うので、body を読むフィルタを取りこぼさない。「判断は型で」の一貫適用。
- **buffer が残る経路（WAF / 変換）も arena 上限で半減。** `/body` 1MB×50 が glibc 326MB → `MALLOC_ARENA_MAX=4` 163MB（**−50%**）。移植可能な既定で、env 上書き可。
- **将来の streaming 増分と直交。** stream<u8> 化（[[000020]]）が入れば buffer 経路の (A) もさらに縮む。今回の bypass はそれと重ねられる。

### Cons / Tradeoffs
- **WIT が二世界になった（guest から見える）。** body を読むフィルタは `world: "filter-body"` を、header-only は `world: "filter"` を選ぶ必要がある。共有形は filter-body 側に重複記述（include が型 `use` を伝播しないため）。
- **host が optional export を manual typed-func で呼ぶ。** bindgen の必須世界に body-hook が無いぶん、`get_typed_func` + `call_async` を host に持つ。**再検討条件:** WIT が optional export を、または wasmtime bindgen が optional 世界を正式サポートしたら、この手動経路を畳む。
- **buffer 経路の (A) コピーは未解決。** body を読むフィルタは依然 body を guest memory に materialize する。**再検討条件:** wasm32-wasip3 / component-model-async が成熟したら `stream<u8>`（`streaming.rs`）を有効化し bounded chunk 化する。
- **arena 上限 N=4 は heuristic 既定。** **再検討条件:** 実デプロイで高RPS の malloc 競合（p99 悪化）や、逆に RSS 高止まりが観測されたら N を上下（worker 数寄り or 2〜1）、あるいは `M_MMAP_THRESHOLD` を別レバーとして検討する。
- **`post-return` は wasmtime 46 で内部処理・no-op 化**されており明示呼び出しはしない。将来の wasmtime 更新で意味論が変わる可能性は追う。
