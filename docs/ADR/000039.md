# 出荷バイナリに SIGHUP reload と graceful shutdown を配線する

## Status
accepted

## Date
2026-07-02

## Affected Components
- fast path（`plecto-server`）— `main.rs` が `serve()` に加えて reload ループを spawn し、シャットダウンシグナルで accept を止めて in-flight を drain する
- control plane（`plecto-control`）— `reload.rs` / `control_reload.rs` の既存実装（content-hash idempotency・fail-closed・`ArcSwap` 原子スワップ）を変更せず消費する

## Context

無停止 reload は [[000008]] が宣言した本プロジェクトの核心的性質であり、reload 機構そのものは control クレートに完成している——`serve_reloads` + `SignalReloadSource::sighup`（`crates/control/src/reload.rs`）、`reload_from_disk`（content-hash で idempotent、検証失敗は fail-closed で現行構成維持、`ArcSwap` による原子スワップ）。TLS 証明書ホットリロード（[[000014]]）や RR カーソル引き継ぎ（[[000024]]）もこの経路を前提に設計されてきた。

しかし 2026-07-02 のギャップ分析（コード実体インベントリ）で、**出荷バイナリ `plecto` がこの機構を一切呼んでいない**ことが確認された。`crates/server/src/main.rs` は `serve()` を呼ぶだけで reload ループを spawn しておらず、唯一の消費者は `examples/hot-reload/main.rs`。さらに accept ループ（`crates/server/src/listener.rs`）には shutdown シグナル処理も connection draining も無く、プロセス終了は in-flight リクエストを切断する。つまり「無停止 reload」は看板と実体が乖離しており、これは機能不足ではなく **tenet（運用の単純さ）と既存 ADR の未履行**である。

外部環境の参考（取得日 2026-07-02、駆動力ではない）: 業界水準は「graceful reload がある」から「reload 自体が不要な動的再構成」へ進んでいる（HAProxy 3.4 の runtime backend 追加/削除等）。Plecto は宣言的マニフェスト + 原子スワップ（[[000008]]）で既にこの方向にあり、欠けているのは配線だけである。

native 採用基準（[[000029]]）上は、プロセスライフサイクル管理は per-request decision ではない横断的データプレーン関心事（基準 (i)）であり、native に置くことに議論の余地はない。

## Decision

出荷バイナリに次の二つを配線する。

1. **SIGHUP reload**: `main.rs` が `serve_reloads`（`SignalReloadSource::sighup`）を spawn し、SIGHUP 受信でマニフェストを再読込・検証・原子スワップする。失敗時は fail-closed で現行構成を維持する（既存実装のまま）。
2. **graceful shutdown**: SIGTERM / SIGINT で (a) 全 listener の accept を停止し、(b) in-flight リクエストを drain 期限まで待ち、(c) 期限超過分を打ち切って終了する。drain 期限のデフォルト値と manifest 設定可否は実装時に確定する（projected）。データプレーン no-panic は維持する。

**admin API 経由の reload は導入しない。** admin listener（`/metrics` `/healthz` `/readyz`）は read-only を保つ。

検討した代替案と却下理由:
- **admin API reload（POST /reload 等）**——運用ツールから叩けて便利だが、read-only admin という攻撃面の最小化を崩す。宣言的ファイル + シグナルで運用要件は満たせる。却下。
- **ファイル監視（inotify）による自動 reload**——シグナル送信すら不要になるが、「いつ構成が変わるか」が暗黙になり、fail-closed 時の運用者の認知（reload が失敗したことに気づく契機）も弱くなる。明示的シグナルの方が運用が単純。却下。
- **example のまま放置し、運用者に自前配線させる**——ライブラリとしての柔軟性は保てるが、`plecto` を単体バイナリとして self-host する主用途で看板（無停止 reload）が成立しない。却下。

本決定は [[000008]]（無停止 reload）の実体化であり、[[000029]] の native 採用基準 (i) に適合する。新しい設計 Fork は増やさない。

## Consequences

### Pros
- [[000008]] の「無停止 reload」が出荷物として初めて成立する。証明書ローテーション（[[000014]]）・weighted split の drain（weight 0）等、reload 前提の既存機能が実運用可能になる。
- graceful shutdown により、デプロイ・再起動時の in-flight 切断がなくなり、外部 ACME クライアント + SIGHUP のような運用パターン（[[000043]] の ACME 後回し判断の前提）が成立する。
- admin が read-only のままなので攻撃面は増えない。

### Cons / Tradeoffs
- シグナルベースは Unix 前提（Windows は対象外のまま。ポータビリティ tenet 上は注記に留める）。
- drain 期限内に終わらない長寿命接続は打ち切られる。**再検討条件**: WebSocket / Upgrade（[[000043]] の deferred 序列）を導入するとき、長寿命接続の drain 設計（打ち切りか、既存接続のみ継続か）を見直す。
- reload 進行中の shutdown、連続 SIGHUP 等の競合はテストで押さえる必要がある（実装時の負担）。
