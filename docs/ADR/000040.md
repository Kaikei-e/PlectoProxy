# OTLP エクスポートを host 側 TelemetrySink 実装で先行し、wasi-otel guest 契約は M3 以降に据え置く

## Status
accepted

## Date
2026-07-02

## Affected Components
- extension plane / host runtime（`plecto-host`）— `TelemetrySink`（既存の同期 seam。実装は Noop/InMemory/Metrics/FanOut）に `OtlpBuffer`（有界キュー = sink の一実装）と手書き OTLP encoder を足す（`otlp.rs`）。host は runtime-free のまま（tokio/hyper は host に入れない）。`plecto:filter` 契約・フィルタから見える世界は変えない
- control plane（`plecto-control`）— `[observability]` セクションに `otlp_endpoint` を追加し、既存 sink の**横に** buffer を fan-in（呼び出し元の sink を置き換えない）
- fast path（`plecto-server`）— export pump（tokio タスク: tick drain → encode → OTLP/HTTP POST、retry / shutdown flush）と、per-transaction の request span（SERVER）送出を `proxy_core` に配線

## Context

[[000009]] は可観測性の契約に wasi-otel を据え、トレース文脈はホストが span state を管理して伝播する方針を定めた。Stage A（[[000029]]）でその大半は実体化している——per-filter span 生成、W3C `traceparent` の受信継続と upstream への伝播（`crates/server/src/proxy.rs` / `crates/host/src/observe.rs`）、`TelemetrySink` seam、手書き Prometheus exposition の `/metrics`、JSON access log。

しかし 2026-07-02 のギャップ分析で、**SpanExporter / OTLP ネットワーク送出 / wasi-otel guest 契約が named-deferred のまま**であることが確認された（`observe.rs` 冒頭に明記）。つまり span は生成されるが**どこにも送られない**。[[000029]] が「Stage A 着地済み」と記録している点と実体に乖離があり、本 ADR はその解消（送出の追加）と、据え置く範囲の明確化を行う。

外部環境（取得日 2026-07-02、参考）: OTel ネイティブ対応は 2025–26 に業界のテーブルステークスとなった（HAProxy 3.4 が `filter opentelemetry` を入れて OpenTracing を廃止、Traefik v3 はデフォルト OTel）。ただし本決定の駆動力は parity ではなく、「span を生成しているのに送出できない」という自分の役割上の不整合である。

## Decision

1. **host 側に OTLP exporter を `TelemetrySink` の一実装として追加する。** フィルタ契約（`plecto:filter`）は不変。sink の差し替えだけで OTLP 送出が有効になる。
2. **transport は OTLP/HTTP（protobuf）とする。** tonic / gRPC スタックを依存に持ち込まない。`/metrics` を手書き exposition で依存レスに保った判断（Stage A）と同じ論理で、必要な protobuf エンコードを自前で持つ。
3. **送出は tokio 側の非同期バッチとする。** 同期 seam（`TelemetrySink`）の背後でバッファし、ホット経路・filter chain 実行（blocking pool）に送出 IO を持ち込まない。実装で確定した形（2026-07-02、当初 projected）: host 側は有界キュー（`OtlpBuffer`、cap 2048 = OTel BSP 仕様デフォルト、溢れは incoming-drop + 計数 + 初回のみ warn）を `TelemetrySink` の一実装として持ち、fast path 側の pump タスクが 250 ms tick で空になるまで drain して batch ≤ 512 を POST する（純粋な 5 s tick は cap 2048 では 410 spans/s 超で恒常 drop になるため短縮。opentelemetry-rust 0.28+ / Envoy / nginx-otel はいずれも size-trigger 併用で、pull 型ではこの短 tick が等価）。retry は仕様の retryable set（429/502/503/504）のみ・最大 3 attempt・`Retry-After`（秒形式）尊重、`partial_success` は retry せず記録。graceful shutdown は有界 deadline（3 s）で flush-until-empty し、残余は drop 計数（仕様の「Shutdown は ForceFlush を含み、有界時間で abort してよい」に準拠）。損失は `/metrics` の `plecto_otlp_dropped_spans_total` で観測できる。
4. **wasi-otel guest 契約（フィルタ自身が span/metric を発行する口）は M3 以降に据え置く。** [[000009]] の「契約は wasi-otel」という方向は維持したまま、host 発の span 送出だけを先行させる。guest 契約は `wasi:http` 収斂（[[000020]] / [[000021]]）と同じ地層の変更であり、束ねると過大になる。
5. **request span（SERVER）を送出に含め、request span id は常にローカル採番とする**（実装時判断、2026-07-02）。従来 `RequestTrace` は継続トレースで inbound `traceparent` の span id を request span id として流用しており、そのまま送出すると (a) 他プロセス所有の id で span を発行して衝突する、(b) filter span の親が exporter に存在せず孤児になる。inbound span を remote parent として別保持し、SERVER kind の request span を per-transaction で 1 本送出することで解消した。副作用として upstream へ伝播する `traceparent` は Plecto の request span id を持ち、トレース木で filter span と upstream が Plecto の SERVER span の下にネストする（標準的なプロキシの形。[[000009]] の「host が span state を管理し伝播する」の具体化）。

検討した代替案と却下理由:
- **opentelemetry-rust SDK の採用**——公式実装で protobuf 追随の保守が不要になるが、依存グラフが大きく API がまだ変動する。`TelemetrySink` seam を保つので、後から SDK ベースの sink 実装へ差し替え可能。現段階では却下（再検討条件あり）。
- **wasi-otel guest 契約と同時に導入**——契約の一貫性は上がるが、M3（ホスト async 化 + `wasi:http` 収斂）と結合して一枚岩のスライスになる。送出の運用価値は今すぐ得られるので分離する。却下。
- **OTLP/gRPC**——エコシステムの既定だが tonic 依存が入る。OTLP/HTTP は Collector 側で普遍的にサポートされており、機能差は本用途で無い。却下。

本決定は [[000009]] の具体化であり、native 採用基準（[[000029]]、observability = 横断的データプレーン関心事 (i)）に適合する。

## Consequences

### Pros
- 生成済みの span が実際に OTLP Collector へ届き、Stage A の可観測性が閉じる。[[000029]] の記録と実体の乖離が解消される。
- フィルタ契約・既存 sink 実装に手を入れないため、変更の爆発半径が `observe.rs` + manifest 配線に収まる。
- 依存レス方針が保たれ、サプライチェーン面（`deny.toml`）の審査対象が増えない。

### Cons / Tradeoffs
- 手書き OTLP encoder の保守コスト（OTLP protobuf 定義への追随）。実装時の確認（2026-07-02）: opentelemetry-proto の trace/common/resource proto は Stable 凍結（フィールド番号・型は不変保証、v1.10.0 時点）なので追随リスクは additive の見落としに限られる。テスト oracle は公式生成型（`opentelemetry-proto` を **dev-dependency** に採用、`gen-tonic-messages` = prost のみで tonic は入らない。出荷バイナリの依存面は不変）。**再検討条件**: encoder 保守が負担化、または OTLP の仕様変更に追随しきれなくなったら、sink 実装を opentelemetry-rust SDK ベースへ差し替える。
- 今回送出するのは traces のみ。metrics は Prometheus scrape、logs は tracing のままで、これらの OTLP 化は未決（必要になった時に個別判断）。
- sampling は W3C `traceparent` の sampled flag を尊重する（sampled=0 の transaction は span を構築せず送出しない）ことまで実装で確定。head-based 比率サンプリング等の自発的戦略は引き続き未決。全量送出はトラフィック次第で Collector 側コストになる。
- 送出先は `http://` のみ（collector はローカル agent / サイドカーが標準形）。TLS 送出（`https://` endpoint）は未対応で、必要になった時に upstream TLS スタックと合わせて判断。
- `otlp_endpoint` は `admin_addr` と同じく起動時捕捉で、変更には restart が要る（reload は filter set のみを swap する）。
