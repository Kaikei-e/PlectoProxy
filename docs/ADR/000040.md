# OTLP エクスポートを host 側 TelemetrySink 実装で先行し、wasi-otel guest 契約は M3 以降に据え置く

## Status
accepted

## Date
2026-07-02

## Affected Components
- extension plane / host runtime（`plecto-host`）— `TelemetrySink`（既存の同期 seam。実装は Noop/InMemory/Metrics/FanOut）に OTLP exporter を足す。`plecto:filter` 契約・フィルタから見える世界は変えない
- fast path（`plecto-server`）— `[observability]` セクションに OTLP 送出先の設定を追加し、`main.rs` で sink を配線

## Context

[[000009]] は可観測性の契約に wasi-otel を据え、トレース文脈はホストが span state を管理して伝播する方針を定めた。Stage A（[[000029]]）でその大半は実体化している——per-filter span 生成、W3C `traceparent` の受信継続と upstream への伝播（`crates/server/src/proxy.rs` / `crates/host/src/observe.rs`）、`TelemetrySink` seam、手書き Prometheus exposition の `/metrics`、JSON access log。

しかし 2026-07-02 のギャップ分析で、**SpanExporter / OTLP ネットワーク送出 / wasi-otel guest 契約が named-deferred のまま**であることが確認された（`observe.rs` 冒頭に明記）。つまり span は生成されるが**どこにも送られない**。[[000029]] が「Stage A 着地済み」と記録している点と実体に乖離があり、本 ADR はその解消（送出の追加）と、据え置く範囲の明確化を行う。

外部環境（取得日 2026-07-02、参考）: OTel ネイティブ対応は 2025–26 に業界のテーブルステークスとなった（HAProxy 3.4 が `filter opentelemetry` を入れて OpenTracing を廃止、Traefik v3 はデフォルト OTel）。ただし本決定の駆動力は parity ではなく、「span を生成しているのに送出できない」という自分の役割上の不整合である。

## Decision

1. **host 側に OTLP exporter を `TelemetrySink` の一実装として追加する。** フィルタ契約（`plecto:filter`）は不変。sink の差し替えだけで OTLP 送出が有効になる。
2. **transport は OTLP/HTTP（protobuf）とする。** tonic / gRPC スタックを依存に持ち込まない。`/metrics` を手書き exposition で依存レスに保った判断（Stage A）と同じ論理で、必要な protobuf エンコードを自前で持つ。
3. **送出は tokio 側の非同期バッチとする。** 同期 seam（`TelemetrySink`）の背後でバッファし、ホット経路・filter chain 実行（blocking pool）に送出 IO を持ち込まない。バッチ間隔・キュー上限・溢れ時の drop 方針（データプレーンを守る方向で drop）は実装時に確定する（projected）。
4. **wasi-otel guest 契約（フィルタ自身が span/metric を発行する口）は M3 以降に据え置く。** [[000009]] の「契約は wasi-otel」という方向は維持したまま、host 発の span 送出だけを先行させる。guest 契約は `wasi:http` 収斂（[[000020]] / [[000021]]）と同じ地層の変更であり、束ねると過大になる。

検討した代替案と却下理由:
- **opentelemetry-rust SDK の採用**——公式実装で protobuf 追随の保守が不要になるが、依存グラフが大きく API がまだ変動する。`TelemetrySink` seam を保つので、後から SDK ベースの sink 実装へ差し替え可能。現段階では却下（再検討条件あり）。
- **wasi-otel guest 契約と同時に導入**——契約の一貫性は上がるが、M3（ホスト async 化 + `wasi:http` 収斂）と結合して一枚岩のスライスになる。送出の運用価値は今すぐ得られるので分離する。却下。
- **OTLP/gRPC**——エコシステムの既定だが tonic 依存が入る。OTLP/HTTP は Collector 側で普遍的にサポートされており、機能差は本用途で無い。却下。

本決定は [[000009]] の具体化であり、native 採用基準（[[000029]]、observability = 横断的データプレーン関心事 (i)）に適合する。

## Consequences

### Pros
- 生成済みの span が実際に OTLP Collector へ届き、Stage A の可観測性が閉じる。[[000029]] の記録と実体の乖離が解消される。
- フィルタ契約・既存 sink 実装に手を入れないため、変更の爆発半径が `observe.rs` + manifest 配線に収まる。
- 依存レス方針が保たれ、サプライチェーン面（`deny.toml`）の審査対象が増えない。

### Cons / Tradeoffs
- 手書き OTLP encoder の保守コスト（OTLP protobuf 定義への追随）。**再検討条件**: encoder 保守が負担化、または OTLP の仕様変更に追随しきれなくなったら、sink 実装を opentelemetry-rust SDK ベースへ差し替える。
- 今回送出するのは traces のみ。metrics は Prometheus scrape、logs は tracing のままで、これらの OTLP 化は未決（必要になった時に個別判断）。
- sampling 戦略（head-based の比率等）は未決のまま実装時判断（projected）。全量送出はトラフィック次第で Collector 側コストになる。
