# host state backend を単一 [state] 設定に束ね、redb を production 経路へ配線する

## Status
accepted

## Date
2026-07-02

## Affected Components
- control plane（`plecto-control`）— manifest スキーマに `[state]` を追加（`deny_unknown_fields` 維持）。reload 時の backend 変更の扱いを定義
- extension plane / host runtime（`plecto-host`）— `host-kv` / `host-counter` / `host-ratelimit` の 3 capability が共有する `KvBackend` を manifest から選択可能にする

## Context

[[000004]] は「フィルタはステートレス、状態はホスト KV（redb）に置く」を定めた。`KvBackend` 抽象は実装済みで、`MemoryBackend` と `RedbBackend` の両方がテスト込みで存在し、`host-kv` / `host-counter` / `host-ratelimit`（[[000026]]）の 3 capability を支えている（`crates/host/src/backend.rs`）。

しかし 2026-07-02 のギャップ分析で、**redb が production 経路から到達不可**であることが確認された。`Host::new` は `MemoryBackend` 固定（`crates/host/src/lib.rs`）で、manifest に backend を選ぶフィールドが無く、`plecto-control` の `build_host_and_store` も `Host::new` を呼ぶ。`RedbBackend` に到達できるのはテスト用の `with_backend` 注入だけである。つまり [[000004]] の「状態は redb」は、出荷経路では一度も成立していない。restart / reload のたびにフィルタの KV・counter・rate limit 窓が消えるのは、rate limit については fail-open 方向の挙動でもある。

なお native front-door rate limit（[[000033]]）の固定サイズテーブルは `KvBackend` と別実装の別関心事であり、本 ADR の対象外。

## Decision

manifest に **ホスト単位の `[state]` セクションを 1 つ**追加する:

- `backend = "memory"`（デフォルト）または `"redb"`。`redb` 選択時は `path` 必須。
- `host-kv` / `host-counter` / `host-ratelimit` の 3 capability は**同一 backend を共有**する。
- reload で `[state]` の変更（backend 種別・path）は認めない（構築時固定。trust root と同じ扱い）。変更にはプロセス再起動を要する。

検討した代替案と却下理由:
- **capability 別に backend を選択可能にする**（kv は redb、counter/ratelimit は memory 等）——柔軟だが設定面が 3 倍になり、運用の単純さ（コア原則）に反する。要求が実証されてから `[state]` を細分化する方が、最初から複雑にするより良い。却下。
- **kv のみ durable にする**——[[000004]] の文言には最も忠実だが、counter / rate limit 窓の restart 越え連続性も運用上の意味を持つ（restart で rate limit がリセットされるのは fail-open 方向）。単一ノブの方が挙動の説明も単純。却下。
- **redb をデフォルトにする**——[[000004]] の宣言に沿うが、ファイルパスが必須になり zero-config 起動（manifest だけで立つ）が壊れる。memory デフォルト + 明示 opt-in が単純で、挙動変更のサプライズも無い。却下。

本決定は [[000004]] の production 経路での成立（実体化）であり、新しい Fork は増やさない。「フィルタはステートレス・状態はホスト側」という Tenet の配線を閉じるだけである。

## Consequences

### Pros
- [[000004]] が出荷物として成立し、`with_backend`（テスト専用経路）と production の乖離が消える。
- rate limit（[[000026]]）・counter の窓が restart / reload を越えて連続し、fail-closed 方向に倒れる。
- 設定は 1 セクション・2 値で、運用者の認知負荷が最小。

### Cons / Tradeoffs
- 3 capability が単一 backend を共有するため、redb 選択時は capability 間で性能が干渉し得る（fsync・ロック競合）。**再検討条件**: capability 間で durability / 性能要求が割れる実例が出たら `[state]` を細分化する。
- `memory` 選択時は従来どおり restart / reload で状態が消える（現状と同じだが、設定として明示化される）。
- redb ファイルの権限・バックアップ・ディスク容量は利用者責務（セルフホスト前提として文書化で対応）。
- 分散 opt-in（[[000008]]）時にはノード横断の状態レイヤを別途設計する必要があり、`[state]` はそのときのローカル層になる（**再検討条件**の一つ）。
