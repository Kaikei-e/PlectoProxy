# upstream への転送を TLS+ALPN で再暗号化し、upstream HTTP/2 と gRPC パススルーを成立させる

## Status
accepted

## Date
2026-07-02

## Affected Components
- fast path（`plecto-server`）— upstream 接続を平文 HTTP/1.1 固定から「per-upstream で TLS+ALPN（h2 / http/1.1）」へ拡張。h2 経路では `TE: trailers` を透過し trailers を転送
- control plane（`plecto-control`）— `[[upstream]]` の manifest スキーマ拡張（TLS 有効化、custom CA 指定）

## Context

2026-07-02 のギャップ分析で、upstream への転送が**平文 HTTP/1.1 固定**であることを確認した——`crates/server/src/upstream_client.rs` は `Client<HttpConnector>`、health probe も HTTP/1.1（`crates/server/src/health.rs` に "upstream TLS is deferred" と明記）。さらに `TE: trailers` を hop-by-hop としてストリップし（`crates/server/src/headers.rs`）、upstream が h1 固定であることと合わさって、**trailers ベースの gRPC が end-to-end で成立しない**。

これは二つの理由で座視できない。第一に、コア原則の最上位は「安全」であり、downstream は rustls で終端する（[[000014]]）のに内側の脚が平文のみというのは暗号境界の穴である。フィルタに貸す outbound HTTP（[[000036]]）には TLS + SSRF ガードがあるため、本体の転送だけが平文という非対称も説明がつかない。第二に、upstream h2 は CONTEXT-MAP が明記する既定の後続スライスであり、gRPC バックエンドを持つセルフホスト構成（本プロジェクトの主用途）で実需がある。

2026-07-02 のグリルで、TLS 再暗号化と upstream h2 を別スライスに割るか一体で入れるかを検討し、**一体化**を選んだ（下記代替案参照）。

## Decision

1. **per-upstream に TLS 設定を追加する。** 有効化した upstream へは rustls client で接続し、**ALPN で h2 / http/1.1 をネゴシエート**する。プロトコル選択は ALPN に委ね、manifest でプロトコルを直接固定しない。
2. **h2c（平文 HTTP/2）は upstream 方向も不採用。** downstream で h2c を採らなかった [[000015]] と対称の判断。HTTP/2 は常に TLS+ALPN の上でだけ話す。
3. **gRPC パススルー**: TLS+ALPN で h2 が成立した経路では `TE: trailers` をストリップせず透過し、response trailers を転送する。h1 upstream への gRPC は非対応のまま（成立し得ないものを黙って通さない）。
4. **server 証明書検証は常時有効。** webpki roots に加え、per-upstream で custom CA（内部 CA / 自己署名運用）を指定できる。**検証を無効化する insecure オプションは設けない**（fail-closed）。
5. **health probe は upstream の scheme に従う**（TLS upstream には TLS で probe）。
6. **upstream mTLS（client 証明書の提示）は本スライスの対象外。** downstream mTLS と併せて後続とする（deferred 序列は [[000043]]）。HTTP/3 upstream も対象外。

検討した代替案と却下理由:
- **h2c prior-knowledge を per-upstream オプションで許す**——内部ネットワークの gRPC backend には簡便だが、[[000015]] の「h2c は採らない」との対称性が崩れ、平文 h2 framing という新しい攻撃面を持ち込む。内部でも TLS を張るのが安全 tenet の順序に忠実。却下（再検討条件あり）。
- **TLS 先行、h2 は別スライス**——スライスは小さくなるが、駆動要件が gRPC である以上 TLS 単独では役割が閉じない。実装面でも ALPN は rustls `ClientConfig` の設定項目で、TLS と h2 は同じ層の変更。一体化の方が総工数が小さい。却下。
- **証明書検証スキップ（insecure / verify=false）オプション**——自己署名運用の導入摩擦は下がるが、fail-closed 原則に反し、「一時的に off」が恒久化する定番の穴になる。custom CA 指定で同じ運用要件を安全に満たせる。却下。

本決定は native 採用基準（[[000029]]、upstream 接続管理 = 横断的データプレーン関心事 (i)）に適合し、コア原則の最上位（安全）を具体化する。

## Consequences

### Pros
- edge から upstream まで暗号化された経路が構成でき、内側の脚が平文のみという穴が塞がる。[[000036]]（filter outbound の TLS）との非対称も解消。
- gRPC バックエンドが end-to-end（h2 + trailers）で成立し、API gateway としての適用範囲が実質的に広がる。
- h2c 不採用が両方向で対称になり、「HTTP/2 は TLS+ALPN の上でだけ」という不変条件が説明しやすい。

### Cons / Tradeoffs
- connection pool が (scheme, ネゴシエート結果) で分かれ、接続管理が複雑化する。
- h2 の multiplexing 導入により、per-try timeout（[[000031]]）・retry（[[000023]] / [[000030]]）・circuit breaker の in-flight cap（[[000028]]）・outlier detection（[[000032]]）の計上単位が「接続」ではなく「stream」であることをテストで再確認する必要がある。
- rustls client 側の設定面（roots / custom CA / ALPN）が増える。custom CA の PEM 管理は利用者責務。
- **再検討条件**: (a) 内部ネットワークで TLS 終端コストが実測で問題化し h2c 要求が支配的になったとき、h2c prior-knowledge オプションを再検討する；(b) upstream mTLS の実需が出たとき（[[000043]] の deferred 序列で downstream mTLS と併せて扱う）；(c) HTTP/3 upstream が水準入りしたとき（調査時点 2026-07-02 では業界でも実験的）。
