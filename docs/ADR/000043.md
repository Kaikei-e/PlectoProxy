# response caching と AI/LLM gateway の native 化を declined とし、残余ギャップに deferred 序列を固定する

## Status
accepted

## Date
2026-07-02

## Affected Components
- roadmap / governance — 役割駆動（[[000029]]）の運用として declined / deferred の区別を記録する。コード変更なし
- fast path（`plecto-server`）— deferred 各スライス（圧縮・mTLS・WebSocket）は着手時に個別 ADR で確定する（本 ADR は序列のみ固定）

## Context

[[000029]] は成熟の駆動軸を役割駆動に固定し、その代償として「何を declined にしたかを都度言語化する負担」を明示的に受け入れた。2026-07-02 のギャップ分析（コード実体インベントリ・業界機能水準・WASM エコシステムの三方向調査。外部事実の取得日はいずれも 2026-07-02）で、判定に足る材料が揃った:

- **response caching**: 業界でもテーブルステークスではない。前段 CDN / Varnish への外出しが設計として許容されており、Traefik はコア OSS に持たず、Envoy / HAProxy も部分的。
- **AI/LLM gateway**（token-based rate limiting / provider routing / guardrails）: 2025 年の業界最大の投資先だが、Envoy（Envoy AI Gateway）・Traefik（Hub）・nginx とも**コアのプロキシは触らず別レイヤー製品**で対応している。機能の形状は「このリクエスト（のトークン消費・行き先）をどう判断するか」という user-policy そのもの。
- **ACME**: 2025 年に nginx と HAProxy が相次いで内蔵し、ネイティブ ACME はテーブルステークス化した（背景に CA/B Forum の証明書短命化）。一方 Plecto は静的 PEM + 証明書ホットリロード（[[000014]]）があり、外部 ACME クライアント + SIGHUP reload（[[000039]] が前提）で運用が成立する。
- **圧縮（gzip/br/zstd）・downstream mTLS・WebSocket/Upgrade**: リバースプロキシ水準の残余。いずれも横断的データプレーン関心事（[[000029]] 採用基準 (i)）で native 適格だが、今回の優先度（運用ギャップ [[000039]]/[[000040]]/[[000041]]、暗号境界 [[000042]]）より後ろに置く。

## Decision

1. **response caching = native declined。** 状態を持つ横断機能だが、運用の単純さ（キャッシュ無効化・整合性の運用負担）および body-opaque 方針（[[000038]] のゼロコピー素通し）と衝突する。セルフホスト構成では前段の CDN / Varnish に外出しする。roadmap の非目標として記録する。
2. **AI/LLM gateway = native declined、reference フィルタの題材として Stage F（[[000029]]）へ。** token rate limit は `host-ratelimit` の `(key, cost)` 契約（[[000026]]）で cost=トークン数として既に表現可能であり、provider routing / failover も filter の `decision` variant で表現できる——つまり Fork 6 の user-policy 形状の証左。native には実装しない。
3. **deferred 序列を固定する: 圧縮（gzip/br/zstd）→ downstream mTLS → WebSocket/Upgrade。** 各スライスは着手時に個別 ADR で確定する（[[000029]] の運用どおり）。序列は優先順位であり実装内容の先取りではない。upstream mTLS は downstream mTLS のスライスで併せて扱う（[[000042]] から送られた宿題）。
4. **ACME = 完全後回し（序列にも載せない）。** テーブルステークス化は認識した上で、現段階は静的 PEM + ホットリロード + 外部 ACME クライアント（certbot / lego 等）で足りる。native 化は横断的関心事として適格だが、優先度を透過性・暗号境界より下に置く。
5. **OIDC RP フロー / developer portal は未判定のまま残す。** declined とも deferred とも記録しない（判定材料不足。JWT 検証がフィルタで表現可能なことだけは Fork 6 から自明）。

検討した代替案と却下理由:
- **caching を deferred に留める**——可逆性は高いが、判定材料が揃った今 declined を記録する方が [[000029]] の governance（declined の言語化）に忠実。再検討条件を残すので declined でも可逆。却下。
- **AI gateway を native rate limiting（[[000033]]）の拡張として扱う**——「グローバルカウンタ」（採用基準 (ii)）に見えるが、トークン計数はプロバイダ応答の usage 等リクエストのセマンティクスに依存する per-request decision であり、汎用の窓/バケットとは形が違う。Fork 6 に従い WASM 側。却下。
- **ACME を deferred 序列に載せる**——セルフホスト性への直撃度は高いが、外部クライアントで代替可能な機能と、代替のない透過性・暗号境界の穴（[[000042]]・WebSocket）では後者が先。ユーザ判断として完全後回しを明示する。却下。

本決定は [[000029]]（役割駆動・Fork 6・declined の言語化）の運用そのものであり、native の採用基準に新しい例外を作らない。

## Consequences

### Pros
- 「持たない機能」が理由付きで記録され、以後の議論（issue / 提案）を本 ADR への参照で短縮できる——[[000029]] が予告した負担の前払い。
- AI/LLM gateway を Stage F の reference フィルタ題材に接続したことで、拡張プレーンの実証テーマ（`host-ratelimit` の cost 次元・outbound HTTP [[000036]] の実用例）が具体化した。
- deferred 序列が固定され、次スライス群（[[000039]]〜[[000042]] 完了後）の着手判断が不要になる。

### Cons / Tradeoffs
- declined は対外的には「無い機能」として効く（役割駆動の既知の代償）。特に caching 不在は、前段 CDN を置けない単体セルフホスト構成では不便が残る。
- ACME 完全後回しは、証明書短命化時代のセルフホスト運用に外部クライアント併用という摩擦を残す。
- **再検討条件**: (a) caching——前段 CDN を置けない単体セルフホスト需要が支配的になり、外出し前提が破綻したとき；(b) AI/LLM gateway——reference フィルタで表現しきれない横断関心事（例: ノード内グローバルな token 計数の一元化）が実証され、採用基準 (ii) に該当する形が特定できたとき；(c) ACME——証明書の外部クライアント運用がセルフホスト採用の実測ボトルネックになったとき、deferred 序列へ昇格する；(d) OIDC RP / developer portal——判定材料（実需・フィルタでの実装実験）が揃ったとき個別 ADR で確定する。
