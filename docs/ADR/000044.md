# upstream の hostname アドレスを interval で定期再解決し、各レコードを LB エンドポイントに展開する

## Status
accepted

## Date
2026-07-03

## Affected Components
- control plane（`plecto-control`）— `[[upstream]]` の manifest スキーマに `resolve_interval_ms` を追加。`UpstreamGroup` の endpoint 集合（instances）と compiled LB state（Maglev table は instances を index する）を一つの `Endpoints` にまとめ、`ArcSwap<Endpoints>` としてグループに載せる。`update_endpoints(resolved)` が health を保存しつつ atomically に差し替える
- fast path（`plecto-server`）— hostname を定期解決する supervisor（`dns.rs`）。health supervisor と同型（Control からグループを毎ループ読み直す＝reload 追従、各グループを自タスクで refresh）

## Context

2026-07-02 の moka-1 ドッグフーディング（field report §2）で、「upstream アドレスに compose のサービス名がそのまま書ける」ことが良い点として挙がった一方、その仕様には穴がある。upstream は静的 `host:port` リストで、hyper の `HttpConnector` は**接続時に毎回解決する**が、health probe は登録時アドレスに対して行われ、round-robin/least-request/Maglev の LB もエンドポイントを静的リストとして扱う。

このため、compose のサービス名や k8s headless Service を upstream に書いた構成では、**コンテナ再作成で IP が変わった瞬間に破綻する**: 古い IP を掴んだ接続はエラーになり、health probe も古いアドレスを叩き続ける。DNS が複数レコード（headless Service の各 Pod）を返しても、Plecto はそのうち一つの名前を一エンドポイントとしてしか扱わないので、レコード単位の負荷分散も health も効かない。これは「コンテナ環境で使えるかどうかを分ける」機能であり、2026-07-03 のギャップ分析（§4.2）が deferred 序列への追加を提案した点。

業界の確立技術は二つ（いずれも一次ソースを 2026-07-03 に確認、確定事実）:
- **nginx `resolver` + `server ... resolve`**: hostname が複数 IP に解決されると各 IP が LB peer になり、再解決で peer 集合を更新する。TTL 準拠（`valid=` で固定 interval 上書き可）。open-source では 1.27.3 以降。
- **Envoy STRICT_DNS**: 各解決 IP がクラスタの明示ホスト。消えた IP は drain、成功して 0 件なら空集合（本 ADR は後述のとおり別方針）、失敗は last-known-good を保持し `dns_failure_refresh_rate` で再試行。

この機能は[[000029]] の native 採用基準 (i)（横断的データプレーン関心事）に該当する。upstream の接続管理は全リクエストが通る経路であり、フィルタ（user-policy）ではなく fast path が持つのが妥当。実装アルゴリズムは特定 OSS の写しではなく、上記二者に共通する**一般的な periodic-DNS endpoint-discovery 技術**を採る（プロジェクト方針: 技術は引用し、実装は一般アルゴリズムで書く）。

## Decision

1. **`[[upstream]]` に `resolve_interval_ms` を追加する（default `0` = 無効）。** `0` は従来どおり（hostname は接続ごとに解決されるが endpoint 集合は宣言のまま固定）。非ゼロなら、その interval で hostname アドレスを再解決し、endpoint 集合を更新する。
2. **各解決レコードを LB エンドポイントに展開する（STRICT_DNS 型）。** hostname が返す各 A/AAAA レコードが、それぞれ自分の health 状態を持つ独立の instance になる。IP リテラルは DNS を経由せず素通し（health 状態が再解決で乱されない）。
3. **endpoint 集合 + compiled LB state を単一の `ArcSwap<Endpoints>` に統合する。** Maglev の lookup table は instances を index するので、両者は必ず一緒に swap しなければならない。`Endpoints`（instances + LbState）を一単位にし、グループに `ArcSwap` として載せる。route は従来どおり `Arc<UpstreamGroup>` を握り続け、**refresher（と、それだけ）が中身を in-place で差し替える**。pick 経路は per-pick で一度 `load()` するので、並行 swap が instance list と Maglev table を desync させることはない。
4. **エンドポイント差し替えの health セマンティクスは reconcile と対称にする（[[000017]]）。** 生存レコード（`address+weight` 一致）は instance `Arc` を保持（health と in-flight を維持）、消えたレコードは drop（in-flight は clone した `Arc` 上で完走）、新規レコードは pessimistic スタート（rotation 入り前に probe 通過が必要）。
5. **失敗解決は last-known-good を保持する（fail-closed だが self-DoS しない）。** 解決が失敗（または成功して空）したら、その hostname の直近成功展開を維持する——flaky な resolver が稼働中の upstream を落とさない。一度も解決できていない hostname は、その名前自身を endpoint として残す（hyper の接続時解決が効くので、起動時に DNS が死んでいても upstream を black-hole しない）。
6. **interval ベースにする（TTL 追従にしない）。** production resolver は `tokio::net::lookup_host`（getaddrinfo）。getaddrinfo は TTL を運ばないが `/etc/hosts` と nsswitch を尊重する——セルフホスト構成に馴染む選択。operator は DNS TTL 以下の interval を選ぶ。
7. **reload は resolving group の現在エンドポイントを丸ごと引き継ぐ。** declared addresses / LB config が不変な resolving group は、reconcile で解決済み集合（IP + health）をそのまま carry する。さもないと reload のたびに解決集合を捨てて次の refresh + probe まで pessimistic 窓に戻ってしまう。

検討した代替案と却下理由:
- **成功して 0 件なら endpoint 集合を空にする（Envoy STRICT_DNS 準拠）**——仕様に忠実だが、Plecto の「fail-closed だが可用性を自己破壊しない」姿勢では、空応答で稼働中 upstream を全滅させるより last-known-good 保持が安全。意図的な逸脱として記録する。
- **hickory-resolver で TTL 追従する**——レコード TTL に正確に追従でき Envoy に近づくが、既存 tokio/rustls スタックに依存 100+ crate を持ち込む（2026-07-03 計測）。かつ getaddrinfo の `/etc/hosts`/nsswitch 尊重を失う。セルフホスト向けには interval + getaddrinfo が実利。却下（再検討条件あり）。
- **フル SD（xDS / Gateway API controller / k8s API 監視）**——最も強力だが、[[000008]] の single-node-first・静的 manifest 方針と衝突し、規模も桁違いに大きい。periodic DNS は「コンテナ環境需要の大半」をはるかに小さいスライスで満たす。却下。
- **endpoint を毎 refresh で無条件再構築する**——単純だが、生存レコードの health が毎 interval リセットされ pessimistic 窓が周期的に開く。`address+weight` 一致で instance を再利用し、変化が無ければ swap 自体をスキップ（idle tick は一度の atomic load と比較のみ）。却下。

本決定は[[000029]] の native 採用基準 (i) を具体化し、[[000017]] の health FSM と[[000035]] の LB を「動的エンドポイント集合の上で」動かす一般化である。[[000042]] の per-upstream TLS client は endpoint 集合とは独立（scheme はグループ単位）なので、この swap と干渉しない。

## Consequences

### Pros
- compose のサービス名・k8s headless Service を upstream に書いた構成が、コンテナ再作成に追従して動く——「読める OSS」から「コンテナで使える OSS」への一段。
- レコード単位の health / 負荷分散が効く（headless Service の各 Pod が独立エンドポイント）。
- `ArcSwap<Endpoints>` 化で、hot path（route が握る `Arc<UpstreamGroup>`）はロックに触れずに動的エンドポイントを読める。refresher だけが writer。
- 失敗時 last-known-good と一度も解決できない場合の hostname 残置で、DNS 障害が upstream を落とさない（fail-closed だが可用性優先）。

### Cons / Tradeoffs
- getaddrinfo は TTL を運ばないので、再解決の粒度は operator が選ぶ interval に依存する（レコード TTL より粗いと古い IP を最大 interval 分掴む）。
- 「成功して 0 件は last-known-good 保持」は Envoy STRICT_DNS からの意図的逸脱。upstream を意図的に 0 台にドレインしたい場合は address をリストから外す（reconcile 経路）必要がある。
- endpoint 集合 + LB state を一単位に再構成したことで、`UpstreamGroup` の内部構造が `ArcSwap` 一段深くなった（`endpoints()` スナップショット経由でアクセス）。
- **再検討条件**: (a) レコード TTL への正確な追従が実測で必要になり、interval 近似では鮮度不足が問題化したとき、hickory-resolver（依存増を受容）へ切り替える；(b) k8s / service-mesh 統合の実需が出て、DNS では表現できない endpoint メタデータ（weight/zone のサーバー主導配布）が要るとき、フル SD を[[000029]] の基準で個別 ADR 化する；(c) 「成功空応答で全ドレイン」の需要が出たとき、last-known-good 保持を manifest オプションで切り替え可能にする。
