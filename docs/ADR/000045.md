# data-plane の bind アドレスと Alt-Svc 広告ポートを manifest の [listen] に宣言する

## Status
accepted

## Date
2026-07-03

## Affected Components
- control plane（`plecto-control`）— `Manifest` に `[listen]`（`addr: Option<String>` / `advertised_port: Option<u16>`）を追加。`[observability]` と同じ扱い: `skip_serializing` で semantic content-hash から除外し、構築時に `Control` へ捕捉。reload はリスナーを再 bind しない
- fast path（`plecto-server`）— `main.rs` の bind 決定を三段優先に。`listener.rs` の Alt-Svc 生成で広告ポートを `advertised_port`（あれば）に差し替え

## Context

2026-07-02 の moka-1 ドッグフーディング（field report §3.2 / §3.4）で、コンテナ運用の摩擦が二点報告された:

- **§3.2 listen アドレスが位置引数のみ**: `plecto <manifest.toml> [listen_addr]`、default `127.0.0.1:8080`。コンテナでは `0.0.0.0` bind がほぼ必須で、毎回引数で渡す（entrypoint スクリプトで吸収する）ことになる。「manifest が静的な単一ソース」という[[000008]] の設計思想なら listen アドレスも manifest に居場所があるはず、という指摘。
- **§3.4 Alt-Svc がリッスンポートを広告する**: `Alt-Svc: h3=":<port>"` はリッスンポート由来（[[000016]]）。コンテナ内部ポートと公開ポートが違う構成（例: 内部 8443 → 公開 443）だと h3 広告がズレて、HTTP/3 アップグレードが黙って失敗する。rootless で 443 を bind できないランタイムでは特に踏みやすい。

両者とも「manifest を単一の静的な設定ソースにする」という[[000008]] の方針の未達部分。listen アドレスと h3 広告ポートは、フィルタ/ルート identity（config version）とは無関係の**リスナー配線**であり、[[000039]] の SIGHUP reload では再 bind されない（リスナーは起動時に一度 bind する）——`[observability] admin_addr` と全く同じ性質。

## Decision

1. **`[listen]` セクションを追加する（`addr` / `advertised_port`、両方 optional）。** `addr` は data-plane の bind アドレス（例 `0.0.0.0:8443`）。`advertised_port` は Alt-Svc が h3 に広告するポート。
2. **bind の優先順位を「CLI 位置引数 > `[listen] addr` > loopback default（`127.0.0.1:8080`）」にする。** manifest を静的な単一ソースにしつつ、明示的な CLI 引数は operator の override として最優先で残す（後方互換）。
3. **Alt-Svc の広告ポートを `advertised_port` で上書きする。** 未設定なら従来どおり bind ポートを広告する（[[000016]]）。コンテナのポートマッピングで公開ポートが内部と違うとき、公開ポートを広告して h3 アップグレードのズレを塞ぐ。
4. **`[observability]` と同じ扱いにする: 構築時に捕捉、content-hash から除外、reload で再 bind しない。** listen 設定は起動時に一度だけリスナーを bind するので、reload はこれを触らない。`skip_serializing` で semantic content-hash から外し、`[listen]` の変更が config-version の変化としてカウントされないようにする（リスナー配線はフィルタ/ルート identity ではない）。
5. **宣言を manifest に集中させる（env var を増やさない）。** listen アドレスは env var（`PLECTO_LISTEN` 等）でも渡せるが、[[000008]] の「manifest が静的な単一ソース」に忠実に、宣言箇所を散逸させない。env は導入しない。

検討した代替案と却下理由:
- **env var（`PLECTO_LISTEN`）で渡す**——コンテナでは env が自然という声はあるが、宣言が manifest / CLI / env の三箇所に散る。単一ソース方針（[[000008]]）を優先し、宣言を manifest に集中。CLI 引数は既存の override として残すが env は増やさない。却下。
- **`advertised_port` ではなく advertised authority（host:port）まで上書き可能にする**——より一般的だが、現状の需要はポートのズレ（コンテナ port mapping）に尽きる。authority 全体の上書きは名前ベース仮想ホスティングの別議論を呼ぶので、必要になってから拡張する。今回はポートのみ。却下（YAGNI）。
- **`[listen]` も content-hash に含めて reload で再 bind する**——リスナーを動的に移せる利点はあるが、in-flight 接続の扱い・ポート衝突・[[000039]] の drain との相互作用が複雑化する。`[trust]`/`[state]`/`[observability]` と同じ「構築時固定・変更は restart」の一貫した契約を選ぶ。却下。

本決定は[[000008]]（静的 manifest = 単一設定ソース）を具体化し、[[000016]]（Alt-Svc 広告）と[[000039]]（reload は起動時配線を触らない）の既存契約に整合する。新しい設計原則は導入しない。

## Consequences

### Pros
- コンテナで `0.0.0.0` bind が manifest 一行で済む（entrypoint スクリプトの引数吸収が不要）——field report §3.2 の摩擦解消。
- 内部/公開ポートが違う構成でも h3 広告が公開ポートを指す——§3.4 の「黙って失敗する HTTP/3」を塞ぐ。
- `[observability]` と同一の「構築時固定」契約なので、reload の意味論に例外を持ち込まない。
- 宣言が manifest に集中し、設定の在処が散逸しない（単一ソース方針の徹底）。

### Cons / Tradeoffs
- listen アドレス変更に restart が要る（reload では効かない）。ただし `[trust]`/`[state]`/`[observability]` と同じ既知の契約で、驚きは無い。
- `advertised_port` はポートのみの上書きで、advertised authority 全体は変えられない（現状の需要には十分だが、将来の名前ベース構成では拡張が要るかもしれない）。
- **再検討条件**: (a) advertised authority（host 含む）の上書き需要が名前ベース仮想ホスティングと共に出たとき、`advertised_port` を authority 上書きに一般化する；(b) リスナーを無停止で移動したい実需が出たとき、`[listen]` を reloadable にする設計を個別に検討する（drain との相互作用込み）。
