# plecto validate サブコマンドと --version を追加し、artifact を読まず非破壊に manifest を静的検証する

## Status
accepted

## Date
2026-07-03

## Affected Components
- control plane（`plecto-control`）— `validate_manifest(&Manifest, base_dir)` と `validate_manifest_path(path)` を公開関数として追加（成功時に config version = semantic content-hash を返す）。`validate_routes` の引数を `&HashMap<String, Arc<LoadedFilter>>` から `&HashSet<&str>`（filter ID）に変更し、静的経路が WASM をロードせず参照検証を共有できるようにした
- fast path（`plecto-server`）— `main.rs` の引数分岐に `validate <manifest.toml>`（`nginx -t` 相当）と `--version` / `-V` を追加。どちらも serve せず即 exit

## Context

2026-07-02 の moka-1 ドッグフーディング（field report §2）は、manifest の strict 検証（`deny_unknown_fields`）と examples の充実で「ドキュメント外の挙動を推測で埋める場面がほぼ無かった」と評価した。一方、[[000039]] の SIGHUP reload は fail-closed（bad edit は現行 set を維持）で**実行時の防波堤**としては正しいが、**デプロイ前に CI で manifest を検証する手段が無い**。operator の第一日目体験（2026-07-03 ギャップ分析 §4.2）として、`nginx -t` 相当のコマンドが欠けている。

検証ロジック自体は `plecto-control` にライブラリとして既にある（`Manifest::from_toml` の strict parse、`validate_routes`、`Upstream::validate_lb`、`FilterEntry::validate`、TLS 証明書ロード等）。ただし既存の `Control::from_manifest_path` はこれらに加えて **`Host` を構築し OCI filter artifact を解決・ロード**するので、「manifest だけ検証したい」経路が無かった。CI では deploy dir（署名済み artifact 一式）がまだ存在しないことが普通なので、artifact ロードを含む検証は CI で回せない。

`--version` も無く（起動時に `config.config_version()`＝content-hash をログするのみ）、`CARGO_PKG_VERSION` の露出が無い。配布物としての最低限（2026-07-03 ギャップ分析 §4.1）。

## Decision

1. **`plecto validate <manifest.toml>` サブコマンドを追加する（`nginx -t` の形）。** strict parse に加え、「server が起動時に fail-closed する検証のうち、artifact を要さず何も破壊しないもの」を全て走らせ、成功時に config version を出力して exit 0、失敗時は原因を stderr に出して非ゼロ exit。
2. **`validate_manifest` が走らせる検証**: strict parse（呼び出し側が済ませる）、`[trust]` 鍵ファイルの読み込み、`[state]` の整合性（redb なのに path 無し等——ただし backend は開かない）、per-filter の metering / rate-limit レンジ、filter ID 重複、chain / route の参照、weighted split、`[[tls]]` 証明書ロード、`[[upstream]]` の LB config と `[upstream.tls]` CA ロード（[[000042]]）。
3. **意図的に含めない**（CI で manifest + 参照設定ファイルだけあれば回るように）: OCI artifact の解決と署名/SBOM ロードゲート（[[000006]]/[[000007]]。deploy dir が検証環境に無いことがある——起動時は依然強制）、`[state]` backend の open（検証が redb ファイルを**作ってはならない**）。
4. **`validate_routes` を filter ID 集合を取る形に変える。** 従来は loaded filter map（`Host` 構築が前提）を取っていた。artifact をロードしない静的経路がこの参照検証を共有できるよう、`&HashSet<&str>`（宣言された filter ID）を取る形に一般化する。`build_active` 側は `filters.keys()` から ID 集合を作って渡す。
5. **`--version` / `-V` を追加する。** `plecto <package version>` を出力して exit。
6. **serve プロセスとは出力を分ける。** validate / version は plain 出力（JSON 構造化ログではない）——operator/CI コマンドであり、serving プロセスではない。

検討した代替案と却下理由:
- **`validate` でも artifact を含めてフル検証する**——「起動時と完全に同じ検証」になる利点はあるが、CI で deploy dir（署名済み component 一式）が必要になり、最大の用途（軽量な pre-deploy チェック）が成立しない。静的検証と artifact ゲートを分け、後者は起動時に依然強制する。却下。
- **`validate` を `Control::from_manifest_path` の dry-run フラグで実現する**——コード重複は減るが、`[state]` backend の open（redb ファイル生成）や artifact ロードといった副作用を条件分岐で抑える形になり、「検証は非破壊」という不変条件がフラグの網羅性に依存して脆くなる。副作用ゼロの専用関数（`validate_manifest`）を別に切る方が、非破壊性がコード構造で保証される。却下。
- **`--version` を JSON ログで出す**——serving プロセスの観測性（[[000009]]）と揃うが、`--version` は人間/CI が叩く一行コマンド。plain 出力が素直。却下。

本決定は[[000039]]（実行時 fail-closed）を**デプロイ前の静的ゲート**で補完し、[[000008]] の strict manifest を CI に接続する。新しい設計原則は導入しない。「検証は非破壊・artifact 非依存」という不変条件を専用関数で構造的に保証する点が要点。

## Consequences

### Pros
- CI / pre-SIGHUP で manifest を検証できる——[[000039]] の実行時防波堤を、デプロイ前ゲートで補完。
- 静的検証が artifact も state ファイルも触らないので、deploy dir が未整備の CI 環境でそのまま回る。redb ファイルを生成しないことをテストで固定。
- `validate_manifest` が config version を返すので、operator が manifest の identity（[[000008]]）を検証時に確認できる。
- `--version` で配布物としての最低限を満たす（2026-07-03 ギャップ分析 §4.1）。
- `validate_routes` の filter-ID 化で、静的検証と `build_active` が同じ参照検証コードを共有（重複無し）。

### Cons / Tradeoffs
- `validate` は artifact の存在・署名・SBOM を検証しない。「manifest は妥当だが filter artifact が欠落/未署名」は起動時まで検出されない（意図的な線引き——起動時ゲートは依然 fail-closed）。この境界はドキュメントで明示する必要がある。
- 静的検証は `Control::from_manifest_path` とロジックが並走する（`validate_manifest` が別関数）。両者が乖離しないよう、共有できる検証（`validate_routes` 等）は関数を共有し、乖離しうる部分（artifact ロードの有無）だけを分ける設計にした。
- **再検討条件**: (a) 「artifact も含めたフル検証」の需要が CI で deploy dir を用意できる運用と共に出たとき、`validate --with-artifacts` のようなオプトインを足す；(b) manifest の JSON Schema 出力（`plecto schema`）の需要が出たとき——`deny_unknown_fields` + serde を基盤にエディタ補完/CI 検証/ドキュメント自動生成が生えるので、別スライスとして検討する（2026-07-03 ギャップ分析 §4.2）。
