# フィルタに課す供給網規律を自らの配布物に適用し、署名付きバイナリ・イメージで v0.1.0 を切る

## Status
accepted（成果物形への named runtime capability profile 追加は [[000079]] が amend。crates.io publish の却下判断は [[000090]] が撤回・amend。供給網規律そのものは不変）

## Date
2026-07-03

## Affected Components
- CI（`.github/workflows/release.yml`）— `v*` タグ push で起動する draft release ワークフロー。バイナリ（x86_64 / aarch64、native runner）と multi-arch コンテナイメージを署名付きで生成
- 配布物（`Dockerfile` / `CHANGELOG.md`）— canonical な reference build（moka-1 field report §3.1）と、バージョニング方針を内包した changelog
- Rust workspace — `version = "0.1.0"`（workspace 共有）を初回リリースとしてタグ付けする

## Context

2026-07-03 のギャップ分析（docs/servey/plecto-gap-analysis-2026-07-03.md §4）が、機能水準の三方向調査（[[000043]]）に欠けていた第三の軸——**採用可能性（どう手に入り、どう運用に載るか）**——を指摘した。検証時点の実態: タグ 0・リリース 0・Dockerfile なし・crates.io 未公開（`publish = false`）・CHANGELOG なし。つまり Plecto を試す方法が `git clone && cargo build` しかなく、compose / k8s のモダンスタックには物理的に参加できない。43 本の ADR と全緑 CI に対して、これは機能ではなく配布の欠落である。

同時に、Plecto はフィルタ配布に cosign 署名 + SBOM の verify-then-load を**要求する**側（[[000006]] / [[000007]]）でありながら、自らのバイナリは無署名という非対称を抱えていた。「フィルタに要求する供給網の規律を、自分が最初に満たすプロキシ」への転換は、セキュリティモデルの一貫性そのもの。

moka-1 ドッグフーディング（field report §3.1）も「per-user Dockerfile の乱立を防ぐ canonical Dockerfile」を要求しており、[[000045]] の `[listen]` 宣言はコンテナ実行を前提に着地済み。受け皿だけが欠けていた。

## Decision

1. **`v*` タグ push で起動する release ワークフローを置き、成果物は必ず draft release に載せる。** 公開（publish）は人間の明示操作として残す——リリースは自動化するが、公開判断は自動化しない。
2. **バイナリは cargo-auditable でビルドする**（依存リストをバイナリ自体に埋め込み、syft ≥ 1.15 がソースツリーなしで読める）。arch ごとに native runner（x86_64 / aarch64）でビルドし、クロスコンパイル toolchain を信頼対象に加えない。tar.gz + sha256 + cosign keyless（Fulcio OIDC）署名 bundle + SPDX SBOM を成果物にする。
3. **コンテナイメージは multi-arch（amd64/arm64）で GHCR に push し、BuildKit の provenance（mode=max）+ SBOM attestation をイメージ manifest に載せ、cosign keyless で digest に署名する。** tag は mutable なので署名対象にしない——署名するのは常に digest。
4. **`Dockerfile` は multi-stage の reference build とする。** build stage は pinned Rust toolchain + `--locked`（committed Cargo.lock が供給網ピン）、runtime は `distroless/cc:nonroot`（shell も package manager もない最小 CVE 表面）。設定は `[listen]` を含め manifest 一枚（[[000045]]）で、ディレクトリごと mount する（SIGHUP reload が editor の inode swap を生き延びる、field report §3.5）。
5. **CHANGELOG は Keep a Changelog 1.1.0 + SemVer とし、pre-1.0 方針を明文化する**: `0.x` の **minor** bump は manifest スキーマ / CLI の破壊的変更を含みうる（必ず Changed/Removed + migration note）。patch は常に安全。WIT 契約は `plecto:filter@<version>` として独立にバージョンし、ホストは出荷済み契約バージョンのロードを維持する（プロキシ更新がデプロイ済みフィルタを黙って壊さない）。
6. **v0.1.0 を初回タグとして切る。** 43 ADR・CI 全緑・E2E 網羅の現状は `0.1.0` を名乗る資格として十分（ギャップ分析 §4.1）。タグ 0 の状態は「まだ触るな」というシグナルとして機能してしまう。

検討した代替案と却下理由:

- **cargo-dist で release ワークフローを生成する**——installer スクリプトまで含む一括生成は魅力だが、生成 YAML への追随が新たな依存になり、既存 `ci.yml` の規律（全 action の SHA ピン留め）や cosign/SBOM ステップの完全制御と噛み合わない。2 ジョブの手書きで足りる規模。却下（成果物の種類が増えて手書きが辛くなったら再検討）。
- **musl 静的リンク + `distroless/static`**——イメージはさらに小さくなるが、sigstore 検証系が引く aws-lc-sys の musl クロスビルドが toolchain-heavy（cmake/clang 依存）。`distroless/cc`（glibc + libgcc + CA 証明書のみ）で CVE 表面は十分小さい。却下（aws-lc-sys が依存から消えたら再検討、Dockerfile にも注記済み）。
- **crates.io への publish**——`publish = false` を維持する。ライブラリ API（plecto-control / plecto-host の公開面）はまだ安定を約束できず、配布はバイナリ / イメージ先行が実態に合う。却下（M3 GA でライブラリ面を安定化するときに再検討）。
- **鍵ベースの cosign 署名**——管理・漏洩リスクのある長期鍵を持たない。keyless（Fulcio OIDC + Rekor 透明性ログ）は GitHub Actions の workflow identity に検証を束ね、「何を信頼するか」が verify コマンドに明示される。フィルタ検証側（[[000006]]）と同じ姿勢。

本決定は[[000043]] のギャップ分析を配布軸へ拡張し、[[000006]] の供給網規律を自家適用するもの。コア原則の「セルフホスト性 × 運用の単純さ」を、機能ではなく配布で具体化する。

## Consequences

### Pros
- `docker pull ghcr.io/kaikei-e/plecto` が存在する——「読める OSS」から「使われうる OSS」への一段（ギャップ分析 §6 の結論）。
- フィルタに要求する規律（cosign + SBOM）を自らの全成果物が満たす。検証コマンドが release notes / workflow コメントに明示され、供給網の信頼根拠が README の一行で説明できる。
- cargo-auditable により、デプロイ済みバイナリ単体から依存監査ができる（ソースツリー不要）。
- draft release により、成果物生成（機械）と公開判断（人間）が分離される。

### Cons / Tradeoffs
- keyless 署名の信頼は GitHub OIDC（Fulcio / Rekor インフラ）に立脚する。GitHub の外でビルドする fork には検証手順がそのまま移植できない。
- base image はタグ参照（可読性優先）。digest ピンは production fork の責務として Dockerfile に注記した——Plecto 自身の規律（ピン留め）からの意図的な緩和。
- リリースは tag push という人間の操作がトリガーで、リリース列車（定期リリース）は組まない。頻度は maintainer の判断に委ねる。
- **再検討条件**: (a) 成果物の種類が増えて手書きワークフローの維持が辛くなったら cargo-dist へ；(b) aws-lc-sys が依存ツリーから消えたら musl 静的 + `distroless/static` へ；(c) M3 GA でライブラリ API を安定化するとき crates.io publish を個別判断する。
