# HTTP/1.1 Upgrade を route ごとの token allowlist で通し、WebSocket トンネルを activity-based idle timeout 付きで成立させる

## Status
accepted

## Date
2026-07-03

## Affected Components
- control plane（`plecto-control`）— `[[route]]` に `[route.upgrade]` セクション（`protocols`: 大文字小文字を区別しない Upgrade token の allowlist、`idle_timeout_ms`: トンネルの無通信タイムアウト）。`h2c` の宣言は build-time validation で拒否
- fast path（`plecto-server`）— `listener.rs`（h1 接続の upgrade 対応）、forward 経路（Upgrade/Connection の controlled re-issue と 101 検証）、トンネル splice（双方向コピー + idle timer + 資源計上 + drain 連動）

## Context

Plecto は `Upgrade` / `Connection` を hop-by-hop として一律剥離するため（RFC 9110 §7.8 準拠の既定動作）、**WebSocket はどの構成でも必ず失敗する**。2026-07-03 のギャップ分析 §2 が指摘したとおり、これは「代替のない透過性の穴」である: 圧縮は前段/後段で代替でき、mTLS は網境界で運用回避できるが、プロキシが Upgrade を剥離する限り WebSocket はどこでも通らない。セルフホスト想定利用者が置きたいアプリ（Grafana Live、Home Assistant、code-server、リアルタイム UI 一般）は WebSocket 必須が多数で、現状の Plecto はこれらの前に置けない。[[000043]] の deferred 序列（圧縮 → mTLS → WebSocket）はこの基準——ACME を却下したときの「代替可能な機能より、代替のない透過性の穴が先」——を一貫適用すると WebSocket が先頭に来る。

業界の有効化モデルは二分している（いずれも一次ソースを 2026-07-03 に確認、確定事実）:
- **default-on**（Caddy「performing the HTTP upgrade request then transitioning the connection to a bidirectional tunnel」・Traefik「out of the box」・HAProxy 自動 tunnel mode・ingress-nginx）——設定ゼロで動く。
- **opt-in**（nginx: `map $http_upgrade` + `proxy_set_header` 2 行を location ごとに・Envoy: `upgrade_configs` の `upgrade_type` token allowlist、per-route override）。

この分岐には確立したセキュリティ根拠がある。**h2c smuggling**（Bishop Fox, 2020）: `Upgrade: h2c` を無差別転送するプロキシは、backend の 101 で「unmanaged TCP tunnel」に化け、攻撃者が proxy のルーティング・認可・WAF を素通しで backend に raw HTTP/2 を話せる。**default-on だった HAProxy / Traefik / Nuster がデフォルト構成で脆弱**であり、推奨緩和は「Upgrade は `websocket` のみ許可」——つまり token allowlist そのもの。Envoy（opt-in allowlist）は safe by default だった。

トンネルの時間境界も全先行事例が持つ: nginx は `proxy_read_timeout`（default 60s）が無通信トンネルを閉じ、Envoy は `stream_idle_timeout`（default 5 分、encode/decode イベントでリセット、per-route `idle_timeout` で上書き、0 で無効）、HAProxy は `timeout tunnel`。共通する一般形は「**どちら向きでもデータが流れるたびリセットされる idle timer**」である。

実装素材は hyper 1.x で確定している（2026-07-03 確認）: server 側は `http1::Builder::serve_connection(..).with_upgrades()`、client 側は既存の pooled legacy Client がそのまま upgrade 対応（hyper-util 0.1.2 以降、lockfile は 0.1.20。101 後の接続は pool に戻らない）、確立後は両端の `Upgraded` を `TokioIo` で橋渡しして双方向コピー。RFC 9110 §7.8 は「server は client が要求しなかった protocol へ switch してはならない（MUST NOT）」を明記しており、unsolicited 101 の扱いを規範づける。

## Decision

1. **`[route.upgrade]` を manifest に追加する（宣言がなければ従来どおり剥離 = deny-by-default）。**
   ```toml
   [route.upgrade]
   protocols = ["websocket"]     # Upgrade token の allowlist（case-insensitive、必須・非空）
   idle_timeout_ms = 300000      # 省略時 300_000（5 分）。0 = 無効（トンネルは close まで生きる）
   ```
   route 単位なのは Plecto の manifest が route を第一級に置くため（`[route.rate_limit]`（[[000033]]）と同じ粒度）。1 行の宣言的 opt-in は nginx の 3 行のヘッダ機構より単純で、Envoy の allowlist と同型——「opt-in は煩雑」という懸念は、煩雑なのが nginx の**表現**（ヘッダ配管の手書き）であって opt-in という**方針**ではない、という調査結果で解消した。
2. **`h2c` トークンの宣言は validate で拒否する。** Plecto は h2c を採らない（[[000015]]）。h2c smuggling の攻撃面を manifest の書き間違いでも開けないようにする（fail-closed）。
3. **ハンドシェイクの転送は controlled re-issue とする。** hop-by-hop 剥離は不変のまま、`[route.upgrade]` を宣言した route 上で allowlist に一致する Upgrade 要求が来たときだけ、upstream 向けに `Upgrade: <token>` + `Connection: upgrade` を再発行する——[[000042]] の `TE: trailers` と同じ「無差別 forward ではなく条件付き再発行」パターン。filter chain はハンドシェイクの headers に通常どおり作用する（request 側・response 側とも）。トンネル payload は不透過（[[000038]] の body-opaque 原則の延長）。
4. **トンネル成立条件は RFC 9110 §7.8 に従う**: (a) client が allowlist 内の token で Upgrade を要求し、(b) upstream が 101 を返し、(c) 101 の `Upgrade` ヘッダが要求 token と case-insensitive に一致する——三条件が揃ったときだけ splice する。upstream の非 101 応答は通常応答としてそのまま client へ転送する（ハンドシェイク拒否はアプリの正当な応答）。**unsolicited 101**（client が要求していない、または token 不一致）は response smuggling として upstream 接続を破棄し 502 で fail-closed。
5. **101 確立後は既存のリクエスト時間境界を解除し、activity-based idle timer に引き継ぐ。** per-try / overall timeout（[[000031]]）はハンドシェイクの完了までを縛り、splice 時点で武装解除する。以後は `idle_timeout_ms` の idle timer（どちら向きのバイト転送でもリセット）がトンネルを閉じる唯一の時計になる。双方向コピーは一般的な bidirectional byte-copy で実装し、特定 OSS の写しは作らない。
6. **トンネルも既存の資源会計に乗せる。** circuit breaker permit（[[000028]]）と LB の `Pick` guard（least-request の in-flight 計上）をトンネル task に move し、長寿命接続が `max_requests` と least-request 負荷に正しく現れるようにする。graceful shutdown（[[000039]]）の drain flip でトンネルは閉じる——無期限のトンネルに drain を無期限に待たせない。
7. **スコープは downstream HTTP/1.1、upstream 側も h1 の leg に限る（slice 1）。** downstream h2/h3 の WebSocket（RFC 8441 / RFC 9220 extended CONNECT）は deferred: hyper の `enable_connect_protocol` と h3 0.0.8 の `enable_extended_connect` で素材の存在は確認済み（確定事実）だが、h2↔h1 の変換ロジックは自作が必要（projected）。upstream が `https` group で ALPN が h2 に倒れる origin は Upgrade が成立せず可視に失敗する——これも extended CONNECT 変換と同じ deferred 面。

検討した代替案と却下理由:
- **default-on 透過（Caddy / Traefik 型）**——設定ゼロは魅力だが、Upgrade を無差別転送した default-on 組は h2c smuggling にデフォルト脆弱だった実績があり、deny-by-default テネットにも反する。却下。
- **bool フラグ（`websocket = true`）**——`h2c` の構造的排除と「websocket 以外の独自 Upgrade token」の表現力を両立できない。token allowlist は Bishop Fox 緩和の素直な形。却下。
- **listener / global 一括 opt-in（Envoy HCM 型）**——global 定義 + per-route override の二層は Plecto の manifest 規模に対して過剰。route 単位一層で足りる。却下（route 数が増えて共通宣言の需要が出たら再検討）。
- **idle timeout なし（無期限デフォルト）**——nginx / Envoy / HAProxy いずれもトンネルに時間境界を持つ。放棄された接続が `conn_limit` と breaker permit を静かに食い潰す。却下（`idle_timeout_ms = 0` で operator の明示無効化は許す）。

本決定は[[000043]] の deferred 序列を（WebSocket → 圧縮 → mTLS）へ更新する（ACME 却下基準の一貫適用、ギャップ分析 §2 の提案どおり）。[[000029]] の native 採用基準 (i)（横断的データプレーン関心事）に該当し、Fork 6 の配置判断に従う。

## Consequences

### Pros
- Grafana Live / Home Assistant / code-server 級の WebSocket 必須アプリの前に Plecto を置ける——「代替のない透過性の穴」が閉じる。
- opt-in は TOML 1 行で、nginx の 3 行呪文（忘れると不可解な 400 で落ちる）より単純。h2c smuggling は allowlist + `h2c` 拒否で構造的に封じられる。
- トンネルが breaker / least-request / drain の既存会計に乗るので、長寿命接続が資源を「見えないまま」占有しない。
- ハンドシェイクは通常のリクエストとして filter chain / retry / timeout の既存規律を全部通る（トンネル化は 101 検証後の最後の一歩だけ）。

### Cons / Tradeoffs
- 未宣言 route への Upgrade 要求は従来どおり剥離され、upstream には通常の HTTP リクエストとして届く（多くのアプリは 400/426 を返す）。「設定忘れ」の失敗は残る——ただし `plecto validate` + JSON Schema（[[000046]] / [[000049]]）で宣言自体の検証はできる。
- downstream h2/h3 の WebSocket は未対応（deferred）。ブラウザは h1 に fallback するため実害は限定的だが、h2-only クライアントは通らない。
- idle timeout の既定 5 分は ingress-nginx が WebSocket 向けに推奨する 1h+ より短い。ping/pong を打たない静かなアプリは `idle_timeout_ms` の引き上げが必要（既定を長くするより、既定で資源が漏れない方を取る）。
- **再検討条件**: (a) downstream h2/h3 WebSocket の実需が出たら extended CONNECT（RFC 8441/9220）変換を個別 ADR で；(b) upstream h2 leg 上の WebSocket（extended CONNECT を上流に話す）の需要も同 ADR で扱う；(c) route 数の増加で allowlist の共通宣言（global default + per-route override）の需要が出たら二層化を再検討；(d) トンネルの転送バイト数 / 滞留時間の observability 需要が出たら metrics を拡張する。
