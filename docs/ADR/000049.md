# plecto schema サブコマンドで manifest の JSON Schema を出力し、エディタ補完と CI 検証を manifest 構造体から自動生成する

## Status
accepted

## Date
2026-07-03

## Affected Components
- control plane（`plecto-control`）— manifest の全 `struct`/`enum` に `JsonSchema` derive を追加し、`manifest_json_schema()`（draft-07 出力）を公開面に載せる
- fast path（`plecto-server`）— `plecto schema` サブコマンド。`validate` / `--version`（[[000046]]）と並ぶ operator CLI の第三の口

## Context

manifest は TOML + `deny_unknown_fields`（[[000008]]）で、`plecto validate` が deploy 前の静的検証を提供する（[[000046]]）。しかし operator の第一日目体験には二つの穴が残る: (1) **エディタで manifest を書くとき補完も inline 検証もない**——typo は `validate` を回すまで見えない；(2) **JSON Schema を前提とする汎用ツーリング**（taplo / Even Better TOML、schemastore 系 CI、ドキュメント生成）に接続する口がない。2026-07-03 のギャップ分析 §4.2 は「TOML + `deny_unknown_fields` は良い基盤で、schema を吐ければエディタ補完・CI 検証・ドキュメント自動生成が全部ここから生える」として `plecto schema` を提案した。

外部事実（2026-07-03 に一次ソース確認、確定事実）:
- **schemars 1.x**（1.0.0 = 2025-06-23、最新 1.2.1 = 2026-02-01）は serde 属性に整合した JSON Schema を derive で生成する。`deny_unknown_fields` → `additionalProperties: false`、`#[serde(default)]` → required から除外 + `default` 値の埋め込み、`rename_all` / enum 表現も追従する。
- 依存フットプリントは小さい: default features の直接依存は serde / serde_json / dyn-clone / ref-cast / schemars_derive のみで、serde derive を既に持つ Plecto への**正味追加は約 5 crate**（dyn-clone, ref-cast(+impl), schemars_derive, serde_derive_internals）。chrono / url 等は opt-in feature で、default では引かれない。
- **「バイナリが自分の schema をサブコマンドで吐き、schemars で生成する」は確立パターン**: Vector（`vector generate-schema`）、taplo 自身（taplo.toml の schema を schemars で生成）、cargo-dist（`cargo-dist-schema`）が同型。
- taplo（Even Better TOML）の schema 対応は公式には Draft 4 を明記。schemars の default 出力（2020-12、`$defs`）より **draft-07 出力**（`SchemaSettings::draft07()`）の方が互換が固い。関連付けは TOML 先頭の `#:schema <url-or-path>` directive が最も単純。

この機能は[[000029]] の native 採用基準でいえばデータプレーン外の operator ツーリングであり、フィルタ境界に一切触れない。

## Decision

1. **`plecto schema` サブコマンドを追加し、manifest の JSON Schema を stdout に出力する。** `validate`（[[000046]]）と同じ「artifact を読まない・非破壊・CI に置ける」性質を保つ。
2. **schemars 1.x の derive で manifest 構造体から生成する（無条件コンパイル、feature gate なし）。** schema は serde モデルから機械的に導出され、構造体と schema の drift が原理的に起きない——`deny_unknown_fields` の厳格 manifest では、手書き schema の同期漏れがそのまま「エディタは通るが validate で落ちる」偽陰性になるため、この保証が本体。正味 +5 crate は `deny.toml` の審査対象として受容する。
3. **出力は draft-07（`SchemaSettings::draft07()`）とする。** 主要な消費者である taplo の公式対応水準（Draft 4 明記、draft-07 は実運用で広く通用）に合わせる。schemars default の 2020-12 は taplo 互換が不確かで、schema の消費側都合を優先する。
4. **schema が記述するのは deserialization 面（operator が書ける TOML の形）とする。** `[observability]` / `[listen]` は content-hash 都合で `skip_serializing`（[[000045]]）だが manifest には書けるフィールドなので、schema には**必ず含める**（schemars の serialize 側追従で脱落する場合は `#[schemars(...)]` で明示補正する）。
5. **エディタ関連付けは TOML 先頭の `#:schema` directive を文書化する**（`docs/writing-a-filter.md` 系の operator ドキュメントに一行）。schemastore への登録や URL ホスティングは release 成果物（[[000047]]）に schema を同梱できるようになってからの後続判断とする。

検討した代替案と却下理由:

- **手書きの静的 JSON Schema ファイル**——依存ゼロだが、構造体との同期を検査する仕組みがなく、フィールド追加のたびに手動ミラーが必要。`deny_unknown_fields` + schema の `additionalProperties: false` は両方が同じ真実から導出されて初めて意味を持つ。却下。
- **schemars を cargo feature で gate する**——正味 5 crate の節約と引き換えに、release バイナリの `plecto schema` が「ビルド構成次第で存在しない」ことになる。operator CLI の口が feature で消えるのは運用の単純さに反する。却下（cargo-deny の審査で問題が出たら再検討）。
- **facet-jsonschema（facet 反射系）**——実在するが pre-1.0 で、serde derive を別の反射システムに置き換える大工事になる。供給網保守の観点で不適。却下。
- **2020-12 のまま出力する**——schemars の default で最も素直だが、主要消費者 taplo の公式対応外。消費できない schema に意味はない。却下（taplo が 2020-12 対応を明記したら default に戻す）。

## Consequences

### Pros
- manifest 編集時の補完・inline 検証がエディタ（taplo / Even Better TOML）で効く。typo が書いた瞬間に見える。
- schema は構造体から自動導出されるので、manifest にフィールドを足せば schema も必ず追従する（review の目視に頼らない）。
- `plecto validate`（意味論まで検証）と `plecto schema`（構造を機械可読に公開）が相補になり、CI は「schema で構造 → validate で意味論」の二段が組める。
- Vector / taplo / cargo-dist と同型の確立パターンで、説明コストが低い。

### Cons / Tradeoffs
- 依存が正味 +5 crate（すべて小粒の proc-macro 系 / ユーティリティ）。cargo-deny の監視対象が増える。
- TOML → JSON Schema の意味論ギャップは残る: TOML datetime は JSON に対応物がなく（Plecto の manifest は文字列 + ms 整数のみなので現状非該当）、`Option<T>` は TOML で null を書けないのに `"null"` 分岐が出る（無害だがノイズ）。
- draft-07 固定は taplo 都合の妥協で、JSON Schema の最新 draft 機能（`prefixItems` 等）は使わない。
- **再検討条件**: (a) taplo が 2020-12 対応を公式化したら default 出力へ戻す；(b) release 成果物に schema を同梱し URL 配布する段（[[000047]] の後続）で schemastore 登録を個別判断する；(c) cargo-deny の supply-chain 審査で schemars 系依存に問題が出たら feature gate 化を再検討する。
