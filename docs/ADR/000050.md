# [upstream.tls] に検証名オーバーライド sni を追加し、IP エンドポイントとの相互作用を閉じる

## Status
accepted

## Date
2026-07-04

## Affected Components
- control plane（`plecto-control`）— `UpstreamTls`（現状 `ca_path` のみ、`manifest/mod.rs`）に optional な `sni` を追加。validate で組合せ検査（警告）
- fast path（`plecto-server`）— TLS leg の ServerName 導出を `sni` 宣言時に差し替える。未宣言時は現挙動のまま（後方互換）

## Context

[[000042]] の upstream TLS は、転送 URI `https://{address}` を組み立てて hyper-rustls に渡す構造であり（`forward.rs` が `pick.address()` から URI を構築）、SNI と証明書検証名は **URI のホスト部から暗黙に導出される**。明示的な ServerName 指定はコードベースのどこにも存在しない。この構造は endpoint が hostname である限り正しく動くが、IP になった瞬間に破綻する:

- addresses に IP リテラルを直書きした場合、rustls は `ServerName::IpAddress` を構築する。RFC 6066 により **SNI 拡張は送信されず**、証明書検証は IP SAN に対して行われる。IP SAN を持たない通常のサーバ証明書では全 handshake が fail-closed で失敗する。
- より深刻なのは [[000044]] との相互作用で、DNS 再解決は hostname アドレスを解決済み IP 文字列に**置換**する（`dns.rs` の `refresh_group`）。つまり `[upstream.tls]` + `resolve_interval_ms > 0` の組合せ——**k8s headless service + 内部 CA という [[000044]] の動機そのものの構成**——では、元の hostname が失われ、全 leg が「SNI なし + IP 検証」で確実に失敗する。

テストは hostname（`localhost`）経路のみで、この面は未カバー。先行事例は同型の解を持つ（2026-07-04 に一次ソースで確認）: nginx は `proxy_ssl_name`（既定 `$proxy_host`）で「証明書検証名と SNI の両方」をオーバーライドでき、Envoy は cluster 単位の固定 `sni` と `auto_sni` / `auto_san_validation` を持つ。

## Decision

1. **`[upstream.tls]` に `sni = "<name>"` を追加する（optional）。** 宣言時、この upstream への全 TLS leg は SNI と証明書検証名の**両方**に宣言値を用いる。未宣言時は従来どおり接続先アドレス由来（hostname アドレスなら現挙動のまま後方互換）。nginx `proxy_ssl_name` と同じく「名前の単一オーバーライド」であり、検証を弱める knob ではない——検証自体は常時 ON（[[000042]] の insecure なし原則は不変）。
2. **validate で組合せを検査する**: `[upstream.tls]` があり、かつ (a) addresses に IP リテラルが含まれる、または (b) `resolve_interval_ms > 0` のいずれかで `sni` 未宣言なら**警告**を出す（拒否はしない——IP SAN 証明書の運用は正当）。
3. **テストに「IP アドレス + sni 宣言」経路と「DNS 展開 + sni 宣言」経路を追加する。** 現状の hostname-only カバレッジの穴を閉じる。

検討した代替案と却下理由:

- **DNS 展開時に元 hostname を検証名として自動継承する（Envoy `auto_sni` 型の暗黙解のみ）**——[[000044]] 経路は救えるが、IP 直書き経路を救えず、「manifest に書いていない名前で検証される」という暗黙の魔法が生まれる。宣言的 manifest の静的な単一ソース原則に合わない。明示宣言 1 キーの方が読み手に対して正直。却下。
- **警告でなく validate エラーにする**——IP SAN 証明書は実在する正当構成であり、静的検査で拒否する根拠がない。fail-closed は runtime handshake が既に担っている。却下。
- **per-address の sni マップ**——現時点の実需がなく、1 upstream = 1 論理サービス = 1 検証名で足りる。却下（再検討条件へ）。

本決定は [[000042]]（常時検証・insecure なし）の原則を維持したまま、[[000042]]×[[000044]] の未定義な相互作用を仕様に変える。

## Consequences

### Pros
- [[000042]]×[[000044]] 同時使用（DNS 展開 + upstream TLS）が設計どおり動く。宣言外の未定義挙動が仕様になる。
- IP 直書き + 通常証明書という構成が、`sni` 1 キーで成立する。
- validate 警告により、失敗する組合せがデプロイ前に可視化される。

### Cons / Tradeoffs
- 検証名がアドレスと乖離しうる knob が増える。ただし誤設定しても検証自体は常時 ON なので、安全性は落ちない（間違った名前なら handshake が fail-closed で落ちるだけ）。
- hyper-rustls の暗黙導出に対する明示オーバーライドの配線（connector 層）が実装コストとして残る。
- **再検討条件**: SNI をエンドポイント単位で変えたい実需が出たとき、per-address の `sni` マップへ拡張する。
