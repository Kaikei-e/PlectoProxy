# TLS crypto provider を aws-lc-rs へ一本化し、cmake declined の判断を撤回する

## Status
accepted

## Date
2026-07-04

## Affected Components
- workspace（`plecto/Cargo.toml` / `deny.toml`）— rustls 系クレートの feature を `aws_lc_rs` に、quinn を `rustls-aws-lc-rs`（quinn 自身の既定は `rustls-ring` のため明示指定が必要）に切替
- control plane（`plecto-control`）— `tls.rs` の 3 注入点（downstream TCP ServerConfig / upstream ClientConfig / QUIC ServerConfig）を aws-lc-rs へ統一
- host（`plecto-host`）— `outbound_http.rs` の provider 注入を aws-lc-rs へ。`ring と aws-lc-rs が同時にリンクされるため process-default が使えない」という既存コメントは、provider が 1 つに統一されることで不要になる

## Context

TLS 経路は計測上の最弱面（kept-alive で平文比 −50%）であり、crypto provider の選択はその改善候補であると同時に、供給網の一貫性（[[000047]]: フィルタに課す規律を自らに適用する）に関わる。本 ADR は前バージョン（2026-07-04 時点の別ドラフト）で「aws-lc-rs は cmake 依存 → declined、ring 既定 + graviola opt-in」と判断したが、**その後のコードベース検証で中核前提が崩れたため撤回・再判断する**。

**発覚した事実（2026-07-04、コードベースを実地検証）**: Plecto は既に `sigstore`（cosign 署名検証、[[000006]] / [[000047]]）経由で **aws-lc-rs（= aws-lc-sys、AWS-LC の C/アセンブリコード）を無条件にビルド・リンクしている**。

- `cargo tree -i cmake` → `cmake`(crate) ← `aws-lc-sys` ← `aws-lc-rs` ← `sigstore` ← `plecto-host`。
- `sigstore` は `default-features = false`（offline crypto module のみに絞り込み済み、`Cargo.toml` のコメントに ADR 000006 の意図が明記されている）だが、aws-lc-rs は sigstore の feature 構成に関係なく必須の依存であり、署名検証プリミティブ（`aws_lc_rs::signature::ECDSA_P256_SHA256_ASN1` 等）に直接使われている——rustls の TLS provider 選択とは無関係。
- **実機検証**: `cmake` バイナリが存在しない環境（`which cmake` → exit 1）で `cargo build -p plecto-host` を実行し**成功**。`aws_lc_sys` / `aws_lc_rs` の `.rlib` が生成された。これは前回裏どりの「非 FIPS aws-lc-rs は C コンパイラのみで足り、cmake は FIPS 変種のみ要求する」という事実と整合する。
- コードベースには既に `crates/host/src/outbound_http.rs:342` に「with both ring and aws-lc-rs linked there is no single process-default provider」という**既存コメント**があり、この共存自体は前バージョンの ADR を書く前から自覚されていた既存事実だった。

**この事実が意味すること**: 前バージョンの中核論拠「vendored C codebase を避けて Rust 完結の供給網を守る」は、**ADR を書く前から既に破れていた**。AWS-LC という大規模 C/アセンブリコードベースは、TLS provider の選択と無関係に、もう Plecto のバイナリの中に存在し、コンパイルされ、deny.toml の監査対象になっている（deny.toml に `[licenses]` セクションはなく、現状 advisories/bans/sources のみが CI でブロッキングされる）。したがって「aws-lc-rs を避ける」という判断が守っていたはずの性質（供給網の Rust 完結性）は、判断の時点で既に存在しなかった。

**追加で裏どりした事実（2026-07-04）**:

- **CPU フロア/フォールバック**: AWS-LC は README で「a portable C implementation of all algorithms is included」と明記し、実際の runtime dispatch（`crypto/fipsmodule/aes/aes.c` 等）は `hwaes_capable()`（AES-NI / ARMv8 crypto extension）→ `vpaes_capable()`（SSSE3 ビットスライス）→ portable C、の 3 段階フォールバックを持ち、**フォールバック不能で panic する分岐がない**。graviola が明示的に除外する Raspberry Pi 4 級のハードウェアでも（低速化はするが）動作する。ただし 32-bit ARM（armv7）の一部 Raspberry Pi 構成では aws-lc-sys の**ビルド時**トラブルが報告されている（`aws-lc-rs` issue #681、`librespot` issue #1447）——これは runtime dispatch とは別の、ビルド toolchain 側の既知の穴であり、64-bit（aarch64）で動く Raspberry Pi 4/5 は対象外。
- **QUIC 対応**: rustls の aws-lc-rs provider は TLS 1.3 スイート全てで `quic` フィールドが `Some(...)`（`tls13.rs`）であり、quinn の rustls バックエンドが要求する QUIC ヘッダ保護/シークレット導出に対応する。quinn 自身の Cargo 既定 feature は `rustls-ring` のため、`default-features = false` + `rustls-aws-lc-rs` の明示指定が必要——ただしこれは feature flag の変更のみで、機能的な障壁ではない。
- **rustls チームの現在の推奨**: rustls は「`rustls-aws-lc-rs` を、その完全な機能集合と性能のために推奨する」と明言し、ring は「ビルドしやすいが機能集合が限定的（PQ なし等）」と位置づける。

品質ターゲットは「SaaS 導入水準」（[[000054]]）であり、FIPS 140-3 は非目標。aws-lc-rs の非 FIPS 既定ビルドを使う限り、cmake/Go は要求されない。

## Decision

1. **aws-lc-rs を、downstream TCP TLS・upstream 再暗号化・QUIC/h3・host outbound-http のすべてに使う単一の crypto provider とする。** sigstore が既に強制している依存を、rustls 全経路の provider として再利用する——新しい依存を追加するのではなく、既に払っているコストを一本化する判断である。
2. **PQ hybrid（X25519MLKEM768）を TCP・QUIC の両方で既定最優先にする**（rustls 0.23.27 以降の `prefer-post-quantum` 既定 feature）。前バージョンの graviola(TCP)+ring(QUIC) ハイブリッド案が抱えていた「h3 経路だけ PQ なし」という非対称は解消される。
3. **FIPS 変種は使わない。** 品質ターゲット（[[000054]]）が FIPS を非目標とする以上、cmake + Go を要求する FIPS ビルドを選ぶ理由がない。既定の非 FIPS aws-lc-rs（C コンパイラのみ、sigstore で既に要求済み）を使う。
4. **provider はビルド時固定の 1 バイナリ 1 選択とし、runtime 切替は作らない。** graviola/ring の opt-in fallback feature は設けない——AWS-LC 自身が CPU フロア未達時にソフトウェアフォールバックで動作するため、graviola が必要としていた `crypto-ring` エスケープハッチや起動時 CPU self-check は不要になる。
5. **deny.toml / ドキュメント**: aws-lc-rs は既に依存グラフに存在し監査済みの扱いを受けているため、deny.toml の追加変更は不要。README / hardening ガイドに「provider は aws-lc-rs に一本化」「32-bit ARM(armv7) の一部 Pi 構成はビルド時に既知の穴がある」ことを明記する。性能計測は [[000052]] の resumption ラングと同一セッションで、旧 ring 既定との A/B として取る。

検討した代替案と却下理由:

- **graviola(TCP) + aws-lc-rs(QUIC) のハイブリッドに修正するだけ**——「供給網から C を完全排除できない」という新事実を踏まえても、graviola を TCP 経路に残す理由（rustc-only ビルド）は、sigstore が既に C toolchain を要求している以上、バイナリ全体では実現しない目標になっている。graviola 特有の CPU フロア無フォールバック（Pi 4 除外）リスクを TCP 経路に残したまま、crypto backend を 2 つ運用する複雑さだけが残る。却下。
- **前バージョンの graviola 既定 + ring(QUIC) 併存を維持し、Context の記述だけ正確化する**——「C を完全には避けられないが、TLS 終端という最大の攻撃表面での AWS-LC 実行は避ける」という論拠は理屈としては成立する（Cons に明記）。しかし graviola の QUIC 非対応・CPU フロア無フォールバック・第三者監査なしという実務コストに対し、得られる利得（TLS 終端で C コードを実行しない）は、sigstore が既に同じ C コードを署名検証に使っている現実の前では限定的と判断した。却下。
- **ring 既定の維持**——ring の実態（単独メンテナ・16 ヶ月無リリース・資金依存、[[000051]] 前バージョンで確認済み）と PQ の恒久的不在は変わらず、aws-lc-rs 一本化の方が SaaS 導入水準（[[000054]]）への回答として強い。却下。

本決定はコア原則「安全 × ポータビリティ × セルフホスト性 × 運用の単純さ」のうち、単純さ（provider 1 つ）とポータビリティ（CPU フロア無フォールバック）を同時に改善する。安全面のトレードオフ（TLS 終端での AWS-LC 実行拡大）は Cons に明記し、再検討条件を残す。

## Consequences

### Pros
- crypto backend が 1 つに統一される（現状 ring + aws-lc-rs の 2 backend 併存、前バージョン案では graviola + ring + aws-lc-rs の 3 backend）——監査対象・バイナリサイズ・「どの backend がどこで動くか」の説明コストが下がる。
- PQ hybrid（X25519MLKEM768）が TCP・QUIC 問わず既定最優先——SaaS 調達の「PQ ready?」に無条件で Yes と答えられる。
- CPU フロア無フォールバック問題が消える。graviola が除外する Raspberry Pi 4 級のハードウェアでも動作し、セルフホスト性との衝突がない。
- quinn の QUIC 対応が標準経路として最初から成立し、graviola のような「対応待ち」の非対称を抱えない。
- rustls チームの現在の推奨に沿う。既に依存しているコードのメンテナンス（脆弱性対応含む）が rustls エコシステムの主流と一致する。

### Cons / Tradeoffs
- **AWS-LC の C/アセンブリコードが、任意の攻撃者が送りつける生の TLS バイト列に常時晒される経路（TLS 終端・upstream 再暗号化・QUIC）で実行される**——これまでは sigstore の狭い・構造化された（operator 提供の artifact に対する）署名検証呼び出しに限られていた。依存自体は新規ではないが、実行される攻撃対象面は拡大する。
- 大規模 C コードベースが Rust 語彙の供給網監査（cargo-deny の advisories/bans/sources、license 検査は現状未配線）の外にある事実は変わらない——本決定で「悪化」はしないが「解消」もしない。
- quinn 自身の既定 feature は `rustls-ring` のため、Plecto は明示的に `default-features = false` + `rustls-aws-lc-rs` を指定し続ける必要がある（quinn の将来のデフォルト変更に追随する保守コストが発生する）。
- 32-bit ARM（armv7）の一部 Raspberry Pi 構成で aws-lc-sys のビルド時トラブルが既知（aarch64/64-bit 構成は対象外）。
- **再検討条件**: (a) TLS 終端経路での AWS-LC 実行がセキュリティインシデントに関与したとき、graviola またはハイブリッド構成へ回帰を個別 ADR で判断; (b) rustls-graviola が QUIC 対応を完了し、かつ CPU フォールバック（無 panic）の道筋が示されたとき、graviola への統一を再評価; (c) FIPS 実需が出たとき、[[000054]] と合わせて FIPS 変種（cmake + Go の新規コスト）を再検討; (d) sigstore への依存自体が将来なくなったとき、aws-lc-rs を TLS provider として使い続ける根拠を単独で再評価する。
