# TLS 1.3 の stateless session resumption を導入し、0-RTT 拒否とチケット鍵ノードローカルを不変条件にする

## Status
accepted

## Date
2026-07-04

## Affected Components
- control plane（`plecto-control`）— `build_server_configs` に stateless ticketer を明示設定、stateful セッションストアを無効化。QUIC ServerConfig にも同方針を適用
- bench — resumed handshake ラングの追加で TLS decomposition を完成させる

## Context

TLS 計測は keep-alive（最良）と handshake/req（最悪 ~30k/s、フル ECDHE ~28µs/req）の両極端のみで、実運用の大半が通る **resumed handshake の中間値が見えていない**。

前提の正確化（2026-07-04、rustls 0.23.41 のソースと docs を一次確認）: Plecto は `tls.rs` で ticketer / session_storage を一切設定していないため、**rustls サーバ既定の stateful resumption が既に暗黙で動いている**——in-memory セッションキャッシュ 256 件、TLS 1.3 NewSessionTicket は 2 枚送信され、チケット値はキャッシュへの lookup key である（既定 ticketer は `NeverProducesTickets` で、自己暗号化チケットは発行されない）。つまり本 ADR は「resumption の新規導入」ではなく、**暗黙の stateful 既定を、明示の stateless ticket 方式に置き換えて仕様化する**判断である。

本判断は 2 つの既存判断と交差する: [[000016]] は 0-RTT（early data）を RFC 8470 根拠で拒否しており（`max_early_data_size` は rustls 既定の 0 のまま）、[[000053]] は全ホスト状態をノードローカルと宣言する。さらに rustls 0.23.41 の確定事実として、**サーバ側 0-RTT は stateful resumption が前提であり、stateless ticketer とは構造的に排他**（early data の受理条件が `max_early_data_size > 0 && !ticketer.enabled()`）——stateless 化は 0-RTT 拒否を構造でも補強する。

## Decision

1. **TLS 1.3 PSK（session ticket）による stateless resumption を明示設定する。** rustls の `TicketRotator`（provider 非依存の鍵ローテーション機構）を用い、鍵は**プロセス内メモリのみ・ノードローカル**（[[000053]] と整合。ディスク永続化しない、manifest 設定なし）。ローテーションは rustls 標準の 6 時間周期・受理窓 12 時間（current + previous の 2 鍵保持）を採用する。チケット構造は RFC 5077 §4 の一般形（AEAD による自己暗号化チケット）に依拠し、特定実装の模倣はしない。provider は [[000051]] により aws-lc-rs に一本化されたため、TCP・QUIC の全経路で `rustls::crypto::aws_lc_rs::Ticketer` を単一の ticket producer として使う——実装選択肢の分岐は生じない。本 ADR が固定するのは不変条件（stateless・プロセス内鍵・ローテーション周期）である。
2. **暗黙の stateful セッションキャッシュを無効化する（`NoServerSessionStorage`）。** per-session のサーバメモリをゼロにし、セッション数に比例する状態を持たない——native rate limit（[[000033]]）が確立した有界メモリ規律（CWE-770）と同じ姿勢。256 件という暗黙の既定キャッシュは、計測に映らない崖として振る舞うため排除する。
3. **0-RTT（early data）の拒否は不変。** `max_early_data_size = 0` を明示的な不変条件としてテストに固定する。stateless ticketer との構造的排他（上記）により二重に保証されるが、rustls の将来バージョンで排他条件が変わっても、テストが単独で防波堤になる。
4. **ベンチの TLS 分解表に resumed handshake ラングを追加し、full / resumed / keep-alive の三点で decomposition を完成させる。** [[000051]] の provider A/B（aws-lc-rs vs 旧 ring 既定）と同一計測セッションで取る。

検討した代替案と却下理由:

- **チケット鍵のレプリカ間共有（クラスタ全体で resumption を効かせる）**——鍵の配布・ローテーション・失効という新しい秘密管理面が生まれ、[[000053]] のノードローカル原則と衝突する。SaaS 前段 LB が sticky / consistent-hash ならノードローカル鍵でもヒット率は実用十分。却下（再検討条件へ）。
  **（2026-07-06 更新）**この再検討は [[000062]] で実行された: 既定ノードローカルは維持したまま、証明書束縛（HKDF）・mTLS 時 fail-closed・鍵ファイル規律つきの**オプトイン**として共有 STEK を導入する。不変条件のうち「チケット鍵ノードローカル」は「既定ノードローカル・共有は制約付きオプトイン」へ改訂（psk_dhe_ke のみ / 0-RTT 拒否は不変のまま）。
- **stateful セッションキャッシュの明文化・拡大（rustls 既定路線の採用）**——forward secrecy はチケット方式より強い（サーバ側状態は鍵より短命）が、セッション数に比例するメモリと、キャッシュサイズという新しい knob が増える。レプリカ非対称（キャッシュはノード間で共有されない）はチケット方式と同じで、利点が代償に見合わない。却下。
- **暗黙の既定のまま放置**——現状は「仕様化されていない stateful resumption が 256 件だけ効く」という中途半端な状態で、計測にも文書にも現れない。却下。

## Consequences

### Pros
- 実運用経路の handshake コストがフル ECDHE から大幅に下がる（定量は resumed ラングの計測で確定）。
- resumption の挙動が暗黙の既定から明示の仕様になり、0-RTT 拒否・鍵のノードローカル性・メモリ有界性がテストと構造の両方で保証される。
- per-session サーバメモリがゼロになる（現状の暗黙キャッシュ 256 件より状態が減る）。

### Cons / Tradeoffs
- **マルチレプリカでは resumption ヒット率がレプリカ跨ぎで落ちる**（ノードローカル鍵の帰結、[[000053]] と整合。前段 LB の affinity で緩和可能）。
- チケットは forward secrecy の窓をローテーション周期分（受理窓 12 時間）開ける——rustls 標準のローテーションで受容する。
- stateful キャッシュ方式より resumption 時の forward secrecy が弱い——鍵ローテーションの短周期化で調整可能な設計余地を残す。
- **再検討条件**: レプリカ跨ぎの resumption miss が実測で問題化したとき、鍵共有を [[000053]] の共有状態の議論（拡張プレーン経由）として再開する。
