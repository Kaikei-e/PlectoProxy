# マルチレプリカ構成の状態意味論をノードローカルと宣言し、共有状態は拡張プレーンへ置く

## Status
accepted

## Date
2026-07-04

## Affected Components
- roadmap / governance — native の分散状態（gossip / 共有ストア依存）を declined として記録。コード変更なし
- docs — README / hardening ガイドに「実効レート = 設定値 × N」の設計指針と、fairness / enforcement 主張のノードローカル・スコープ明記を追加

## Context

Plecto のホスト状態はすべて**ノードローカル**である: native rate limit（[[000033]]、`ratelimit.rs` 自身が "node-local and ephemeral" と明記）、`host-ratelimit` / host-kv / host-counter（[[000026]]）、redb バックエンド（[[000041]]、redb は単一プロセス設計）、そして [[000052]] の session ticket 鍵。用語は [[000033]] が既に導入した「ノードローカル（node-local）」に統一する。

SaaS 導入（[[000054]]）の標準形は「前段 LB + N レプリカ」であり、「N 台ではバケットが N 倍緩む」は導入審査で必ず問われる。この判定が未記録のまま実装が進むのは、[[000029]] が要求する「declined の言語化」の運用に反する。

先行事例（2026-07-04 に一次ソースで確認）: Envoy は分散レートリミットを native 実装せず、**外部の global rate limit service（gRPC、参照実装は Go + Redis）へ委譲**し、ノードローカルな local rate limiting（プロセス内 token bucket）と併用する二層構成を公式に案内している。「分散状態はプロキシ本体の外」という配置は業界の確立形であり、Plecto では Fork 6（user-policy はフィルタへ）がその受け皿になる。

## Decision

1. **Plecto の全ホスト状態はノードローカルであると宣言する（native の分散状態は declined）。** 分散合意・共有ストアへの native 依存は、self-hostable single binary という核（コア原則）と fail-closed の単純さを壊す。
2. **運用ガイドを提供する**: (a) 前段 LB が均等分配なら、レプリカ数 N での実効レートは設定値 × N（目標値から逆算するなら「設定値 = 目標 / N」）; (b) consistent-hash な前段（または Plecto 自身の Maglev）でキー単位の局所性を作れば、ノードローカルなバケットが実質グローバルに近づく。この 2 パターンを README / hardening ガイドに文書化する。
3. **共有状態が必要なケースは拡張プレーン（Fork 6）で表現する**: フィルタが `outbound-http`（[[000036]]）経由で外部ストア（Redis 等）を叩く reference 実装を Stage F 題材に加える。Envoy の external rate limit service と同型の配置だが、実装は特定 OSS の模倣ではなく一般的なアルゴリズム（token bucket / 分散カウンタ）に立脚する。native には持ち込まない。
4. **ベンチ・README の fairness / enforcement 主張にノードローカル・スコープを明記する**: 「noisy tenant の封じ込め」は 1 ノード内の主張であると正確化する。

検討した代替案と却下理由:

- **native 分散レートリミット（gossip / 中央カウンタ）**——一貫性モデル・分断時挙動・依存追加のコストが役割宣言（[[000029]]）の外。Envoy ですら分散版は外部サービスへ外出ししており、これは Fork 6 の user-policy 形状の傍証。却下。
- **redb を共有ボリュームで複数レプリカから共用**——redb は単一プロセス設計であり、そもそも成立しない。却下。
- **宣言を保留し実装の現状を暗黙のまま残す**——[[000029]] の governance に反し、導入審査への回答が「未定」のままになる。却下。

本決定は [[000029]]（役割駆動・declined の言語化）の運用であり、native の採用基準（横断的データプレーン関心事 / グローバルカウンタ）に新しい例外を作らない——「グローバル」の範囲が 1 ノード内であることを明文化するだけである。

## Consequences

### Pros
- 導入審査への回答が「未定」から「設計上ノードローカル、運用パターンはこれ」に変わる。
- ベンチ・README の主張のスコープが正確になり、誇大主張のリスクが消える。
- [[000052]]（ticket 鍵 per-node）や将来の状態設計が、同じ物差し（ノードローカル宣言）で判断できる。

### Cons / Tradeoffs
- 厳密なグローバル制限を求める利用者には、外部ストア（フィルタ経由）という追加部品が要る。
- レプリカ跨ぎの一貫性を要する機能（グローバル quota 等）は native では永続的に持たない。
- **再検討条件**: reference フィルタで表現しきれない横断関心事（ノード内グローバルな計数の一元化など、[[000043]] 再検討条件 (b) と同型）が実証されたとき、[[000029]] 採用基準 (ii) への該当を個別 ADR で判断する。

## 実装ノート（着地確認・2026-07-05）

Decision の docs 側（1・2・4）を本日付で実装した。3（外部ストア経由の reference filter）は
affected_components の宣言（コード変更なし）どおり、Stage F 題材としてロードマップに記録するに留め、
コードは書いていない——実装するなら Stage F 着手時の個別 ADR に従属させる（ADR-first 運用、[[000029]]）。

- **`docs/hardening.md` ＋ `docs/hardening.ja.md` を新設した。** ノードローカル状態の一覧（native rate
  limiter [[000033]]・`host-ratelimit`/`host-kv`/`host-counter` [[000026]]・redb [[000041]]・TLS session
  ticket 鍵 [[000052]]）、実効レート ≈ 設定値 × N の式とその逆算、consistent-hash / weighted Maglev で
  キー単位の局所性を作りノードローカルをほぼグローバルに近づけるパターン、それでも足りない場合は
  `outbound-http`（[[000036]]）経由の外部ストアへ委譲する Fork 6 の受け皿を文書化した。
- **README.md / README.ja.md の Rate limiting 行**に「node-local」を明記し、hardening ガイドへリンクした。
- **`performance/README.md` の Host-enforced rate limiting セクション**冒頭に「Scope: single node」の
  注記を追加し、enforcement / fairness の数値が単一ノードの計測であって多レプリカ・フリートの集約挙動を
  保証しないことを明記した。
- **`docs/ROADMAP.md` の M2 エントリ**に、native の分散状態が declined であることと hardening ガイドへの
  リンクを追記した（[[000029]] の「declined の言語化」運用）。
