# 品質ターゲットを SaaS 導入水準と定義し、mTLS を deferred 序列の先頭へ引き上げる

## Status
accepted（mTLS のスコープ・STEK 交差・deferred 線は [[000078]] が amend。品質ターゲット定義は不変）

## Date
2026-07-04

## Affected Components
- roadmap / governance — 到達品質の物差しを記録し、[[000043]] の deferred 序列を WebSocket（完了 [[000048]]）→ **mTLS** → 圧縮 に改訂。各スライスは着手時に個別 ADR で確定（[[000029]] の運用どおり）

## Context

[[000029]] は成熟の駆動軸を役割駆動に固定したが、**到達品質の水準**は未定義だった。その暗黙の水準に依存した判断が既に複数走っている（provider 選択 [[000051]]、状態意味論 [[000053]]）ため、目標を言語化する:

**SaaS 企業のプロダクション導入に耐える水準**——具体的には、(a) セキュリティ審査票（依存の保守状況・脆弱性対応プロセス・監査ログ・TLS 設定）に空欄なく答えられる、(b) 複数レプリカ運用の意味論が文書化されている（[[000053]]）、(c) 供給網が検証可能である（[[000047]]）。**金融・政府調達水準（FIPS 140-3、形式的コンプライアンス認証）は非目標**とする。

この定義は優先度を 2 箇所動かす:

1. **mTLS**——個人セルフホストでは WireGuard 等のネットワーク層で代替可能と評価してきたが、SaaS のゼロトラスト内部網では client cert の提示/検証が審査の定番項目であり、「代替可能性」の評価が目標変更で反転する。現状は downstream / upstream とも `with_no_client_auth`（`tls.rs`）で、upstream mTLS は [[000042]] から mTLS スライスへ送られた宿題。
2. **FIPS**——[[000051]] の provider 判断（aws-lc-rs への一本化）から FIPS 変種固有の切り札を外す（FIPS ビルドは cmake + Go を要求するため非採用、同 ADR に反映済み）。[[000051]] の aws-lc-rs 一本化により、既定ビルドは TCP・QUIC 問わず PQ hybrid（X25519MLKEM768）を持つ——SaaS 調達の「PQ ready?」への回答は無条件で Yes。

## Decision

1. **品質ターゲット「SaaS 導入水準」を上記 (a)(b)(c) の定義で記録し、FIPS / 金融調達水準を非目標とする。** 以後の judgment call（provider・状態意味論・監査機能・優先度）はこの物差しで下す。
2. **[[000043]] の deferred 序列を改訂する: WebSocket（完了 [[000048]]）→ mTLS（downstream + upstream client cert を同一スライスで）→ 圧縮（gzip/br/zstd）。**
   **（2026-07-06 更新）**序列は [[000056]] で再改訂: mTLS →（同一スライスか直後に）PROXY protocol v2 → h3 graceful drain（GOAWAY）→ 圧縮。本 ADR が宣言した「前段 LB + N レプリカ」標準形を実際に成立させる前提部品（クライアント同一性の復元）が序列に欠けていたため。 圧縮は前段/後段で代替可能、mTLS は SaaS 文脈で代替困難——[[000043]] 自身の「代替可能な機能より、代替のない穴が先」基準を、目標変更後に再適用した結果である。[[000043]] にはこの改訂を注記する。
3. **SaaS 水準の残余（fuzz、h2spec/h3spec 適合、hardening ガイド、サポート方針文書）は roadmap 項目として本 ADR から参照する。** 個別 ADR は着手時に書く（[[000029]] の運用どおり）。

検討した代替案と却下理由:

- **品質目標を定義しないまま個別判断を続ける**——[[000051]] / [[000053]] のような判断が目標の暗黙仮定に依存しており、暗黙のままだと将来の再検討条件が書けない（「FIPS 実需が出たら」は FIPS が非目標であると明文化されて初めて意味を持つ）。却下。
- **金融・政府調達水準まで目標に含める**——FIPS は provider 選択（[[000051]]）を強制的に反転させ、vendored C declined と両立しない。実需もない。却下。
- **mTLS の序列を据え置く（圧縮が先のまま）**——「代替可能性」の評価が目標変更で反転した以上、据え置きは [[000043]] の自基準に対して不整合。却下。

本決定は [[000029]]（役割駆動）に「どこまでやるか」の軸を追加する governance ADR であり、native/WASM の配置基準そのものは変えない。

## Consequences

### Pros
- 以後の judgment call（provider、状態意味論、監査機能）が同じ物差しで下せ、再検討条件が書けるようになる。
- mTLS の序列根拠が明文化され、次スライスの着手判断が不要になる。
- 導入審査で問われる項目群（依存保守・レプリカ意味論・供給網・TLS 設定）への回答が ADR 参照で完結する。

### Cons / Tradeoffs
- 非目標宣言は該当顧客層（規制産業）を明示的に外す——役割駆動（[[000029]]）の既知の代償と同型。
- mTLS 前倒しの分、圧縮の着地が後ろにずれる（前段/後段での代替は引き続き可能）。
- **再検討条件**: 規制産業からの実需が出たとき、[[000051]] 再検討条件 (d) と連動して FIPS / aws-lc-rs を再評価する。
