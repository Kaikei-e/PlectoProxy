# ロードマップを Multi-Replica Readiness / Filter Ecosystem の二大プログラムへ再編し、deferred 序列に PROXY protocol v2 を挿入する

## Status
accepted

## Date
2026-07-06

## Affected Components
- roadmap / governance — [[000054]] の deferred 序列を mTLS → **PROXY protocol v2** → h3 graceful drain（GOAWAY）→ 圧縮 に改訂。以後のロードマップを Program R（Multi-Replica Readiness）/ Program F（Filter Ecosystem）の二大プログラムで編成する。各スライスは着手時に個別 ADR で確定（[[000029]] の運用どおり）

## Context

[[000053]] はマルチレプリカ構成の状態意味論をノードローカルと宣言し、[[000054]] は品質ターゲットを SaaS 導入水準と定義した。両 ADR が「前段 LB + N レプリカ」を標準運用形として名指しした瞬間に、実装との間に**構造的な穴が一つ**露出した（2026-07-06 のコードレビューで特定、コードで裏取り済み）:

- native rate limit の per-client-IP キーは `route.check_rate_limit(peer.ip())`（`proxy.rs`）、Maglev の `SourceIp` アフィニティも `HashInput::Ip(peer.ip())` を使う。
- edge モデル（[[000018]] / [[000022]]）が inbound `X-Forwarded-*` を「偽装可能」として剥がすのは、Plecto が真のエッジである前提では完全に正しい。しかし前段が L4 LB のとき `peer.ip()` = LB の IP であり、(a) per-client-IP rate limit が全クライアント共有のバケット 1 個に退化し（noisy-tenant 封じ込めの主張と正面衝突）、(b) `SourceIp` Maglev が単一 instance 固定に退化し、(c) hardening ガイドの「key-consistent routing で node-local ≈ global」パターンが成立しなくなる。

つまり **PROXY protocol v2 の受信対応（または同等のクライアント同一性復元）が、[[000053]] / [[000054]] の宣言と実装の間に落ちている唯一の前提部品**である。

外部事実（2026-07-06 に一次ソースで検証済み・いずれも確定事実）:

- PROXY protocol v1/v2 は HAProxy が保守する公開仕様（`proxy-protocol.txt`、Willy Tarreau）であり、仕様文書のみから実装可能。受信側は Envoy（listener filter）・HAProxy（`accept-proxy`）・NGINX（`listen ... proxy_protocol`、v2 は 1.13.11 以降）のいずれも公式対応しており、審査票の定番項目でもある。
- Kubernetes では `externalTrafficPolicy: Local` が client source IP を保全する（クラウド LB の実転送に依存する条件付き）。PROXY v2 とこの設定の使い分けは運用ガイド側に置く。
- レートリミットの業界確立形は **local + global の二層併用**である。Envoy Gateway は同一ルートに local（各インスタンス独立の token bucket、「N レプリカで実効 N 倍」を公式に明文化）と global（外部 Rate Limit Service への照会、Redis バックエンド）を同時設定するパターンを公式化し、v1.3 以降は cost 指定（global 限定）も持つ。→ Plecto の native floor（[[000033]]）= local、拡張プレーンの分散レートリミット filter（[[000053]] Fork 6）= global という既存判断は業界形と同型であり、足りないのは二層を一枚の運用モデルとして提供・実測・文書化することだけである。

一方、[[000055]] が polyglot 主張を反証可能化したことで、フィルタエコシステム側も「証明済み」から「実務で選ばれる」への昇格が次の課題として明確になった（Go 解禁 = 最小 WasiCtx 貸与の個別 ADR、reference filter、conformance の外部開放）。

## Decision

1. **ロードマップを二大プログラムで編成する**: **Program R — Multi-Replica Readiness**（PROXY v2 / 二層レートリミット / 共有 STEK オプトイン / Maglev 意味論固定 / h3 GOAWAY drain / N-replica ベンチ / 運用ガイド昇格の 7 柱）と **Program F — Filter Ecosystem**（Go 解禁 / WIT の wkg・OCI 配布 / reference filter 三部作 / conformance CLI 外部開放 / scaffolding / dogfooding の 6 柱）。両者の交点（Redis 分散レートリミット filter = R の global 側 = F の reference）を最優先成果物とする。各柱は着手時に個別 ADR で確定する（[[000029]] の運用）。プログラムの詳細は内部計画文書（untracked）にあり、本 ADR は採択と序列だけを正式記録する。
   **（2026-07-06 追記）**主要柱の実装判断は同日の grill セッションで確定し、[[000057]]（PROXY v2）・[[000058]]（buffered-body retry）・[[000059]]（h3 GOAWAY drain + readiness + tunnel 可観測性）・[[000060]]（outbound-tcp capability）・[[000061]]（二層レートリミット + Redis reference）・[[000062]]（共有 STEK オプトイン）・[[000063]]（Go 解禁 / Tier B）・[[000064]]（WIT の wkg/OCI 配布 + 互換ポリシー）・[[000065]]（Filter Dev Kit + TTFVF）として起票済み。

2. **[[000054]] の deferred 序列を改訂する**: mTLS →（同一スライスか直後に）**PROXY protocol v2** → h3 graceful drain（GOAWAY）→ 圧縮。挿入根拠は Context の構造的穴 — PROXY v2 は [[000053]] が宣言した運用パターンを実際に成立させる前提部品であり、[[000043]] / [[000054]] 自身の「代替可能な機能より、代替のない穴が先」基準の再適用である。

3. **PROXY v2 の設計制約を今固定する**（実装 ADR は着手時に別途）: manifest の `[listen]` で宣言したときのみ有効。宣言なしの listener に PROXY ヘッダが来たら fail-closed で切断、宣言ありの listener でヘッダ欠落・不正も fail-closed。edge モデル（[[000018]]）は変えない — クライアント同一性は接続層の**下**で復元し、inbound `X-Forwarded-*` の剥がし・再発行はそのまま維持する。実装は公開仕様書（proxy-protocol.txt）のみから行い、特定実装のポートはしない。

4. **二層レートリミットを推奨運用形として文書化する**: native floor（[[000033]]）を「外部呼び出しゼロの即時 flood 遮断（local）」と再定義し、全レプリカ合算の厳密上限（global）は拡張プレーンの分散レートリミット filter に置く（[[000053]] Fork 6 の実物）。「実効レート = 設定 × N」の但し書きは local 単独時の性質として hardening ガイドに残す。cost ベース制限は native 化せず（[[000043]] の declined を維持）、global filter の契約に `cost` 引数を予約する形で受ける（projected）。

検討した代替案と却下理由:

- **trusted-hops モデル（XFF を N ホップ分信頼して剥く）** — HTTP ヘッダ層での復元は、ホップ数の設定ミスがそのまま偽装可能性になり、[[000018]] の「inbound XFF は無条件に剥がす」という単純さを壊す。PROXY v2 は HTTP を解さない L4 LB 配下でも機能し、接続層の下で復元するため edge モデルと直交する。主要 3 実装が収斂している形でもある。却下（k8s の `externalTrafficPolicy: Local` は補完パターンとしてガイドに記載する）。
- **序列を据え置く（mTLS → 圧縮のまま）** — [[000053]] / [[000054]] が宣言した標準運用形の下で per-client 機能群の意味論が壊れたまま SaaS 水準を名乗ることになり、[[000054]] の自基準（審査票に空欄なく答えられる）と不整合。却下。
- **native の分散状態でマルチレプリカの限界を解く** — [[000053]] で declined 済み。Program R の全項目は「運用パターン + 拡張プレーン + 実測」で構成し、gossip・共有ストア・consensus を native に入れる項目は一つもない。却下。
- **プログラム編成をせず個別機能列挙のまま進める** — R と F の交点（Redis filter が multi-replica 実証・Fork 6 の実物・reference filter の三役を一本で兼ねる）が見えず、優先順位が机上の列挙に戻る。削る順序（縮退線）も事前確定できない。却下。

本決定は [[000029]]（役割駆動）・[[000054]]（品質ターゲット）に「どの順で到達するか」の軸を足す governance ADR であり、native/WASM の配置基準・品質ターゲットそのものは変えない。

## Consequences

### Pros
- [[000053]] / [[000054]] が宣言した「前段 LB + N レプリカ」標準形が、宣言どおりに機能する経路（PROXY v2 → per-client rate limit / SourceIp Maglev の復元）を持つ。
- 二層レートリミットの配置（local = native floor / global = filter)が業界確立形（Envoy Gateway の公式パターン）と同型であることが記録され、導入審査での説明が一次ソース参照で完結する。
- Program R/F の交点が最優先成果物として名指しされ、削る順序（縮退線）も事前確定できる。

### Cons / Tradeoffs
- PROXY v2 の受信は listener に untrusted 入力のパース面を一つ増やす — 有界読み・fail-closed を実装 ADR の必須制約とする。
- 序列挿入の分、圧縮の着地がさらに後ろにずれる（前段/後段での代替は引き続き可能 — [[000043]] の判断を維持）。
- 二本柱化でフェーズの物量が増える。**再検討条件**: フェーズ出口条件の過半が期末未達なら、事前確定した順（scaffolding → 共有 STEK → reference の第 3 言語)で後送する。Redis filter と N-replica 実測は削らない。
- **再検討条件（序列）**: mTLS スライスの実装中に PROXY v2 との干渉（listener 設定の交差）が見つかった場合、同一スライス化か分離かを実装 ADR で確定する。
