# PROXY protocol v2 の受信を trusted CIDR 必須のオプトインとして TCP listener に導入する

## Status
accepted（判断は 2026-07-06 の grill セッションで合意、同日実装着地。実装時の確定事項: untrusted peer の通常トラフィックは素通し（ミックスモード、検出は `TcpStream::peek` の非消費読み）、`trusted` は CIDR 表記のみ受理（裸 IP は validation error）、fuzz 基盤は本件で `fuzz/` に新設（cargo-fuzz、`proxy_v2` が初のターゲット））

## Date
2026-07-06

## Affected Components
- server/listener — `listener.rs` の per-connection セットアップ（accept 後・`TlsAcceptor::accept` / plaintext `serve_conn` の前）に PROXY v2 パースを挿入し、`serve_conn(..., peer, ...)` に渡す peer を復元値に差し替える
- control/manifest — `[listen]` に `proxy_protocol` サブセクションを追加。`plecto validate` で未設定 CIDR を fail-closed に弾く
- server/proxy — 変更なし（`peer.ip()` 消費点である rate limit / `set_forwarded` / Maglev `SourceIp` は、復元済み peer を透過的に受け取る）

## Context

[[000056]] が特定した構造的穴の実装解である。前段 L4 LB 配下では `peer.ip()` = LB の IP に潰れ、per-client-IP rate limit（`proxy.rs` の `route.check_rate_limit(peer.ip())`）と Maglev `SourceIp`（`HashInput::Ip(peer.ip())`）の意味論が崩れる。edge モデル（[[000018]] / [[000022]]）は inbound `X-Forwarded-*` を無条件に剥がすため、HTTP ヘッダ層での復元は設計上使えない——復元は**接続層の下**で行う必要がある。

確定事実（2026-07-06 検証済み）: PROXY protocol v1/v2 は HAProxy が保守する公開仕様（`proxy-protocol.txt`）であり、仕様文書のみから実装可能。受信側は Envoy / HAProxy / NGINX（v2 は 1.13.11 以降）が公式対応。Kubernetes では `externalTrafficPolicy: Local` が代替経路（クラウド LB の実転送に依存する条件付き）。

現行コードの挿入点は一意である: accept ループ（`listener.rs`）が `(TcpStream, SocketAddr)` を得た後、spawn される per-connection クロージャ内で raw stream がそのまま `TlsAcceptor::accept`（TLS）または `serve_conn`（平文）へ流れる。PROXY v2 ヘッダはこの間——**TLS handshake の前**——に read する以外の位置がない。

## Decision

1. **manifest**: `[listen.proxy_protocol]` セクションを新設する。`trusted = ["<CIDR>", ...]` は**必須**（空・欠落は `plecto validate` / 起動時 validation error）。セクションが存在するときのみ有効。「宣言なしに信用しない」（deny-by-default, P4）の一貫形であり、有効化 = 全 peer 受入（NGINX 同型）は却下した。
2. **受理規則（すべて fail-closed = 接続切断）**:
   - trusted CIDR **外**の peer からの PROXY ヘッダ → 切断（ログに fault コード）。
   - trusted CIDR **内**の peer でヘッダ欠落・署名（`\r\n\r\n\0\r\nQUIT\n`）不一致・パース不能・長さ超過 → 切断。トランスポート素通しはしない。
   - v2 の `LOCAL` コマンド → 接続の実 peer をそのまま使用（LB health check 互換。仕様どおり）。`PROXY` コマンドで TCP over IPv4/IPv6 以外のプロトコル → 切断。
3. **v2 バイナリ形式のみ受理する（v1 テキスト形式は不採用）。** 判別が確実（12 バイト固定署名）、長さが自己記述（16 バイト固定部 + 宣言長）で有界読みが単純、主要 LB は v2 送信可能。読み込みは宣言長を上限 2 KiB で clamp し、既存の header-read timeout 系と同じ deadline 下で行う。パーサは I/O を持たない純関数（`parse_proxy_v2(&[u8]) -> Result<..., ...>`）として切り、**fuzz ターゲットに追加する**（[[000054]] の残余 4 ターゲット + 本件で 5）。データプレーン panic 禁止（P11）。
4. **peer 差し替えの意味論**: 復元した src address を `serve_conn` 以降の `peer: SocketAddr` として渡す。これにより (a) native rate limit の client-IP キー、(b) `set_forwarded` が再発行する `X-Forwarded-For` / `X-Real-IP`、(c) Maglev `SourceIp`、(d) access log の peer、が一括で real client に揃う。edge モデルの剥がし・再発行ロジック自体は無変更。
5. **スコープは TCP listener（h1/h2）のみ。** HTTP/3 の独立 UDP listener は対象外とし、h3 を前段 LB 配下で使う構成は「LB を経由しない直接受け」または k8s `externalTrafficPolicy: Local` を運用ガイド（Multi-Replica Operations Guide, [[000056]] R7）に明記する。
6. **実装は公開仕様書（proxy-protocol.txt）のみから行う。** 特定実装（HAProxy / Envoy 等のソース）の移植はしない。

検討した代替案と却下理由:

- **trusted-hops 型 XFF 復元** — [[000056]] で却下済み（ヘッダ層・設定ミス = 偽装可能性・edge モデルの単純さを破壊）。
- **有効化 = 全 peer 受入（NGINX 同型）** — 防御を規約（ネットワーク層で LB 以外が届かない前提）に置くことになり、構造で強制する原則（P4）と不整合。却下。
- **v1 テキスト形式の併載** — パース面がもう一つ増える。送信側で v2 を選べない主要 LB は確認できない。却下（**再検討条件**: v1 しか送れない実需 LB が現れたら追補 ADR）。
- **UDP(QUIC) への PROXY v2 適用** — 前段 LB 側の実務サポートが一般的でなく、検証できない主張になる（P9 / §3.4 反証可能性と衝突）。却下し、TCP 側の実測（R6 ハーネス）で主張を固定する。

## Consequences

### Pros
- [[000053]] / [[000054]] の「前段 LB + N レプリカ」標準形で、per-client-IP rate limit・`SourceIp` Maglev・access log・XFF 再発行が宣言どおりの意味論を取り戻す。
- 審査票の定番項目（PROXY protocol 対応）に Yes で答えられる。
- 復元は接続層で完結し、chain・filter・route 選択のコードは無変更。

### Cons / Tradeoffs
- listener に untrusted 入力のパース面が一つ増える（緩和: 純関数化 + fuzz + 有界読み + fail-closed 切断）。
- `[listen]` は現在単一のため、有効化は listener 全体に効く（TLS/平文の別なし）。マルチ listener 化の際に per-listener 化する。
- **再検討条件**: (a) v1 形式の実需、(b) h3 側の identity 復元手段の業界標準化（QUIC-aware LB / PROXY over UDP の普及）、(c) mTLS スライス実装中に listener 設定と交差した場合の同一スライス化判断（[[000056]] に既記）。
