# バッファ済みリクエストボディを再生可能（replayable）として retry 対象に含める

## Status
proposed（判断は 2026-07-06 の grill セッションで合意済み・**未実装**。着手時に accepted へ更新する。実装は Phase 1、[[000056]] の序列に従う）

## Date
2026-07-06

## Affected Components
- server/forward — `forward_with_retry(..., bodyless: bool, body: Option<ReqBody>, ...)` の一回限りボディ（`body.take()` で一度だけ move）を、再生可能ボディを別変種として持つ形へ拡張
- server/retry — `may_retry` / `should_attempt_retry` の `bodyless` パラメータを `replayable`（`bodyless || buffered`）に一般化
- server/proxy — `on-request-body` 通過後のボディ（現在 `Vec<u8>` → `req_full` で `ReqBody` 化）を `Bytes` として保持し replayable で渡す

## Context

[[000023]] / [[000030]] の retry 設計は「streamed body は一度 upstream へ流れ始めたら再生不能」という前提の上に立ち、retry 可否の述語に `bodyless`（`body.size_hint().exact() == Some(0)`）を用いてきた。これは streaming 経路では今も正しい。

しかし [[000025]] / [[000038]] で `filter-body` ルートのボディは **16 MiB 上限でメモリに完全バッファ**されるようになった（buffer-then-decide）。バッファ済みボディは定義上再生可能であるにもかかわらず、retry 述語は冒頭で計算した `bodyless` に固定されたままのため、**connect failure（upstream に届いてすらいない失敗）ですら別 instance への retry が行われない**。前提（再生不能）が buffered 経路では成立しなくなっており、2026-07-06 のコードレビューが P1 として特定した（`forward_with_retry` は現在 `body.take()` により bodied リクエストを一度だけ move する構造）。

## Decision

1. **ボディの受け渡しを三値にする**: `None`（bodyless）/ `OneShot(ReqBody)`（streamed・再生不能、現行と同じ一度限り move）/ `Replayable(Bytes)`（バッファ済み）。`reads_body` 経路では filter 通過後の `Vec<u8>` を `Bytes` に変換して `Replayable` で渡し、attempt ごとに `Full<Bytes>` を組み立てる。`Bytes` は参照カウント共有のため**再送でボディのメモリコピーは発生しない**（借用優先・無駄 clone 禁止の規約と整合）。
2. **retry 判断表は変えない。述語だけを差し替える**: `bodyless && …` を `replayable && …` に一般化する（`replayable = bodyless || buffered`）。すなわち——
   - connect failure（upstream 未達）: 全 method で retry 可（[[000023]] どおり。replayable なら body 再送可）。
   - per-try timeout: idempotent method のみ（RFC 9110 準拠、[[000031]]）。
   - retriable-5xx（502–504 gateway-error class）: idempotent method のみ・別 healthy instance へ（[[000030]] どおり。outlier detection と軸を混ぜない原則も不変）。
3. **streamed（`OneShot`）経路の挙動は一切変えない。** 変更は「メモリに載っているものを使う」だけであり、新たなバッファリングは導入しない（[[000038]] のゼロコピー原則を破らない）。
4. Upgrade（tunnel）リクエストは従来どおり retry 対象外（handshake の意味論、[[000048]]）。

検討した代替案と却下理由:

- **現状維持** — 「ボディは既に replayable なのに connect failure でも捨てる」非対称は、retry 層（[[000023]] / [[000030]]）が自ら定義した判断基準（届いていない失敗は安全に再試行できる）に照らして説明がつかない。却下。
- **retry のために全ルートでボディをバッファする** — [[000038]] が契約の形（`on-request-body` export の不在）から導出したゼロコピー継続を破壊する。却下。
- **replayable でも timeout / 5xx retry まで広げる（non-idempotent 含む）** — 再生可能性と冪等性は別問題。upstream に届いた可能性がある失敗の再送は method の冪等性が支配する。却下（判断表不変の理由）。

## Consequences

### Pros
- `filter-body` ルート（認証・検査系フィルタの主戦場）が、pure-proxy ルートと同等の connect-failure 耐性を得る。
- 追加メモリ・追加コピーなし（`Bytes` 参照共有）。streamed 経路への影響ゼロ。
- 述語の一般化は型で表現され（三値 enum）、`match` 網羅性でコンパイル時に検査される。

### Cons / Tradeoffs
- 再送は upstream への負荷再印加である——既存の bounded retry（max_retries・full-jitter backoff・overall deadline）の枠内に収まるため新たな bound は不要だが、buffered ボディの再送がこの枠を共有することをテストで固定する。
- **再検討条件**: M3 で `stream<u8>` ボディが既定化した際、buffer-then-decide 経路自体が縮小する。その時点で `Replayable` 変種の存続を再評価する（streaming 側に「先頭 N バイトまでは replayable」の概念を持ち込むことはしない）。
