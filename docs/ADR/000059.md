# HTTP/3 に GOAWAY graceful drain を実装し、readiness 契約と tunnel 可観測性を同梱する

## Status
accepted（判断は 2026-07-06 の grill セッションで合意、同日実装済み。実装は Phase 1、[[000056]] R5。
実装時の確定事項: drain window も `[listen.drain] window_ms` として設定化（[[000039]] の projected を履行、
既定 30 s は従来定数のまま）、readiness 猶予は `[listen.drain] readiness_grace_ms`（既定 0）、GOAWAY は
single-step `shutdown(0)`、運用ガイドは `docs/operations.md`（+ `.ja.md`）を新設して判断表を収載。
h3 0.0.8 の実装都合として、server 発 `shutdown` 後の `accept()` は client 側の close なしには
`Ok(None)` に到達しない（done 判定が `recv_closing` をゲートにする）ため、in-flight 完走の検知は
接続タスク自身の JoinSet 追跡で行う）

## Date
2026-07-06

## Affected Components
- server/h3/endpoint — drain flip での即時 close（現行: accept loop return + リクエストループ break で QUIC 接続ごと落とす）を、GOAWAY 送出 + in-flight 完走に変更
- server/admin — `/readyz`（現在は無条件 "ready" の静的 200）を drain watch に接続
- server/metrics — `plecto_tunnels_active` ゲージと tunnel 転送バイト counter を追加
- server/proxy — tunnel spawn 部に RAII ゲージガードを追加

## Context

[[000039]] で SIGTERM/SIGINT の drain は導入済みだが、コード内コメントが自認するとおり **h3 は「新規拒否 + 既存接続 close」であり、TCP 側のような in-flight 完走の drain window がない**（per-request GOAWAY draining は follow-up と明記されている）。SaaS 水準（[[000054]]）ではローリングデプロイのたびに h3 クライアントの瞬断として観測される。

実装手段は既に依存の中にある: h3 crate はサーバ側 Connection に `shutdown(max_requests)`（RFC 9114 の GOAWAY frame 送出、以後の新規リクエストを拒みつつ in-flight を完走させる）を公開しており、Plecto は現在これを呼んでいない。

併せて二つの隣接ギャップを同一スライスで閉じる:
1. **readiness 契約の不在** — `/readyz` が常に 200 のため、front LB がレプリカを外すより先に drain が始まり得る。無瞬断は「not-ready 先行 → LB が外す → drain」の順序があって初めて成立する。
2. **tunnel の不可視性** — WebSocket tunnel（[[000048]]）は 101 応答時点で in-flight ゲージと RED tally から消えるが、breaker permit と LB pick は tunnel 生存期間中専有し続ける（リソース会計は正しい）。運用者は「なぜ breaker が開くのか」を tunnel 専有と紐づけられない。drain 判断にも tunnel 数の可視化が要る。

## Decision

1. **h3 GOAWAY drain**: drain flip を受けたら、各 h3 Connection に対し `shutdown(0)`（GOAWAY 送出・新規リクエスト受付 0）を呼び、in-flight リクエストの完走を **TCP 側と同一の drain window** 内で待つ。window 満了後に残る接続は close（fail-closed）。accept loop の新規接続拒否は現行どおり。
2. **readiness 契約**: `/readyz` を drain watch に接続し、drain 開始と同時（またはそれ以前）に 503 を返す。**契約として文書化する順序**: SIGTERM 受領 → `/readyz` = not-ready → （front LB の health check 間隔ぶんの猶予）→ GOAWAY / TCP drain → window 満了で exit。猶予は `[listen]` 系設定で宣言可能にする（既定は即時 not-ready + drain 並行——LB の除外が間に合わない構成では猶予を明示設定する。運用ガイド R7 に判断表を置く）。`/healthz`（liveness）は drain 中も 200 のまま（restart loop を誘発しない）。
3. **tunnel 可観測性**: `ServerMetrics` に `tunnels_active`（ゲージ、RAII ガードで increment/decrement——`InFlight` と同じキャンセル安全パターン。h2 RST_STREAM での future drop でも Drop で確実に減算）と、tunnel 終了時に加算する方向別転送バイト counter（`plecto_tunnel_bytes_{down,up}_total`）を追加する。ガードは tunnel task に `_permit` / `_pick` と並べて move する。
4. drain の全経路（TCP in-flight / TCP tunnel / h3 in-flight）が同じ window 設定を共有することをテストで固定する。tunnel は drain で能動的に close される現行挙動を維持（長寿命接続に無期限の window は与えない）。

検討した代替案と却下理由:

- **h3 は Alt-Svc fallback（TCP へ再ダイヤル）に任せて据え置く** — 現行コメントの立場だが、クライアント側の再試行実装に依存し、[[000054]] の「無瞬断ローリングデプロイ」を自分の実装で保証できない。却下。
- **readiness を LB 側設定（外部 health check の工夫）だけで解決** — Plecto 側に not-ready を表明する口がない限り、LB は drain 開始を知る手段がない。却下。
- **tunnel を RED メトリクスに合算し続ける（handshake 1 リクエスト扱いのまま可視化しない）** — リクエスト計数としては正しいので変えない。不足しているのは専有の可視化であり、別ゲージが正しい配置。現行の記録方式は維持。

## Consequences

### Pros
- ローリングデプロイで h3 クライアントも無瞬断になり、[[000054]] (b) の「複数レプリカ運用の意味論」が h3 込みで成立。R6 ベンチハーネスの「1 レプリカ kill 時の無瞬断」実測（[[000056]]）で反証可能な形になる。
- front LB との連携順序が推測でなく契約（文書 + 設定）になる。
- breaker 開放・LB 偏りの診断に tunnel 専有が使えるようになる。

### Cons / Tradeoffs
- drain window の間、旧プロセスは h3 接続を保持し続ける（メモリ・fd 専有。window は有界なので bound は明確）。
- readiness 猶予の既定値は「即時 not-ready」であり、health check 間隔が長い LB では設定なしだと瞬断が残り得る——判断表を運用ガイドに置くことで対処（設定の自動検出はしない。宣言的静的設定の原則 P7）。
- **再検討条件**: h3 crate の shutdown API が per-request draining の粒度を変えた場合（現在 0.0.x 系）、追従を Cargo.lock pin + パッチ追従方針（[[000021]] と同型）で吸収する。
