# filter に outbound TCP capability（wasi:sockets）を deny-by-default ＋ per-filter allowlist ＋ SSRF ガードで貸与する

## Status
accepted（判断は 2026-07-06 の grill セッションで合意、2026-07-07 実装着地。実装は Phase 2 の Redis global filter（[[000061]]）の前提スライス。実装時の精緻化は「実装追補」節）

## Date
2026-07-06

## Affected Components
- host — feature `outbound-tcp`（off-by-default）: `wasi:sockets/{tcp,network,instance-network,ip-name-lookup}` を、接続先検問（`socket_addr_check` 相当）と名前解決検問を差し込んだ WasiCtx で貸与
- control/manifest — `[filter.outbound_tcp]`: `allow = [{host, port}]`（完全一致、deny-by-default）、`allow_private`（CIDR オプトイン）、`connect_timeout_ms` / `max_connections` の資源上限
- host/outbound — 既存の resolved-IP 分類（`classify`: OWASP アルゴリズム準拠、metadata/loopback/link-local はオプトイン不能の床）を共用

## Context

[[000056]] / ロードマップ Program R の要石である「global 分散レートリミット filter」（[[000061]]）は Redis を正とするが、Redis は RESP/TCP を話す。extension plane の外向き能力は現在 `outbound-http`（[[000036]]、wasi:http、off-by-default）**のみ**であり、**現行の能力面では Redis filter は構造的に書けない**——本 ADR はこの矛盾（grill セッションで検出）の解消である。

前例は二つ揃っている:
1. **検問付き外向き能力の型**は [[000036]] が確立済み: deny-by-default + per-filter allowlist + ホスト側 DNS 解決 + resolved-IP 分類 + IP ピン（rebinding 遮断）+ 資源上限 + feature gate。
2. **既知の契約形に合わせる**原則は host-counter（`wasi:keyvalue/atomics` 同形、[[000002]] 系）で確立済み: 独自 interface を切らず標準語彙を使えば、ゲスト言語側の既存ライブラリ（Redis クライアント等）が std → wasi:sockets 経由でそのまま動く可能性を得る。

## Decision

1. **wasi:sockets の標準語彙をそのまま貸す**（独自 `plecto:outbound-tcp` interface は切らない）。貸与は off-by-default の feature `outbound-tcp` に隔離し、manifest に `[filter.outbound_tcp]` を宣言した filter にのみ Linker へ追加する（能力の存在自体が per-filter。1 capability = 1 貸与単位の原則は「wasi:sockets 一式」を単位として維持）。
2. **検問は三層**（[[000036]] と同型・実装は同じ `classify` を共用）:
   - **名前解決検問**: `ip-name-lookup` は allowlist 内の host 名のみ解決を許し、解決結果はホスト側で分類してから返す。
   - **接続先検問**: connect ごとに宛先 IP:port を検査——(a) reserved floor（cloud metadata / loopback / link-local / v4-mapped 正規化）は**オプトイン不能で常時拒否**、(b) private レンジは `allow_private` CIDR オプトイン、(c) IP は「allowlist 内 host をホスト自身が解決した結果」に一致すること（IP ピン——guest が解決を迂回して任意 IP へ connect する経路を閉じる）。
   - **資源上限**: 接続数 `max_connections`・`connect_timeout_ms`。超過・タイムアウトは fail-closed でエラー返却（P4／P11: filter の判断へ、worker は巻き込まない）。
3. **feature 無効ビルドで `[filter.outbound_tcp]` が宣言されたら validate で fail-closed に拒否**（`outbound-http` の既存規則と同一）。
4. wasi:sockets を import する guest は zero-WASI（Tier A、[[000055]]）ではなくなる——この能力を使う filter は wasm32-wasip2 ビルドとなり、最小 WASI 貸与（[[000063]]）と同じ貸与基盤に乗る。Tier 表（writing-a-filter.md）にこの帰結を明記する。
5. **UDP・listen（受動 open）は貸さない。** 貸与は能動 TCP connect のみ。

検討した代替案と却下理由:

- **独自 `plecto:redis` capability** — 用途特化の interface は「役割が要求する床」ではなく「特定製品への追従」であり [[000029]] の役割駆動に反する。汎用 TCP + allowlist の方が WAF 連携・memcached 等にも同じ検問で応用が利く。却下。
- **outbound-http + HTTP ブリッジ（webdis / REST 型 Redis）** — 能力面は増えないが、セルフホスト利用者に追加コンポーネントの常駐を強いる。単一バイナリ・運用の単純さの価値序列に照らして本末転倒。却下（構成としては可能な旨をガイドに残す）。
- **global レートリミットの native 実装** — [[000053]] で declined 済み。却下。
- **wasi:sockets を検問なしで貸す** — deny-by-default（P4）の放棄。論外として記録のみ。

## Consequences

### Pros
- [[000061]] の Redis filter が標準語彙の上で書ける。ゲスト言語の既存 TCP/Redis クライアントライブラリが再利用できる可能性が高い（polyglot の実用性に直結）。
- SSRF ガードが outbound-http と単一実装に集約され、検問の抜けが二系統に分散しない。
- 能力の追加が [[000036]] で確立したパターンの二例目となり、「危険な能力は feature gate + allowlist + 床」の型がプロジェクトの再利用可能な形式として固まる。

### Cons / Tradeoffs
- 能力面の実面積は増える（緩和: off-by-default feature + per-filter 宣言 + 三層検問 + 「署名ゲート・SSRF 床の迂回不能性」を E2E テストで固定——§3.4 反証可能性）。
- TLS over TCP（rediss:// 等）はゲスト側実装に委ねる（ホストは平文 TCP の検問のみ。guest 内 TLS はサンドボックス内で完結し検問と直交）。
- **再検討条件**: (a) `wasi:sockets` の 0.3 系での形状変化（wasip3 収斂時に追補）、(b) UDP 需要（DNS/QUIC を filter から話したい実需）が出たら別 ADR、(c) [[000036]] の wasi:http 収斂ゲートと同時に、outbound 系 feature の既定ビルド入りをまとめて再判断。

## 実装追補（2026-07-07、実装時 grill で確定）

実装に先立つ wasmtime-wasi 46.0.1 の実地調査で、提案時の記述を四点精緻化した。いずれも三層検問の**不変条件は不変**で、実現機構と資源上限の意味論だけを確定させたものである。

1. **名前解決検問は自前実装**: upstream の `wasi:sockets/ip-name-lookup` ホスト実装にはホスト名フィルタのフックが存在しない（bool 許可のみ）。よって host が同 interface の Host trait を自前実装して Linker に差す——allowlist 内ホスト名のみ解決を許し、ホスト側 Resolver で解決し、全結果を `classify` で検問（一つでも blocked なら解決ごと失敗＝rebinding/混在レコード遮断）した上で per-Store の **pinned set**（IP → 解決元ホスト名）へ記録する。
2. **IP ピンは connect 検問側で照合**: `socket_addr_check` は接続先 `SocketAddr` しか見えない（ホスト名不可視）ため、ピンは「宛先 IP が (a) allowlist の IP リテラル、または (b) そのポートの entry のホスト名に対して host 自身が解決・記録した pinned IP であること」を connect ごとに照合する形で実現する。`TcpBind` と全 UDP 用途は拒否し、UDP interface はそもそもリンクしない（能力の不在 > 実行時 deny）。
3. **`connect_timeout_ms` は `io_deadline_ms` に置換**: p2 の TCP connect は guest が pollable を駆動するため、per-connect timeout の正確な実装には tcp interface 全体の delegation wrapper を要し、しかも connect だけ bound しても「connect 成功後の無応答 read での無限ブロック」（HTTP では `total_timeout` が担った穴）を塞げない。よって manifest は `io_deadline_ms`（既定 5000ms・上限 30000ms）とし、outbound-tcp filter の **hook 呼び出し全体を wall-clock timeout で bound** する（epoch はホスト内 I/O ブロックに届かない）。超過時は呼び出しをエラー化して instance を破棄（fail-closed、pool の trap 経路に合流）。
4. **`max_connections` は per-request の connect 予算**: `socket_addr_check` からは close が見えず「今生きている接続数」は正確に追えない。意味論は「1 リクエスト中に許可される connect 数」（`begin_request` でリセット、既定 4・上限 64）と定め、リクエスト内ファンアウトを bound する。累積 FD は pool の `max_requests_per_instance` recycle が bound する。また **pooled instance の跨リクエスト接続保持は許容**する——「フィルタはステートレス」（[[000002]]）が禁じるのは判断状態であり、検問済み接続は recycle が回収する通信資源（[[000061]] の Redis filter が毎リクエスト connect+AUTH を払わないための前提。用語は host CONTEXT.md「Persistent connection」）。

併せて、命名の対称性のため既存の `[filter.outbound]`（[[000036]]）を `[filter.outbound_http]` へ改名した（`LoadOptions::with_outbound` → `with_outbound_http` 同様）。旧キーは `deny_unknown_fields` により fail-closed にパースエラーとなる（v0.1.x・invitation-only の現段階で唯一安く直せるタイミングと判断）。

検証（§3.4 反証可能性の固定）: E2E（実 wasip2 guest `filter-tcp-gate`）で (a) 未許可名の解決前拒否、(b) 許可名→loopback 解決の floor 遮断、(c) 名前解決を迂回した IP リテラル connect の遮断、(d) 検問通過チェーン（name→resolve→pin→connect）の成功と connect 予算の per-request リセット、を固定。unit で floor のオプトイン不能・pin 必須・port 跨ぎペアリング拒否・bind/UDP 拒否・予算消費を固定。CI は `--features outbound-tcp` でこのスイートを常時実行する。
