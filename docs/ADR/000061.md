# レートリミットを local floor × global filter の二層モデルとして完成させ、Redis reference filter を提供する

## Status
proposed（判断は 2026-07-06 の grill セッションで合意済み・**未実装**。着手時に accepted へ更新する。実装は Phase 2、[[000060]] に依存。[[000056]] R2×F3 の交点＝最優先成果物）

## Date
2026-07-06

## Affected Components
- docs/hardening（→ Multi-Replica Operations Guide, [[000056]] R7）— 「local で各レプリカを守り、global で合算上限を守る」を第一推奨に。「実効レート = 設定 × N」は local 単独時の性質として存置
- examples/filters/filter-ratelimit-redis — reference 実装（第一弾 Rust。多言語版は Program F3 で拡充）
- performance/bench — N-replica ハーネス（[[000056]] R6）で (i) local 単独 ≈ 設定×N、(ii) 二層併用 ≈ 設定値、の実測を回帰対象に追加

## Context

Plecto のレートリミットは既に二つの部品を持つ: native floor（[[000033]]、route / client-IP 粗粒度、`apply_bucket` による host-native token bucket）と、「共有状態は拡張プレーンへ」という宣言（[[000053]] Fork 6）。欠けているのは、この二つを**一枚の運用モデル**として提供・実測・文書化することと、宣言の「実物」となる global filter である。

確定事実（2026-07-06 検証済み）: 業界確立形は **local + global の二層併用**である。Envoy Gateway は同一ルートに local（各インスタンス独立、「N レプリカで実効 N 倍」を公式明文化）と global（外部 Rate Limit Service + Redis バックエンド）を同時設定するパターンを公式化しており、v1.3 以降の cost 指定は **global 限定**。Plecto の既存判断はこの業界形とちょうど同型であり、かつ Plecto では filter そのものが RLS になる（別プロセス不要——単一バイナリ原則の勝ち筋。「外部 RLS を別バイナリで作らない」は不変）。

なお契約側の cost 対応は**既に済んでいる**: `host-ratelimit.try-acquire(key: string, cost: u64)`（[[000026]]）。

## Decision

1. **役割の再定義（コード変更なし）**: native floor（[[000033]]）を「**外部呼び出しゼロの即時 flood 遮断（local 層）**」と定義し直す。全リクエストが通る床（テナント非依存・WASM CPU を消費する前に落とす）であり、レプリカ間合算の厳密性は**求めない**。「実効レート = 設定 × N」は仕様として存置し文書化を維持する。
2. **global 層は reference filter `filter-ratelimit-redis` で提供する**（native 化しない）:
   - トランスポートは [[000060]] の outbound-tcp（RESP をゲスト内で実装 or 既存クライアント crate）。
   - **アルゴリズムは一般形の fixed-window counter**（atomic `INCRBY cost` + 初回 `EXPIRE`）を v1 とする。races は Redis の原子性で閉じ、実装は一般的・教科書的手法として自前で書く（特定 OSS 実装の移植はしない）。sliding-window / GCRA への差し替えは filter 内部の判断であり契約に影響しない。
   - **cost 対応を最初から持つ**: 控除量は filter config（固定値）または指定ヘッダ由来の動的値。AI/LLM トークン課金型の用途は「native 化せず filter の応用例として受ける」（[[000043]] の declined を維持）。
   - **key 設計**: route 識別子 + クライアント同一性。クライアント同一性は edge モデルが再発行する `X-Forwarded-For`（実 peer 由来、[[000018]] / [[000057]] 後は前段 LB 配下でも real client）を読む——filter から見て信頼できる値であることが edge モデルの帰結。
3. **バックエンド障害時の挙動は宣言必須（既定なし）**: filter config の `on_backend_error = "deny" | "allow"` を必須とし、未宣言は設定エラーで load 拒否。fail-closed の原則（P4）は「未宣言を通さない」で表現し、deny（Redis 断 = 全遮断）/ allow（local floor が残る前提の可用性優先）の実トレードオフは運用者の明示判断に委ねる。判断表は運用ガイドへ。
4. **二層併用を第一推奨形として文書化・実測する**: hardening ガイド（→ R7 Guide）の推奨を「local floor で各レプリカを守り、global filter で合算上限を守る」に昇格。R6 ハーネスで (i) local 単独 = 実効 ≈ 設定×3（3 レプリカ）、(ii) 併用 = 実効 ≈ 設定値、を数値で取り、主張と実測をリンクさせる（[[000055]] が polyglot で行った aspirational → verified の転換をレートリミットの主張で反復する）。

検討した代替案と却下理由:

- **global を native に持つ（共有ストアへホストが直結）** — [[000053]] で declined 済み。ノードローカル宣言は本 ADR でも一切緩めない。却下。
- **外部 RLS プロセス（Envoy の ratelimit サービス相当）を別バイナリで提供** — 単一バイナリ原則に反し、filter が RLS になれる Plecto の差別化を自ら捨てる。却下。
- **fail 挙動に既定値を置く（deny 既定 or allow 既定）** — deny 既定はキャッシュ障害を全停止に化けさせ、allow 既定は P4 の例外を暗黙に作る。どちらの既定も「運用判断の省略」であり、宣言必須が最も Plecto 的。却下。
- **sliding window / GCRA を v1 から** — 精度は上がるが実装・説明の複雑さが増す。v1 は fixed window で反証可能な単純さを取り、精度要件は filter 差し替えで受ける。却下（filter 内部の進化として開放）。

## Consequences

### Pros
- [[000053]] の宣言（共有状態は拡張プレーンへ）が reference 実装という実物を持ち、導入審査の追問に「実装があります」で答えられる。
- マルチレプリカ対応・Fork 6 の実証・reference filter（Program F3）の三役を一本の成果物で兼ねる。
- 業界確立形（Envoy Gateway の公式パターン）と同型であることを一次ソース参照で説明できる。

### Cons / Tradeoffs
- global 層の精度は fixed window の粒度（窓境界のバースト）を受け入れる——local floor が瞬間バーストの床を別途抑えるため、二層併用時の実害は有界。
- Redis という外部依存はあくまで**利用者の選択**（filter を載せなければ存在しない）だが、reference の運用文書・監視指針まで含めて提供する責任を負う。
- **再検討条件**: (a) 一般的アルゴリズムの選択（fixed window）は R6 実測でバースト特性が問題化したら filter 内で GCRA へ、(b) cost の標準形（ヘッダ名等）は利用実態が出てから互換ポリシー（[[000064]]）の枠で固定、(c) wasi:keyvalue 系の標準が Redis 接続を覆う日が来たら outbound-tcp 直結と比較再評価。
