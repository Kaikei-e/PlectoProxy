# TLS session ticket 鍵（STEK）のレプリカ間共有を、証明書束縛と mTLS 時 fail-closed 付きのオプトインとして導入する

## Status
accepted（2026-07-08 実装完了。実装時の精密化・逸脱は本文末尾「実装追記」参照）

## Date
2026-07-06

## Affected Components
- control/tls — 現行: process-lifetime の `OnceLock` ticketer（aws-lc-rs `Ticketer` = RFC 5077 型自己暗号化、6h ローテーション/12h 受理窓、鍵はプロセスメモリのみ・ノードローカル）。共有モードでは file-based 鍵素材から派生する自作 `ProducesTickets` 実装に差し替え
- control/manifest — `[resumption] stek_file = "<path>", max_age_hours = 24`（オプトイン）。TCP/QUIC 両 config に一括適用（現行の共有 ticketer 構造を踏襲）

## Context

[[000052]] は stateless resumption を導入し（full handshake 比 +35% の実測）、**チケット鍵のノードローカル性を不変条件**とした。しかし前段 LB がラウンドロビンする N レプリカ構成（[[000053]] / [[000054]] の標準形）では resumption ヒット率が 1/N に落ち、+35% の性能を取り逃す。nginx / NGINX Unit は共有鍵ファイル + 外部ローテーションを公式手順としており、レプリカ間共有は確立慣行である。

一方で 2025 年に落とし穴が体系化された（2026-07-06 一次ソース検証済み）:
- **nginx CVE-2025-23419**: name-based virtual host 間で SSL セッションが再利用され client cert 認証をバイパス。**TLSv1.3 固有**・resumption 有効時（fixed 1.26.3 / 1.27.4）。
- **Apache CVE-2025-23048**: mod_ssl の vhost 間 session resumption による access control バイパス（2.4.35–2.4.63、fixed 2.4.64）。
- Hebrok et al.「We Really Need to Talk About Session Tickets」（**USENIX Security 2023**）が STEK 起因の復号・なりすましリスクを大規模実証。
- 業界の修正形: Fastly は「チケットを発行証明書に束縛」、Cloudflare は「client 認証有効時は resumption 無効化」。psk_dhe_ke 強制（resumption でも ECDHE 毎回）は harvest-now-decrypt-later 文脈で重要。

つまり「共有するか否か」より先に「**共有しても資格が越境しない構造**」を先に固定するのが 2026 年水準であり、Plecto は rustls の性質（psk_dhe_ke のみ / 0-RTT 拒否 = [[000052]] の不変条件）で土台を既に持つ。

## Decision

1. **既定は現行どおり per-node**（プロセスメモリ内・6h ローテーション）。front LB に session affinity があれば per-node で足りる——共有はあくまで**オプトイン**であり、判断表（affinity か共有か）を Multi-Replica Operations Guide（[[000056]] R7）に置く。
2. **オプトイン共有モード**: `[resumption] stek_file` に鍵素材ファイル（外部ローテーション前提）を宣言。実装は rustls の `ProducesTickets` を自作し、AEAD（AES-256-GCM）鍵を **HKDF(file 鍵素材, info = 発行 listener の証明書 identity)** で派生する。
   - **(a) 証明書束縛**: HKDF info に cert fingerprint を含めることで、vhost/listener を跨いだチケット受理を**暗号学的に**遮断する（Fastly 型。CVE-2025-23419 / 23048 の越境形を構造で殺す）。
   - **(b) mTLS 交差の fail-closed**: downstream client cert 検証（mTLS スライス、[[000054]] 序列先頭）が有効な listener では共有 STEK を **validation エラーで拒否**（Cloudflare 型緩和を既定に。mTLS 実装が先に来ても後から来ても、この交差規則が先に効く）。
   - **(c) 不変条件の維持**: psk_dhe_ke のみ / 0-RTT 拒否 / stateful session cache 無効（`NoServerSessionStorage`）は共有モードでも一切変えない。
   - **(d) 鍵ファイル規律**: パーミッション検証（group/other 読取可なら fail-closed 起動拒否）、mtime 監視による再読込、`max_age_hours`（既定 24、RFC 5077 の運用推奨に整合）超過で fail-closed に resumption 停止（full handshake へ縮退——可用性は落ちない）。
3. [[000052]] の「チケット鍵ノードローカル」不変条件は、本 ADR により「**既定はノードローカル。共有は (a)–(d) の制約下のオプトイン**」へ改訂される（[[000052]] に注記を追加する）。

検討した代替案と却下理由:

- **stateful session cache のレプリカ間共有（Redis 等）** — 共有ストアへの TLS 秘密の常駐は攻撃面・運用面ともチケット鍵ファイルより広い。rustls 構成も `NoServerSessionStorage` を前提に固めてある。却下。
- **resumption 自体を N レプリカ構成で無効化** — +35% の実測性能を恒久放棄する。affinity 構成という無料の代替がある以上、既定 per-node + 判断表で足りる。却下。
- **証明書束縛なしの単純共有（nginx 同型）** — 2025 年の CVE 群はまさにこの形の帰結。確立慣行であっても、判明済みの落とし穴を新規実装が再現する理由はない。却下。
- **Plecto 自身が鍵ローテーションを配る（ノード間配布機構の内蔵）** — native の分散機構を持たない原則（[[000053]]、single-node first）に反する。ローテーションは外部（cron / operator）責務とし、ファイルという最小インターフェースだけを持つ。却下。

## Consequences

### Pros
- N レプリカで resumption ヒット率が回復し、[[000052]] の性能便益がマルチレプリカ標準形でも成立（R6 ハーネスで「共有 STEK 有無のヒット率・handshake スループット差」を実測、[[000056]]）。
- 2025 年 CVE 群の越境形を「設定を正しく書く」ではなく鍵派生の構造で遮断——審査票の TLS 設問に一次ソース付きで答えられる。
- 共有しない構成が引き続き正当な既定であり、運用の単純さを損なわない。

### Cons / Tradeoffs
- 自作 `ProducesTickets` は暗号実装面が増える（緩和: AEAD/HKDF は aws-lc-rs のプリミティブのみ使用、鍵スケジュールを property テストで固定、越境不能テスト——別 cert の listener が発行したチケットを拒否する——を E2E で固定）。
- 外部ローテーションの品質は運用者依存（緩和: max_age fail-closed が上限を強制、ガイドに手順例）。
- **再検討条件**: (a) 実装時に rustls の `ProducesTickets`/cert identity の取り回しが証明書束縛と噛み合わない場合はスライスを落とし per-node + affinity 推奨に一本化（事前確定済みの縮退線）、(b) rustls 本体が cert-bound ticket を標準装備したら自作実装を退役、(c) mTLS スライス側の設計確定時に (b) 規則の粒度（listener 単位）を再確認。

## 実装追記（2026-07-08、grill セッションで確定）

実装は `control/src/stek.rs`（`SharedStekTicketer`）+ `control/src/manifest/resumption.rs`。本文の決定を
次のとおり精密化・一部改訂した:

1. **AEAD → AES-256-CBC + HMAC-SHA-256（Encrypt-then-MAC）へ改訂**。本文の AES-256-GCM は採用しない。
   根拠は 2 点（2026-07-08 一次ソース検証）: (a) 複数レプリカが同一派生鍵でランダム 96-bit nonce を打つ
   構成では GCM の nonce 衝突が鍵回復級に壊滅的で、発行量の上限管理が外部ローテーション品質に依存して
   しまう。CBC のランダム IV に同種の崖はない。(b) rustls 本体も partitioning oracle 対策（rustls#2023、
   PR #2066）で GCM から key-committing な CBC+HMAC へ移行済みであり、鍵素材ファイルが規律違反で弱い
   場合の oracle 耐性も得られる。チケット構造は RFC 5077 §4 の一般形
   `key_name(16) ‖ IV(16) ‖ ct ‖ HMAC-SHA-256(先行全体)`（MAC 範囲を全レコードに拡張、EtM）。
2. **証明書束縛の粒度は cert セット**（`[[tls]]` 全 cert の SPKI SHA-256 fingerprint をソート・連結した
   ハッシュを HKDF info に投入）。rustls `ProducesTickets` は config 単位で handshake ごとの SNI cert
   選択を知り得ないため（再検討条件 (a) が想定した噛み合わせ）、per-cert ではなくセット束縛とした——
   越境遮断という目的はこの粒度で達成される。同一 config 内の SNI 越境は rustls 自身の resumption 時
   SNI 照合（0.23.41 `server/hs.rs:57`）が遮断し、E2E テストで outcome を固定済み。
3. **fingerprint 基準は SPKI**（RFC 7469 と同根拠）。同一鍵ペアでの証明書更新でチケットが継続し、鍵
   不一致は full handshake への安全縮退。SPKI は `SigningKey::public_key()` から取得（`keys_match()` が
   cert との一致を保証済みのため X.509 パーサ依存が不要）。
4. **鍵ファイルは生バイト 64B 固定**（`openssl rand 64`）。長さ不一致・group/other 可読は起動拒否。
   key_name も HKDF で派生させるため、ファイル内に構造は持たない（レプリカ間は無調整で一致）。
5. **mtime 監視は lazy throttled 方式**: ticketer 内で handshake 経路に相乗りして最大 10 秒に 1 回
   再読込（content digest で変化検出、mtime granularity に依存しない）。current + previous の 2 epoch
   保持で外部ローテーションの混在期間を吸収。control は sync のまま（監視スレッドなし）。
6. **ticket_lifetime ヒント = max_age_hours**（既定 24h、validation 上限 168h = RFC 8446 の 7 日）。
7. **(b) mTLS 交差規則はスキーマ canary テストで担保**: downstream client-auth 設定は未実装のため、
   manifest JSON Schema に client-auth 系フィールドが出現したら fail するテスト
   （`no_downstream_client_auth_exists_to_cross_shared_stek_yet`）を置き、交差 validation の先行実装を
   強制する。
8. **reload 意味論**: `[resumption]` は content hash に載り、reload で再構築される。鍵が (ファイル,
   cert セット) の純関数なので、入力不変の reload では発行済みチケットが失効しない（[[000052]] の
   OnceLock が担っていた性質を決定性が代替）。previous epoch の記憶だけは reload で失われ、ローテー
   ション直後のチケットが full handshake に縮退する（安全側）。
