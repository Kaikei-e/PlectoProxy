# fat guest への最小 WASI 貸与（Tier B）を導入し、Go（TinyGo）フィルタを解禁する

## Status
proposed（判断は 2026-07-06 の grill セッションで合意済み・**未実装**。着手時に accepted へ更新する。実装は Phase 1 後段〜Phase 2 —— reference 三部作（[[000056]] F3）を Go でも書くための先行投資）

## Date
2026-07-06

## Affected Components
- host — feature `fat-guest`（off-by-default）。`[filter] wasi = "minimal"` の filter にのみ、既存の outbound-http 貸与基盤（proxy interfaces + cli runtime、`host.rs` に実在）を土台に最小 WasiCtx を貸す
- control/manifest — filter エントリに `wasi` フィールド（enum、既定 `none`）
- examples/filters/filter-hello-go — TinyGo `-target=wasip2` + `wit-bindgen-go`（bytecodealliance/go-modules）ビルド。`build.sh` は「wasi: import は許可リスト内のみ」を assert する **Tier B 変種**
- crates/host/tests/polyglot.rs — Go fixture を同一アサーションスイートに追加

## Context

[[000055]] は polyglot 主張を Tier A（zero-WASI: Rust / MoonBit / C / JS）で反証可能化し、**Go は TinyGo の必須 `wasi:cli` import のせいでブロック**——「host が fat guest に最小 `WasiCtx` を貸すか」を独立のセキュリティ判断として名指しで繰り延べた。本 ADR がその判断である。

確定事実（2026-07-06 検証済み）:
- TinyGo の `wasip2` ターゲット + `wit-bindgen-go`（bytecodealliance/go-modules。旧 TinyGo 用ジェネレータは deprecated）で Go の Component は今日ビルドできる。TinyGo は `wasi:cli/command` 系 world を前提とするため、guest 側 world に `include wasi:cli/imports@0.2.0` を足すのがコミュニティ確立のワークアラウンド。
- Go 本体（gc）の Component 対応は wasip3 文脈の projected であり、待つ理由にはしない（TinyGo 経路で先に出す）。
- Go は proxy-wasm 時代からフィルタ作者人口が最も厚い言語であり、解禁はエコシステム上のリターンが最大の一手（[[000056]] F1）。

前例: **貸与基盤は既に実在する**。outbound-http feature は `wasmtime_wasi::p2::add_to_linker_proxy_interfaces_async`（io / clocks / random / stdio。fs なし・sockets なし）+ `add_cli_runtime`（空 WasiCtx で不活性な wasi:cli env/exit）を Linker に足している。本 ADR はこれを独立 feature に切り出し、per-filter 宣言で制御可能にする形式化である。

## Decision

1. **貸与面（deny-by-default の最小集合）**: `wasi:io`、`wasi:clocks`（real monotonic + wall——ランタイム都合の時刻。**契約上の決定的時刻は従来どおり host-clock が担う**旨を writing-a-filter.md に明記。outbound-http 経路の既存貸与と同一の線）、`wasi:random`（real CSPRNG）、`wasi:cli`（env = 空、args = 空、exit）。**fs は不貸与**（preopen なし）、**sockets は不貸与**（outbound-tcp [[000060]] / outbound-http [[000036]] は別能力・別宣言）。
2. **stdout/stderr は host-log へブリッジする**: stdout → `debug`、stderr → `warn`。行単位で配管し、1 行 4 KiB・リクエストあたり 64 KiB を上限（超過は切り捨て + 警告 1 回、CWE-770 系の log flood を bound）。TinyGo の panic メッセージが filter 別ログに乗り、trap の原因が見える——TTFVF（[[000065]]）の三大壁対策と直結する。捨てる（既存の空 WasiCtx と同型）案は「Go filter のデバッグ体験が最悪になる」ため却下した。
3. **契約は変えない**: `plecto:filter` world に `include wasi:cli/imports` は**足さない**（足すと Tier A の zero-WASI 意味論が壊れる）。TinyGo guest は自分の bindgen 用 world（filter + wasi imports の合成）を guest 側テンプレートで持つ。ホストから見れば「filter world を export し、許可リスト内の wasi を import する component」である。
4. **Tier B conformance**: `build.sh` の assert を「`wasi:` import は許可リスト（io / clocks / random / cli）内のみ」に緩和した変種を Go 例に適用し、polyglot スイート（単一アサーション）に Go fixture を追加。許可リスト外 import（fs / sockets / http）は従来どおり instantiate 失敗（構造的 deny）であることをテストで固定する。
5. **trusted / untrusted 両ライフサイクルで許可**（WasiCtx は Store ごと・untrusted は fresh-per-request で毎回新品——[[000011]] / [[000012]] の分岐と直交）。
6. 言語マトリクスを二層で文書化: **Tier A**（zero-WASI: Rust / MoonBit / C / JS）と **Tier B**（minimal-WASI: Go、および wasi を要する将来言語）。サイズ・init コスト・実用度の対照表を writing-a-filter.md に置く。

検討した代替案と却下理由:

- **wasip3 の Go 本体対応を待つ** — projected に依存し、最大の作者人口を持つ言語を無期限にブロックする。TinyGo 経路は今日動く。却下。
- **フル WasiCtx（fs / sockets 込み）を貸す** — deny-by-default（P4）の放棄。論外として記録のみ。
- **`plecto:filter` world 側に wasi imports を include** — Tier A の「zero-WASI = deny-by-default Linker がそのまま instantiate できる唯一の形」（[[000055]]）を全言語で壊す。契約は共有財であり、一言語の都合で汚さない。却下。
- **stdio を捨てる（既存同型）** — 実装最小だが、Go filter の trap 原因が不可視になり「初見者が最初に踏む壁」を一つ増やす。却下。

## Consequences

### Pros
- 最厚の作者人口（Go）に polyglot が開き、reference 三部作（[[000056]] F3）を Rust + Go を含む 3 言語で出せる。
- 貸与面が outbound-http の既存実装と共通基盤になり、「fat guest への貸与」が場当たりでなく単一の検問点を通る。
- stdout/stderr ブリッジで、WASM フィルタのデバッグ体験（三大壁）が言語を問わず改善する。

### Cons / Tradeoffs
- sandbox の面は増える（緩和: off-by-default feature + per-filter 宣言 + fs/sockets 不貸与 + 許可リスト assert + 「許可外 import は instantiate 失敗」の E2E 固定）。wall-clock の real 貸与により、Tier B filter は host-clock 経由でない時刻を読める——決定性が要る policy は host-clock を使う規約とし、conformance で host-clock 使用例を示す。
- TinyGo toolchain は CI での supply-chain pin 対象が増える（MoonBit と同じ論点、[[000055]] 注記の系譜）。
- **再検討条件**: (a) Go 本体（gc）の wasip2/p3 Component 対応が Tier 相当に成熟したら TinyGo 前提を再訪、(b) 貸与面の audit で懸念が出たら Tier B を feature-gate のまま維持し Tier A のみで F3 を成立させる（縮退線、[[000056]] 既記）、(c) wasi:cli の後継（wasip3 の形状変化）で include ワークアラウンドが不要になったら guest テンプレートを簡素化。
