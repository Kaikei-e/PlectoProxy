# WIT 契約を wkg / OCI Artifact で標準配布し、契約互換ポリシーを発効する

## Status
accepted・実装済み（判断は 2026-07-06 の grill セッションで合意済み。実装は 2026-07-09 に着手・完了。
[[000056]] F2 / [[000068]] で Program F の F1 最優先スライスへ再編。reference filter **実体**の OCI 配布は [[000080]] が、互換ポリシーの対外宣言の段階化と契約 1.0 恒久保証は [[000085]] が amend）

実装時点（2026-07-09）で確定した詳細:

- 配布パスは `ghcr.io/kaikei-e/wit/plecto/<package>:<version>`（`namespace_registries` で `plecto`
  namespace を `kaikei-e/wit/` prefix 配下へ再配置。WASI 公式 WIT が `ghcr.io/webassembly/wasi/...`
  へ同じ機構で配布されているのと同型）。
- `wkg` は CI で v0.15.1（sha256 固定、`ci.yml` の既存 polyglot-guest-go job と同一バイナリ・同一
  ピンで共用）。publish はタグ付きリリースごとに独立 job（`gate` のみに従属、binaries/container
  job とは非同期）として走らせ、対象バージョンが同一 digest で既発行なら push をスキップする
  （wkg 自身の再 publish 挙動に依存しない設計）。
- 発行 digest の記録先は **GitHub Release notes**（`gh release edit`）。CHANGELOG.md 本体は従来
  どおり人間が手で編集し、CI からのコミット差し戻しは行わない（push は明示指示があるときのみ、
  というグローバル規律に合わせた選択）。
- 初回 publish 後、ghcr.io パッケージの可視性（Public 化）は手動での一度きりの作業として残る
  （GITHUB_TOKEN 経由の初回 push は既定で private になりうるため）。

## Date
2026-07-06

## Affected Components
- CI（release workflow）— タグ時に `wkg publish` ステップを追加し、発行 digest を CHANGELOG に記録
- wit/ — 配布対象は `plecto:filter@x.y.z`（`filter` / `filter-body` 両 world を含む単一 package）。`plecto:filter-streaming` は実験の旨を明示した別 package として併載
- docs/writing-a-filter.md — 互換ポリシー節を新設（契約の SemVer 運用・併存期間）

## Context

`plecto:filter` の WIT を取得する現行手段は「リポジトリを clone して `plecto/wit/` を読む」しかなく、多言語のフィルタ作者（[[000055]] / [[000063]] で拡大）に Plecto 固有の手順を強いる。

確定事実（2026-07-06 検証済み・ロードマップ §0-4）: WIT パッケージ / コンポーネントを OCI で配布する標準経路は整備済みである——CNCF Wasm OCI Artifact layout と `wkg`（wasm-pkg-tools）により ghcr.io 等へ publish でき、WASI 公式 WIT 自体が ghcr.io/webassembly 配下で同方式配布されている。名前空間→レジストリの解決は `/.well-known/wasm-pkg/registry.json` で宣言できる。

また、フィルタ作者にとって**契約の安定性宣言は最重要のインフラ**である。CHANGELOG に萌芽のある「host は出荷した全契約バージョンを読み続ける」を、配布形の SemVer 運用として独立させる時機でもある。digest 固定の WIT 配布は、Plecto が filter 本体に課している供給網規律（[[000006]] / [[000047]]）の**契約側対応物**という位置づけを持つ。

## Decision

1. **配布**: タグ付きリリースごとに `plecto:filter@x.y.z` を CNCF Wasm OCI Artifact layout で ghcr.io（Kaikei-e 名前空間配下）へ `wkg publish` する。実験系 `plecto:filter-streaming` も「off-by-default feature の実験契約」である旨をアノテーションに明記して併載する。発行した OCI digest は CHANGELOG のリリースエントリに記録する（バイナリの cosign/SBOM 記録と並ぶ、契約の供給網記録）。
2. **初期スコープは ghcr.io のみ**。`/.well-known/wasm-pkg/registry.json` によるプロジェクトドメインでの名前空間解決は、docs サイト / ドメイン整備後の後続スライスとする（利用者は `wkg get` にレジストリ指定で到達可能——ドメイン確保をこのスライスの前提にしない）。
3. **契約互換ポリシーの発効**（writing-a-filter.md に正式節として）:
   - 契約の変更は additive を基本とし、**additive = minor / breaking = major** の SemVer で version を進める。
   - **host は出荷した全契約バージョンを読み続ける**（load 時に component の world バージョンで分岐）。major 更新時も旧 major の受理を最低 2 リリース系列は併存させ、廃止は declined/deferred と同様に ADR で宣言する。
   - `filter` と `filter-body` の二 world 構成・「export の不在が意味を持つ」性質（[[000038]]）は互換性判定の一部である（body world の追加 export は minor、既存 export の署名変更は major）。
4. scaffolding（[[000065]] の `plecto new-filter`）とドキュメントは `wkg get plecto:filter` を契約取得の正規手順として案内し、リポジトリ内 `wit/` は開発の正文（source of truth）として従来どおり維持する（配布物は正文の写し。食い違いはあり得ない構造——publish は CI がタグの `wit/` から行う）。

検討した代替案と却下理由:

- **git submodule / raw URL での配布** — 言語横断の標準ツールチェーン（wkg / wit-bindgen 系）に乗らず、作者に Plecto 固有手順を残す。却下。
- **crates.io のみ（Rust クレートに WIT を同梱）** — Rust 作者にしか届かない。polyglot を一次価値とする以上、配布も言語中立であるべき。却下。
- **well-known registry.json まで一気に** — ドメイン・docs サイトが前提になりスライスが重くなる（grill で確認済み）。ghcr.io 先行で価値の大半（標準ツールでの取得）が出る。後送。
- **契約互換ポリシーを配布と切り離して先に文書だけ発効** — 「どの座標の何を読み続けるか」は配布形が決まって初めて具体になる。同一スライスが正しい。却下。

## Consequences

### Pros
- 任意言語の作者が標準ツールチェーン一発（`wkg get`）で契約を取得でき、Plecto 独自の配布物を読む必要がなくなる（TTFVF [[000065]] の前提部品）。
- 契約の安定性が宣言され、フィルタ資産の寿命が作者から見て予測可能になる。
- 契約にも digest 固定の供給網記録が付き、[[000047]]（自己適用）の一貫性が契約まで届く。

### Cons / Tradeoffs
- リリース CI の供給網対象に wkg toolchain が増える（sha256 pin 対象。MoonBit と同じ管理系譜）。
- 互換ポリシーは将来の設計自由度を縛る約束である——それが目的だが、breaking を選ぶ日のコスト（major 併存期間）を先に引き受ける。
- **再検討条件**: (a) wasi:http 収斂（[[000020]]）実施時は契約 major の最初の実戦——併存運用をこのポリシーどおり回せるかで表の妥当性を検証、(b) wkg / OCI layout の仕様変動（CNCF 側）への追従、(c) ドメイン・docs サイト整備が済んだら registry.json スライスを起票。
