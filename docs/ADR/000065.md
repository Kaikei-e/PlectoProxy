# Filter Dev Kit を確立し、TTFVF（Time To First Verified Filter）15 分を North Star に据える

## Status
accepted（判断は 2026-07-06 の grill セッションで合意済み。2026-07-09 の実装 grill セッションで着手し、
本文に実装スコープの絞り込みを追記。実装は Phase 1〜2（90 日計画）。プロモーション計画側の詳細は内部文書、
本 ADR はプロダクト判断のみを記録）

### 実装スコープ（2026-07-09 grill セッションでの絞り込み）
フルスコープ（`--lang` 5 言語 + エラーコード全面刷新）は一度の実装では終わらないため、最初の縦串を
**Rust 一本**に絞る: `plecto conformance` の抽出 → `plecto new-filter --lang rust` → dev 鍵 →
`plecto dev` の Rust フィルタでの一気通貫。go/moonbit/c/js の `new-filter` テンプレートは同型の後続作業として
明示的に積み残す（サイレントな縮小にしない）。PLECTO-E は decision 5 が名指す三大壁（署名検証失敗・quota
超過・path 正規化拒否）のみを初回対象とする。

以下、各サブ決定の具体化:

- **`plecto conformance` の判定対象は汎用プロパティのみ**——既存 `tests/polyglot.rs` の 7 アサーションは
  `filter-hello` フィクスチャ固有の振る舞い（`x-block` ヘッダで 403 等）に依存しており、任意の第三者フィルタに
  この固有振る舞いを要求できない。CLI が「conformant」と呼ぶのは (1) `plecto:filter` world の構造的妥当性、
  (2) 自己署名ロードゲート通過（component/SBOM binding の健全性）、(3) 汎用 request/response への応答で
  trap しない、(4) 宣言 deadline 内に応答する（host 側 epoch 強制の健全性）——の四点。フィクスチャ固有アサーションは
  Rust 内部の回帰テストのまま残し、CLI には持ち出さない（用語は host/CONTEXT.md「Conformant（component）」参照）。
- **`plecto dev` はプロセス内蔵 + 自己 SIGHUP**——`plecto dev <filter-dir>` は `plecto serve` と同じ起動経路
  （Control 構築 + `serve_with_shutdown`）を 1 プロセス内で実行し、並行して file-watcher スレッド
  （`notify` + `notify-debouncer-mini`、8.x 系固定、~300ms debounce）を追加する。変更検知 → cargo build
  wasm32-unknown-unknown --release → wit-component encode → dev 鍵で署名 → `write_layout` →
  conformance 実行 → **合格したときだけ** 自プロセスへ実 SIGHUP（`libc::raise`）。conformance 失敗時は
  reload をブロックし、dev ループ自体が fail-closed になる。既存の `ReloadSource` / `serve_reloads` は無変更で
  再利用する。
- **dev 鍵は sigstore-rs の `SigStoreKeyPair` を流用**——Web 調査は新規 `p256` crate 追加を示唆したが、
  `TestSigner` が内部で使う `sigstore::crypto::signing_key::{ECDSAKeys, SigStoreKeyPair}` に
  `private_key_to_pem`/`from_pem`/`to_sigstore_signer` が既にあり、新規依存ゼロで永続化できる。本番検証経路
  （`sigstore::crypto::CosignVerificationKey`）と同一 crate・同一型を使うため、署名フォーマット
  （PKCS8 PEM・ASN.1 DER）の不一致事故が構造的に起きない。`TestSigner` 自体は無変更（テスト専用のまま）、
  host crate に production コードとして新設する `DevSigner`（仮称）がこの系譜を継ぐ。
- **PLECTO-E の四点セット型は `plecto-control` に置く**——依存グラフは host ← control ← server
  （host は control に依存できない）。初回の三大壁はいずれも control 層以上で人間向けに整形される
  （署名失敗は `ControlError::Load` が host の `LoadError` をラップ済み、quota 超過・path 正規化拒否は
  server の `fault` モジュール、server は production で control に依存済み）。新規クレートは切らず、
  host 自身は PLECTO-E を知らない薄い `thiserror` 型のままにする。
- **CLI 引数パースは手書き match のまま拡張**——clap は新規追加しない。今回追加するフラグ面積
  （`--lang rust` 固定 + filter-dir 位置引数）はまだ手書きで十分小さく、依存最小化方針とも整合する。
  既存の `serve`/`--version`/`schema`/`validate` の挙動は変更しない。

### 実装完了（2026-07-09）
Rust 一本の縦串を実装・実機検証済み: `plecto new-filter --lang rust` でスキャフォールド生成（wkg 経由
WIT 取得含む）→ `plecto dev` が実際に `cargo build --target wasm32-unknown-unknown --release` +
`wit-component` で componentize → conformance 実行 → dev 鍵署名 → OCI layout 書き込み →
manifest.toml の digest を format-preserving に書き換え（`toml_edit`）→ 自己 SIGHUP reload、という
一連の流れを実ファイルシステム・実ビルドで完走させ、ソース編集 → 自動リロードまで確認した。
`plecto conformance <component.wasm> [--json]` も実コンポーネントで検証済み。

実装中に見つかった非自明な罠: `notify` で単一ファイル（`Cargo.toml`）を watch すると、少なくとも
inotify バックエンドではその親ディレクトリ単位で登録され、同ディレクトリ内の別ファイルの変更
（`plecto dev` 自身が書く `target/`/`artifacts/`/`manifest.toml`）が誤って「Cargo.toml changed」
として報告される——自己を再帰的に rebuild し続ける無限ループを実機で確認した。対策は `Cargo.toml`
の watch を廃止し `src/` のみを watch 対象にすること（`Cargo.toml` 編集時は `plecto dev` の再起動が
必要、というスコープの明示的な割り切り）。

積み残し（明示的、サイレントな縮小ではない）: `new-filter --lang go|moonbit|c|js`（`--lang` に渡すと
明確なエラーで拒否）、PLECTO-E は初回三大壁の 4 コード（E0001 署名失敗・E0002 quota 超過・E0003 path
正規化拒否・E0004 dev 鍵混入警告）のみ。TTFVF 15 分の外部協力者による実測は本 ADR のスコープ外
（内部計画側）。

### 実装追記（2026-07-10 監査）

実装監査（コード突き合わせ + upstream ソース確認）で以下を確認・修正した:

- **conformance の実行順序**: 上のサブ決定の具体化に書いた「dev 鍵で署名 → `write_layout` →
  conformance 実行」は実装と順序が異なる。実装は **conformance を署名より前**に走らせ、非合格
  ビルドは署名も OCI layout 書き込みも manifest 書き換えも一切しない（`dev_cmd.rs` の
  `build_sign_and_gate`）。fail-closed としてより強い順序であり、コード側を正とする（実装完了の
  節の記述はもともとこの順序）。
- **dev 鍵の実装型**: 上で名指した `SigStoreKeyPair` ではなく、同じ sigstore crate の
  `ECDSAKeys` + `SigStoreSigner` を使った。crate・署名スキーム・鍵フォーマット（PKCS8 PEM /
  ASN.1 DER）は判断どおり検証経路と同一で、「同一 crate による構造的なフォーマット不一致の排除」
  という趣旨は変わらない。
- **inotify の機構説明を撤回**: 上の「単一ファイル watch はその親ディレクトリ単位で登録される」
  という説明は、notify 8.2.0 の inotify バックエンドのソース（単一ファイル watch はファイル自体に
  登録される）と矛盾するため、機構の説明としては撤回する。観測事実（`Cargo.toml` の単一ファイル
  watch 下で、同ディレクトリ内の別ファイルの変更が「Cargo.toml changed」として誤報告され、自己
  リビルドの無限ループになった）と対策（watch を `src/` のみに限定）は変わらない。upstream が
  文書化する既知の脆さは「rename-replace 保存で inode 単位の watch が壊れる（堅牢なのは
  ディレクトリ watch）」であり、「単一ファイル watch の正確な意味論に依存しない」という結論は同じ。
- **エラーコード表と E0001 の配線を補完**: affected_components に掲げた「エラーコード表」が docs
  に未着地で、かつ E0001 の四点セットを表示する経路（`diagnose()`）が本番コードから呼ばれて
  いなかった（マッピングとテストのみ存在）。`docs/writing-a-filter.md` §9 に表を追加し、起動
  load 失敗（`plecto serve` / `plecto dev`）と SIGHUP reload 失敗ログに診断を配線した。なお HTTP
  レスポンス上の E0002/E0003 は `x-plecto-error-code` ヘッダで**コードのみ**を返す——修正提案や
  docs リンクを任意クライアントへ送らない情報最小化は意図的な設計であり、四点セットの全文表示は
  操作者向けサーフェス（起動・reload ログ・`plecto validate`）の責務とする。

## Affected Components
- server/main — サブコマンド追加（現行: serve / validate / schema / --version）
- host — `signed_load` + 共有アサーション群（現行 `tests/polyglot.rs` 内のテスト専用構造）を、`plecto conformance <component.wasm>` が使える形へ抽出
- host/test_support — TestSigner の系譜に連なる **dev 鍵**（プロジェクトローカル・永続）機構を production コードとして新設（テスト専用の TestSigner はそのまま）
- docs — writing-a-filter.md / チュートリアル層 / エラーコード表

## Context

Plecto の first success を他プロキシと同じ「設定が通る」に置くと、中核価値（あなたのコードが、**信頼されて**、ゲートウェイに載る）が体験に現れない。一方で現状、署名検証（[[000006]]）は初見者にとって最大の摩擦要因であり、開発ループの各工程（componentize → 署名 → 検証 → 配置 → reload）が手作業で分断されている。開発用署名の道具は test-support feature の TestSigner（テスト専用・ephemeral・`#[doc(hidden)]`）しかない。

方針の型は確立している: 勝ったプロキシは一つの DX フックで勝つ（Caddy = 自動 HTTPS が「TLS の規律を DX に変換」した）。Plecto の対応物は「**署名 + conformance という規律を、開発時は自動化して摩擦ゼロ、本番では規律のまま**」の二相化である。conformance の共有アサーション（[[000055]]）と `plecto validate`（[[000046]]）という部品は既に在る。

## Decision

1. **North Star Metric = TTFVF（Time To First Verified Filter）**: 「初見の開発者が、自分の言語でフィルタを書き、署名し、conformance を緑にし、ローカルの Plecto で実トラフィックを判定するまで」の時間。目標 **15 分**（公式ドキュメントのみで）。first success の定義に**署名と conformance を含める**こと自体をプロダクトの主張とする。KPI 実測（外部協力者による完走テスト）は内部計画の運用へ。
2. **`plecto dev <filter-dir>`（新設）**: watch → componentize → dev 鍵署名 → conformance → SIGHUP reload をローカルで一括するインナーループ。
   - **dev 鍵はプロジェクトローカル・永続**: 初回実行時に `.plecto/dev-key`（ECDSA-P256）を生成し `.gitignore` へ追記。dev 用 manifest の `[trust]` にはその公開鍵**のみ**を注入する。
   - **署名ゲートは一切緩めない**（P5 不侵）: 検証経路は本番と同一コード。dev と本番の差は trust root の中身だけであり、"allow unsigned" の類は導入しない。**本番 manifest には触れない**（dev manifest は生成物）。
3. **`plecto conformance <component.wasm>`（新設）**: 現在 host の内部テスト（polyglot スイート）にある契約行使・fail-closed 挙動・deadline 遵守・quota 遵守のアサーション群を、フィルタ作者が自分の成果物へ回せる CLI に外部開放する。「うちのフィルタは Plecto conformant か?」に一コマンドで答えられる状態が、エコシステムの品質床（`plecto validate` = manifest 側との対で、CLI の語彙も揃う）。
4. **`plecto new-filter --lang rust|go|moonbit|c|js`（新設）**: 言語別テンプレート生成（filter-template の CLI 化）+ WIT 取得（[[000064]] の wkg 経路）+ import assert 込み build.sh 同梱。scaffold → 実装 → 署名 → 配置 → 計測が公式ドキュメントだけで完走できることを出口条件とする。
5. **エラーを教材にする**: fail-closed が売りである以上、拒否メッセージの品質がそのまま DX である。既存の fault 定数を安定エラーコードに昇格し、**四点セット**（`PLECTO-EXXXX` + 一行の原因 + 修正提案 + docs への参照）で返す。初見者が最初に踏む三大壁——**署名検証失敗・quota 超過・path 正規化拒否**——から整備する。`plecto validate` の警告には根拠 ADR への参照を含める（ADR 資産の DX への還流）。docs 参照は当面リポジトリ内相対（専用ドメインは後続）。
6. これら（wkg 配布 [[000064]] / conformance CLI / scaffold / 言語コスト表 [[000063]]）を **「Filter Dev Kit」の一語**で束ね、ドキュメント・発信で常に同じ名で呼ぶ。

検討した代替案と却下理由:

- **開発時は署名検証をスキップ可能にする（--insecure 等）** — "allow unsigned" の escape hatch を作らない構造（[[000006]]）を DX の名目で破る最悪手。二相化（鍵を替える、ゲートは同じ）で同じ摩擦削減が達成できる。却下。
- **dev 鍵をグローバル（ホームディレクトリ）に一つ** — 全プロジェクト共有の鍵は「うっかり本番 trust に入れる」事故の射程が長い。プロジェクトローカルで爆風半径を切る。却下。
- **実行ごとの ephemeral 鍵** — dev セッションを跨いで trust 設定が毎回変わり、「鍵が固定で検証が通る」という本番と同型の体験から遠のく。却下。
- **conformance を docs の手順（cargo test の呼び方）として公開** — 「一コマンドで答えが出る」から「リポジトリを clone して環境を作る」へ後退する。CLI 化が要点。却下。
- **first success を「プロキシが動く」に置く（業界標準形）** — 差別化の核（検証可能な信頼）が体験から消える。却下。

## Consequences

### Pros
- 最大の摩擦要因（署名）が、価値を落とさずに開発ループから消える——「規律を DX に変換する」Caddy 型の構図を供給網検証で再現。
- conformance の外部開放により、外部作者フィルタの品質床と「conformant」という語の意味が機械検証可能になる（§3.4 反証可能性のエコシステム版）。
- エラーコード四点セットは運用者・初見者・AI アシスタントのいずれに対しても効く（構造化された拒否理由）。

### Cons / Tradeoffs
- CLI 面積が増える（現行の手書き arg match から成長する。既存 validate/schema と同じ「fail-closed を前倒しする」思想で動詞を統制し、際限ない多機能化はしない）。
- conformance アサーションの公開面化は、その後方互換も背負う（契約互換ポリシー [[000064]] と同じ SemVer 系譜で扱う）。
- dev 鍵ファイルはローカル秘密である——`.gitignore` 自動追記 + ファイル権限 + 「dev 鍵を本番 trust に入れない」旨の validate 警告（dev 鍵にマーカーを持たせ検出可能にする）で事故の芽を摘む。
- **再検討条件**: (a) TTFVF 実測が 15 分を大きく超え続ける場合、ボトルネック（署名以外の壁）を計測で特定して優先度を再配分、(b) cargo-generate 等テンプレート機構の言語横断限界が見えたら scaffold の実装手段を再選定。
