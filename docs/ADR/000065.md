# Filter Dev Kit を確立し、TTFVF（Time To First Verified Filter）15 分を North Star に据える

## Status
proposed（判断は 2026-07-06 の grill セッションで合意済み・**未実装**。着手時に accepted へ更新する。実装は Phase 1〜2（90 日計画）。プロモーション計画側の詳細は内部文書、本 ADR はプロダクト判断のみを記録）

## Date
2026-07-06

## Affected Components
- server/main — サブコマンド追加（現行: serve / validate / schema / --version）
- host — `signed_load` + 共有アサーション群（現行 `tests/polyglot.rs` 内のテスト専用構造）を、`plecto conformance <component.wasm>` が使える形へ抽出
- host/test_support — TestSigner の系譜に連なる **dev 鍵**（プロジェクトローカル・永続）機構を production コードとして新設（テスト専用の TestSigner はそのまま）
- docs — writing-a-filter.md / チュートリアル層 / エラーコード表

## Context

Plecto の first success を他プロキシと同じ「設定が通る」に置くと、中核価値（あなたのコードが、**信頼されて**、ゲートウェイに載る）が体験に現れない。一方で現状、署名検証（[[000006]]）は初見者にとって最大の摩擦要因であり、開発ループの各工程（componentize → 署名 → 検証 → 配置 → reload）が手作業で分断されている。開発用署名の道具は test-support feature の TestSigner（テスト専用・ephemeral・`#[doc(hidden)]`）しかない。

方針の型は確立している: 勝ったプロキシは一つの DX フックで勝つ（Caddy = 自動 HTTPS が「TLS の規律を DX に変換」した）。Plecto の対応物は「**署名 + conformance という規律を、開発時は自動化して摩擦ゼロ、本番では規律のまま**」の二相化である。conformance の共有アサーション（[[000055]]）と `plecto validate`（[[000046]]）という部品は既に在る。

## Decision

1. **North Star Metric = TTFVF（Time To First Verified Filter）**: 「初見の開発者が、自分の言語でフィルタを書き、署名し、conformance を緑にし、ローカルの Plecto で実トラフィックを判定するまで」の時間。目標 **15 分**（公式ドキュメントのみで）。first success の定義に**署名と conformance を含める**こと自体をプロダクトの主張とする。KPI 実測（外部協力者による完走テスト）は内部計画の運用へ。
2. **`plecto dev <filter-dir>`（新設）**: watch → componentize → dev 鍵署名 → conformance → SIGHUP reload をローカルで一括するインナーループ。
   - **dev 鍵はプロジェクトローカル・永続**: 初回実行時に `.plecto/dev-key`（ECDSA-P256）を生成し `.gitignore` へ追記。dev 用 manifest の `[trust]` にはその公開鍵**のみ**を注入する。
   - **署名ゲートは一切緩めない**（P5 不侵）: 検証経路は本番と同一コード。dev と本番の差は trust root の中身だけであり、"allow unsigned" の類は導入しない。**本番 manifest には触れない**（dev manifest は生成物）。
3. **`plecto conformance <component.wasm>`（新設）**: 現在 host の内部テスト（polyglot スイート）にある契約行使・fail-closed 挙動・deadline 遵守・quota 遵守のアサーション群を、フィルタ作者が自分の成果物へ回せる CLI に外部開放する。「うちのフィルタは Plecto conformant か?」に一コマンドで答えられる状態が、エコシステムの品質床（`plecto validate` = manifest 側との対で、CLI の語彙も揃う）。
4. **`plecto new-filter --lang rust|go|moonbit|c|js`（新設）**: 言語別テンプレート生成（filter-template の CLI 化）+ WIT 取得（[[000064]] の wkg 経路）+ import assert 込み build.sh 同梱。scaffold → 実装 → 署名 → 配置 → 計測が公式ドキュメントだけで完走できることを出口条件とする。
5. **エラーを教材にする**: fail-closed が売りである以上、拒否メッセージの品質がそのまま DX である。既存の fault 定数を安定エラーコードに昇格し、**四点セット**（`PLECTO-EXXXX` + 一行の原因 + 修正提案 + docs への参照）で返す。初見者が最初に踏む三大壁——**署名検証失敗・quota 超過・path 正規化拒否**——から整備する。`plecto validate` の警告には根拠 ADR への参照を含める（ADR 資産の DX への還流）。docs 参照は当面リポジトリ内相対（専用ドメインは後続）。
6. これら（wkg 配布 [[000064]] / conformance CLI / scaffold / 言語コスト表 [[000063]]）を **「Filter Dev Kit」の一語**で束ね、ドキュメント・発信で常に同じ名で呼ぶ。

検討した代替案と却下理由:

- **開発時は署名検証をスキップ可能にする（--insecure 等）** — "allow unsigned" の escape hatch を作らない構造（[[000006]]）を DX の名目で破る最悪手。二相化（鍵を替える、ゲートは同じ）で同じ摩擦削減が達成できる。却下。
- **dev 鍵をグローバル（ホームディレクトリ）に一つ** — 全プロジェクト共有の鍵は「うっかり本番 trust に入れる」事故の射程が長い。プロジェクトローカルで爆風半径を切る。却下。
- **実行ごとの ephemeral 鍵** — dev セッションを跨いで trust 設定が毎回変わり、「鍵が固定で検証が通る」という本番と同型の体験から遠のく。却下。
- **conformance を docs の手順（cargo test の呼び方）として公開** — 「一コマンドで答えが出る」から「リポジトリを clone して環境を作る」へ後退する。CLI 化が要点。却下。
- **first success を「プロキシが動く」に置く（業界標準形）** — 差別化の核（検証可能な信頼）が体験から消える。却下。

## Consequences

### Pros
- 最大の摩擦要因（署名）が、価値を落とさずに開発ループから消える——「規律を DX に変換する」Caddy 型の構図を供給網検証で再現。
- conformance の外部開放により、外部作者フィルタの品質床と「conformant」という語の意味が機械検証可能になる（§3.4 反証可能性のエコシステム版）。
- エラーコード四点セットは運用者・初見者・AI アシスタントのいずれに対しても効く（構造化された拒否理由）。

### Cons / Tradeoffs
- CLI 面積が増える（現行の手書き arg match から成長する。既存 validate/schema と同じ「fail-closed を前倒しする」思想で動詞を統制し、際限ない多機能化はしない）。
- conformance アサーションの公開面化は、その後方互換も背負う（契約互換ポリシー [[000064]] と同じ SemVer 系譜で扱う）。
- dev 鍵ファイルはローカル秘密である——`.gitignore` 自動追記 + ファイル権限 + 「dev 鍵を本番 trust に入れない」旨の validate 警告（dev 鍵にマーカーを持たせ検出可能にする）で事故の芽を摘む。
- **再検討条件**: (a) TTFVF 実測が 15 分を大きく超え続ける場合、ボトルネック（署名以外の壁）を計測で特定して優先度を再配分、(b) cargo-generate 等テンプレート機構の言語横断限界が見えたら scaffold の実装手段を再選定。
