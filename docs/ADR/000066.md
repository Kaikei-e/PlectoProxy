# フィルタ業務設定を manifest から貸す host-config capability を導入する

## Status
accepted（判断は [[000061]] 実装準備の grill セッション、2026-07-08 に合意・実装済み。WIT 契約 /
host 実装 / manifest スキーマ / `filter-ratelimit-redis` からの利用まで green）

## Date
2026-07-08

## Affected Components
- wit — `host-config` interface（`get: func(key: string) -> option<string>`）を新設し、`filter` /
  `filter-body` 両ワールドに他の基本 capability と並んで `import` する
- control/manifest — `FilterEntry` に `config: Option<BTreeMap<String, String>>` を追加。
  `[filter.config]` という任意キーの inline table としてパースする
- host — `HostState` が config map を保持し `host_config::Host::get` を実装する。生成済みの
  `Filter::add_to_linker` がそのまま配線するので、Outbound 系のような専用 Linker 配線コードは不要

## Context

[[000061]]（レートリミット local floor × global filter 二層モデル）の reference filter
`filter-ratelimit-redis` を設計する過程で、フィルタ自身の業務設定——接続先 Redis の host:port、
window 秒数、limit 数、cost の取得元ヘッダ名、`on_backend_error = deny | allow`、複数ルートに同じ
フィルタを配る場合の route 識別子タグ——を manifest からフィルタへ渡す一般的な経路が存在しないと
判明した。既存の reference filter（`filter-apikey` 等）は例外なく Rust のビルド時定数として
ハードコードしており、これは「不要な抽象化を入れない」（CLAUDE.md）の帰結として妥当だったが、
`filter-ratelimit-redis` は事情が異なる:

- `on_backend_error` は運用者が**環境ごとに明示的に選ぶべき**可用性トレードオフであり、
  「フィルタを fork してソースを書き換える」以外に値を変える手段がないのは、この判断を manifest
  の外（コードレビュー）に押し出してしまう。
- 同じ reference filter を複数ルートに配る場合、route ごとに異なる Redis 接続先や limit を
  持たせたい——ビルド時定数では「ルートごとに別バイナリを焼く」ことになり、運用の単純さ
  （CLAUDE.md コア原則）に反する。

一方で、`[filter.ratelimit]`（[[000026]]）や `[filter.outbound_tcp]`（[[000060]]）のような
**host が意味を理解し検問・強制する**設定と違い、Redis host:port や window 秒数はフィルタの内部
ロジックが解釈するだけの不透明な値であり、host 側に新しい強制ロジックを持たせる理由がない。

## Decision

**manifest 由来の読み取り専用キー・バリュー文字列マップを、新しい基本 capability `host-config`
として全フィルタに無条件で貸す。**

1. **WIT契約**（`plecto/wit/world.wit`）:
   ```wit
   interface host-config {
       // Read a manifest-declared config value for this filter (ADR 000066). Absent = the
       // key was not declared in `[filter.config]`. Read-only: a filter cannot write its own
       // config (the operator owns it, same ownership model as host-ratelimit's bucket spec).
       get: func(key: string) -> option<string>;
   }
   ```
   `filter` / `filter-body` 両ワールドに `import host-config;` を追加する。`host-log` /
   `host-clock` / `host-kv` / `host-counter` / `host-ratelimit` と同格の**基本 capability**——
   `[filter.config]` の宣言有無に関わらず全フィルタに無条件で貸す。`outbound-http` /
   `outbound-tcp` のような feature-gate 付き per-filter オプトイン能力とは性質が異なる（あちら
   は「危険な外向き能力」、こちらは「操作者が既に manifest に書いた値の読み取り」）。既存フィルタ
   は `host-config` を import しなくても挙動は変わらない——値を要求するかどうかはフィルタ側の
   選択であり、deny-by-default の建て付け自体は変わらない（そもそも要求しなければ何も得られない）。

2. **manifestスキーマ**（`plecto/crates/control/src/manifest/filter_entry.rs`）: `FilterEntry` に
   `config: Option<BTreeMap<String, String>>` を追加し、`[filter.config]` という**任意キーの
   inline table**としてパースする。キー・バリューとも文字列——型付け（整数へのパース等）は
   受け取ったフィルタ自身が行う。host はキーの意味を一切解釈しない、純粋なパススルーである。
   他の `[filter.*]` セクション（`ratelimit` / `outbound_http` / `outbound_tcp`）が
   `deny_unknown_fields` の型付き構造体であるのとは対照的に、`config` は自由形式のまま——両者を
   混同しない。

3. **host実装**（`plecto/crates/host/src/state.rs` / `host.rs`）: `HostState` に config map
   （`Arc<BTreeMap<String, String>>` 等）を追加し、`host_config::Host for HostState { fn get(&mut
   self, key: String) -> Option<String> { ... } }` を実装する。`wasmtime::component::bindgen!`
   が生成する `Filter::add_to_linker`（host.rs）は宣言された import を自動的に配線するため、
   `outbound_http` / `outbound_tcp` のような専用 Linker 配線コードは不要——`host-kv` /
   `host-counter` と全く同じ扱いになる。

4. **設定エラーの扱いは host-config の外に置く**: `host-config::get` は「値が manifest に存在
   するか」を機械的に返すだけで、**特定のキーが必須かどうかを host は一切知らない**（汎用パス
   スルーである以上、意味は持たせられない）。フィルタごとの必須キー検証は各フィルタの `init()`
   内で行う責務とし、検証に失敗したら `init()` を意図的に trap させる。

   これは既存の `isolation = "trusted"` の挙動と組み合わさることで、追加の host 側検証ロジック
   なしに「必須設定が無ければ load 時点で fail-closed」という運用上の効果を得られる。
   `plecto/crates/host/src/host.rs` の trusted 分岐は trusted filter について「manifest 適用時に
   eager に1インスタンスを instantiate + init し、それが失敗したら `LoadError::Instantiate` として
   manifest 適用そのものを失敗させる」ことをすでに実装済みである（コメント: "Eager-build ONE
   instance now so a broken `init` surfaces at load (not on the first request)"）。したがって
   `on_backend_error` のような「宣言必須」設定を持つフィルタは **`isolation = "trusted"` で運用
   しなければならない**——`untrusted` は instantiate が毎リクエストなので、同じ trap が
   「load 拒否」ではなく「毎リクエスト 503」に化けてしまう。この要件は host-config 自体の制約
   ではなく、host-config を使う個々のフィルタ（例: [[000061]] の `filter-ratelimit-redis`）の
   ドキュメントに明記する。

検討した代替案と却下理由:

- **(A) host-kv 流用** — manifest の `[filter.config]` を host が `init()` 呼び出し前にフィルタ
  自身の `host-kv` 名前空間へ事前 seed する。WIT 契約変更が不要という利点はあるが、「host-kv は
  フィルタが自分の**可変業務状態**を持つ場所」（[[000002]] / [[000011]]）という既存の意味論に、
  「host が外から書き込む設定注入経路」という別の役割を重ねることになり、責務が混同する。読み手が
  `host_kv::get("config:x")` を見たとき、それが「フィルタ自身が過去に書いた状態」なのか「host が
  起動時に注入した設定」なのか区別できない。却下。
- **(C) init の引数化** — `export init: func(config: string)` に変える。意味論的には素直だが、
  `plecto:filter` の**基本契約そのもの**を破壊的に変更し、既存の全フィルタ（`filter-hello` 系
  polyglot 実装を含む）の `init` エクスポート署名を書き換えることになる。1個の reference filter
  のために契約全体を割るのは影響範囲に見合わない。却下（`init` への引数追加が将来 general に必要
  になったら、その時点で改めて評価する）。
- **host 側でキー必須性を検証する仕組みを新設する**（例: manifest に「このフィルタが要求する
  必須キー一覧」を宣言させ、`FilterEntry::validate()` で機械的にチェックする）— host がフィルタ
  ごとの意味を知る必要が生じ、`host-config` を「意味を持たない汎用パススルー」にする本判断の
  趣旨と矛盾する。「必須キーの検証は `init()` 内でフィルタ自身が行う」の方が capability の境界を
  綺麗に保つ。却下。

本判断は [[000002]] / [[000011]]（フィルタは可変業務状態を持たずステートレス、状態は host-kv
に置く）を変えない——`host-config` は読み取り専用であり、フィルタはこの値を書き換えられない。
[[000006]] の deny-by-default 原則にも反しない——基本 capability として無条件配線される点は
`host-kv` / `host-counter` と同型であり、新しい特権を作るものではない。

## Consequences

### Pros
- [[000061]] の `filter-ratelimit-redis` が、Redis 接続先・window・limit・`on_backend_error`
  ・route タグを manifest 上で環境ごとに変更できるようになる——フォークやリビルドが不要になる。
- 将来の reference filter（WAF のしきい値、ext_authz のエンドポイント名等）も同じ capability を
  再利用でき、フィルタごとにアドホックな設定注入手段を発明せずに済む。
- host 側の実装コストが小さい（`host-kv` と同型の read-only map、Linker 配線は自動）。

### Cons / Tradeoffs
- 汎用文字列マップである以上、型安全性はフィルタ側の実装に委ねられる——host は値の妥当性を一切
  保証しない。
- 「宣言必須」設定を持つフィルタは `isolation = "trusted"` を要求される、という制約がやや
  非自明——ドキュメント（各フィルタの README・`docs/writing-a-filter.md`）で明示する必要がある。
- **再検討条件**: (a) 型付き値（整数・真偽値等）を要求する filter が増え、文字列パースの重複が
  目立つようになったら、host 側で型付きアクセサを足すか検討する、(b) 複数フィルタが同じ設定キー
  を共有したいという要求が出たら、フィルタ横断の設定名前空間を別途検討する、(c) `init()` trap に
  よる「load 拒否」が非同期的すぎる（エラーメッセージが伝わりにくい等）と判明したら、host 側で
  検証結果を manifest 適用時のログへ明示的に伝える仕組みを別 ADR で検討する。
