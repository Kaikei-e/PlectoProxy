# 差別化の重心を検証可能な信頼×契約駆動の性能×宣言された意味論の交差に置き直す

## Status
accepted（近期顧客記述は [[000076]] が、対外メッセージの前面順序・従属関係は [[000083]] が amend。三本柱そのものは不変）

## Date
2026-07-09

## Affected Components
- roadmap / governance — 対外的な差別化の説明軸を「四条件の同時充足」（[[000001]]）単体から、その上に立つ三本柱（Verifiable Trust / Contract-Derived Performance / Declared Semantics）へ更新する。コード変更は伴わない
- positioning（README / docs のポジショニング記述）— 今後の発信・ドキュメント冒頭はこの三本柱を軸に書く。個別文書（README 等）の実際の書き換えは本 ADR のスコープ外で、必要になった時点で別途行う

## Context

[[000001]] は「プロセス内の速さ・サンドボックスの安全・言語の自由・無停止差し替えを同時に満たす事実上唯一の技術が WASM」という認識のもとで拡張モデルを確定した。この四条件同時充足は Plecto の存在理由の土台であり続けるが、2026 年前半の外部動向を整理した開発者調査（本 ADR 執筆にあたり主要な事実関係は一次情報で再確認済み）を踏まえると、**四条件の同時充足は入場券であって差別化ではなくなった**と判断する。

確認した外部動向の要旨（個別プロダクト名・定量比較は開発者調査に譲り、本 ADR では設計判断に必要な示唆のみを扱う）:

- 「Rust 製・安全・プログラマブルなセルフホストプロキシ」という定義を掲げ、別プロセス隔離型の拡張モデル（拡張をプロセス外で実行し、OS プロセス境界で障害を隔離する設計）で実績を積んでいる新規プロダクトが、直近半年で多数登場している。
- WASM 拡張の商用化に成功している実装エコシステムが別途存在し、拡張機構そのものよりも「すぐ使える reference 実装群 + 道具立て + 牽引ユースケース」のパッケージが需要を生んでいることを示している。
- AI/LLM ゲートウェイに特化した新興プロジェクトが認知を広げている。
- WASM 標準仕様の一部（module ABI 系）は策定が停滞している一方、Component Model 側の基盤整備（wasmtime のリリースで WASI 0.3 + component-model-async が default-on になるなど）は進んでいる。

これらを重ねると、「Rust 製・安全・プログラマブルなセルフホストプロキシ」という定義の席には既に他プロダクトが座っており、「WASM 拡張の棚」という主張も先に商用化した実装エコシステムがあり、「Rust 製新世代ゲートウェイ」の認知も AI ゲートウェイに特化した製品が持っていく可能性が高い。**四条件の同時充足だけでは、もはや固有の立ち位置を説明できない。**

一方で、[[000001]]（Component Model + WIT）・[[000006]]（deny-by-default capability・OCI 署名・pooling ゼロ化）・[[000038]]（body 契約由来の zero-copy）・[[000047]]（供給網規律の自己適用）・[[000053]]／[[000029]]／[[000043]]／[[000056]]（宣言された意味論・declined の言語化運用）を重ね合わせると、**型付き契約 × 拡張側の供給網検証 × fail-closed 不変則を同時に満たすゲートウェイ**は、開発者調査が調査した限定的な範囲（悉皆調査ではない）では他に確認できていない。この交差点にこそ差別化を置き直す必要がある。

## Decision

差別化の重心を、**P3（判断も不在も型で表す）× P4（deny-by-default / fail-closed 不変則）× P5（検証してからロードする）**（`docs/design-principles.md`）の交差に置く三本柱として明文化する。四条件の同時充足（[[000001]]）は **supersede せず**、その上に立つ差別化レイヤーとして本 ADR を位置づける。

1. **柱 1: 検証可能な信頼（Verifiable Trust）。** 「あなたがロードする拡張」の署名・SBOM 束縛を必須ゲートにし、unsigned の逃げ道を構造的に持たない（[[000006]] / [[000047]]）。同種のセルフホスト・プログラマブルプロキシの多くは自身の配布物には署名するが、**ロードする拡張側**の供給網検証を必須ゲートにする例は、開発者調査の調査範囲では確認できていない。
   - 顧客の時系列を正直に置く: SBOM/SLSA を審査票で要求する組織は、0.x の個人開発プロジェクトを署名機構の有無に関わらず単独では採用しない（バス係数・サポート・実績の問題）。柱 1 の**近期の顧客はセルフホスター・データ主権志向のチーム**であり、審査言語（SBOM/attestation）との互換性はその将来価値として維持する。「SaaS 導入水準」（[[000054]]）は品質の物差しとして維持しつつ、販売の約束としては掲げない。
2. **柱 2: 契約駆動の性能（Contract-Derived Performance）。** 「body フックの不在 = zero-copy」（[[000038]]）は、運用者の注意力ではなく契約の形から性能を導く設計言語。module ABI 系の拡張機構には「不在の型」という概念が無く、境界を越えるデータは常にコピーが発生しやすい。これに対する Component Model + pooled dispatch の優位は、µs 単位の実測で語れる具体的な材料として保持する。
3. **柱 3: 宣言された意味論（Declared Semantics）。** node-local 宣言（[[000053]]）、「実効レート = 設定値 × N」の但し書き、declined の言語化運用（[[000029]] / [[000043]] / [[000056]]）。曖昧になりがちな境界を仕様として宣言する規律を、今後も維持・拡張する。

一文に圧縮したポジショニング: **Plecto Proxy は「あなたのコードが、検証されて、ゲートウェイに載る」ことを構造で保証する L7 プロキシである。** 拡張が速い・安全・多言語・差し替え可能なことは前提条件（[[000001]]）。その上で、何がロードされているかを暗号学的に言え（digest + cosign + SBOM 束縛）、拡張が何に触れられるかを型で言え（WIT 能力貸与）、失敗したとき何が起きるかを一意に言える（fail-closed 不変則）——この三つの「言える」が製品である。

対競合カテゴリのポジショニング文（発信・docs で再利用する。個別プロダクト名は挙げず、拡張モデルの型で語る）:
- 対 module-ABI 型の拡張プラットフォーム: 「先行する実装エコシステムが実証した知見を、策定が停滞した module ABI とそのコピー税の上ではなく、Component Model の型と性能の上で継ぐ」
- 対 別プロセス隔離型の拡張モデル: 「別プロセス隔離ではなく、プロセス内の速さとサンドボックスの安全を同時に。fail 挙動は選択肢ではなく不変則。そしてロードする拡張自体の供給網まで検証する」
- 対 サンドボックスを持たない native 拡張: 「速さのためにサンドボックスを捨てない。untrusted なコードでも載せられる in-process 拡張」
- 対 AI ゲートウェイに特化した製品群: 「AI ゲートウェイの席は争わない。AI 向けポリシーはフィルタとして書ける・提供される基盤である」（[[000043]] の native declined と整合）

**「唯一の」という強い主張はしない。** 開発者調査が確認した限定的な範囲で三点交差の反例が無い、という限定付きの主張に留める。反例の発見は敗北ではなく、ポジショニングを更新する一次情報として扱う。

検討した代替案と却下理由:

- **機能網羅性で対抗する**（先行プロダクトの機能一覧に数で追いつく）——却下。コア原則の順序（機能網羅性は後位）・[[000029]] の役割駆動と正面衝突する。「広さ」ではなく「言えることの強さ」で立つ。
- **「WASM で拡張できる」こと単体を差別化として維持する**——却下。別プロセス型・module ABI 型のいずれも同種の主張を先に広めており、もはや固有の立ち位置ではない。
- **別プロセス隔離モデルへ転換する**——却下。[[000001]] が確定した四条件同時充足（特に「プロセス内の速さ」）を放棄することになり、Component Model 採用の根拠そのものと矛盾する。
- **調査を悉皆にしてから確定する**——却下。全数調査は単独開発のコストに見合わず、「限定的な調査範囲では」という限定付き主張と再検討条件（Consequences 参照）で十分に誠実である。

本決定は [[000001]]（四条件）を supersede せず、[[000029]]（役割駆動）・P3/P4/P5（`docs/design-principles.md`）を差別化の言葉として具体化する上位のポジショニング ADR である。

## Consequences

### Pros
- 対外説明（README・発信等）に一貫した一文が持て、新規参入プロダクトが増えても守れる立ち位置が明確になる。「機能で負けたら終わり」という不安な比較軸から離れられる。
- 柱 2（契約駆動の性能）は [[000038]] の既存実測（body 税 −76%、arena 上限 −50% 等）をそのまま証拠として転用できる。
- 柱 1 の近期顧客をセルフホスター側に正直に置いたことで、「SaaS 導入水準」（[[000054]]）と「単独開発の実態」の間の緊張を、誇張せず直視する形で運用できる。

### Cons / Tradeoffs
- 機能表の広さで同種のプロダクトに劣後する状態は今後も続く可能性が高く、外部からの「機能が少ない」という比較に対し、本 ADR の三本柱を根拠に応答し続ける発信規律が要る。
- 調査網羅性の限界（開発者調査は限定的な範囲の調査で、悉皆ではない）に依存した主張であり、反例が出た時点でポジショニングの更新が要る。
- wasip3 依存（streaming の遅れが柱 2 の説得力を削り得る）・wasmtime 追随コスト（default-on 化する async・API 変化）・新規参入プロダクトの機能追随速度、という 3 つの外部要因に、この立ち位置の持続性が依存する。
- **再検討条件**: (a) 三点交差（型付き契約 × 拡張側供給網検証 × fail-closed 不変則）を満たす反例プロダクトが確認されたとき、ポジショニングを更新する；(b) wasip3 のゲートが 6–12 ヶ月以上滑り、streaming 面の遅れが柱 2 の説得力を実質的に損なったとき、[[000068]] の優先順位を再検討する；(c) 新規参入プロダクトの機能追随ペースが持続し、機能表の差が「言えることの強さ」だけでは対外的に説明しきれなくなったとき。
