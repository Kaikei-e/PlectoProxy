# Program F の優先順位を Filter Dev Kit と reference filter 棚の完遂へ置き直し、比較ベンチハーネスを格下げする

## Status
accepted（F2 近期必須項目と発信ゲートは [[000076]] が amend。棚の出荷成熟定義は [[000080]] が amend。F1 最優先と比較ベンチ格下げは不変）

## Date
2026-07-09

## Affected Components
- roadmap / governance — [[000056]] が編成した Program F（Filter Ecosystem、6 柱: Go 解禁 / WIT の wkg・OCI 配布 / reference filter 三部作 / conformance CLI 外部開放 / scaffolding / dogfooding）の内部優先順位を改訂する。Filter Dev Kit（[[000065]]）と wkg 配布（[[000064]]）を最優先スライスとし、reference filter 棚（WAF・JWT/OIDC・AI 文脈）をそれと対等の優先度へ引き上げる。比較ベンチハーネス（v1.0 Gap 5 相当の提案）は優先リストから外し将来の余力案件へ格下げする。あわせて Program F の外に残る提言（初日体験レシピ / mTLS 復帰順 / streaming 待ち時間仕事）の位置づけを確認する。コード変更は伴わない
- 発信計画 — Show HN / Zenn 等の対外発信のタイミングを F1（FDK）+ F2（reference filter）完了に同期させる

## Context

開発者調査（[[000067]] が採用した分析）は、コア価値の柱 1（検証可能な信頼）を体験させる経路が現状どこにも無いことを指摘した:

- **Filter Dev Kit（[[000065]]）は 2026-07-09 時点では proposed のまま未実装だったが、以降スキャフォールド・dev 鍵・conformance CLI の実装が進行中である（2026-07-10 時点の一次確認）。** `plecto new-filter` → `plecto dev`（watch → componentize → dev 鍵署名 → conformance → reload）→ `plecto conformance` という TTFVF（Time To First Verified Filter）15 分の North Star は定義されたが、計測できる実装が無い。WIT の wkg/OCI 配布（[[000064]]）も未実装。
- **reference filter 棚が薄い。** 現状の棚は `filter-ratelimit-redis` / `filter-extauthz` / `filter-apikey` 程度で、WAF reference（[[000037]] が「拡張プレーンに置く」と配置決定した当のもの）と JWT/OIDC 級 auth が不在。

これに加え、開発者調査が確認した外部動向のうち、優先順位の判断材料になるものが二点ある（個別プロダクト名・定量比較は開発者調査に譲る）:

- **拡張機構そのものを提供するだけでは需要にならず、すぐ使える reference 実装群 + 道具立て + 牽引ユースケースをセットで提供した場合に限って需要が実証されている**、という業界一般の観察。拡張機構を裸で出しただけの前例は定着しにくい一方、「棚 + 道具立て + ユースケース」をパッケージした前例では拡張エコシステムが実際に育っている。
- **フィルタを自分で書きたい層は運用者全体の少数派であり、多数派は書かれたフィルタを使いたい**という需要側の構造。Program F の重心は SDK（FDK）だけでなく reference filter の質と数に同じ比重を置くべきである。また、需要が実証されているのは AI/inference 文脈のポリシー（token rate limit・PII・guardrails）であり、Plecto はこれを native declined（[[000043]]）とした——declined の判断そのものは維持しつつ、「フィルタとして提供する」の実物を早期に示すことが、需要側への最短の応答になる。

一方、v1.0 で挙げられていた「比較ベンチハーネス（数値ではなく再現可能な計測環境を出荷する）」提案は、敵対的に見るとコストを過小評価していた: 複数プロキシの公平な設定を維持する作業は単独開発では重く、同種の比較ベンチを既に公開している先行事例がある状況で後追いの期待値は低い。

## Decision

1. **F1（最優先）: Filter Dev Kit を実装する。** [[000065]] を最優先スライスとして実行対象に確定する。[[000064]]（wkg/OCI 配布）を同一スライスの前提部品として含める。scaffold（`plecto new-filter`）→ dev 鍵二相化（`plecto dev`）→ conformance CLI 外部開放 → エラーコード四点セット（`PLECTO-EXXXX`）を出口条件とする。`plecto dev` のローカル自己署名証明書対応（[[000043]] 2026-07-09 追記の (b)）もこのスライスに含める——署名まわりの摩擦削減と TLS 証明書まわりの摩擦削減は、どちらも「初日体験を DX レイヤで吸収する」という同じ性質の作業であり、まとめて一つのスライスで進める。
2. **F2（F1 と対等の優先度に昇格）: reference WAF filter + JWT/OIDC 級 auth + AI 文脈 reference を一本ずつ用意する。** WAF は [[000037]] が配置決定済みの実装手段（(a) componentize / (b) host-side engine / (c) bridge）のいずれかで出荷可能な実物を用意する。JWT/OIDC 級 auth は既存 `filter-apikey` より一段本格的な reference とする。AI 文脈 reference（token-cost rate limit）は [[000043]] Decision 項目 2 が示した `host-ratelimit` の `(key, cost)` 契約（[[000026]]）をそのまま使い、需要側実証（Context 参照）への直接の応答とする。FDK テンプレートの最初の顧客を自分（reference filter 実装者）が務め、TTFVF のドッグフーディングを兼ねる。
3. **比較ベンチハーネスを優先リストから外し、将来の余力案件へ格下げする。** 複数プロキシ横並びの公平な計測環境を整備・出荷する提案は取り下げる。代わりに、**module ABI のコピー税 vs Component Model + pooled dispatch の一点比較**（[[000067]] 柱 2 の証拠として最小構成）だけを候補として残す——全面横並びベンチより小さく、柱 2 の主張を直接補強する。
4. **発信を F1 + F2 完了に同期させる。** reference filter 棚と FDK が埋まる前に発信を最大化しても着地体験が摩擦で潰れる。対外発信の「弾」は「15 分で署名され conformance を通ったフィルタが実トラフィックを判定する」体験と、動く WAF/auth reference の実物であるべきで、それ以前の発信は最大化しない。
5. **Program F の外に残る提言の位置づけを確認する（deferred 序列は変えない）。**
   - **初日体験レシピ**: 外部 ACME クライアント（certbot / lego 等）+ SIGHUP reload の一級公式レシピ（compose 例込み、[[000043]] 2026-07-09 追記の (a)）は、F1/F2 の完了にゲートされない独立した docs 作業として随時実施する。同追記のうち F1 スライスに編入されるのは (b)（`plecto dev` のローカル証明書）のみで、(a) の実施先は本項とする。
   - **mTLS（R1）**: F1 + F2 の完了後、native 側の次スライスとして [[000056]] deferred 序列の先頭どおり mTLS へ復帰する。本 ADR は序列そのものを変えず、F1/F2 との相対順序だけをここで確定する。
   - **streaming の待ち時間仕事**: 実験系 `plecto:filter-streaming`（[[000064]] で実験 package として併載）のサーバ側配線と E2E を、外部ゲート（`wasm32-wasip3` の Rust Tier 2 昇格・component-model-async の安定化）の解除後に速やかに有効化できる状態まで進める。あわせて、buffer-then-decide（[[000025]]）で今日カバーできる範囲（16 MiB 以下の判定・書換系ユースケース）と真 streaming の境界を docs で線引きし、「できない」と「まだしない」を区別して記述する。いずれも F1/F2 をブロックしない背景作業とする。

検討した代替案と却下理由:

- **Program F の 6 柱を均等に並行進行する**——却下。[[000056]] は「R/F の交点を最優先成果物とする」とだけ定めており、Program F 内部での明示的な優先順位は未確定だった。開発者調査の分析（コア価値の柱 1 を体験させる入口と棚が無ければ他の柱が実証されない）を踏まえ、均等進行ではなく FDK + reference filter を明示的に最優先へ固定する。
- **比較ベンチハーネスを予定通り進める**——却下。単独開発でのコストが P9（性能測定）の価値に見合わず、かつ同種のベンチを既に公開している先行事例があり後追いの期待値が低い。一点比較という最小構成に絞る。
- **reference filter を FDK の後に回す（v1.0 のギャップ順のまま）**——却下。需要側の構造（Context 参照）を踏まえ、「拡張機構だけでは需要にならない」という洞察を優先順位に反映する。FDK と reference filter は両輪であり、順番ではなく同時に進める。

本決定は [[000056]]（Program F 編成そのものは変えない）に「Program F の中で何を先にやるか」を足す governance ADR であり、[[000037]] / [[000043]] / [[000064]] / [[000065]] の各判断内容（配置・declined・実装方針）は一切変えない。[[000067]] が定めた三本柱、特に柱 1（検証可能な信頼）を実体験として提供する最短経路を優先順位として固定する。

## Consequences

### Pros
- TTFVF 15 分という North Star（[[000065]]）が計測可能な実装を持つようになる——同 ADR の再検討条件 (a) がようやく機能する状態に近づく。
- reference filter 棚が「WAF は拡張プレーンに置く」（[[000037]]）という設計判断の説得力を実物で示せるようになる。
- 発信タイミングを F1 + F2 完了に同期させることで、着地体験の摩擦で発信効果が潰れるのを防ぐ。

### Cons / Tradeoffs
- 比較ベンチハーネスの格下げにより、性能面の対外説明は当面 µs 単位の一点比較のみに絞られ、横並びベンチほどの説得力は持たない。
- mTLS（R1、[[000054]] / [[000056]] の deferred 序列先頭）は F1 + F2 の後に回るため（Decision 5）、審査票の空欄は当面埋まらないまま。F1/F2 を優先する判断そのものは「機能で見劣りする」という外部批判への直接の反論にはならない。
- **再検討条件**: (a) TTFVF 実測（[[000065]] の外部協力者による完走テスト）が計測可能になった後、目標 15 分から大きく外れ続ける場合、ボトルネックを計測で特定し優先度を再配分する；(b) reference filter 棚を揃えてもなお発信への転換が乏しい場合、認知獲得の優先度を単独で引き上げる；(c) mTLS（R1）の欠如が実際の導入見送り理由として顕在化した場合、F1/F2 完了を待たず割り込ませる。
