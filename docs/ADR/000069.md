# Kubernetes controller / xDS 相当のフル対応を declined とし、Gateway API は成熟度の物差しに限定する

## Status
accepted

## Date
2026-07-09

## Affected Components
- roadmap / governance — Kubernetes 向けのフル controller 実装（CRD 監視・Gateway API controller・xDS 相当の動的 config push）を **native declined** として明文化する。コード変更は伴わない
- control plane（`plecto-control`）— k8s 環境での実務上の代替（[[000044]] の periodic DNS 解決、[[000056]]/[[000057]] の PROXY protocol v2）が既に実装済みであることを、本 ADR から一箇所で辿れるようにする

## Context

[[000008]] は single-node-first・静的宣言的設定 + 無停止 reload を確定し、xDS 的な動的 push を明示的に不採用とした。[[000034]] は Kubernetes **Gateway-API の conformance level（Core / Extended）**を、match semantics・precedence・weighted `backendRefs` の**設計物差し（projected）**として引用し、実際の conformance 認証ではないと明記した。[[000044]] は代替案検討の中で「フル SD（xDS / Gateway API controller / k8s API 監視）」を、[[000008]] の single-node-first 方針との衝突・規模の大きさを理由に却下している。

これらは実質的に Kubernetes フル対応（controller 実装 / Gateway API のサーバーサイド実装）を declined としているが、**いずれも他の決定の副次的な記述**（[[000034]] は「物差しとして使うだけ」の注記、[[000044]] は代替案却下の一項目）に留まり、単体で発見できる明文化がされていない。開発者調査（[[000067]] が採用した分析）が「意図的空白の再確認」として指摘した通り、[[000029]] が求める「何を declined にしたかの言語化」（P8: 運用としてのやらないことの言語化）を満たすには、Kubernetes / Gateway API 非対応そのものを主題とする ADR が一本要る。

一方で、Plecto は Kubernetes 環境を積極的に**無視している**わけではない。実務上の代替は既に個別 ADR で着地している:

- upstream にコンテナオーケストレータのサービス名（headless Service 等）を書いた構成は [[000044]] の periodic DNS 解決で追従する。
- 前段が L4 LB の k8s 環境で client IP を保全する経路として、`externalTrafficPolicy: Local`（クラウド LB の実転送に依存する条件付き）または PROXY protocol v2（[[000056]] / [[000057]]）を運用ガイドに記載する方針が既に確定している。

つまり Plecto の立ち位置は「Kubernetes を無視する」ではなく「**Kubernetes の controller/API を Plecto 自身が話すことはしないが、Kubernetes 環境で運用できる個別の運用パターンは提供する**」であり、この区別を一本の ADR として立てる。

## Decision

1. **Kubernetes 向けのフル controller 実装を native declined とする。** CRD 監視、Gateway API controller のサーバーサイド実装、xDS 相当の動的 config push は実装しない。[[000008]] の single-node-first・静的宣言的設定という上位方針と構造的に衝突し、規模も他の native 機能とは桁が異なる（[[000044]] の却下理由を正式に継承・一般化する）。
2. **Gateway API 仕様は今後も設計の物差し（projected）としてのみ引用する。** [[000034]] が実践した「conformance level を機能選定の参考にする」使い方は維持する。**Gateway API 準拠（conformance 認証）を名乗ることはしない**——実際に満たしているのは Core の一部（path-prefix / header exact match / weighted backendRefs 相当）に留まり、フル conformance テストは追っていない。
3. **k8s 環境での実務上の代替は個別の運用パターンでカバーする。** サービス名前提の upstream 追従は periodic DNS 解決（[[000044]]）、LB 配下での client IP 保全は `externalTrafficPolicy: Local` または PROXY protocol v2（[[000056]] / [[000057]]）。いずれも Multi-Replica Operations Guide（[[000056]] R7）に記載する。
4. **本 ADR は declined の明文化そのものであり、新たな実装を伴わない。** 審査票や issue で「Kubernetes / Gateway API 対応」を問われた場合、以後は本 ADR を参照して「controller/xDS 相当は non-goal、Gateway API は設計物差し、運用パターンで k8s 上の実利用は成立する」と答える。

検討した代替案と却下理由:

- **明文化せず現状（[[000034]]/[[000044]] への副次的な記述）のままにする**——却下。開発者調査が指摘した通り、他の ADR に埋め込まれた却下は「意図的空白」として対外的に説明する材料にならない。[[000029]] が求める declined の言語化には単体の ADR が要る。
- **将来的な Gateway API controller 実装の方向で検討を残す（deferred にする）**——却下。single-node-first（[[000008]]）は「分散はオプトイン」という上位方針であり、フル SD / controller 実装はその方針の根本的な転換を要する規模の変更である。曖昧な deferred ではなく明確な declined として線を引く方が [[000029]] の役割駆動運用に忠実。
- **Gateway API 準拠を対外的に謳う**——却下。実際には Core 相当の一部（[[000034]]）のみで、フル conformance は追っていない。誤認を招く主張はしない（[[000067]] 柱 3: 宣言された意味論、と整合）。

本決定は [[000008]]（single-node-first）・[[000029]]（役割駆動・declined の言語化）を Kubernetes/Gateway API という具体的な領域について明文化するもので、[[000034]] / [[000044]] の判断内容を変えず、両者に散在していた declined を一箇所に集約する。

## Consequences

### Pros
- 「意図的空白」が理由付きで一箇所に記録され、審査票や issue での議論を本 ADR への参照で短縮できる（[[000029]] が予告した負担の前払い）。
- k8s 環境での実務上の代替（[[000044]] / [[000056]] / [[000057]]）が本 ADR から一枚で辿れるようになり、「非対応」と「運用パターンでカバー」の区別が対外的に伝わる。

### Cons / Tradeoffs
- k8s ネイティブ統合（CRD・controller・Gateway API サーバーサイド実装）を求める運用者には引き続き届かない。
- Gateway API 準拠を審査項目にする組織には「非対応」と明確に答えることになる。柱 1（[[000067]]）の近期顧客（セルフホスター・データ主権志向チーム）には影響しないが、将来的な審査部門顧客との間には摩擦が残る。
- **再検討条件**: (a) k8s 関連の実需（開発者調査の需要側反証と同種の実証）が強く出た場合、[[000044]] の再検討条件 (b) に沿って個別 ADR でフル SD を再検討する；(b) Gateway API 仕様側に単一ノード運用にも適用可能な軽量プロファイルが登場した場合、再評価する。
