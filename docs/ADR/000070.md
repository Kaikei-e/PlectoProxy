# Program F2 の初出荷を JWT reference に絞り、鍵供給・検証意味論・trusted 必須を確定する

## Status
accepted（設計判断は 2026-07-10 の grill セッションで合意済み。**実装ノート 2026-07-10:**
`plecto/examples/filters/filter-jwt` を追加。静的 PEM 経路は host テスト
`cargo test -p plecto-host --features outbound-http --test jwt` で green。JWKS 経路は init 時
outbound 失敗が load 失敗になることまで検証（loopback は SSRF 床で常時拒否のため、成功取得の
E2E は CI では扱わない）。verify-only の `p256`/`rsa` で JWS を実装（ECDSA/RSA の検証は RNG を
要さないため、この guest は `getrandom`/`wasi:random` を一切 import しない）。`jsonwebtoken` は
その `getrandom` バックエンドが固定する `wasi` クレートのリリースが outbound-http host の
wasmtime-wasi 供与と component-link 段階で噛み合わず、採用を見送った。空の `outbound_http.allow`
を deny-all として許可するよう control の validate を更新）

## Date
2026-07-10

## Affected Components
- roadmap / governance — [[000068]] が並べた F2 三部作（WAF / JWT・OIDC 級 auth / AI token-cost）のうち、**本スライスの出荷対象を JWT 検証 reference のみに絞る**。AI token-cost と WAF は先送り（[[000068]] の F2 編成そのものは supersede しない）
- examples/filters（projected）— `filter-jwt`（名称は実装時に確定可）を、一般的な JWS 検証アルゴリズム（RFC 7515 / 7519）＋ RFC 8725 BCP に沿った allowlist で実装する設計を確定
- control / host（運用契約・projected）— `isolation = "trusted"` 必須、静的 JWK または init 時 JWKS 取得のハイブリッド鍵供給、RFC 6750 準拠の 401 応答、成功時の identity stamp を実装時の受け入れ条件とする

## Context

[[000067]] は差別化を「検証可能な信頼 × 契約駆動の性能 × 宣言された意味論」の交差に置き、[[000068]] はその実体化として F1（Filter Dev Kit）と F2（reference filter 棚）を最優先にした。F1 の Rust 縦串は [[000065]] で実装済み。F2 は WAF・JWT/OIDC 級・AI 文脈を「一本ずつ」と書いたが、一度に三本を設計・実装すると縦切りが膨らみ、柱 1（検証可能な信頼）を **一つの署名済みフィルタ体験**として早く見せる目的と衝突する。

2026-07-10 の grill では当初 AI token-cost と JWT の両方を縦切りに載せる案（B→A）まで進んだが、AI 側は header-only 契約下での cost 推定（`max_tokens` 予約 vs body/tiktoken vs reserve→reconcile）が本体議論を食うと判断し、**AI token-cost を本スライスから先送り**し、JWT に集中する方針へ更新した。OIDC RP フローは [[000043]] どおり未判定のまま触れない。

JWT reference が埋める空白は明確である: 現状の auth 棚は `filter-apikey`（共有秘密ヘッダ）と `filter-extauthz`（outbound 委譲）に留まり、**ゲートウェイ自身が Bearer JWT を検証する**一段本格的な reference が無い（[[000068]] Context）。outbound HTTP（[[000036]]）は JWKS 取得を動機例に挙げており、能力面の前提は既にある。

本決定は Fork 6（user-policy は拡張プレーン）・P3（判断も不在も型で）・P4（fail-closed）・P5（検証してからロード）を、F2 の初出荷形として具体化する。[[000067]] / [[000068]] を supersede しない。

## Decision

### 1. F2 初出荷スコープ

1. **本スライスの出荷物は JWT 検証 reference（`filter-jwt`）のみとする。**
2. **AI token-cost reference は先送り。** [[000043]] / [[000026]] の `(key, cost)` 契約はそのまま有効だが、本 ADR の実装対象に含めない。再開条件は Consequences を参照。
3. **WAF reference は [[000068]] どおり F2 の残件として残し、本スライスでは着手しない。**
4. **OIDC RP / developer portal は [[000043]] の未判定のまま。** 本フィルタは Resource Server としての JWT 検証に限定する。

### 2. 鍵供給（ハイブリッド）

運用者は `[filter.config]`（[[000066]]）で次のいずれか一方を**必須**とする（両方欠如・両方同時指定は `init` trap → load 失敗）:

- **静的パス**: 検証用公開鍵（JWK または PEM）を config に置く。outbound 不要。オフライン / FDK ドッグフード向き。
- **JWKS パス**: `jwks_url` を宣言し、`init` 時にだけ outbound HTTP（[[000036]]）で取得して host-KV 等にキャッシュ。reload まで固定（リクエスト経路での TTL 再取得は初版に含めない）。

これは業界で広く見られる **「インライン鍵」と「リモート JWKS」のどちらか一方**という一般形（XOR）であり、標準の JWK Set（RFC 7517）取得と静的鍵配置の組み合わせとして採用する。

### 3. アルゴリズムとクレーム（RFC 8725 に整合）

- **許可 alg**: `ES256` と `RS256` のみ。`none`・HS\*・その他は拒否（ヘッダの `alg` を信頼して鍵種を切り替えない）。
- **必須クレーム**: `exp` 必須（`host-clock` で検証）。`iss` / `aud` は config で期待値を必須化し一致検証。`sub` 必須（後段フィルタの主体キー用）。`nbf` があれば検証。
- 時刻ソースはゲスト `SystemTime` ではなく **`host-clock`**（契約済み）。

### 4. 失敗時 decision と成功時 stamp（RFC 6750）

- トークン欠落: **401** short-circuit。`WWW-Authenticate: Bearer realm="…"`（error 属性なし）。
- 不正・期限切れ・alg 拒否等: **401** + `error="invalid_token"`（必要なら短い `error_description`）。
- 初版は scope 検証をしないため **403 / `insufficient_scope` は使わない**。
- 成功: `modified` で `x-authenticated-user` = `sub`、`x-jwt-issuer` = `iss` を stamp（クライアント spoof は上書き。`filter-apikey` と同型の防御）。`Authorization` は除去しない。

### 5. `isolation = "trusted"` を必須とする

**結論: 静的パス・JWKS パスの両方で `isolation = "trusted"` を必須とする。**

根拠は、公開されているゲートウェイ実装に共通して見られる運用パターンと、自リポの不変則の交差である（個別製品名は挙げない）:

| 観察（確定事実として一般化） | Plecto への写像 |
|------------------------------|-----------------|
| リモート JWKS を採る構成では、鍵集合の初回取得を **リスナー起動前／ロード時** に置き、リクエスト経路の関心事にしないことが多い | JWKS 準備は **ロード/起動の関心事** |
| JWKS クライアントはプロセス起動時に一度初期化し、共有キャッシュへ載せる形が一般的 | init-once + キャッシュ（trusted プール） |
| [[000066]] — 必須業務設定の欠如は `init` trap。**trusted なら load 失敗、untrusted なら毎リクエスト 503** | issuer/aud/鍵の必須検証を「undeclared fails load」にするなら trusted 必須 |
| Plecto untrusted ライフサイクル — 毎リクエスト instantiate + init。init deadline は untrusted=250ms（[[000027]]） | JWKS outbound を毎リクエスト init で行うのは構造的に不適（レイテンシ・期限・IdP 負荷） |

したがって:

- **JWKS パス**: init-once 取得＋キャッシュが本質 → trusted 一択。
- **静的パス**: outbound は不要だが、必須 config 欠如を **load 時 fail-closed** にするには [[000066]] と同じく trusted が必要。untrusted でも動かし得るが「毎リクエスト 503」に化け、運用上の宣言意味論（柱 3）が崩れるため採用しない。

`filter-ratelimit-redis`（[[000061]] / [[000066]]）と同じ文書規律: README と example manifest に `isolation = "trusted"` を明記する。

### 6. 実装メモ

- ゲストは `wasm32-wasip2`（JWKS が `wasi:http/outgoing-handler` を要するため）。静的 PEM 経路でもコンポーネントが同 interface を import するため、manifest では `[filter.outbound_http]` を宣言し、静的のみなら `allow = []`（deny-all）でよい。
- 署名検証は RFC 7515 の compact JWS 手順（分割 → 署名検証 → クレーム検証）を `p256` / `rsa` の verify-only で実装する（ECDSA/RSA の署名検証自体は RNG を要しないため、この経路は `wasi:random` を import しない）。フル機能の JWT クレート（`jsonwebtoken` 等）はその `getrandom` バックエンドが固定する `wasi` クレートのリリースが wasip2 上で outbound-http host の wasmtime-wasi 供与と component-link 段階で噛み合わなかったため採用していない。
- FDK（[[000065]]）でスキャフォールド→署名→conformance→reload をドッグフードする。

検討した代替案と却下理由:

- **AI token-cost を同スライスで並行** — 却下。cost 推定境界（header-only 制約）が JWT 設計を遅らせる。
- **JWKS をリクエスト経路で TTL 更新（初版）** — 却下。レイテンシ・失敗時 fail-closed・キャッシュ無効化が本体を食う。reload 固定で足りる。
- **HS256 をデモ用に許可** — 却下。RFC 8725 の alg 混同攻撃面を reference の既定に入れない。必要なら後続で明示 opt-in。
- **静的パスだけ untrusted 可** — 却下。必須設定の load 時 fail-closed（[[000066]]）と運用の単純さが壊れる。
- **JWT 検証を native fast path へ** — 却下。[[000029]] / Fork 6 の user-policy 配置に反する。

## Consequences

### Pros
- [[000068]] F2 の「JWT/OIDC 級」を、OIDC RP に踏み込まず **検証可能な最小の本格 auth reference** として切れる。
- 柱 1（検証可能な信頼）を、FDK 経由の署名済み JWT フィルタという一本の体験に落とせる。
- 鍵供給ハイブリッドにより、オフライン完走と JWKS 本番形の両方を同じフィルタで示せる。
- trusted 必須を、業界で一般的な「起動／ロード時 JWKS ゲート」パターンと自リポ不変則（[[000066]]）の両方から根拠付けでき、後から「なぜ untrusted で動かない」と聞かれにくい。

### Cons / Tradeoffs
- AI token-cost / WAF の先送りにより、F2 三部作の完了は遅れる。需要側（AI 文脈）への直接応答も遅れる。
- JWKS 初版は reload まで鍵固定のため、IdP 側の緊急鍵ローテーションは **manifest reload** に依存する。
- trusted 必須は「第三者の untrusted JWT フィルタをそのまま載せる」物語とは緊張する——本 reference は **運用者所有の auth ゲート**として位置づける。
- **`isolation = "trusted"` は manifest validation では強制されない。** [[000066]] の一般則は「必須 config の欠如」を load 時 trap にするだけで、config が揃った状態で運用者が `isolation` を書き忘れた（既定値は `untrusted`）場合までは検出しない。JWKS パスを untrusted のまま動かすとリクエスト毎に live fetch が走り、untrusted の tight な init deadline に収まらず 503 が続く——実害は `pool.rs` の untrusted breaker（連続 init 失敗でクールダウンに入る既存の回路遮断）が下限を作るため無制限の連打にはならないが、恒常的な認証不能という形で気づかれる。filter 単位で必須 isolation を宣言・検証する仕組みは本 ADR の範囲外とし、READMEでの明記に留める。
- **再検討条件**: (a) wasm 上の暗号依存が成立せずビルド不能になったら、host-side 検証 capability を再評価する；(b) IdP 鍵ローテーション頻度が reload 運用で足りないと実測されたら、init 外の安全な JWKS 再取得（TTL + fail-closed）を増分 ADR で足す；(c) AI token-cost の header-only 縮小形（例: `max_tokens` 予約）だけで需要実証に足りると判断したら、本 ADR を待たず別スライスで再開する；(d) 静的パスのみの軽量デプロイで trusted が過剰摩擦になった実測があれば、静的パス限定の untrusted 許容を再検討する（JWKS パスは維持）。
