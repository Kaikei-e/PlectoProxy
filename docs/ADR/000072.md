# plecto new-filter の WIT 取得をネットワーク wkg からバイナリ同梱（self-vendoring）へ切り替える

## Status
accepted（判断は 2026-07-11 の grill セッションで合意。実装は後続スライス。暫定の 1 行 bump は先行可）

## Date
2026-07-11

## Affected Components
- server（CLI） — `plecto/crates/server/src/new_filter.rs`: `fetch_wit()` の `wkg` 子プロセス起動を廃し、`include_str!` で同梱した契約テキストをスキャフォールドへ書き出す
- docs — `docs/writing-a-filter.md` の外部作者向け `wkg get` 手順は配布チャネルとして残す（バージョン表記は現行契約へ追随）

## Context

開発者調査（Alt 導入トライアル反映、2026-07-10）が、[[000065]] の North Star である TTFVF 15 分を壊す欠陥を実地で露出させた:

- `new_filter.rs:115` が `wkg get plecto:filter@0.1.0` をハードコードしており、[[000071]] が 0.2.0 を契約正文にした後も **スキャフォールドは deprecated 契約を生成する**。作った直後のフィルタがロード時 deprecation warning を出す体験は、「15 分で署名済みフィルタが実トラフィックに載る」の中に置けない。
- 根本原因は文字列 1 箇所ではなく構造にある。契約バージョンがコード中の文字列として存在する限り、契約リリースごとに人間が bump する**運用ルール**（binary と契約は同一コミットで揃える、いわゆる same-commit rule）に依存する。このルールは今回すでに一度破れた。
- `wkg` ネットワーク取得は、スキャフォールドの成立を **レジストリ到達性と publish 完了**に依存させる。トライアル時点で `wit-publish` job のグリーンは未確認のままであり、「0.2.0 が publish される前のウィンドウでは正しいバージョンへ bump してもスキャフォールドが壊れる」という時間窓が構造的に存在する。

前提の確定事実: [[000064]] により WIT は wkg/OCI Artifact でレジストリへ publish 済みの配布チャネルを持つ。[[000065]] は `wasm-pkg-core` を**ライブラリとして**組み込む案を「stable な公開 API を確認できない」ため見送り、`wkg` 子プロセスを選んだ経緯がある（`new_filter.rs` モジュールコメントに記録）。

## Decision

1. **暫定（即時・1 行）**: `new_filter.rs` の取得対象を `plecto:filter@0.2.0` へ bump する。`writing-a-filter.md` の該当記載も追随する。
2. **恒久（本 ADR の主判断）**: `plecto new-filter` は WIT をネットワークから取得せず、**ビルド時に同一ソースツリーの `plecto/wit/` から `include_str!` で同梱した契約テキスト**をスキャフォールドへ書き出す（self-vendoring）。ホストの bindgen が読むのと同じファイルを同梱するため、「バイナリが実行できる契約」と「バイナリが scaffold する契約」の一致が**運用ルールではなく構造**で担保される。バージョン文字列の bump 作業自体が消滅する。
3. **[[000064]] の wkg/OCI 配布チャネルは無変更で維持する。** 外部・polyglot 作者が任意バージョンの契約を digest 固定で取得する経路はレジストリのまま。本 ADR は `new-filter` がその経路に**依存しない**ようにするだけで、配布ポリシーを変えない。

検討した代替案と却下理由:

- **wkg 取得を維持し、リリース手順書に bump チェックを追加する** — 運用ルールの強化は再発防止にならない（今回の欠陥そのものが運用ルールの破れ）。テンプレート（`TEMPLATE_CARGO_TOML` / `TEMPLATE_LIB_RS`）は既に `include_str!` 同梱であり、WIT だけネットワークに残す一貫性もない。却下。
- **build.rs でビルド時に wkg 取得して埋め込む** — bump 忘れは防げるが、ビルドがネットワークとレジストリ publish 順序に依存するようになり、オフラインビルド・再現ビルド（開発者調査の非再現ビルド懸念）と衝突する。同一ツリーに正文（`plecto/wit/`）があるのに外から取る理由がない。却下。
- **`wasm-pkg-core` をライブラリとして組み込む** — [[000065]] 時点の調査で stable な公開 API を確認できず見送った経緯があり、self-vendoring はそもそも取得処理自体を不要にする。却下。
- **スキャフォールドが clone 済みリポジトリの `wit/` を参照する** — 「外部作者は Plecto を clone しない」という [[000064]] / [[000065]] の目標に反する。却下。

本決定は P5（自分自身にも同じ規律を適用する）の適用である: フィルタ供給網には署名で same-commit 相当の束縛を要求しながら、自身のスキャフォールドが契約バージョンを運用ルールで揃えているのは非対称だった。

## Consequences

### Pros
- 契約バージョン不一致のスキャフォールドが**構造的に発生しなくなる**（バイナリと契約が常に同一コミット）。
- `plecto new-filter` がオフラインで完結し、レジストリ障害・publish 前ウィンドウ・GHCR レート制限の影響を受けない。TTFVF の分散が減る。
- `wkg` 子プロセスへの実行時依存が消え、FDK の前提ツールが減る。

### Cons / Tradeoffs
- 同梱されるのはバイナリが積んだ 1 バージョンのみ。旧契約や実験契約（`plecto:filter-streaming`）でスキャフォールドしたい場合は従来どおり wkg 経路を手で使う（docs に明記する）。
- バイナリサイズが契約テキスト分（数 KB）増える。無視できる。
- **再検討条件**: (a) 複数契約バージョンの scaffold 需要が実測で出たら、`--contract-version` フラグ + wkg 経路の明示オプションとして再導入する；(b) WIT が複数ファイル / 外部 deps を持つ構成に育ったら、単一 `include_str!` では足りないため同梱方式を manifest 化する実装 ADR を切る。
