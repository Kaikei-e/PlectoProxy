# on-response に request context を渡し response-decision に replace を加える plecto:filter@0.3.0 を導入する

## Status
accepted（判断は 2026-07-11 の grill セッションで合意。実装は後続スライス。[[000071]] の移行レール一式を踏襲する）

## Date
2026-07-11

## Affected Components
- wit — `plecto/wit/v0.3.0/world.wit`（新設・凍結）と開発デフォルト `plecto/wit/world.wit` の 0.3.0 化
- host — トリプル bindgen（0.1 / 0.2 / 0.3）、ロード時バージョン判定の `@0.3.` 対応、`replace` 出力の fail-closed 検証
- fast path — response フェーズへの request スナップショット供給、`replace` の合成応答送出
- examples/filters — in-tree フィルタの 0.3 移行。F2 reference 棚 3 本目として `filter-cors` が書ける前提を作る

## Context

開発者調査（Alt 導入トライアル反映、2026-07-10）が、現行契約の **response 側の構造的欠落** を 2 点確定させた。いずれも実装不足ではなく契約の形の問題である:

1. **`on-response: func(resp: http-response)` にはリクエスト側コンテキストへの経路がゼロ。** CORS の dynamic origin echo（`Origin` リクエストヘッダを見て `Access-Control-Allow-Origin` を応答に反映する）が契約上表現できない。guest グローバルで持ち越す裏技も成立しない — pool は `on_request` / `on_response` を**独立に checkout** し（`pool.rs` の `run_pooled`、untrusted は毎フック fresh インスタンス）、同一リクエストの両フックが同一インスタンスに載る保証は仕様上存在しない（[[000011]] のステートレス原則の帰結でもある）。
2. **`response-decision` は `continue | modified(response-edit)` のみで、合成ボディへの差し替えが契約レベルで不可能。** `response-edit` に body フィールドは無い。一方 fast path 内部には既に「編集後 body が非空なら合成応答として送出し upstream ストリームを捨てる」という**帯域内シグナル**が存在する（`proxy.rs`、trap 時の fail-closed 合成 5xx 用）。つまり機構はあるのに guest からは型で到達できない。

一般パターンの調査（取得日 2026-07-11）では、response フックへの request context の渡し方は 3 形に大別される:

- **(a) per-request インスタンスがフェーズ跨ぎで状態を保持** — プール再利用と敵対する（in-flight リクエストがインスタンスを request→response 間ピン留めし、インスタンス再利用時の状態残留はゼロ化事故のリスク類型 CVE-2022-39393 と同族）。P6（フィルタはステートレス）にも反する。
- **(b) 明示パラメータとして渡す** — response フックが引数の純関数になり、プールされたどのインスタンスでも応答フェーズを処理できる。契約シグネチャ自体が「response フィルタが見られる request データ」を宣言する。
- **(c) host-API で query する** — シグネチャは不変で済むが、「どのフェーズでどの値が引けるか」という暗黙の可用性マトリクスが契約の細目と化す。vendor 中立の proxy-wasm ABI 仕様はまさにこの形で、request ヘッダ map は request フェーズのコールバックに限定され、フェーズ跨ぎのプロパティ可用性はホスト実装依存の未規定領域として実運用の混乱が報告されている。柱 3（Declared Semantics、[[000067]]）と正面衝突する。

合成差し替えについては、proxy-wasm ABI 仕様が response フェーズでも `send_local_response`（応答ヘッダ送出前に限り有効）を許しており、「modify とは別の第一級 decision としての replace」は業界横断で確立した形である。

移行レールの前提（確定事実、2026-07-11 時点）: wasmtime の semver 自動解決は **import 側のみ**で、guest が export する world（`plecto:filter` はこれ）には効かない（wasmtime issue #8395、未解決）。したがって [[000071]] が敷いたロード時バージョン判定 + 明示的 dual-bind がエコシステム上も正攻法であり、0.3 でも同じレールに `V03` arm を足すだけでよい。なお WASI 0.3.0 は 2026-06-11 に正式リリース、wasmtime 46 で component-model-async がデフォルト有効（確定事実）だが、`wasm32-wasip3` は Rust Tier 3 のまま（Tier 2 化は 2026 年の projected）。よって 0.3.0 契約は sync・header-only を維持し、`stream<u8>` 化は従来どおり別スライス（[[000020]] / [[000038]] 再検討条件）に残す。

## Decision

1. **`plecto:filter@0.3.0` を契約正文として追加する。** 変更は response 側の 2 点のみ:
   - `on-response: func(req: http-request, resp: http-response) -> response-decision` — 上記 (b) の明示パラメータ形。
   - `variant response-decision { %continue, modified(response-edit), replace(http-response) }`。
2. **`req` は as-forwarded スナップショットである（Declared Semantics として WIT コメントに明記する）。** request チェーン適用後、実際に upstream へ送った header-only の形を渡す。auth フィルタが stamp した `x-authenticated-user` 等を response フィルタが利用でき、ホストの保持は canonical request 1 系統で済む。`Origin` のような通常無編集のヘッダは as-forwarded にもそのまま残るため、CORS ユースケースも同時に満たす。
3. **型は既存 `http-request` record を再利用する。** WIT の record は値渡しであり構造的に read-only。内容（method / path / authority / scheme / headers）は必要十分で、専用 record の新設は型の重複と変換コードを増やすだけ。
4. **arm 名は `replace` とする（P12）。** request 側 `short-circuit` は「upstream を経ずに返す」、response 側は「届いた応答を差し替える」で意味が異なる。variant 名が意味論を宣言する。`replace` は**合成**であって upstream ボディの読み取りではない — `on-response` に渡る `resp.body` は従来どおり常に空（header-only）で、upstream ボディは zero-copy でストリームされ続けるため、[[000038]] の不変則は無傷。guest が返した `replace(http-response)` は request 側 `short-circuit` 出力と同一の fail-closed 検証（[[000071]] のヘッダ検証規則）を通す。これにより `proxy.rs` の帯域内シグナル（body 非空 = 合成）は契約上の型付き decision に昇格する（P3）。
5. **移行は [[000071]] のレールを踏襲する。** 0.2 を `wit/v0.2.0/` に凍結（済み）、host は `BoundFilter::V03` を追加してロード時に decoded imports の `@0.3.` で判定。0.2 guest への呼出は `req` 引数を落とすだけ（投影不要の薄いアダプタ。0.1 guest は既存の lossy 投影のまま）。[[000064]] の互換ポリシーどおり host は出荷した全契約バージョン（0.1 / 0.2 / 0.3）を読み続け、新規・in-tree のデフォルトは 0.3、0.2 ロード時は stderr に 1 回 deprecation warning（0.1 と同じ流儀）。

検討した代替案と却下理由:

- **host-API `get-request-context()` で query（上記 (c)）** — 可用性マトリクスが暗黙契約化し、フェーズごとの「いつ呼べるか」が仕様の細目になる。proxy-wasm の実運用がこの形の脆さの前例。ホット経路のホスト呼び出しも増える。却下。
- **per-request インスタンスのフェーズ跨ぎピン留め（上記 (a)）** — プールの checkout モデルの根本変更で、P6 と衝突し、in-flight リクエスト数ぶんのインスタンスがピン留めされ、状態残留リスク類型を抱え込む。却下。
- **arm 名を `short-circuit` で対称にする** — 字面の対称性は上がるが「何を短絡するのか」が response 側では不正確になる。P12 により却下。
- **専用 `request-context` record の新設** — 「ビューであって編集対象でない」ことを名前で言えるが、値渡しの時点で構造的に read-only であり、型の重複に見合わない。却下。
- **downstream 原形と as-forwarded の両方を渡す** — 保持コスト 2 倍・契約肥大。必要になれば minor 追加で足せる。却下。
- **`wasi:http` 収束（[[000020]]）を待って解決する** — wasip3 が Tier 3 のままで時期が読めず、待つ間 F2 reference（`filter-cors`）が書けない。却下（従来どおり別スライス）。

本決定は P3（判断は型で）を response 側に対称化するものであり、[[000029]] の配置基準（CORS / error-page 型はいずれも per-request の user-policy = 拡張プレーン）に新しい例外を作らない。

## Consequences

### Pros
- CORS dynamic origin echo と error-page 型の自己修復（status とヘッダだけ見て合成応答へ差し替える）が**フィルタとして書ける**。F2 reference 棚 3 本目 `filter-cors` の前提が整う。
- fast path 内部の帯域内シグナルが型付き decision に昇格し、暗黙の機構が契約に宣言される（P3 / 柱 3）。
- 移行レールは [[000071]] で一周実証済みで、host 側は `V03` arm の追加が中心。起票から実装までの距離が短い。

### Cons / Tradeoffs
- ホストはフィルタ付き route で as-forwarded リクエストの header スナップショットを response フェーズまで保持する（既存ヘッダ上限に拘束されるため有界）。`on-response` 呼出ごとに request ヘッダのコピーが guest へ渡る — フィルタが使わなくても払うコストで、`on-request` と同じコスト階級。
- トリプル bindgen（0.1 / 0.2 / 0.3）がホストに常駐する。
- pre-1.0 のため 0.3 デフォルト化で全フィルタの再ビルドが必要（0.2 は警告付き猶予、[[000064]] ポリシーどおり）。
- **再検討条件**: (a) [[000071]] の条件どおり 0.1 ロードが 6 ヶ月ゼロになったら `V01` arm の削除を ADR 化する（0.2 にも同じ物差しを適用する）；(b) `wasi:http` 収束スライスで record / header 型を再統合する（[[000071]] (a) を継承）；(c) request スナップショットのコピーコストが実測で問題化したら、縮小ビューや lazy 化を再検討する（可用性マトリクスを作らない形に限る）。
