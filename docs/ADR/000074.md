# deferred 序列の圧縮を h3 graceful drain より前へ引き上げ、caching declined に初の一次データを記録する

## Status
accepted

## Date
2026-07-11

## Affected Components
- roadmap / governance — deferred 序列のみ改訂。各スライスは着手時に個別 ADR で確定する（[[000029]] の運用どおり）
- docs/ADR — [[000043]] への dated 追記（序列 4 回目改訂の参照、caching 再検討条件 (a) の一次データ）

## Context

[[000067]] は柱 1（Verifiable Trust）の近期顧客を**セルフホスター**と名指しした。一方、現行の deferred 序列 mTLS → PROXY protocol v2 → h3 graceful drain → 圧縮 は、[[000054]]（SaaS 導入水準の品質ターゲット）と [[000056]]（前段 LB + N レプリカ標準形）の文脈で組まれたものであり、**名指しした顧客に最も効く項目（圧縮）が最後尾**にいる。序列の根拠とポジショニングの根拠がずれた状態である。

開発者調査（Alt 導入トライアル反映、2026-07-10）が、このずれの実害を初めて一次データとして示した:

- **caching declined（[[000043]]）と圧縮不在は複合する。** キャッシュも gzip も無い構成では全アセットが毎リクエストでフルサイズのまま流れる。セルフホスト（家庭・VPS 回線）では帯域が最も希少な資源であり、複合の実害が直撃する。
- **「前段 CDN に外出しする」という caching declined の代替は、セルフホスト志向と矛盾する。** [[000043]] 自身の判断基準「代替可能な機能より代替のない穴が先」をセルフホスター視点で適用すると、圧縮にはプロダクト内の代替が存在しない。
- 一方、h3 graceful drain（GOAWAY）はローリングリスタート・マルチレプリカ文脈（Program R）の品質項目であり、単体セルフホスト構成での欠如の実害は相対的に小さい。

一般的手法の調査（取得日 2026-07-11）では、response 圧縮は境界の確立した native 横断関心事であることを確認した: RFC 9110 §12.5.3 の content negotiation（`Accept-Encoding` + qvalue）、zstd（RFC 8878）は 2026 年時点で evergreen ブラウザ横断対応（確定事実。サーバ側実装は薄く差別化余地）、安全既定は content-type 許可リスト・最小サイズ閾値（実務水準 ~1KB）・既 encode / `Cache-Control: no-transform`（RFC 9110）/ 206 のスキップ・`Vary: Accept-Encoding` 付与・strong ETag の弱化・SSE / 非有限ストリーミングの除外（sync-flush 問題）・BREACH への route 単位オプトアウト、として業界横断で収斂している。圧縮は response フィルタチェーンの**後段**（フィルタは常に identity エンコーディングを見る）に置くのが確立形で、[[000073]] の response 契約とも干渉しない。

## Decision

1. **deferred 序列を再改訂する（4 回目）: mTLS → PROXY protocol v2 → 圧縮（gzip/br/zstd）→ h3 graceful drain（GOAWAY）。** mTLS（[[000054]] の品質ターゲット確定による暗号境界）と PROXY v2（[[000056]] が認定した `peer.ip()` 意味論の構造的穴の前提部品）の先行理由は不変のため追い越さず、圧縮を h3 drain の前へ引き上げる。序列は優先順位であり実装内容の先取りではない（[[000043]] の運用どおり）。
2. **caching = native declined は維持する。** Alt 導入トライアルを [[000043]] 再検討条件 (a)「前段 CDN を置けない単体セルフホスト需要」の**初の一次データ**として記録する。ただしトライアル自身が単一ユーザーなら許容と評価しており、declined を覆す閾値（需要が支配的）には未達。[[000043]] 本文に dated 追記を行う。
3. **圧縮スライスの実装 ADR への引き継ぎ制約を記録する**（本 ADR では決定しない）: native fast path 配置（[[000029]] 採用基準 (i) の横断的データプレーン関心事）、response フィルタチェーンの後段、RFC 9110 準拠の negotiation と no-transform 尊重、ETag 弱化 / `Vary` 付与、SSE・非有限ストリーミングの既定除外、BREACH への route 単位オプトアウト、codec 集合（gzip 基線 + zstd / brotli）の選定。

検討した代替案と却下理由:

- **序列先頭（mTLS より前）へ** — mTLS は SaaS 導入水準（[[000054]]）の確定判断かつ暗号境界の穴で、追い越すには顧客名指しの再変更級の材料が要る。トライアル 1 件では足りない。却下。
- **PROXY v2 の前へ** — PROXY v2 は「前段 LB 配下で per-client-IP rate limit / SourceIp Maglev の意味論が崩れる」構造的穴の前提部品（[[000056]]）であり、穴埋めが利便に先行するのは P4（fail-closed）系の一貫適用。却下。
- **序列を動かさない（再検討条件の追記のみ）** — [[000067]] が顧客を名指しした以上、序列の根拠がポジショニングとずれたまま放置するのは柱 3（宣言された意味論）の精神に反する。圧縮の順位変更に必要な材料（顧客適合の一次データ）と caching declined を覆す材料（需要が支配的）は閾値が異なり、前者は既に揃っている。却下。

## Consequences

### Pros
- deferred 序列が [[000067]] のポジショニング（セルフホスター名指し）と整合し、「なぜこの順か」を柱 3 の言葉で説明できる。
- caching declined の再検討条件 (a) に初の物差し（一次データ 1 件目）が付き、以後の判断が「件数と質の積み上げ」として評価可能になる。
- 圧縮スライスの実装 ADR が引き継ぎ制約から書き始められる。

### Cons / Tradeoffs
- h3 graceful drain の後送で、ローリングリスタート品質（Program R）の完成がさらに遅れる。マルチレプリカ文脈の顧客には不利な取引。
- 序列改訂はこれで 4 回目であり、序列自体の安定性への信頼を毀損しうる。以後の改訂は「顧客の名指し（[[000067]]）级の前提変化」を要求する運用とする。
- **再検討条件**: (a) SaaS / マルチレプリカ導入案件が h3 drain 欠如を実測の見送り理由として挙げたら、圧縮との順序を再逆転する；(b) caching — 一次データが複数ユーザー・実測帯域の水準で積み上がり、単体セルフホスト需要が支配的と言えるとき [[000043]] (a) の本判定を行う；(c) 圧縮スライス着手時に BREACH / ストリーミング制約でスコープが肥大するなら、実装 ADR でスライス分割を確定する。
