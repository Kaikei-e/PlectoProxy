# response 圧縮を native fast path に [route.compression] opt-in として導入し、RFC 9110 negotiation と安全既定一式を確定する

## Status
accepted

## Date
2026-07-11

## Affected Components
- server (fast path) — `crates/server/src/compression.rs`（negotiation・適格判定・ヘッダ書換・`CompressBody`）と `proxy.rs` の response 側 2 経路への適用。h1/h2/h3 は同一の `ResponseBody` ラップを継承（各 1 本の E2E で確認）
- control — `[route.compression]` スキーマ + validation + `RouteInfo.compression`（`Arc<CompressionConfig>`）
- 依存 — flate2 / brotli / zstd を server crate に追加（テスト側は同 crate を decode oracle として使用）

## Context

[[000074]] が deferred 序列の圧縮を h3 graceful drain より前へ引き上げ、実装 ADR への引き継ぎ制約
（native fast path 配置・response チェーン後段・RFC 9110 準拠 negotiation・no-transform 尊重・
ETag 弱化・`Vary` 付与・SSE 既定除外・BREACH の route 単位オプトアウト・codec 集合の選定）を記録した。
本 ADR はそのスライスの実装決定である。

実装前調査（取得日 2026-07-11、いずれも確定事実。2026-07-11 レビューで一部訂正）:

- **zstd の web 相互運用は RFC 9659 で確定**: decoder は 8 MiB window までサポート必須、encoder は
  8 MiB 超の frame を生成してはならない（MUST NOT）。ブラウザ対応は Chrome/Edge 123+ / Firefox 126+ /
  Safari 26+（macOS は partial 表記）。Chrome は HTTPS でのみ `zstd` を広告する、との報告が多い
  （公式一文の一次確認は弱い — 運用上は TLS 終端前提で問題にならない）。
- **Rust codec 事情**: gzip（flate2/miniz_oxide）と brotli（dropbox/rust-brotli）は pure Rust。
  **production 級の pure-Rust zstd encoder は未成熟**: ruzstd の encoder は速度・比率・設定性で
  libzstd 未達と自認。c2rust 由来の pure-Rust 移植も encoder は動くが unsafe が多く、安定
  リリース前。帯域（セルフホスターの最希少資源）を直接左右するため、当面は libzstd bindings。
- **libzstd（C）のセキュリティ実績**: ライブラリ本体のメモリ安全 CVE は 7 年で実質 1 件
  （CVE-2019-11922、**圧縮側**・推奨未満の出力バッファという API 誤用前提・1.3.8 で修正済）。
  CVE-2021-24031/24032 と CVE-2022-4899 は CLI のみ。2023–2026 の公開 CVE ゼロ、RUSTSEC advisory
  ゼロ、oss-fuzz 常時 + Meta bug bounty 対象。著名な伸長側メモリ破壊はサードパーティ再実装
  （7-Zip 独自デコーダ、ruzstd &lt;0.7.3）に出ており、本体ではない。
- **コード形状**: response フィルタチェーンは header-only（`on-response` の body は常に空、
  [[000073]]）であり、WASM フィルタには圧縮が原理的に書けない。upstream body は streamed
  `BoxBody<Bytes>` として `proxy_core` の単一点で client response になるため、そこでラップすれば
  h1/h2/h3 全てが一度の実装で継承する。[[000029]] 採用基準 (i)（横断的データプレーン関心事 = native
  適格）の教科書例。

## Decision

1. **配置: `proxy_core` の response 終端で `ResponseBody` をラップする単一挿入点。** filterless
   直通と チェーン後 `Forward` の 2 経路に適用し、`replace` / fail-closed の合成 response は対象外
   （host が framing を所有する小さな合成 body を圧縮する価値はなく、チェーン後段原則とも整合）。
   フィルタは常に identity を見る（[[000074]] 制約の充足）。
2. **opt-in は `[route.compression]` のみ（listener 既定層は作らない）。** ブロック不在 =
   一切変換しない deny-by-default。BREACH へのオプトアウトは「その route で有効にしない」ことで
   充足する — 反射型 secret（CSRF token 等をレスポンスに埋め、リクエストにエコーバックさせる
   経路）を持つ route は enable しない。運用警告は README（EN/JA）とスキーマコメントに置く。
3. **codec 集合は gzip + br + zstd の全部、zstd は C bindings（`zstd` crate）を条件付き採用。**
   条件: (a) `default-features = false` で最小 feature、(b) compress-only（decompress API には
   一切触れない — 攻撃面の大半である「攻撃者制御の圧縮フレームのパース」が発生しない）、
   (c) `window_log ≤ 23` を明示設定（RFC 9659 の encoder MUST NOT を宣言された意味論として固定）。
4. **negotiation は RFC 9110 §12.5.3 を自前実装**（qvalue 3 桁・`*`・`q=0` 除外・`x-gzip` 別名・
   明示 `identity` の qvalue 競合）。クライアントの非ゼロ最大 qvalue が勝ち、同点は route の
   `algorithms` 順（server preference、既定 `["zstd", "br", "gzip"]`）。明示 `identity` の q が
   選ばれた coding の q より高いときは identity（圧縮しない）。同点は server preference で圧縮
   してよい。`Accept-Encoding` 欠如時は identity — RFC は「any coding acceptable」だが、選好未
   表明での変換は保守的に見送る（proxy は変換 MAY、406 は返さない）。acceptable な coding が
   無ければ identity。負う相手は**元のクライアント要求**（チェーン編集後の forward ではない）。
5. **安全既定一式**（業界横断の収斂点、[[000074]] 制約の具体化）:
   - 適格 skip: 既 `Content-Encoding` / `Cache-Control: no-transform`（RFC 9110 §7.7 MUST NOT）/
     204・206・304 / HEAD / content-type allowlist 外 / 宣言長 < `min_length`（既定 1024 B。
     宣言長なしの streamed body は適格）
   - content-type allowlist 既定はテキスト系 web type + `application/wasm`。`text/event-stream` は
     既定に含めない（SSE 除外の充足）
   - 変換時: `Content-Length` 除去・`Accept-Ranges` 除去（byte range は identity 表現を指す —
     RFC 9110 §14。content-coded body に残すと Range 数学がずれる）・strong `ETag` は **W/ 前置
     で弱化**（RFC 9110 §8.8.3。削除より revalidation が生き、suffix 方式より単純）・
     `Vary: Accept-Encoding` を適格 response に常時付与（identity を選んだ場合も — 共有キャッシュ
     の正しさのため）
   - encoder レベル固定: gzip 5 / brotli q4 (lgwin 22) / zstd 3 — on-the-fly 帯の一般的既定。
     v1 では manifest から変更不可（knob は需要が出てから）
6. **streaming 形状: frame 単位の sync flush。** 各 upstream frame を圧縮して即 emit し、trailers は
   coded stream の終端後に素通し（gRPC 互換）。数 % の圧縮率と引き換えに「trickle する upstream は
   trickle のまま client に届く」ことを選ぶ（codec バッファ内でのストール禁止 — P4 系の一貫）。
   encoder は非 Sync（zstd context）のため `Mutex` + `get_mut()` の零コスト Sync ラッパで
   `BoxBody` 制約を満たす。commit 前（適格判定・negotiation・codec init）の失敗は identity。
   `Content-Encoding` 設定後の encode 失敗はストリーム失敗のみ可能（既に宣言した coding を
   identity に書き換えられない）。
7. **観測は現状維持。** 既存 metrics に response バイト系カウンタが無いため、圧縮比カウンタの追加は
   本スライスに含めない（Cons に残す）。

検討した代替案と却下理由:

- **`async-compression` crate でブリッジ** — 実装は薄くなるが、`AsyncRead` 変換で trailers が落ち、
  flush 境界の制御が間接化する。frame→frame の直接変換の方が streaming 意味論を宣言どおりに
  保てる。却下（依存も 1 層減る）。
- **pure-Rust 縛りで zstd を ruzstd encoder に** — encoder が本家未達と自認する段階で、圧縮率 =
  帯域（セルフホスターの最希少資源、[[000074]]）を直接劣化させる。却下。
- **zstd を落として gzip + br のみ** — FFI ゼロにはなるが、[[000074]] が「サーバ側実装は薄く差別化
  余地」と名指しした項目を落とす。libzstd の実績（上記 Context）は compress-only なら十分守れる
  水準。却下（再検討条件は Cons に明記）。
- **listener 既定 + route override の 2 層 config** — セルフホスターの 1 行有効化には便利だが、
  どちらが勝つかの宣言が増える。route 単位 opt-in は BREACH オプトアウトと同じ粒度で、宣言された
  意味論（柱 3）が最も単純。需要が見えたら追加できる（後方互換）。却下。
- **`identity;q=0` で 406 を返す** — origin server 向けの SHOULD であり、proxy が配達可能な
  response を負わせる理由がない。常に identity fallback。却下。
- **`Accept-Encoding` 欠如時に RFC どおり任意 coding を選ぶ** — 選好未表明での変換は観測・デバッグ
  を難しくし、業界の保守的慣行（欠如 = identity）とずれる。欠如は identity、明示選好のみ変換。却下。

## Consequences

### Pros
- セルフホスト構成（前段 CDN なし）の帯域実害（[[000043]] 2026-07-11 追記の一次データ）に
  プロダクト内の代替が入る。caching declined（[[000043]]）の複合が半減する。
- 契約無風: `plecto:filter@0.3.0` に変更なし。フィルタは identity を見る、という [[000073]] の
  response 意味論がそのまま成立し続ける。
- h1/h2/h3 で実装一度・挙動同一（E2E で各プロトコル確認済み）。
- 全 492 テスト green、clippy `-D warnings` / fmt / cargo-deny（advisories・bans・sources）通過。

### Cons / Tradeoffs
- **ワークスペース初の C コード（libzstd）**。再検討条件: (a) production 級の pure-Rust zstd
  encoder が安定リリースされた時点で移行を評価する、(b) `zstd` crate（gyscos/zstd-rs）が
  upstream libzstd リリースへの追随を 12 ヶ月止めた時点で bundled 版の鮮度を監査する。いずれも
  compress-only 縛りは維持する。
- frame 単位 sync flush は大きな転送で数 % の圧縮率を捨てる。バッファリング型（サイズ/時間閾値で
  flush）への切替は、実測でこの数 % が問題になったときの再検討条件。
- 圧縮比・圧縮 CPU の観測が無い（既存 metrics にも response バイト系が無い）。metrics 拡張時に
  `tunnel_bytes_*` パターンで追加する。また既存の duration histogram は response head 時点で確定する
  ため、body streaming 中の圧縮 CPU はどのみち現行 latency 指標に載らない点は記録しておく。
- encoder レベル・`Vary` 挙動は非設定式。knob 要求が来たら manifest 拡張で受ける（後方互換）。
- 上流が `Accept-Encoding` を自力で尊重した場合（既 encode）はスキップして素通しする —
  二重圧縮はしないが、上流の codec 選択には介入しない。
- `Content-Encoding` commit 後の encode 失敗はストリーム失敗になる（identity への事後フォールバック
  は coding 宣言と矛盾する）。init 前の失敗のみ identity。
