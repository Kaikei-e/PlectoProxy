# 近期ユーザーを SaaS/platform team × filter author の獲得ループへ置き直し、発信ゲートを外部 TTFVF 証拠へ切り替える

## Status
accepted（発信ゲートの hard-gate 解釈は [[000077]] が段階的発信へ amend。近期ユーザー定義は不変）

## Date
2026-07-11

## Affected Components
- roadmap / governance — [[000067]] / [[000068]] を本 ADR が amend。実装コード変更なし
- positioning — 近期の需要側 / 供給側の定義だけを更新。self-hostability・data sovereignty・三本柱は維持
- 発信計画 — ゲート条件の差し替え

## Context

2026-07-11 の戦略敵対的レビュー（一次資料: WIT `@0.3.0`、[[000037]]、[[000065]]、[[000067]]、[[000068]]、[[000070]]、EU CRA 公式、wasmtime PR、NIST usability）が、次の確定事実と矛盾を示した。

1. **顧客記述の誤り**: [[000067]] は柱 1 の近期顧客を「セルフホスター」と名指しした。しかし self-hosting は顧客セグメントではなく **deployment property** である。「セルフホスト開発チーム」を自己認識の集団として置く根拠の利用者証拠は無く、一方で品質ターゲットは既に [[000054]] が **SaaS 導入水準**と定義している。記述が二重になっていた。
2. **発信ゲートの誤り**: [[000068]] は発信を F1 + F2（WAF を含む reference 棚）完了に同期した。その後 FDK 骨格と jwt/cors 等は着地したが、(a) WAF reference は [[000037]] どおりエンジン未選定の deferred、(b) AI token-cost の「response body から usage を読む」形は `@0.3.0` の `on-response` が body を常に空とするため**契約上不可能**（[[000073]] / [[000038]]）、(c) 「棚 5 本」のうち複数は off-by-default capability 依存で出荷成熟と数えてはならない。機能本数をゲートにすると、契約不可能・security-critical な中途半端実装へ圧力がかかる。
3. **TTFVF 計測の誤り**: [[000065]] の North Star は「初見の開発者」の完走時間である。maintainer によるクリーン環境の自己計測は install-path smoke に過ぎず、representative users を要する usability 計測（ISO 9241-11 / NIST）を満たさない。
4. **部分 WAF の再提案は却下済み**: 「CRS 思想の代表則をフィルタ 1 本で」は [[000037]] が既に「false sense of protection」として却下した経路。外部エンジン事情（例: TinyGo 系 WASM WAF の公式既知問題）はフルエンジンのコストを補強するが、部分自作を正当化しない。

確定事実と projected の区別: CRA 報告義務 2026-09-11 / 本則 2027-12-11、WASI 0.3.0 正式リリース、`on-response` body 空、は確定事実。SaaS/platform 需要側への到達可能性は **検証対象の仮説**である。

## Decision

1. **[[000067]] を amend する — 近期の届け先。** 三本柱（Verifiable Trust × Contract-Derived Performance × Declared Semantics）と「唯一の」強い主張をしない限定は不変。変更するのは柱 1 の近期顧客記述のみ:
   - **需要側**: self-managed infrastructure 上で L7 gateway を運用し、トラフィックと秘密を自組織境界内に置きつつ、審査票（TLS / クライアント認証 / 供給網 / multi-replica 意味論）に答えたい **SaaS / platform / security team**。
   - **供給側**: 組織内または外部の **filter author**。別市場セグメントではなく、検証済み policy を需要側へ供給する獲得ループの第二の主体。
   - **self-hosting**: 製品原則（self-hostability / data sovereignty）としては維持するが、顧客セグメント名としては使わない。
   - 「SaaS 導入水準」（[[000054]]）は品質の物差しとして維持する。販売の過大約束にはしない（既存どおり）。

2. **[[000068]] を amend する — F2 の近期必須項目。** F1（Filter Dev Kit）最優先は維持。F2 の「WAF + JWT/OIDC + AI token-cost を一本ずつ」のうち:
   - **JWT/OIDC 級 auth** は既着地分を認め、出荷成熟度（capability 依存の明示・docs）の向上を継続する。
   - **WAF reference** は近期必須から外し、[[000037]] の research / deferred を再確認する。再検討条件: production-grade な拡張プレーン向け engine（pure-Rust SecLang/CRS 相当、または Component-native）が embeddable に成熟したとき。部分ルール自作の「WAF-lite」は採用しない。
   - **AI token-cost（response body usage 読取り）** は近期必須から外す。再検討条件: response-body 契約（または真 streaming）が増分として着地し、[[000070]] の再開条件を満たすとき。response **header** のみで成立する限定デモは、一般形と混同せず別扱いとする。

3. **[[000068]] を amend する — 発信ゲート。** 「F1 + F2 完了に同期」を廃止し、**外部の初見 filter author による TTFVF 完走証拠**に同期する。
   - プロトコル既定: 代表タスクを同一にした外部初見者 **n≥5**。完走率、全試行の所要時間（失敗・離脱を含む）、エラー種別、環境を記録する。失敗を中央値計算から除外して「成功した数字」だけを出さない。
   - maintainer のクリーン環境自己計測は install-path smoke とし、発信ゲートに使わない。
   - North Star の数値目標 **≤15 分**（[[000065]]）は維持する。未達なら発信日を動かさず DX backlog を優先する。
   - 規制カレンダー（CRA 報告義務日など）を発信期限の根拠にしない。柱 1 の背景説明には使えるが、適合主張は禁止。

4. **採用縦串の相対順序。** 「導入して検証済み policy を載せる」ループを閉じるため、需要側の **mTLS（[[000054]] deferred 先頭）**と供給側の **外部 TTFVF 計測可能化**を一つの採用縦串として扱う。どちらか片方だけの完了を発信準備完了とみなさない。[[000068]] Decision 5 の「F1+F2 完了後に mTLS」は、本項により **「発信ゲート通過に F2 の WAF/token-cost 完了を要求しない」**および **「mTLS を縦串の需要側脚として並行し得る」**へ緩める。欠如が導入見送り理由として顕在化した場合の割り込み条件は継承する。

検討した代替案と却下理由:

- **「セルフホスター」セグメントを維持する** — 却下。deployment property と顧客像の混同が [[000054]] と衝突し、利用者証拠も無い。
- **filter author だけを第一対象にする** — 却下。書く人がいても載せる運用者がいなければループが閉じない。
- **顧客セグメントを置かず技術原則だけで優先順位を決める** — 却下。原則は不変層として既にある。優先順位には「誰のどの要求か」が要る。
- **旧発信ゲート（F1+F2 完了）を維持する** — 却下。契約不可能・security-critical deferred をゲートに残すと、部分 WAF や偽の token-cost へ圧力がかかる。
- **内部 smoke 3 回で発信する** — 却下。[[000065]] の「初見者」定義と representative-user 計測に反する。
- **ゲートなしで直ちに発信する** — 却下。柱 1 の体験主張が反証不能な宣伝になる。
- **WAF-lite を教材として近期出荷する** — 却下。[[000037]] の false sense of protection 却下を覆す証拠が無い。非セキュリティの request-inspection example なら WAF と呼ばない別物として検討可能（本 ADR の必須ではない）。

本決定は P5（Verify before load）・P8（declined の言語化）・P9（正直な計測）・§3.4（反証可能な主張）を具体化し、[[000067]] の三本柱を supersede しない。

## Consequences

### Pros
- 品質ターゲット（[[000054]]）とポジショニングの「誰に届けるか」が一致する。
- 発信が契約不可能な F2 残件にブロックされなくなる一方、体験の反証可能性（外部 TTFVF）は上がる。
- WAF / token-cost を無理に近期実装しないことで、[[000037]] / WIT との HRT（Transparent）を回復する。

### Cons / Tradeoffs
- WAF 非在と token-cost 一般形の欠如は、当面「無い機能」として効く（役割駆動の既知の代償）。
- SaaS/platform 需要側仮説が外れれば、優先順位の組み直しが必要。
- mTLS と TTFVF の並行は単独開発の帯域を圧迫する。どちらかが顕著な阻害要因として観測されたら、trial データで再配分する。
- **再検討条件**: (a) 需要側仮説が trial / interview で反証されたとき、届け先記述を再 amend する；(b) 外部 TTFVF が計測可能になった後も 15 分から大きく外れ続けるとき、[[000065]] (a) に従い DX 優先度を再配分する；(c) production-grade 拡張プレーン WAF engine が成熟したとき F2 WAF を再検討する；(d) response-body / streaming 契約が着地したとき token-cost 一般形を再検討する；(e) mTLS 欠如が導入見送り理由として顕在化したとき、他スライスを待たず割り込む。
