# 発信を段階化し、外部 TTFVF を hard gate から供給側 DX 指標へ格下げする

## Status
accepted（2026-07-12 grill セッションで承認。実装着手は [[000078]] の mTLS スライスから。technical preview 床への到達性項目追加と operator TTFV は [[000084]] が amend）

## Date
2026-07-12

## Affected Components
- roadmap / governance — [[000076]] Decision 3（発信ゲート）と Decision 4（採用縦串の「どちらか片方だけでは発信準備完了とみなさない」）を本 ADR が改正。実装コード変更なし
- 発信計画 — 一度きりの「第一波」前提をやめ、段階ごとの主張境界を定義する

## Context

[[000076]] は機能本数ゲートの誤りを正し、外部初見者による TTFVF 完走証拠へ発信を同期した。これは供給側 DX の反証可能性を上げる正しい修正だった。しかし 2026-07-12 の需要側カタログ床レビュー（一次資料: release workflow の default-feature ビルド、[[000054]] / [[000061]] / [[000062]]、RFC 9846、Redis security docs、CNCF TAG Runtime Wasm OCI Artifact layout）は、別の過補正を示した。

1. **TTFVF を hard gate にすると、需要側へ力点を移すと言いながら供給側計測が発信の拒否権を持ち続ける。** 「試させるカタログが無ければ誰も測りに来ない」は正しい一方、床が揃うまで一切語れないのも過剰である。
2. **発信は一度きりではない。** recruiting、engineering log、technical preview、production-readiness claim は主張の強さが違う。同一ゲートで全部を縛ると、弱い主張まで遅延するか、強い主張を床未達で行ってしまう。
3. **需要側 floor は TTFVF と直交する五責務である**（mTLS、runtime capability profile、署名済み filter OCI、secure global state、実行可能 topology）。これらを「TTFVF ∧ 床」の連言に押し込むと、床の未達が TTFVF 未達と区別できなくなる。

確定事実: 現行 release は capability feature 無し、reference filter の多くはテスト fixture 止まり、Redis reference は AUTH/TLS 未実装、downstream mTLS × shared STEK は同一 listener で非両立（[[000062]] / RFC 9846）。需要側仮説の到達可能性は引き続き検証対象（projected）。

## Decision

1. **[[000076]] Decision 3 を amend する — 発信の段階化。** 「外部 TTFVF 完走証拠に同期してから発信する」を廃止し、次の三段階に分ける。
   - **Recruiting / engineering log（今すぐ可）**: TTFVF 募集、設計・実装経過、既知の欠落。production / SaaS 導入水準の主張はしない。
   - **Technical preview**: prebuilt な capability 付き runtime profile、署名済み reference filter OCI artifact、compose skeleton が再現可能であること。mTLS や secure Redis が未了なら**未了と明記**する。preview は体験可能性の主張であり、production floor の主張ではない。
   - **Production-readiness campaign**: 両方向 mTLS（[[000078]]）、global rate limit を弾にする場合は secure global state（[[000081]]）、外部 operator trial の完走。ここで初めて需要側カタログ床を主張する。

2. **TTFVF は hard gate ではない。** [[000065]] の North Star（≤15 分）と [[000076]] が定めた計測プロトコル（代表タスク同一・出自開示・介入時 DNF・active/elapsed 二時計・事前登録・失敗を隠さない）は**供給側 DX の継続指標**として維持・公開する。未達は DX backlog 優先の根拠にはなるが、recruiting / engineering log / technical preview を拒否しない。

3. **[[000076]] Decision 4 を amend する — 採用縦串の読み方。** 「mTLS と TTFVF のどちらか片方だけの完了を発信準備完了とみなさない」は、**production-readiness claim** に限定して継承する。technical preview は床の部分集合と制約の明示で足りる。欠如が導入見送り理由として顕在化したときの割り込み条件は継承する。

検討した代替案と却下理由:

- **[[000076]] の TTFVF hard gate を維持する** — 却下。需要側主線への移動と矛盾し、床の不足と DX の不足を区別できない。
- **`TTFVF ∧ 需要側床` の連言ゲートにする** — 却下。連言でも TTFVF が拒否権を持ち、段階的な弱い主張まで遅延する。
- **ゲートなしで直ちに production 主張する** — 却下。柱 1（検証可能な信頼）と [[000054]] の SaaS 導入水準に対して Transparent でない。
- **床だけ揃えて TTFVF を捨てる** — 却下。供給側ループ（filter author）の反証可能性を失う。指標としては残す。

本決定は P8（declined / 未達の言語化）・P9（正直な計測）・§3.4（反証可能な主張）を具体化し、[[000067]] の三本柱と [[000076]] の近期ユーザー定義は supersede しない。

## Consequences

### Pros
- 需要側へ主線を移しても、弱い発信（募集・経過）を不当に止めない。
- production claim と preview を分離し、満たした性質だけを主張できる（HRT: Transparent）。
- TTFVF の計測品質要求は残るため、供給側 DX の自己欺瞞（maintainer smoke を実績扱い）は防げる。

### Cons / Tradeoffs
- 段階定義の運用纪律が要る。preview を production 言い換えする圧力への防波堤は文書とレビューに依存する。
- production-readiness まで「完成した製品」という単純な物語は作れない。
- **再検討条件**: (a) 需要側仮説が trial / interview で反証されたとき届け先と床の定義を再 amend する；(b) technical preview が production と誤読される実害が出たとき、preview の主張境界をさらに狭める；(c) 外部 TTFVF が計測可能になった後も 15 分から大きく外れ続けるとき、[[000065]] / [[000076]] に従い DX 優先度を再配分する。
