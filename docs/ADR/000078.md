# mTLS のスコープを両方向 floor と STEK 交差不変条件で固定する

## Status
accepted（2026-07-12 実装前 grill 完了 → 同日実装着地。確定事項は「grill 確定」、着地後の精密化は「実装追記」）

## Date
2026-07-12

## Affected Components
- roadmap / governance — mTLS を「審査票の空欄」から「需要側カタログ床の TLS 責務」へ具体化
- control/tls / manifest — downstream client certificate 検証と upstream client certificate 提示（本 ADR の語彙・不変条件どおり実装済み）

## Context

[[000054]] は品質ターゲットを SaaS 導入水準とし、mTLS（downstream + upstream client cert）を deferred 序列の先頭へ引き上げた。実装前の現状は両方向とも client authentication 無し（`with_no_client_auth`）だった。2026-07-12 の需要側床レビューは、次を確定事実として示した。

1. **upstream client cert 提示**と**downstream client cert 検証**は同じ PKI 部品を共有しうるが、利用者要求の性質が異なる。前者は backend が client certificate を必須にする環境での配備ブロッカーになりうる。後者は gateway 上で certificate-bound identity を確立する機能であり、network policy や ext-authz 系 filter は**同じ認証性質の代替ではない**。
2. **TLS 1.3 の認証形（Humble に分ける）**:
   - **サーバ認証**: resumption PSK と certificate によるサーバ認証は標準上排他（RFC 9846、RFC 8446 を obsolete。「either a PSK or a certificate … but not both」）。
   - **クライアント認証**: PSK 使用時サーバは CertificateRequest を送らない（RFC 8773 がこの制限を明示）。したがって resume では client certificate の**再検証は走らず**、identity は初回 handshake 時に ticket へ保存された状態に依存する（rustls は `ServerSessionValue.client_cert_chain` を encode/decode し、resume 時に `peer_certificates` へ復元する）。
   - Plecto は既に [[000062]] で downstream mTLS listener と shared STEK の併用を validation fail-closed している。標準形リファレンスを「一つの完成形」に押し込むと、この不変条件と衝突する。共有 STEK を禁じる直接の標準条文は無く、根拠は「再検証なし」＋「共有鍵による ticket のレプリカ間伝播（CVE-2025-23419 級の blast radius）」である。
3. **失効（CRL/OCSP）と identity の filter 伝搬**まで同一スライスに含めると、床の着地が無限に遅れる。

確定事実（実装前）: [[000062]] の交差規則、上記 RFC / rustls の認証形。着地順は grill で upstream 先行に確定（「grill 確定」1）。

## Decision

1. **需要側カタログ床の TLS 責務として、両方向 mTLS を floor とする。**
   - **upstream**: Plecto が upstream へ接続するとき client certificate を提示できること。
   - **downstream**: listener が peer の client certificate を検証できること。
   - [[000054]] の「同一スライスで」は、**共有する PKI primitive・設定語彙・E2E の切り方を一つのスコープ ADR（本 ADR）で先に固定する**意味に精緻化する。バイナリの単一 PR 必須までは拘束しない。

2. **着地順は本 ADR 本文ではロックしない**（実装前）。「upstream 先行」は配備ブロッカー仮説として合理的だが、trial 証拠が無い時点では Humble でない。実装着手前の短い grill で順序を決める（結果は「grill 確定」1）。どちらが先でも、**両方が production-readiness claim（[[000077]]）の前提**である。

3. **TLS topology の不変条件（[[000062]] を再確認）**:
   - downstream mTLS が有効な listener では shared STEK を使わない（per-node resumption または resumption 無効）。
   - non-client-auth listener では [[000062]] の制約下で shared STEK を選択可能。
   - upstream mTLS は downstream listener の resumption mode と独立。

4. **declared deferred（本スライスに含めない）**:
   - 証明書失効確認（CRL / OCSP）の本番級配線。
   - 検証済み client identity の filter への伝搬契約（ヘッダ注入方針を含む）。
   - SPIFFE 等の特定 identity 文書形式への深い追従。verification-name / SNI 既存機構との整合言及に留める。

検討した代替案と却下理由:

- **downstream のみ / upstream のみを floor とする** — 却下。[[000054]] の審査票項目と配備ブロッカーの両方を片肺にする。
- **network policy や ext-authz を downstream mTLS の代替とみなす** — 却下。認証の確立点が異なり、certificate-bound identity という性質を満たさない。
- **shared STEK と downstream mTLS を同一 listener の完成形にする** — 却下。resume は client-cert を再検証せず、共有鍵は stolen-ticket / 将来の多 CA 越境の blast radius を広げる（[[000062]] (b)、CVE-2025-23419 級）。RFC 9846 のサーバ認証排他を「共有 STEK 禁止」の直接根拠にはしない。
- **CRL/OCSP と identity 伝搬を同一スライス必須にする** — 却下。床の定義を実装不能な完全性へ膨らませる。
- **着地順を upstream 先行で即ロックする** — 却下（本文時点）。仮説としては残すが、証拠前の確定は Humble でない（grill で後から確定）。

本決定は [[000054]] の品質ターゲットと序列を supersede せず、スコープと不変条件を具体化する。P4（fail-closed）・安全優先のコア原則に従う。

## Consequences

### Pros
- mTLS 実装が「全部入り」か「片方で宣称」かの揺れを、着手前に止められる。
- STEK 交差を再検証ギャップと既存 ADR / CVE 級リスクに紐づけて説明でき、reference topology（[[000082]]）のシナリオ分割が書ける。
- deferred 線が明示され、失効・伝搬の議論がスライスを詰まらせない。

### Cons / Tradeoffs
- deferred の失効・伝搬は審査票の追問で「未実装」と答え続ける必要がある（Honest gap）。
- per-node resumption でも ticket 有効期間内は証明書の期限・失効の再検証が走らない（shared STEK 禁止はこのギャップを消さない）。
- **再検討条件**: (a) operator trial で片方だけが導入阻害として顕在化したとき、順序をロックして割り込む；(b) rustls / 証明書 API の制約で共有 primitive が成立しないとき、方向別に実装境界を再切る；(c) 失効確認が導入見送りの主因になったとき、deferred を解除する個別 ADR を書く。

## grill 確定（2026-07-12、実装前 grill セッション）

Decision 2 が実装着手前に残した論点を、コード探索（変更縫い目 3 箇所の特定）と公式ドキュメント調査
（rustls 0.23 系 / quinn 0.11 系 / hyper-rustls 0.27 系、調査日 2026-07-12）に基づき確定した。

1. **着地順は upstream 先行、同一縦串内で downstream を続行する。** 根拠: upstream 側は既存の
   per-upstream `ClientConfig` + connector 分離構造がそのまま identity 分離を解決し（connection pool
   への identity 混入なし）、STEK 交差・schema canary との相互作用が無い。配備ブロッカー仮説
   （backend が client certificate を必須にする環境）にも先に応える。
2. **downstream の設定語彙は `[listen.client_auth]`（`ca_path` = trust anchor の PEM bundle）。**
   粒度は listener 単位（現行は単一 listener のため全体と一致）。セクションの存在 = client
   certificate 必須化。TCP / QUIC の両 ServerConfig に同一 verifier を適用する。中間 CA は
   クライアントが提示し、`ca_path` には trust anchor のみを置く。
3. **検証モードは required のみ。** optional（要求するが未提示も通す）は、検証結果を消費する
   identity の filter 伝搬（Decision 4 の declared deferred）とセットでなければ死に設定になる
   ため、本スライスに含めない。
4. **client auth 有効時も per-node resumption は維持する。** rustls は client-auth 済みセッション
   の certificate chain を ticket に保存・復元するため identity は resumption を跨いで保たれる
   （再検証は走らない）。0-RTT 拒否 / psk_dhe_ke 限定 / stateless ticketer は不変。shared STEK
   （`[resumption]`）と `[listen.client_auth]` の同時指定のみ validation で fail-closed
   （[[000062]] (b) の実装形）。ticket 有効期間内は証明書の期限・失効の再検証が走らないギャップは、
   失効確認自体が declared deferred であることと整合させ、ドキュメントに明記する。
5. **upstream の設定語彙は `[upstream.tls] client_cert_path` + `client_key_path`。**
   both-or-neither を validation で強制（片方のみは起動時エラー）。health probe は同一 connector
   を通るため client certificate を自動的に提示する。
6. **鍵規律: 新規に導入する mTLS 秘密鍵ファイルには STEK 同型のパーミッション検査**（group/other
   読取可なら fail-closed 起動拒否）を適用する。既存 `[[tls]]` サーバ鍵への検査拡張は既存構成を
   壊す破壊的変更のため本スライスに含めず、backlog に残す（非対称は既知として文書化）。
7. **E2E の切り方**: downstream は h1 + h2 + h3 の三系統で固定（tokio-rustls acceptor と quinn
   endpoint の二実装 + h2 経路）。異常系は「certificate 未提示 → handshake 拒否」
   「untrusted certificate 提示 → handshake 拒否」「client_auth × shared STEK 同時指定 → ロード拒否」
   「resumption 継続性 + ticket からの `peer_certificates` 復元」。upstream は
   「提示 → 200」「backend が要求するのに未設定 → fail-closed」「health probe も提示」の三本。

## 実装追記（2026-07-12）

1. **着地**: `control/src/tls.rs` + manifest 語彙 + h1/h2/h3 / upstream E2E。別実装 ADR は起こさず、
   本 ADR のスコープ確定に直結して GREEN した。
2. **STEK 交差の説明を精密化**: 「ticket が認証前に mint される」ではない。成功 handshake の後に
   mint され、accept 時に client-cert 検証を再実行しない。共有鍵はその状態をレプリカ間へ運ぶ。
3. **identity 復元のピン**: `client_auth_peer_certificates_survive_per_node_resumption` が full /
   resumed の両方で `peer_certificates` 一致を固定（HTTP E2E は伝搬 deferred のため server 側
   chain を見られない）。
4. **運用ギャップの文書化**: per-node resume でも期限・失効は再検証しない旨を `CONTEXT.md` と
   design-principles に明記（審査票の Honest gap）。
