# release に named runtime capability profile を導入し、ソースビルドのみを production floor としない

## Status
proposed

## Date
2026-07-12

## Affected Components
- CI / release（`.github/workflows/release.yml` ほか）— 現行は `cargo auditable build --release --locked -p plecto-server` で feature 指定なし＝default のみ。本 ADR 採択後、named profile の prebuilt 成果物を定義する
- host — feature `outbound-http` / `outbound-tcp` の off-by-default 隔離（[[000036]] / [[000060]]）は維持。昇格の是非は本 ADR の Decision で枠を固定
- docs / README — どの profile を試す導線の既定にするかを明示

## Context

需要側が global rate limit reference（outbound-tcp）や JWKS / ext-authz（outbound-http）を試すには、capability 付きランタイムが手に入る必要がある。確定事実（2026-07-12）:

1. **出荷バイナリは default features のみ**であり、当該 reference はソースビルド + feature flag 無しでは動かない。
2. **Cargo の features は additive であるべき**であり、組合せはテスト構成を指数的に増やす（Cargo Book Features 章）。複数 profile は「小さい packaging 変更」ではなく support matrix 案件である。
3. **compile-time にコードが含まれることと、runtime に capability が貸与されることは別境界**である。後者は引き続き manifest の deny-by-default + allowlist + SSRF 床（[[000036]] / [[000060]]）が強制する。

[[000047]] は署名付きバイナリ・SBOM・cosign を自らの配布物に課した。capability 無しの単一成果物のままでは、供給網規律を満たしても**需要側カタログの蛇口が閉じたまま**になる。

## Decision

1. **ソースビルド手順の一級文書化だけを production floor としない。** technical preview（[[000077]]）および production-readiness の需要側主張には、**prebuilt な capability 付き runtime**が必要である。

2. **named runtime capability profile を導入する。** 少なくとも次を区別して名付ける（最終名は実装時に確定してよい）:
   - **minimal** — 現行 default に相当。outbound 系を compile しない（または含めない）最小攻撃面。
   - **capabilities** — `outbound-http` と `outbound-tcp` を compile-time に含む profile。実行時は従来どおり filter 単位の宣言と SSRF 床が全て効く。

3. **単一 canonical 成果物か dual 成果物かは、実装着手前の grill で support matrix コストを見て選ぶ。** 本 ADR が禁止するのは次のみ:
   - 議論なしに default features へ outbound を silently 昇格すること（[[000036]] / [[000060]] の隔離根拠を黙って覆すこと）。
   - profile を増やしたのに test / clippy / SBOM / cosign / vulnerability response を minimal だけに残すこと。

4. **各 profile は同一の供給網規律（[[000047]]）に服する。** auditable build、SBOM、cosign、draft release 経由の公開判断。profile ごとに digest を追跡する。

5. **compile-time inclusion ≠ runtime grant を文書の一等市民にする。** capabilities profile のバイナリを配ることと、任意 filter が任意ホストへ出られることを同義にしてはならない。

検討した代替案と却下理由:

- **ソースビルド手順だけを床とする** — 却下。需要側ペルソナに Rust toolchain とビルド時間を強いる。
- **default features へ即昇格** — 却下（現時点）。ビルド隔離の根拠（攻撃面・依存量）を覆すには [[000036]] / [[000060]] を正面から amend する論証が要る。grill なしの昇格は禁止。
- **profile 名だけ決めて成果物は一つ・ドキュメントで feature を指示** — 却下。prebuilt が無い点でソースビルド床と同型。
- **HTTP と TCP で別々の第三・第四成果物を最初から増やす** — 却下。組合せ爆発を早める。まずは outbound 系を束ねた capabilities profile を単位にする。

本決定は [[000047]] の供給網規律を拡張し、[[000036]] / [[000060]] の capability 境界は維持する。P4（deny-by-default）と運用の単純さの緊張を、profile という明示選択で扱う。

## Consequences

### Pros
- 需要側が「動くバイナリ」で capability 依存 reference に届く。
- minimal を残せる選択を捨てない（隔離根拠の保存）。
- テスト / 署名 / SBOM の matrix を意識せざるを得なくなり、条件付きコンパイルの暗黙負債が可視化される。

### Cons / Tradeoffs
- dual を選ぶとリリース資産・CVE 応答・ドキュメント導線が倍近くになる。
- single capabilities canonical を選ぶと最小攻撃面バイナリを失う（または別チャネルへ退避）。
- **再検討条件**: (a) grill で matrix コストが単独開発の帯域を超えると判明したとき、capabilities のみの単一成果物へ縮退する；(b) outbound 系の依存 CVE が持続的に minimal 利用者へ無関係な騒音を与えるとき dual を強制する；(c) WASI / host 側の能力モデルが変わり feature 境界そのものが不要になったとき、本 ADR を再評価する。
