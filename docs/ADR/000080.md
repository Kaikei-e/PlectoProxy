# reference filter を署名付き OCI artifact として runtime とは別に配布する

## Status
accepted（2026-07-12 実装前 grill で Decision の留保点を解消し、同日実装。確定内容は Decision 末尾「実装前 grill による確定」。同日の敵対的レビュー後、SBOM 対象を component バイトへ修正・CI/release の encode パス共有・版据え置き時の content mismatch fail-closed・import 床の CI 断言・用語の Transparent 化を追記）

## Date
2026-07-12

## Affected Components
- CI / release — [[000047]] の host バイナリ / イメージと [[000064]] の WIT 配布に加え、**filter component 本体**の OCI 配布を定義する
- host/build.rs / test_support — capability feature 有効時の guest ビルドは当面テスト fixture として存続しうるが、**release 成熟の定義から除外**する
- docs — 棚の「本数」ではなく digest・署名・required profile・互換ポリシーで成熟を語る

## Context

[[000064]] は `plecto:filter` **契約（WIT）**を wkg / OCI で配布した。[[000047]] は **host runtime** に供給網規律を課した。欠落している第三の成果物が **reference filter の component 実体**である。

確定事実（2026-07-12）:

1. capability feature を有効にしても、`filter-ratelimit-redis` 等は `build.rs` 経由の**テスト fixture**としてビルドされ、release assets には載らない。
2. CNCF TAG Runtime の Wasm OCI Artifact layout は、Wasm component を runtime バイナリとは別の OCI artifact（`application/vnd.wasm.config.v0+json` + `application/wasm` layers）として扱い、imports / exports / target world を metadata に載せられる。
3. [[000076]] / [[000068]] が戒めた「棚 N 本」勘定は、リポジトリ存在と出荷成熟を混同していた。

runtime profile（[[000079]]）を配るだけでは filter は届かない。逆に filter だけ配っても required imports を満たす runtime profile が無ければ載らない。

## Decision

1. **host runtime artifact と filter OCI artifact を別成果物として扱う。** 一つの tarball / イメージに黙って同梱して「出荷完了」と数えない。対応関係は互換表で明示する。

2. **reference filter ごとに次を release 定義に含める（floor）**:
   - OCI digest と version タグ方針
   - cosign 署名（[[000047]] と同型の verify-then-load 前提に耐えること）
   - 出荷 component バイトを対象とした SPDX SBOM を OCI digest に attest したもの（下記 grill §4。host の keyed SBOM↔component-byte 束縛 [[000006]] とは分界）
   - required imports / target world（`plecto:filter@x.y.z` および outbound 系）
   - 対応する Plecto **runtime capability profile**（[[000079]]）
   - compatibility / deprecation policy（契約版と host 版の行列）
   - conformance / release-parity の記録（default-branch CI green を記録実体とする——別台帳は作らない）

3. **テスト fixture ビルドは release 成熟に数えない。** CI 内で feature 付き guest を建てることは継続してよいが、対外の「棚」や technical preview の弾には、署名済み OCI を要求する。

4. **[[000068]] / [[000076]] の reference 棚の読みを精緻化する。** 成熟の単位は「リポジトリにディレクトリがある」ではなく Decision 2 のチェックリストである。capability 依存 filter は、対応 runtime profile 無しでは preview 弾に使わない。

5. **filter-jwt の static-key 経路と JWKS 経路の分離は消さない。** capabilities profile を配ることは、静的検証まで WASI / outbound import を要求する理由にならない。least privilege な Tier A reference として [[000079]] の grill と独立に評価する。

検討した代替案と却下理由:

- **runtime イメージに filter .wasm を同梱して一件化する** — 却下。更新周期・署名対象・SBOM 境界が溶ける。CNCF の artifact 分離とも逆行する。
- **WIT（[[000064]]）配布だけをもって filter 配布完了とする** — 却下。契約と実装は別物。
- **ソースツリーの examples/filters を clone して使えと案内する** — 却下。需要側 floor と供給網規律（verify-then-load）に対して Transparent でない。
- **全 reference を単一 multi-layer OCI にまとめる** — 却下。個別の deprecation と least privilege な取得ができない。

本決定は [[000064]] の契約配布と [[000047]] の供給網規律を filter 実体へ延長する。P5（Verify before load）を具体化する。

### 実装前 grill による確定（2026-07-12）

Decision 2–5 が実装に委ねた選択を次のとおり確定した。

1. **初期の棚は filter-jwt / filter-cors / filter-apikey / filter-extauthz の四本。** zero-WASI の
   Tier A 二本（cors / apikey）が minimal profile で完結する導線を、outbound-http 依存の二本
   （jwt / extauthz）が capabilities profile（[[000079]]）の導線を担う。**filter-ratelimit-redis は
   棚に載せない**——[[000081]] の AUTH / ACL / TLS 条件が未達のまま署名済み artifact を配ることは
   production 誤読リスク（[[000077]] 再検討条件 b と同型）であり、除外理由を互換表に明記する。
   fixture 群（hello / quickstart / streaming / polyglot）は Decision 3 のとおり対象外。
2. **レジストリパスは `ghcr.io/kaikei-e/plecto/filters/<name>`。** runtime イメージ
   `ghcr.io/kaikei-e/plecto` の直下に入れ子で揃え、`filters/` が名前空間として filter を言うため
   crate 名の `filter-` 接頭辞は落とす（jwt / cors / apikey / extauthz）。crate 名との対応は互換表が持つ。
3. **タグは filter 自身の SemVer（不変）。** 各 guest の Cargo.toml version を source of truth とし、runtime の
   v-tag とも `plecto:filter` 契約版とも独立に版を刻む。publish は常にローカル rebuild し、同版が既に在れば pull した component とローカルを `wasm-tools strip` 後の sha256 で照合——一致なら push を skip、
   不一致なら fail-closed（版を上げよ。ソース変更を同タグで黙って上書きしない。custom section の揺らぎは
   strip で吸収）。署名・attest は常に digest に対して行い、tag には行わない。
4. **署名は keyless cosign、SBOM は出荷 component バイトの SPDX attestation。** Fulcio OIDC keyless で
   artifact digest に署名し、SBOM は syft が **component `.wasm` ファイル**をスキャンした SPDX を
   `cosign attest --type spdxjson` で digest に紐付ける（`cosign attach sbom` は deprecated）。
   これは OCI-digest-bound な component 成果物 attestation であり、host ロード時の keyed
   SBOM↔`sha256(component_bytes)` 束縛（[[000006]]）とは分界する——後者は operator が自身の
   `[trust]` で再署名するときに成立する。SBOM は release asset には置かず attestation のみ。
   部分失敗からの自己修復は verify-before-sign。publish job の権限は `packages: write` +
   `id-token: write`。
5. **verify-then-load への「耐え方」は責任分界の一級文書化。** host のロード経路（keyed offline
   cosign、manifest `[trust].keys`、digest pin、fail-closed）は変更しない。Plecto の release
   identity が保証するのは「出荷したもの」であり、operator は cosign CLI で registry 署名と SBOM
   attestation を検証してから pull し、**自身の trust root** で署名・OCI layout 化・digest pin を行う
   （writing-a-filter §5 の既存経路）。registry の keyless 署名を host が直接 `[trust]` の根として
   検証する拡張は trust model の変更であり、**本 ADR とは別の decision record を要する後続スライス**
   として明示的に切り出す。
6. **互換表は `docs/reference-filters.md` に新設。** Decision 2 の床を一枚で持ち、writing-a-filter §8 と
   README の Prebuilt 節から参照する。encode パスは `scripts/build-reference-filters.sh` で CI と
   release が共有し、import 床（zero-WASI は `wasi:http` 禁止、capabilities は必須）を毎 PR で断言する。
   conformance / release-parity 記録は default-branch CI green（棚の release-parity ビルド + cors/apikey
   の host 振る舞いテスト + jwt/extauthz の outbound-http suite）を実体とし、別台帳を作らない。
7. **filter-jwt の static-key 経路の現実を隠さない。** 現行 component は static 経路でも
   `wasi:http/outgoing-handler` を import するため、instantiate には capabilities profile が必要
   （`allow = []` で貸与ゼロは維持）。Decision 5 の least-privilege Tier A 評価は「import 自体を
   落とした zero-WASI build 変種」の是非として将来判断し、互換表に現状を明記する。

## Consequences

### Pros
- 「棚に在る」の主張が digest / 署名 / 互換表で反証可能になる。
- runtime profile と filter の責任分界が明確になり、[[000079]] との同時審議が噛み合う。
- capability 依存と zero-WASI reference を同じ棚勘定で混ぜない。

### Cons / Tradeoffs
- 発行パイプライン・レジストリ権限・ドキュメント行列が増える。
- preview 前に「動く filter」を配る作業が、host リリースだけでは終わらなくなる。
- **再検討条件**: (a) レジストリ / wkg ツールチェーンが layout を破壊的に変えたとき配布手順を再amendする；(b) 全 reference が同一 profile / 同一 world に収束し個別 artifact の運用コストだけが残ったとき、束ね配布を再評価する；(c) host が signed filter を同梱する公式「batteries」イメージを別製品として出す判断が必要になったとき、本 ADR の分離原則との関係を明示する追加 ADR を書く。
