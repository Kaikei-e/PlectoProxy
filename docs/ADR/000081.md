# Redis reference の production 昇格条件を AUTH・ACL・TLS 付き経路として定め、現状を demo-only と宣言する

## Status
proposed

## Date
2026-07-12

## Affected Components
- `filter-ratelimit-redis` — [[000061]] 実装追補 4 が RESP を `INCRBY` / `EXPIRE NX` / `PING` に限定し、AUTH・TLS をスコープ外とした。本 ADR はそのまま **demo 成熟**と再分類し、production 昇格の条件を別途固定する
- docs/hardening / operations — 二層併用の意味論は維持しつつ、global 層の輸送・認証前提を Honest に書き分ける

## Context

[[000061]] は local floor × global filter の二層モデルと Redis reference を導入した。これは [[000053]]（共有状態は拡張プレーンへ）の実物として必要だった。一方、実装追補は意図的に次を外した: `AUTH`、ACL、TLS over TCP。前提は trusted network 内の平文 TCP である。

2026-07-12 の公式資料確認（Redis security docs）は、次を別々の防御層として示す:

1. **trusted network**（非公開網への配置）
2. **認証**（現行推奨は ACL による named user と細粒度権限。レガシーな単一 `requirepass` は補助層）
3. **TLS**（通信路の暗号化。平文上の AUTH は盗聴を防がない）

確定事実: 現状 reference は (1) のみを前提にし、(2)(3) を実装していない。これを production 標準形と呼ぶと、公式 guidance と自らの hardening 文書に対して Transparent でない。需要側カタログ床の「global state の安全な経路」は、global rate limit を **production claim の弾**にする場合に限り必須である（[[000077]]）。

## Decision

1. **現行 `filter-ratelimit-redis`（AUTH / ACL / TLS なし）を demo-only と宣言する。** trusted local / private network 上の学習・technical preview 補助には使ってよい。production-readiness campaign や「マルチレプリカ標準形」の security 主張には使わない。

2. **production 昇格条件（すべて必須）**:
   - ACL user + `AUTH`（または同等の現行推奨認証）。必要 command と key prefix のみに制限する。
   - TLS（または同等に検証可能な暗号化 transport）。平文 AUTH のみは昇格条件を満たさない。
   - credential / CA / client key を manifest の平文文字列に埋め込まず、**secret file 能力**（または同等のホスト管理シークレット）として渡す設計。
   - backend failure policy（[[000061]] の `on_backend_error` 宣言必須）と local floor の相互作用を E2E で固定したまま維持する。

3. **trusted network だけを production security の代用にしない。** 公式が層として並べる認証・暗号化を、「網が信頼できるから不要」と省略した構成は、demo 以上の呼称を与えない。

4. **実装タイミング**: global rate limit を [[000077]] の production-readiness の弾に使うとき、または operator trial が secure backend を導入阻害として記録したときに着手する。それ以外では demo-only 表記の維持を優先し、スコープ膨張を避ける。

5. **[[000053]] / [[000061]] の配置判断は不変。** global 層の native 化や外部専用レートリミットプロセスの新設は行わない。昇格は **同一 reference filter の輸送・認証硬化**である。

検討した代替案と却下理由:

- **現状のまま production 標準形と呼ぶ** — 却下。公式 security guidance と HRT（Transparent）に反する。
- **global を native 共有ストアへ戻す** — 却下。[[000053]] で declined。
- **AUTH だけ足して TLS を永続 deferred とする** — 却下。平文 AUTH は盗聴耐性を与えない、と公式が明記する。
- **昇格前にアルゴリズムを GCRA 等へ刷新する** — 却下。本 ADR の対象は輸送・認証の床であり、[[000061]] の fixed-window 再検討条件とは独立。

本決定は [[000061]] の二層意味論を維持し、成熟度の呼称だけを改正する。P4（fail-closed）と安全優先を具体化する。

## Consequences

### Pros
- demo と production の言葉が一致し、審査・発信での過大表明を防げる。
- 昇格条件がチェックリスト化され、部分実装（AUTH だけ等）で床完了と誤認しにくい。
- [[000082]] の compose が insecure Redis を載せる場合の注記根拠になる。

### Cons / Tradeoffs
- production claim に global rate limit を使うと、filter 実装と secret 配布設計の追加コストが発生する。
- TLS をゲスト実装に委ねる既存方針（[[000060]]）との接点で、host がどこまで助け船を出すかの設計が残る（本 ADR は「検証可能な暗号化」を要求し、実装分担は後続 grill）。
- **再検討条件**: (a) RESP 互換ストア側が TLS / ACL を提供しない運用が主流と判明したとき、代替バックエンドまたは別 transport を評価する；(b) wasi 標準が key-value / セキュア接続を十分に覆い outbound-tcp 直結が冗長になったとき、[[000061]] 再検討条件 (c) と合わせて再評価する；(c) secret file 能力の一般化が他 filter（mTLS 鍵、STEK）と重複するとき、共通ホスト能力へくくり出す ADR を書く。
