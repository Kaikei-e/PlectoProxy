# マルチレプリカ標準形の実行可能リファレンスを compose 先行で提供し、TLS シナリオを分割する

## Status
proposed

## Date
2026-07-12

## Affected Components
- examples/ — 現状 `plecto/examples/production` は単一ノード実バイナリ demo。本 ADR は **compose skeleton** を需要側床の実行可能 topology とする
- docs/operations — readiness / drain / PROXY v2 / STEK の散文を、シナリオ付きの「一発で起動できる形」へ接続する
- k8s 素マニフェスト — 初期スコープ外。operator trial 後の第二成果物候補

## Context

Program R（[[000056]]）の主要プリミティブ（PROXY v2 [[000057]]、graceful drain [[000059]]、STEK 共有 [[000062]]、二層 rate limit [[000061]]）は landed している。欠落は「知識が散文で在るが、走る形が無い」ことである。

2026-07-12 のレビューが示した制約:

1. **downstream mTLS と shared STEK は同一 listener で両立しない**（[[000062]] / [[000078]] / RFC 9846）。標準形を一構成に押し込むと、どちらかの不変条件を偽ることになる。
2. **global rate limit の現行 Redis reference は demo-only**（[[000081]]）。compose に載せるなら呼称を分ける。
3. **Kubernetes Secret volume の file mode 既定は `0644`** であり、[[000062]] の鍵ファイル権限検査（group/other 読取可なら fail-closed）にそのままでは不適格。k8s 版を出すなら `defaultMode: 0400`、readOnly mount、最小 RBACが必須。
4. **controller / Gateway API controller / xDS 相当は [[000069]] で declined。** 素マニフェストは controller ではないが、最初から compose と k8s を同時保守すると運用コストが床の定義を食い潰す。

## Decision

1. **最初の実行可能リファレンスは compose 版を一つに絞る。** 骨格: 前段 L4 LB（PROXY v2 送出）+ Plecto ×2 + backend ×2 + `/readyz` を LB ヘルスに配線。`docker compose up` と数発の検証脚本で、「1 台 drain してもリクエストが落ちない」を体験できること。

2. **TLS / resumption はシナリオ分割する（同一 compose 内の profile 切替または別 override でよい）**:
   - **Scenario A**: non-client-auth listener + shared STEK（[[000062]] オプトイン）。
   - **Scenario B**: downstream mTLS + per-node resumption（または resumption off）。shared STEK は載せない。
   - **Scenario C**: upstream mTLS（downstream の resumption mode と独立、[[000078]]）。

3. **global rate limit を骨格に含める場合**は、[[000081]] 未達の間 `demo-only / trusted local network` と明記し、production claim に含めない。secure Redis 着地後に同じ骨格へ昇格脚を足す。

4. **k8s 素マニフェストは初期 floor に含めない。** operator trial（[[000077]]）で需要が確認されてから第二成果物とする。出す場合の不変条件: STEK・client key・Redis credential は ConfigMap ではなく Secret volume、`defaultMode: 0400`、readOnly、最小 RBAC。**controller 化はしない**（[[000069]] 不変）。

5. **未実装を偽装しない。** mTLS 未着地の間は Scenario B/C を「将来脚」またはスタブとして明示し、technical preview が完成形を名乗らない（[[000077]]）。

検討した代替案と却下理由:

- **単一の「完全な標準形」compose に mTLS + shared STEK + secure Redis を全部載せる** — 却下。TLS 標準と既存 ADR の不変条件に矛盾するか、未実装を隠す。
- **最初から compose と k8s を同時出荷する** — 却下。保守コストが床の価値を上回る。需要未確認。
- **素の k8s を controller へ育てる** — 却下。[[000069]]。
- **実行可能形を作らず operations 散文の充実だけにする** — 却下。需要側 floor の「試せる」要件を満たさない。

本決定は [[000056]] の Multi-Replica Readiness を「読める」から「走れる」へ具体化する。単一ノード first と declined controller は維持する。

## Consequences

### Pros
- 散文の運用知識が体験可能な形になり、technical preview の骨格になる。
- TLS シナリオ分割により、STEK と mTLS のどちらも嘘なく示せる。
- k8s を後続に回すことで、Secret 権限などの落とし穴を「出すとき」に集中扱いにできる。

### Cons / Tradeoffs
- compose 固有の前提（ローカル DNS、LB 実装の選択）がそのまま本番トポロジではない——「reference」であり「唯一の正しい本番」ではない旨を文書化する必要がある。
- Scenario が増えるとドキュメントの分岐コストが上がる（三つまでに上限）。
- **再検討条件**: (a) operator trial が k8s 成果物を導入阻害の主因として記録したとき第二成果物を起こす；(b) 前段 LB の PROXY v2 挙動差で骨格が不安定なとき、LB を固定実装にピンするか script レベルで差を吸収する；(c) [[000078]] / [[000081]] 着地で Scenario 注記が陳腐化したとき、昇格脚へ更新する。
