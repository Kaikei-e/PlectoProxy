# 対外メッセージの第一看板を供給網検証つき拡張性へ固定し、mTLS を mesh 非導入環境の補完第二看板とする

## Status
accepted（2026-07-13 grill セッションで承認）

## Date
2026-07-13

## Affected Components
- roadmap / governance — [[000067]] が確定した三本柱（Verifiable Trust × Contract-Derived Performance × Declared Semantics）と [[000076]] の近期ユーザー定義は変えず、**前面に出す順序と従属関係**だけを本 ADR が amend する
- positioning — 一文ポジショニング（[[000067]]）へ補完第二文を追加。README 等の実書き換えはスコープ外

## Context

2026-07-13 の開発者調査（先行事例の実証研究＋2026-07 時点の技術的空白調査。本 ADR 執筆にあたり規制関連の主要事実は一次情報で再確認済み）が、[[000067]] の三本柱に対して「柱は正しいが、どの順で・どの従属関係で前面に出すかが未確定」という残余を示した。空白評価の要旨（個別プロダクト名・定量比較は開発者調査に譲る）:

1. **実行する拡張（filter）の署名・SBOM・capability 検証を必須ゲートにするプロキシは、調査範囲で他に確認できていない。** 既存の拡張実行系で確認できたのは、事前コンパイル済み拡張の検証を運用者の責任として委ねる設計（各自のドキュメントで透明に宣言されている）か、capability 制限設定が未実装のまま残っている形である——欠陥ではなく設計上の選択であり、その選択が空白を生んでいる。第三者スクリプト供給網の乗っ取り事件（CDN ドメイン買収→改竄注入型。被害ホスト数十万規模）は、この空白の痛みが実在することを示した。
2. **規制の構造的追い風は確定事実である。** EU Cyber Resilience Act（Regulation (EU) 2024/2847）は 2024-12-10 発効、Article 14 の報告義務は 2026-09-11 から適用、全面適用は 2027-12-11（2026-07-13 時点で日程変更なし・一次確認済み）。SBOM は Annex I Part II(1)（機械可読形式・最低 top-level 依存）、第三者・OSS コンポーネント統合時の due diligence は Article 13(5) が要求する。一方、SBOM 形式の実装法や OJ 引用済み整合規格は未成立（projected）。
3. **「mesh 未満 mTLS 以上」の空白は実在するが縮小傾向。** コンテナオーケストレーション内では node 単位 mTLS・CNI 統合が急速に埋めつつあり、Plecto の mTLS（[[000078]]）が刺さる射程は **mesh を持ち込めない／持ち込まない環境**（VM・bare metal・エッジ・小規模 self-managed）に限定される。
4. **WIT 型付き契約への顕在需要の一次証拠は薄い。** module ABI 系拡張機構の摩擦（バージョン非互換・仕様策定の停滞）は実在するが、それが「型付き契約に移りたい」という需要に直結する証拠は限定的。供給側の技術優位であり、単独の看板にすると反証可能性を欠く。
5. **memory-safe 実装言語は前提であって看板にならない。** 開発者調査には、実装言語を前面に掲げた技術的に健全なプロキシが長期間牽引力を得られなかった事例や、強力な支援体制の下でも開発停止に至った事例が記録されている。停止・停滞の因果を外部から断定することはできないが、いずれも「実装言語それ自体は採用理由にならない」という需要側の受け止めと整合する観測であり、批判の対象は当該プロジェクトではなく「言語を看板にする戦略」である。

[[000067]] は三本柱を、[[000076]] は近期ユーザー（SaaS/platform/security team × filter author）を確定済み。欠けているのは順序の宣言であり、これが無いと発信のたびに「何を最初に言うか」が揺れる。

## Decision

三本柱（[[000067]]）と近期ユーザー定義（[[000076]]）は不変のまま、対外メッセージの構造を次のとおり固定する。

1. **第一看板 = 柱 1（検証可能な信頼）を「供給網検証つき拡張性」として前面に出す。** 「あなたがロードする拡張の署名・SBOM・capability 契約を必須ゲートで検証する」ことを、機能一覧ではなく最初の一文で言う。
2. **WIT 型付き契約と柱 2（契約駆動の性能）は第一看板の実現手段・根拠として語り、単独の看板にしない。**「WIT だから、何を検証しどの能力を貸すかを型で言える」という従属関係で述べる。型付き契約そのものへの顕在需要は薄い（Context 4）ため、これを主語にした発信は反証可能性を欠く——declared deferred と同じ誠実さで、需要の一次証拠が出るまで従属に置く。
3. **mTLS は補完第二看板とし、射程を「mesh 非導入環境」と明示する。** 一文ポジショニングは [[000067]] の「**Plecto Proxy は『あなたのコードが、検証されて、ゲートウェイに載る』ことを構造で保証する L7 プロキシである**」を維持し、補完第二文として「**そして mesh を持ち込まずに、両方向 mTLS の床を張れる**」を追加する。オーケストレーション内の mTLS 需要を争わない（[[000069]] と整合）。
4. **Rust / memory safety は土台であり、看板にしない。**「なぜ信頼できるか」の裏付けとして背後に置く（Context 5 の観測に照らす）。
5. **CRA の扱いは [[000076]] Decision 3 の規律を継承する。** 適合主張はしない・規制カレンダーを発信期限の根拠にしない。背景文脈として Article 13(5) / Annex I Part II(1) を引き、「検証可能なコンポーネントの需要が構造的に高まる」ことの説明にのみ使う。

検討した代替案と却下理由:

- **柱 2 を「mesh 未満 mTLS 以上」へ差し替える**（開発者調査の三本柱案そのまま）——却下。[[000067]] の柱 2 には body 税 −76% 等の実測資産が既に紐付いており、柱の再編は append-only の決定履歴上の必然性がない。順序と従属の宣言で調査の意図（供給網検証を第一に、WIT を手段に、mTLS を補完に）は完全に満たせる。
- **一文ポジショニングを「供給網検証済みのセルフホスト型 L7 リバースプロキシ」型へ全面差し替える**——却下。「セルフホスト」を前面へ戻すのは、self-hosting を顧客セグメント名として使わないと定めた [[000076]] と緊張する。既存の一文は既に供給網検証中心であり、差し替えは連続性を失うだけ。
- **Rust / memory safety を前面に出す**——却下。Context 5 の観測。政府系の memory-safety 推奨は土台の裏付けとしてのみ使う。
- **供給網検証を CRA 適合とセットで売る**——却下。[[000076]] が適合主張を禁止済み。整合規格が OJ 引用されていない現時点で適合は主張不可能であり、主張すれば柱 3（宣言された意味論）に反する。
- **順序を宣言せず発信ごとに判断する**——却下。単独開発では発信の一貫性こそが認知の資産になる。揺れは費用。

本決定は [[000067]] を supersede せず、その対外運用（順序・従属・射程）を確定する amend である。P4（deny-by-default / fail-closed）・P5（Verify before load）を看板の言葉として具体化し、P9（正直な計測・誇張しない主張）に従属する。

## Consequences

### Pros
- 発信のたびに「何を最初に言うか」が揺れなくなり、README・発信・審査票回答が同じ順序で書ける。
- WIT を従属に置いたことで、「型付き契約の需要が薄い」という調査の Caveat に対して誠実な位置取りになる（需要が顕在化したら格上げすればよい）。
- mTLS の射程明示により、mesh 系との比較質問に「争わない領域」を先に言える（[[000069]] の declined と同じ運用短縮効果）。

### Cons / Tradeoffs
- 第一看板は**需要の顕在化タイミングへの賭け**を含む（開発者調査 Caveat 継承）。CRA 全面適用（2027-12-11）まで需要が本格化せず、当面は供給網事件の記憶に依存する可能性がある。
- mesh 非導入環境という射程は、市場規模の上限を自ら明示する行為でもある（誠実さの費用）。
- **再検討条件**: (a) 三点交差（型付き契約 × 拡張側供給網検証 × fail-closed 不変則）の反例プロダクトが確認されたとき（[[000067]] 継承）; (b) mesh 系の node 単位 mTLS が非オーケストレーション環境（VM・bare metal）へ実用的に拡大し、空白が消えたとき、補完第二看板を降格する; (c) WIT 型付き契約への需要の一次証拠（移行事例・issue 等）が確認されたとき、従属からの格上げを再評価する; (d) 第一看板への牽引（問い合わせ・trial の理由）が一定期間観測されないとき、順序を再検討する。
