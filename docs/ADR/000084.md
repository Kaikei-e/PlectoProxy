# 到達性を機能追加より優先する床と定め、operator TTFV 5 分以内を需要側 North Star に置く

## Status
accepted（2026-07-13 grill セッションで承認）

## Date
2026-07-13

## Affected Components
- roadmap / governance — [[000077]] の technical preview 床に到達性項目を追加し、preview 着地までの優先規則を定める
- docs / README — 英語 quick-start（署名検証込み・1 コマンド導線）の一級文書化。実装は本 ADR のスコープ外
- examples — [[000082]] の compose reference を到達性項目として扱う

## Context

2026-07-13 の開発者調査（先行事例の実証研究）が横断的に示したパターン: **使われ始めた瞬間に存在した最小構成は「1 コマンド / 1 ファイル / 賢いデフォルト」であって機能網羅ではない**。逆に、技術的に健全でも牽引力を得られなかった事例には、エコシステム統合（イメージ配布・compose・パッケージ）の不足が共通して観測されている——因果の断定ではなく相関の観測だが、到達性（reachability）が埋まらない限り技術的優位は誰にも観測されない、という本 ADR の前提とは整合する。

同日のリポジトリ実査による正直な棚卸し（開発者調査はリポジトリ実態未検証の前提で書かれており、「コンテナイメージ公開 未了（要確認）」は実態と相違する）:

- **済**: 署名付き multi-arch コンテナイメージ（minimal / capabilities profile）は v0.1.0 以降、タグごとに release workflow が ghcr.io へ公開している（[[000047]] / [[000079]]）。英語 README。WIT 契約の OCI 配布（[[000064]]）。reference filter の署名付き OCI 配布（[[000080]]）。
- **未**: 署名検証済みイメージの取得から最初のプロキシ応答までを **1 コマンド級の導線**として案内する英語 quick-start の一級文書化と、その外部初見者による検証。マルチレプリカ compose reference の実装（[[000082]] は proposed）。

指標の非対称も残っている。供給側（filter author）には TTFVF ≤15 分（[[000065]]、計測規律は [[000076]] / [[000077]]）があるが、**需要側（operator）には到達時間の指標が無い**。[[000077]] は technical preview の床（prebuilt capability profile・署名済み filter OCI・compose skeleton の再現可能性）を定義したが、床項目に quick-start 導線は含まれず、「到達性と新規機能のどちらを先に作るか」の優先規則も暗黙だった。

## Decision

1. **優先規則: technical preview（[[000077]]）着地まで、到達性項目を新規機能スライスより優先する。** 到達性項目 = 署名検証込み quick-start 導線・compose reference（[[000082]]）・英語 first-run 文書。例外は security fix と、既に accepted な ADR の完了スライス（[[000078]] mTLS 等）。preview 着地後、本優先規則は失効し通常の役割駆動（[[000029]]）へ戻る。

2. **[[000077]] の technical preview 床へ追加する**: 「初見の運用者が、公開手順のみを頼りに、**署名検証済みイメージから最初のプロキシ応答**へ到達できる英語 quick-start が再現可能であること」。

3. **operator TTFV（Time To First Verified proxy response）を需要側 North Star として定義する: ≤5 分。** 計測区間は「公開 quick-start を開いてから、イメージの署名検証（cosign verify）を経て、最初のプロキシ応答（例: ローカル backend への 200）を得るまで」。供給側 TTFVF ≤15 分（[[000065]]）と対を成す。

4. **計測規律は [[000076]] / [[000077]] を継承する。** 外部初見者による計測のみを実績とし、失敗・離脱を含めて記録する。maintainer のクリーン環境自己計測は install-path smoke であり実績にしない。**hard gate にはしない**（[[000077]] と同型の継続指標。未達は DX backlog 優先の根拠になるが、recruiting / engineering log を拒否しない）。

5. **署名検証を導線から省かない。** 「速いが unverified」な quick-start を既定にすると柱 1（[[000067]] / [[000083]]）と自己矛盾する。cosign verify を含めて 5 分に収めることが、第一看板と DX の交差点であり差別化そのものである。検証手順を optional な補足に落とさない。

検討した代替案と却下理由:

- **数値を置かず定性床のみにする**——却下。反証可能性を欠き、「maintainer には簡単に見える導線」を外部初見者が完走できない事態を検出できない（[[000076]] の教訓）。
- **TTFVF に統合して一本の指標にする**——却下。ペルソナ（operator / filter author）も導線（イメージ取得 / フィルタ開発）も別物。混ぜると [[000077]] が確立した「床の不足と DX の不足の区別」を再び失う。
- **検証なしの最短導線（署名検証を省いた 1 コマンド）を既定にする**——却下。Decision 5 のとおり第一看板との自己矛盾。到達性のために信頼の看板を下ろすなら、それは別プロダクトである。
- **operator TTFV を hard gate にする**——却下。[[000077]] が hard gate 方式の欠陥（弱い主張まで遅延する）を既に確定している。
- **k8s 向け到達性（Helm 等）を床に含める**——却下。[[000069]] の declined と [[000082]] の「k8s 素マニフェストは operator trial 後」を維持する。

本決定は [[000077]] の段階的発信を supersede せず、technical preview の床項目と優先規則を amend する。P9（正直な計測）を需要側へ拡張し、[[000065]] の North Star 運用を対称化する。

## Consequences

### Pros
- 「何を先に作るか」が preview 着地まで一意に決まり、単独開発の帯域配分の議論が短縮される。
- 需要側・供給側の両方に反証可能な到達時間指標が揃い、[[000076]] の獲得ループ（運用者が導入し、filter author が policy を供給する）の両脚を同じ規律で計測できる。
- 「署名検証込みで 5 分」が達成されれば、それ自体が第一看板（[[000083]]）の体験可能な証拠になる。

### Cons / Tradeoffs
- 指標が二本（TTFVF / operator TTFV）になり、計測・公開の運用負担が増える。
- ≤5 分は未計測の仮説である。署名検証 UX（cosign インストール等）が構造的な律速になる可能性がある。
- 到達性優先の期間、機能表の差は開き得る（[[000067]] の発信規律で応答する既知の代償）。
- **再検討条件**: (a) 外部初回計測で ≤5 分が構造的に不可能と判明したとき、目標値の再設定か導線改善（例: 検証手順の同梱スクリプト化）の優先かを判断する; (b) [[000082]] の operator trial が別形の到達性（k8s 素マニフェスト等）を主要阻害として記録したとき、床項目を再構成する; (c) technical preview 着地により本優先規則は失効する（明示の失効条件）。
