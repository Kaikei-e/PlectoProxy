# WIT 契約互換の対外宣言を段階化し、契約 1.0 以降の出荷 world 読み込みを security 例外つき恒久保証とする

## Status
accepted（2026-07-13 grill セッションで承認）

## Date
2026-07-13

## Affected Components
- docs / README / writing-a-filter — 互換ポリシー（[[000064]] で発効済み）を「探せば見つかる一節」から「対外の看板」へ格上げする
- wit / governance — 契約 1.0 の意味を「恒久保証の発効」と定義する。0.x の契約進化の自由度は維持する

## Context

[[000064]] は契約互換ポリシー——「host は出荷した全契約バージョンを読み続ける」「additive = minor / breaking = major」「major 併存は最低 2 リリース系列、廃止は ADR で宣言」——を発効し、実装もこれを実践している（`plecto:filter` 0.1 / 0.2 は凍結＋ロード時アダプタで現行 host が読み続けている。[[000071]] / [[000073]]）。しかしこのポリシーは writing-a-filter の一節にあり、**運用者・filter author が探しに行かないと見つからない**。

2026-07-13 の開発者調査は、単独〜極少人数のメンテナで長期にミッションクリティカル採用へ至った OSS の共通条件として「**互換性の明示的・強制的保証**」を挙げた。本 ADR 執筆にあたり、その種の宣言の原文を一次情報で再確認した（2026-07-13 取得）ところ、実形は次の三型であり、いずれも**反証可能なヘッジ付き**だった:

1. 「**原則として** ABI を破らない。SONAME bump は可能な限り稀にし、**理想は二度と行わない**」型——最初期に数回の破壊があった履歴を明示した上での宣言。「絶対に破らない」という断言はしていない。
2. 「20XX 年まで支援する**つもりで計画する**（そこまでの存続を絶対には約束できないが、そのつもりで計画することは約束する）」型＋ API / ファイル形式の後方互換の明示。
3. 「**security 問題を除き**、仕様に非互換変更を加えない」型。

対照的に、module ABI 型の拡張機構ではバージョン非互換がフィルタ作者の実運用摩擦として頻発している（開発者調査）。フィルタ資産は「一度書いて長く載せる」性質を持つため、**契約の安定性宣言は拡張エコシステムの最重要インフラ**であり（[[000064]] Context）、その宣言が「将来も続くのか」に答えることが差別化になる。

一方で確定している将来: wasi:http への収斂（[[000020]]）は契約 major の最初の実戦であり、[[000064]] の再検討条件 (a) はその実戦で併存運用の妥当性を検証するとしている。0.x の現段階で「もう二度と破らない」型を誓うと、この予定された進化と正面衝突する。

## Decision

互換宣言を **0.x（現行）と契約 1.0 以降（先行コミット）の二段**に分ける。

1. **0.x の間: [[000064]] の現行ポリシーを対外の一級文書へ顕在化する。** README と writing-a-filter の冒頭級の節で「host は出荷した全契約バージョンを読み続けている（0.1 / 0.2 凍結＋アダプタが現に動いている実績）」を、実装済みの事実として掲げる。**新たな約束は増やさない**。0.x world の廃止は [[000064]] どおり（major 併存 ≥2 リリース系列＋ADR 宣言）。

2. **契約 1.0 到達時に発効する恒久保証を、今、先行コミットする**:
   > `plecto:filter` 1.0 以降に出荷したすべての world は、以後の host が読み続ける。例外は「当該 world を読み続けること自体が安全上維持不能」な場合に限る。その場合も、単独の ADR、最低 24 ヶ月の事前告知、移行文書を必須とする。

   これは Context の型 3（security 例外）と型 1（履歴と限界を隠さない誠実さ）の合成である。「絶対」という語ではなく、**機構（ADR による反証可能な宣言）と告知期間（24 ヶ月）**で担保する。

3. **契約 1.0 を切る行為は、この保証の発効を意味する。** したがって 1.0 の前提条件に「wasi:http 収斂（[[000020]]）の帰趨が確定していること」を含める——major の実戦（[[000064]] 再検討条件 (a)）を 0.x のうちに済ませ、恒久保証はその学習の後に入る。1.0 は機能の節目ではなく**約束の節目**である。

4. **宣言文はヘッジの形まで含めて誠実に書く。** 対外文書では「0.x での実績（読み続けている）」と「1.0 以降の保証（読み続けることを約束する）」を混同させない。[[000077]] の主張境界の規律（満たした性質だけを主張する）に従う。

検討した代替案と却下理由:

- **今すぐ恒久（ever）宣言する**——却下。0.x で予定される契約進化（wasi:http 収斂・body / streaming [[000003]] / [[000025]]）を 24 ヶ月告知の対象にしてしまい、「進化を止める」か「宣言を破る」かの二択に自分を追い込む。最初期に破壊があり得ることを隠して強い宣言をするより、段階を明示する方が Context の先行事例の実形にも近い。
- **現行ポリシーの顕在化だけで済ませる**——却下。「いま読み続けている」は言えても「将来も読み続けるのか」に答えず、開発者調査の信頼条件（明示的・強制的保証）を満たさない。フィルタ資産の寿命を作者から予測可能にする、という [[000064]] の目的を対外に完結させられない。
- **例外なしの純粋 ever 宣言**——却下。sandbox 境界の契約という性質上、「旧 world の意味論を安全に実装し続けられなくなる」事態（capability 意味論の致命的欠陥等）を排除できない。そのとき宣言違反か危険な互換維持かの二択になる宣言は、fail-closed（P4）と両立しない。
- **期限付き保証（出荷から N 年）**——却下。フィルタは「作って忘れる」資産であり、期限切れの管理を作者側に課す保証は約束の価値の大半を失う。
- **告知期間 12 ヶ月**——却下。年次程度の周期でしか触らないフィルタ資産に対して短い。24 ヶ月は「最低 2 リリース系列併存」（[[000064]]）の実運用時間とも整合する。

本決定は [[000064]] の互換ポリシーを supersede せず、その対外宣言の形（段階化）と 1.0 の意味（恒久保証の発効）を amend する。P4（fail-closed）・P9（正直な計測・誇張しない主張）を契約の約束の言葉として具体化する。

## Consequences

### Pros
- フィルタ作者・運用者から見て、契約資産の寿命が「現在の実績」と「将来の保証」の二層で予測可能になる。module ABI 型のバージョン摩擦（開発者調査）への最も構造的な差別化。
- 1.0 に「約束の節目」という明確な意味が与えられ、バージョン付けの判断基準が機能量から独立する。
- 宣言の型（security 例外＋ADR＋告知期間）が先に決まっているため、将来の廃止判断が場当たりにならない。

### Cons / Tradeoffs
- 1.0 を切る判断が重くなる（それ自体が恒久保証の発効であるため）。1.0 が遅れる圧力になり得る。
- security 例外の該当判定は maintainer の説明責任であり、単独 ADR で反証可能に書く規律が要る。
- 読み続ける major が増えるたび、ロード時アダプタ（[[000071]] / [[000073]] の系譜）の維持・テストコストが累積する。
- **再検討条件**: (a) wasi:http 収斂 major の実戦で併存運用が回らないと判明したとき、1.0 発効前に保証の形を再設計する（[[000064]] 再検討条件 (a) の継承）; (b) Component Model / WIT 側の仕様変動で「world を読み続ける」の技術的意味（アダプタ実現可能性）が変わったとき、保証の定義を精密化する; (c) 24 ヶ月告知が実運用の観測（フィルタ更新周期の実データ）と乖離したとき、期間を再評価する。
