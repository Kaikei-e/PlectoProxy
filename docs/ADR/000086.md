# 単独メンテナの信頼代替を明文化する——検証文化の可視化・最終リリースの再現可能性・EOL 撤退プロトコル

## Status
accepted（2026-07-13 grill セッションで承認）

## Date
2026-07-13

## Affected Components
- governance / docs — 意図宣言・撤退プロトコル・検証文化一覧を対外文書化する（本 ADR は方針の確定。文書実装は後続スライス）
- CI / release — 変更なし。既存の検証・供給網規律を可視化の対象として参照する

## Context

2026-07-13 の開発者調査は、単独〜3 人規模のメンテナで数十年級・ミッションクリティカル採用に至った OSS の共通条件を三つ抽出した: **(a) 異常なテスト・検証文化、(b) 互換性の明示的・強制的保証、(c) 長期コミットメントの宣言**。逆に、技術的に健全でも牽引力が観測されなかった事例（長期の停滞・開発停止）も記録されている——因果の断定ではなく観測であり、批判の対象は個別プロジェクトではなく「健全性だけに頼る戦略」である。単独メンテナは「制約」ではなく、この三条件で信頼を代替できる設計対象である。

(b) は [[000085]] が確定した。本 ADR は (a) の可視化と (c) の形を定める。

現状の棚卸し:

- Plecto には検証の実体が既にある——PR-light / merge-heavy に分割された CI、fuzzing、polyglot conformance、release-parity ビルド、cosign 署名＋SBOM attestation＋cargo-deny の供給網検証（[[000047]] / [[000080]]）、ADR-first governance（[[000029]]）。しかし**一覧で見える場所が無く**、審査する側は個々の workflow と ADR を掘らないと検証文化の全体像に到達できない。
- 長期コミットメントの宣言は**欠落している**。一次情報で再確認した先行の年限型宣言（2026-07-13 取得）も「20XX 年まで支援する**つもりで計画する**——絶対の約束はできないが、そのつもりで計画することは約束する」というヘッジ付きの意図宣言であり、断言型ではない。Plecto は 0.x・外部採用実績ゼロの段階で、年限を置く根拠となる履歴が無い。
- 単独メンテナの離脱（bus factor）は審査票の定番懸念であり、[[000067]] 柱 1 の顧客（審査言語で問う組織）に対して、機能ではなく **governance で答える**必要がある。

## Decision

長期コミットメントを「年限の約束」ではなく、**意図の宣言＋守れる機構**の組で宣言する。機構は二層——意図的な終了には告知期間が、不可抗力には再現可能性が効く。

1. **検証文化の可視化。** 何をどこで検証しているか（CI ジョブ構成と PR-light / merge-heavy の分割方針、fuzzing、polyglot conformance、release-parity、供給網検証 [cosign / SBOM / cargo-deny]、ADR グラフ検証）を docs の一枚に地図として一覧する。**記録の実体は default-branch CI green であり、別台帳は作らない**（[[000080]] Decision 2 と同じ規律——一覧は実体への地図に徹し、二重帳簿化による drift を避ける）。

2. **長期コミットメントは意図宣言の形をとる。** 「Plecto を長期に支援するつもりで計画する」——年限は置かない。宣言の信頼性は言葉ではなく、下記 3 / 4 の機構と、積み上がる履歴（互換実績 [[000085]]・リリース実績）が担保する。

3. **EOL 撤退プロトコル（意図的な終了判断の場合）。** メンテナンス終了を決定した場合、**最低 12 ヶ月前に告知**し、告知期間中は security fix を継続する。廃止対象・移行先の選択肢・最終リリース計画を告知文書に含める。

4. **最終リリースの再現可能性（不可抗力もカバーする層）。** すべてのタグ付きリリースは、ソース・署名済み成果物・SBOM attestation・pin 済み依存・ビルド手順の組として自己完結しており、**メンテナ不在でも第三者が検証・再現・fork できる**状態を維持する。これは新規の仕組みではなく、既存の供給網規律（[[000047]] / [[000064]] / [[000080]]）が既に生んでいる性質を「継続の保険」として明文化・維持対象化するものである。告知（Decision 3）が事故・離脱等の不可抗力に無力であることを隠さず、その場合の答えはこの層である、と対外文書に明記する。

5. **掲載先と主張境界。** README / docs の governance 節に置く。SaaS 導入水準（[[000054]]）は品質の物差しとして維持しつつ、本宣言を販売の約束（サポート契約の代替等）として語らない（[[000076]] / [[000077]] の主張境界の規律を継承）。

検討した代替案と却下理由:

- **年限明示型の宣言（「20XX 年まで」）**——却下。0.x・採用実績ゼロの単独メンテナが年限を置くのは、先行事例がヘッジしてなお持っていた履歴の裏付けを欠く過大約束（P9 違反）。1.0 到達・採用実績の後の再検討条件とする。
- **宣言なし（検証文化の可視化だけ）**——却下。三条件のうち (c) を欠き、bus factor 懸念に governance で答えるという本 ADR の目的を果たさない。
- **後継メンテナ探索の意図表明を含める**——却下（今回は）。招待制コントリビューション方針（[[000029]] 系譜の governance）との整合が未設計であり、安易に書くと「PR を歓迎する」と誤読される。必要になったら別 ADR で設計する。
- **撤退プロトコルに告知期間を含めない（再現可能性のみ）**——却下。意図的終了は単独メンテナで最も蓋然性の高い終了形であり、そこに時間の約束が無い撤退プロトコルは審査懸念への答えとして半分しか機能しない。守れる範囲（意図的終了時）に限定した 12 ヶ月なら反証可能な約束にできる。
- **検証文化の一覧を独立の台帳（カバレッジ表等）として維持する**——却下。CI 実態との drift が不可避で、[[000080]] が確立した「CI green を実体とし別台帳を作らない」規律に反する。

本決定は [[000029]]（役割駆動・declined の言語化）の governance 面を対外信頼の言葉へ延長し、P9（正直な計測・誇張しない主張）を「プロジェクトの寿命」という主題に適用する。

## Consequences

### Pros
- 審査票の bus factor 懸念に、機能ではなく governance（可視化された検証・告知期間・再現可能性）で答えられるようになる。
- 新規の仕組みをほぼ増やさない——既にある検証と供給網規律を「見える」ようにするだけで三条件の (a)(c) を満たす。単独開発の帯域に優しい。
- [[000085]]（互換保証）と合わせ、開発者調査の信頼三条件が ADR 三点で完結する。

### Cons / Tradeoffs
- 12 ヶ月告知は不可抗力（事故・疾病等）には無力である。この限界は対外文書に明記する（隠すと宣言全体の信頼を毀損する）。
- 「意図の宣言」は法的拘束のない約束であり、信頼は履歴でしか積み上がらない。宣言直後に効果を期待しない。
- 可視化文書は CI 変更のたびに更新が要る（地図に徹することで軽くするが、ゼロにはならない）。
- **再検討条件**: (a) 契約 1.0 到達・外部採用実績が積み上がったとき、年限型宣言への格上げを再検討する; (b) コントリビューション方針（招待制）が変わったとき、後継探索条項の追補を検討する; (c) 可視化一覧が CI 実態と drift していると観測されたとき、一覧の粒度を下げて実体参照へ寄せる。
