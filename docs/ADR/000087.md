# 署名検証込み operator quick-start を docs/quickstart/ の一級文書として実装し、検証導線をコンテナ実行の cosign と digest 解決に固定する

## Status
accepted（2026-07-13 grill セッションで承認）

## Date
2026-07-13

## Affected Components
- docs/quickstart/ — `README.md`（英語・正）+ `README.ja.md`（ミラー）を新設。署名検証 → 起動 → 最初のプロキシ応答までの全手順を自己完結で置く
- README — `## Quick start` 節を新設し、要約 3 手順と一級文書へのリンクを置く。既存の「Try it」（ソースビルド導線）は filter author 向けとして残す

## Context

[[000084]] は「初見の運用者が、公開手順のみを頼りに、署名検証済みイメージから最初のプロキシ応答へ到達できる英語 quick-start」を technical preview の床に加え、operator TTFV ≤5 分を需要側 North Star と定めた。導線の実装形はスコープ外だったため、本 ADR が確定する。

2026-07-13 のリポジトリ実査と Web 調査（一次情報で確認）で判明した実装制約:

1. **「tag で verify」は現行 release 設計では不可能。** release workflow は cosign keyless 署名を **multi-arch index digest に対してのみ**行い、mutable な tag には署名しない（[[000047]] / [[000079]] の設計どおり）。quick-start は tag→digest の解決を導線に含める必要がある。
2. **plain Docker には pull 時の署名検証が存在しない**（確定事実、2026-07-13 取得: Docker Content Trust は 2025 年に廃止告知済み。pull 時強制は Kubernetes admission や別コンテナランタイムの世界）。検証は `docker run` の前に明示コマンドで行うしかない。
3. **cosign のインストールが律速になり得る**（[[000084]] Cons が明示した懸念）。一方、cosign 公式は署名済みコンテナイメージ（`ghcr.io/sigstore/cosign/cosign`）を配布しており、keyless 検証はローカル状態を要しないため **`docker run` だけで検証が完結**する。前提ツールが Docker のみに縮む。
4. **`gh attestation verify` は現行 release では使えない。** release workflow は GitHub artifact attestations（`actions/attest-build-provenance`）を発行しておらず、導入には workflow 変更と gh CLI 認証前提が要る。
5. コンテナイメージは nonroot・manifest（TOML）必須のため、quick-start には最小 manifest と backend スタンドインの用意が要る。

## Decision

1. **置き場所は `docs/quickstart/` を新設し、英語 `README.md` を一級文書とする**（`README.ja.md` ミラー併設、docs の既存慣行どおり）。repo README には要約と一級文書へのリンクのみを置く。到達性の入口は README で、深さは docs 側で受ける。

2. **検証導線は「digest 解決 → コンテナ実行の cosign で verify → その digest で run」の 3 手に固定する**:
   - tag→digest 解決は `docker buildx imagetools inspect` の 1 行で行う（release notes の digest は照合先として言及）。
   - 検証は `docker run --rm ghcr.io/sigstore/cosign/cosign:<pinned> verify <image>@<digest> --certificate-identity-regexp ... --certificate-oidc-issuer ...` を主導線とし、ローカル cosign のインストール手順は代替として併記する。
   - 起動は検証済み digest を明示指定して `docker run ghcr.io/<owner>/plecto@<digest> ...` で行う。**検証した対象と実行する対象を digest で一致させる**（解決結果が改竄されていれば verify が fail する。fail-closed）。

3. **手順は clone 不要・コピペ完結とする。** 最小 manifest（`[[upstream]]` + `[[route]]` + `[observability] admin_addr`）は heredoc で書き出し、backend スタンドインは `traefik/whoami` 1 コンテナを使う。backend はユーザ自身のサービスの代役であり、Plecto の信頼主張の対象外である旨を 1 行明記する（検証済みなのは Plecto イメージ、という主張境界。[[000077]] の規律）。

4. **署名検証を optional に落とさない。** [[000084]] Decision 5 のとおり、verify は手順の中核であり補足へ降格しない。

5. **TTFV の計測規律は [[000084]] を継承する。** 文書には計測区間（quick-start を開いてから最初のプロキシ応答まで）を明記するが、maintainer の自己計測は実績と主張しない。

検討した代替案と却下理由:

- **ローカル cosign インストールを主導線にする**——却下。パッケージマネージャ差・PATH 問題が 5 分予算の律速になる（[[000084]] Cons の懸念そのもの）。代替として併記に留める。
- **`gh attestation verify` を主導線にする**——却下。release workflow への attest ステップ追加と gh 認証前提が必要で、検証手順が二系統になる保守コストも乗る。需要が観測されたら再検討。
- **digest を release notes からコピペさせる**——却下。ページ往復が入り 5 分予算と写し間違いリスクに響く。照合先としての言及に留める。
- **verify + run を同梱スクリプト（curl 一発）にする**——却下。「検証を隠すスクリプト」は透明性と緊張し、curl|bash 型の導線は供給網検証を第一看板とする [[000083]] と相性が悪い。全コマンドが見えるコピペ手順の方が第一看板の体験そのものになる。
- **repo README に全手順を直書きする**——grill で「新設ディレクトリに置く」を選択。README の肥大を避け、ja ミラー慣行にも乗る。

本決定は [[000084]] の床項目（quick-start 導線）の実装形を確定するものであり、優先規則・North Star・計測規律には触れない。P5（Verify before load）を運用者の最初の 5 分の体験として具体化し、P9（誇張しない主張）に従属する。

## Consequences

### Pros
- 前提ツールが Docker のみになり、cosign インストールという構造的律速（[[000084]] Cons）を消せる。
- 全コマンドが可視のコピペ手順なので、「署名検証込みで 5 分」という第一看板の体験可能な証拠（[[000084]] Pros）がそのまま成立する。
- 検証対象と実行対象が digest で一致し、tag の可変性に起因する TOCTOU を導線レベルで塞ぐ。

### Cons / Tradeoffs
- cosign コンテナの pull が加わる（初回のみ・数十 MB 級）。TTFV 予算内に収まるかは外部初回計測で確認する。
- backend スタンドイン（`traefik/whoami`）は未検証のサードパーティイメージであり、その旨の明記を怠ると主張境界が曖昧になる（文書側で 1 行明記して受ける）。
- cosign のメジャーアップ（現行 3.x、v4 で旧 flag 削除予定 = projected、2026-07-13 取得）で検証コマンドの表記が変わり得る。pin したバージョン表記の追従が要る。
- **再検討条件**: (a) 外部初回計測で cosign コンテナ pull が律速と判明したとき、代替導線の昇格（ローカル cosign / 検証スクリプト同梱）を再判断する; (b) release に GitHub artifact attestations が加わったとき、`gh attestation verify` の併記を再検討する; (c) cosign v4 で verify UX が変わったとき、pin と表記を更新する。
