# マルチレプリカ compose reference を PROXY v2 送出 L4 LB 前段の骨格として実装し、TLS シナリオ A/B を override ファイル切替で同梱する

## Status
accepted（2026-07-13 grill セッションで承認。[[000082]] の実装形を確定する）

## Date
2026-07-13

## Affected Components
- plecto/examples/multi-replica/ — `compose.yaml`（骨格 + profiles）、LB 設定、シナリオ別 manifest、証明書/STEK 生成スクリプト、drain・resumption 検証スクリプト、README
- README — Prior art & acknowledgements 節へ PROXY protocol 仕様の提供元帰属を 1 行追加

## Context

[[000082]] は compose 先行の実行可能リファレンス（前段 L4 LB + Plecto ×2 + backend ×2 + `/readyz` 配線、TLS シナリオ A/B/C 分割）を決定したが、status: proposed のまま実装形（LB の選定、シナリオの実装深度、配置）が未確定だった。2026-07-13 の実査と Web 調査（一次情報、取得日明記）で確定した前提:

1. **必要プリミティブはすべて landed かつ v0.3.2 の公開署名イメージに含まれる**——`/readyz`（admin listener、drain で 503）、PROXY v2 受信（trusted CIDR 必須）、graceful drain（SIGTERM → readiness flip → window）、共有 STEK（`[resumption]`、0600 検査 + mTLS 排他 fail-closed）、downstream mTLS（`[listen.client_auth]`）。ソースビルド不要で骨格が組める。
2. **前段 L4 LB の要件は「PROXY v2 を送出でき、トラフィックと別ポートの readiness を active check できる」こと。** compose で常用される OSS のうちこの 2 条件を今日満たす実装として HAProxy を確認した（`send-proxy-v2`、`option httpchk` + `check port`。現行 LTS 3.4、2026-07-13 取得）。
3. **隣接プロキシを reference に同梱する慣行の先例調査**（2026-07-13 取得）: クラスタ管理 OSS が競合他社プロキシの設定ファイルをリポジトリ直下に同梱する例、コンテナオーケストレータの公式 HA ガイドが同 LB を設定例つきで提示する例、TLS を終端しないキャッシュプロキシが競合プロキシの前段配置を公式文書化する例を確認。成熟プロジェクトの標準形であり、共通規律は (a) 役割の中立的説明、(b) 商標帰属 + 非提携の明記、(c) 代替の併記、(d) 設定ファイルと `image:` 参照のみで再頒布しない、の 4 点。HAProxy 自身の商標ポリシーも参照的使用を明示許容し、帰属明記と endorsement 示唆の禁止を条件とする。`.cursor/rules/rfc-first-hrt.mdc` は「subject がそのプロダクト自体」の名指しを許しており（[[000057]] が先例）、この 4 規律はその運用形である。
4. compose の変種構成には公式に使い分けがある（2026-07-13 取得）: `profiles:` は任意サービスの追加/除外に、複数 `-f` の merge（override ファイル）は**同一サービスの属性変更**に向く。本 reference のシナリオは常在サービス（plecto / LB）の manifest・port・user を変えるので後者に該当する。スケールは `container_name` と併用不可のため、drain デモで特定ノードを止める本 reference には明示 2 サービスが適する。

## Decision

[[000082]] の骨格・シナリオ分割・呼称規律を維持したまま、実装形を次のとおり確定する（Decision 2 のシナリオ実装深度と Decision 3 の LB ピンが [[000082]] への amend 点）。

1. **配置は `plecto/examples/multi-replica/`。** 中身は `compose.yaml`（骨格 + profiles）、`lb/haproxy.cfg`、`manifests/`（シナリオ別 plecto.toml）、`scripts/`（証明書・STEK 生成、drain 検証、resumption 検証）、`README.md`。Plecto は公開署名イメージ（`ghcr.io/<owner>/plecto:0.3` 系、env で差し替え可）を使い、検証手順は quick-start（[[000087]]）へリンクする。

2. **シナリオは A/B を実装し、C は将来脚として文書化する**（[[000082]] は「A 実装、B/C はスタブ可」だったが、`[listen.client_auth]` が landed 済みのため B も嘘なく実装できる。grill で実装を選択）:
   - **既定（骨格）**: plaintext HTTP。L4 LB（PROXY v2 送出）→ Plecto ×2（PROXY v2 受信、trusted は LB の固定アドレスのみ）→ backend ×2（round-robin）。`/readyz` を LB の active check に配線し、`scripts/verify-drain.sh` が「1 台 drain してもリクエストが落ちない」を連続リクエストで実証する。
   - **Scenario A（`compose.scenario-a.yaml` override）**: TLS 終端 + 共有 STEK（[[000062]] オプトイン）。自己署名証明書と STEK の生成スクリプト同梱（demo-only 明記）。`scripts/verify-resumption.sh` が「レプリカ 1 で得たチケットがレプリカ 2 で resume する」を実証する。
   - **Scenario B（`compose.scenario-b.yaml` override）**: downstream mTLS（`[listen.client_auth]`）+ per-node resumption。共有 STEK は載せない（[[000062]] / [[000078]] の排他は build が fail-closed で強制済み）。client 証明書生成スクリプト同梱。
   - **Scenario C（upstream mTLS）**: manifest 面（`[[upstream]] client_cert_path`）は存在するが、本 reference では将来脚として README に明記するのみ（初回スコープの上限。[[000082]] の「三つまで」の分岐コスト上限に従う）。
   - global rate limit 脚は [[000081]] 着地まで含めない（[[000082]] Decision 3 の「載せるなら demo-only 明記」よりさらに保守的に、載せない側を選ぶ。骨格の主題は multi-replica readiness であり、demo-only 注記の混入で主題を濁さない）。

3. **L4 LB は HAProxy（保守中の LTS 系列に pin）を「例」としてピンし、先例の 4 規律を守る**:
   - README に役割を 1 行で中立に説明する（クラウド NLB / 既存 L4 LB のスタンドインであり、Plecto が争わない層。[[000069]] と整合）。
   - 商標帰属と非提携を明記する（HAProxy is a trademark of HAProxy Technologies; not affiliated / not endorsed 型）。repo README の Prior art & acknowledgements にも PROXY protocol 仕様提供元としての 1 行を追加する（[[000057]] の敬意ある出典明示の対外面への延長）。
   - 代替を併記する——「PROXY v2 送出 + 別ポート active check ができる任意の L4 LB で骨格は成立する。本 reference は例として HAProxy を使う」。他実装との比較・除外理由はコミット文書に書かない。
   - 同梱は設定ファイルと `image:` 参照のみ（バイナリ/ソースの再頒布なし）。

4. **compose の実装規約**: 明示 2 サービス（`plecto-1` / `plecto-2`）。LB には Docker 内蔵 DNS への `resolvers` 指定と `init-addr last,libc,none` を設定し、レプリカ再作成での IP 変化に追従させる。シナリオ切替はシナリオごとの小さな override ファイル（`docker compose -f compose.yaml -f compose.scenario-X.yaml`）で行う——シナリオは常在サービスの属性変更であり、compose の使い分け（Context 4）では merge が該当する。`profiles:` は採らない: 全 plecto サービスに profile を振ると素の `docker compose up` で骨格が起動しなくなり、「一発で起動できる」要件（[[000082]]）を壊す。

検討した代替案と却下理由:

- **LB を汎用スロット化して設定を隔離する（bring your own LB を主語にする）**——却下。実際に走るのは結局 1 実装であり、間接化で「一発で起動できる」要件（[[000082]]）が弱る。代替併記の 1 文で中立性は担保できる。
- **HAProxy を避けて PROXY v2 なし骨格へ後退する**——却下。[[000082]] が骨格に据えた client IP 保存の実証を落とす。先例調査により同梱は標準慣行と確認済み。
- **toy L4 LB を自作して同梱する**——却下。非保守のデモ用ネットワークコードを自ら抱えるのは第一看板（検証された供給網）と矛盾する。
- **Plecto イメージをソースビルド（`build:`）にする**——却下。wasmtime を含む workspace のビルドは分単位で到達性を毀損し、公開署名イメージを使う方が供給網検証の導線（[[000087]]）とも接続する。
- **`--scale` による複製**——却下。特定ノードだけを止める drain デモと相性が悪く、`container_name` 併用不可の制約もある。
- **Scenario B もスタブに留める**——却下（grill で実装を選択）。実装が landed している機能をスタブと呼ぶのは [[000077]] の主張境界（満たした性質は満たしたと言う）に照らして過小主張であり、STEK×mTLS 排他という設計上の要点（[[000078]]）を走る形で示す機会を失う。

本決定は [[000082]] を supersede せず、その実装形（配置・LB ピン・シナリオ深度・同梱規律）を確定する amend である。single-node first・declined controller（[[000069]]）は不変。P4（fail-closed）を drain / 排他の検証スクリプトとして、P9 を demo-only 注記として具体化する。

## Consequences

### Pros
- [[000056]] の Multi-Replica Readiness が「読める」から「走れる」になり、technical preview（[[000077]] / [[000084]]）の骨格が揃う。
- STEK 共有と mTLS の排他という不変条件を、注記ではなく profile の切替と fail-closed な build 失敗として体験できる。
- 先例準拠の 4 規律により、隣接プロジェクトへの敬意と中立性が文書構造として担保される。

### Cons / Tradeoffs
- HAProxy の設定・LTS 追従という保守面が増える（設定は最小限に保ち、LB 差し替え可能性を明記して受ける）。
- 自己署名証明書・compose ローカル DNS 等、demo 固有の前提はそのまま本番トポロジではない（[[000082]] Cons の継承。README に明記）。
- Scenario B まで実装したぶん初回の検証スクリプト面積が増える（A/B 各 1 本に抑える）。
- **再検討条件**: [[000082]] の (a)(b)(c) を継承する。加えて (d) pin した LB の LTS 系列が EOL したとき、または PROXY v2 送出 + active check を満たす、より軽量な標準 LB が一般化したとき、pin を更新する。
