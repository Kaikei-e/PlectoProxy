# fuzz smoke を週次 schedule の独立 workflow として CI に組み込み、検証文化の地図を docs/verification.md に一級文書化する

## Status
accepted（2026-07-13 grill セッションで承認）

## Date
2026-07-13

## Affected Components
- .github/workflows/fuzz.yml — 新設。`schedule`（週 1）+ `workflow_dispatch`。nightly toolchain + cargo-fuzz で既存ターゲットを committed corpus 起点の短時間 smoke として実行し、crash 発見時は artifact を保存して赤くする
- docs/verification.md（+ .ja ミラー） — [[000086]] Decision 1 の「検証文化の可視化」の実装。CI 実体への地図に徹する
- README — governance 節相当から verification.md へのリンクを追加

## Context

[[000086]] は検証文化の可視化を決定し、可視化対象として fuzzing を「既にある検証の実体」に数えた。しかし 2026-07-13 のリポジトリ実査で、**fuzz ハーネス（`plecto/fuzz/`、`proxy_v2` ターゲット + corpus）は存在するが、どの workflow・justfile からも呼ばれていない**ことが判明した。このままでは:

- 地図に fuzzing を載せれば「CI で回っている」と誤読され、[[000086]] 自身が依拠する「記録の実体は default-branch CI green」の規律と矛盾する（P9 違反の芽）。
- 載せなければ、第一看板（[[000083]]）を支える検証文化の一覧から、untrusted 入力パーサ（PROXY v2 は fail-closed が要件の代表格）への fuzz が欠ける。

つまり [[000086]] の「CI / release — 新規の仕組みは追加しない」という前提が、fuzzing に関してだけ実態と合っていなかった。地図を弱めるか、実態を直すかの二択であり、本 ADR は後者を選ぶ。

制約: CI は PR-light / merge-heavy の分割で wall-time を厳しく管理している（[[000048]] 系譜、release-parity は main 限定）。fuzz を PR や merge 経路に乗せる余地はない。cargo-fuzz は nightly toolchain を要する。fuzz 対象（PROXY v2 パーサ）は仕様凍結度が高く変更頻度が低い。

## Decision

1. **fuzz を独立 workflow `.github/workflows/fuzz.yml` として新設し、週次 `schedule` + `workflow_dispatch` で回す。** PR / merge 経路の wall-time への影響はゼロ。ジョブは nightly toolchain + cargo-fuzz を導入し、`plecto/fuzz/` の各ターゲットを committed corpus を起点に**ターゲットあたり数分の `-max_total_time` 付き smoke**として実行する。crash / 新規 finding があれば入力を artifact として保存し、workflow を赤くする（発見の記録は workflow の履歴 = CI 実体であり、別台帳は作らない）。

2. **検証文化の地図を `docs/verification.md`（英語正 + `.ja.md` ミラー）として新設する。** 内容は [[000086]] Decision 1 の列挙——CI ジョブ構成（PR-light / merge-heavy の分割方針）、fuzzing（本 ADR で CI 化）、polyglot conformance、release-parity、供給網検証（cosign / SBOM attestation / cargo-deny）、ADR graph 検証——を「何を・どこで・どの workflow が」の 3 列で実体へポイントする地図に徹する。カバレッジ数値や結果の転記はしない（drift 防止、[[000080]] / [[000086]] の規律）。fuzz の深さは「週次・短時間 smoke」と正直に書き、継続 fuzzing と誇張しない。

3. **README の governance 節相当から verification.md へリンクする。** [[000086]] の意図宣言・撤退プロトコルと同じ場所から、検証文化の地図に一跳びで届くようにする。

検討した代替案と却下理由:

- **fuzz を CI 化せず、地図に「ローカルハーネスのみ・CI 未組込」と正直に載せる**——却下。正直ではあるが、第一看板を審査する読者に対して untrusted 入力パーサの fuzz が「あるが回していない」と読ませるのは、数分の週次ジョブで解消できる欠落を放置する選択であり、割に合わない。
- **merge-heavy（main push）ジョブに fuzz を足す**——却下。merge ごとに数分が上乗せされ、wall-time 管理方針と衝突する。fuzz 対象の変更頻度に対して過剰でもある。
- **毎日 cron**——却下。ターゲットの変更頻度（PROXY v2 パーサはほぼ凍結）に対して通知ノイズと Actions 消費が過剰。週次で十分であり、パーサ変更時は `workflow_dispatch` で任意実行できる。
- **外部の継続 fuzzing サービスへの登録**——今回は見送り。外部サービス依存と運用導線の追加はまず週次 smoke の実績を積んでから判断する（再検討条件に置く）。
- **地図を独立カバレッジ台帳として持つ**——却下済み（[[000086]] が同趣旨の代替案を却下済み。本 ADR はそれを継承する）。

本決定は [[000086]] の「CI / release — 新規の仕組みは追加しない」という前提を fuzzing に限って amend し、Decision 1（可視化）の実装形を確定する。可視化の規律（CI green が実体・別台帳なし）は不変。P9（正直な計測・誇張しない主張）を fuzz の深さの表記にまで適用する。

## Consequences

### Pros
- 検証文化の地図から「実態と異なる行」が消え、fuzzing が反証可能な形（workflow 履歴）で第一看板の資料に載る。
- PR / merge の wall-time への影響ゼロで、単独開発の帯域を守ったまま [[000086]] の三条件 (a) が完成する。
- crash 発見時の入力が artifact として残り、修正 → corpus 追加の再発防止ループが CI 上で閉じる。

### Cons / Tradeoffs
- 週次・短時間 smoke は継続 fuzzing に比べ発見力が浅い（地図に深さを明記して受ける）。
- nightly toolchain 依存のジョブが増え、nightly 側の破損で週次が赤くなる運用ノイズがあり得る（fuzz 本体の失敗と区別できるようジョブを分けて受ける）。
- 可視化文書は CI 変更のたびに更新が要る（[[000086]] Cons の継承。地図に徹して軽くする）。
- **再検討条件**: (a) 週次 smoke で crash が継続的に出るなど発見力が不足と観測されたとき、実行時間の延長または外部継続 fuzzing サービスの導入を再検討する; (b) fuzz ターゲットが増え週次枠に収まらなくなったとき、頻度・分割を再設計する; (c) 地図が CI 実態と drift したとき、粒度を下げて実体参照へ寄せる（[[000086]] 再検討条件 (c) の継承）。
