# crates.io publish の却下を撤回し、ライブラリ3クレート + バイナリの公開準備を進める

## Status
accepted（2026-07-13 のセッションで maintainer が判断・実装を承認。`cargo publish --dry-run` で3クレート全ての検証ビルドが成功することを確認済み。同日中に maintainer の明示指示で `publish = true` へ切り替え、`plecto-host` / `plecto-control` / `plecto-server` v0.3.5 の初回 `cargo publish` を maintainer 自身が実行——crates.io の API で3クレートとも `newest_version: 0.3.5` を確認済み。初回公開は完了）

## Date
2026-07-13

## Affected Components
- Rust workspace — crates.io 向けメタデータの整備、repository/homepage URL 修正、内部 path 依存への version 付与
- crates/host — WIT 契約の vendoring（`wit/`, `wit-streaming/`）、`FILTER_WIT` 公開定数の追加
- crates/control — `FILTER_WIT` の re-export
- crates/server — filter-template guest の vendoring、`new_filter.rs` の embed 元差し替え
- scripts/check_wit_vendoring.py・.github/workflows/ci.yml — vendored コピーのドリフト検査を CI 化

## Context

[[000047]] は 2026-07-03 時点で crates.io への publish を明示的に却下し、再検討条件を「M3 GA でライブラリ API を安定化するとき個別判断する」としていた（同 ADR §Decision の却下代替案一覧）。この条件を正式に満たしたと確認できる、M3 GA 完了を宣言した別 ADR は存在しない（`grep -rl "M3 GA" docs/ADR/*.md` は [[000047]] 自身のみヒット）。

2026-07-13、maintainer（Kaikei）が `docs/plans/crates-io-publishing-guide-plecto.md` に沿って crates.io publish 準備を明示的に指示した。作業中に本 ADR 相当の論点が浮上したため、Claude が確認したところ、maintainer は「M3 GA 相当の判断として今進めてよい」と回答した——つまり本決定は、ライブラリ API が実際に安定版としてフリーズされたという技術的事実の確認ではなく、**maintainer の明示的な早期判断**である。この区別を隠さず書く（P9: 誇張しない主張）。

早期判断の実質的な根拠は2つ:
1. **crates.io の名前空間は先着順・恒久占有**（`plecto` / `plecto-host` / `plecto-control` / `plecto-server` の4名とも 2026-07-13 時点で未登録）。API 安定化を待つ間に第三者が名前を取る恒久的リスクがあり、待つこと自体にコストがある。
2. **crate 構成（host / control / server の3分割 + `plecto` バイナリ）は M2 slice 群の着地以降、実質的に安定している**——直近の変更は WIT 契約バージョン追加（0.1→0.3、後方互換アダプタ付き）であり、crate 境界自体の破壊的再編ではない。

一方で、crates.io packaging には Plecto 固有の構造的な障害があった: `cargo package` は**パッケージディレクトリ内のファイルしか .crate に含めない**。実地検証（`cargo publish --dry-run` を試行）で判明した具体的な破損:
- `crates/host` の `bindgen!({ path: "../../wit" })` 系3箇所（`lib.rs` / `contract.rs` ×2）と `streaming.rs` の `../../wit-streaming` が、パッケージ化後に読めなくなる（`failed to read path for WIT` で 13 件のコンパイルエラー）。
- `crates/server/src/new_filter.rs` の `include_str!("../../../wit/world.wit")` と、guest テンプレート2ファイル（`examples/filters/filter-template/{Cargo.toml, src/lib.rs}`）の embed も同型の破損（`new_filter` は ADR [[000072]] の self-vendoring がリポジトリ内の相対パスに依存していたため）。
- 内部 path 依存（`plecto-control` → `plecto-host`、`plecto-server` → `plecto-control`）に `version` が無いと `cargo package` がマニフェスト検証の時点で拒否する。

## Decision

1. **[[000047]] の crates.io 却下を撤回し、`plecto-host` / `plecto-control` / `plecto-server` の3ライブラリクレートを公開対象として準備を進める。** ただし本 ADR は「ライブラリ API の semver 安定性を約束する」宣言ではない——0.x の間は [[000047]] が定めた pre-1.0 バージョニング方針（minor bump が破壊的変更を含みうる）がそのまま適用される。「crates.io に載せる」ことと「API を凍結する」ことを混同しない。
2. **パッケージ名は現状維持**（`plecto-host` / `plecto-control` / `plecto-server`）。`docs/plans/crates-io-publishing-guide-plecto.md` §0 が挙げる「`plecto-server` → `plecto` への package 改名」は本 ADR では判断しない——独立した後続判断として残す（同計画のチェックリスト item 2 のまま）。
3. **WIT 契約は `plecto/wit/` を正文として維持し（`CLAUDE.md` の既存規約）、crates.io 公開に必要な範囲だけ各クレート内へ vendoring する**: `crates/host/wit/{world.wit, v0.1.0/, v0.2.0/}` と `crates/host/wit-streaming/`。`bindgen!` のパスをクレートルート相対（`"wit"` 等）に付け替え、正文からのドリフトは `scripts/check_wit_vendoring.py`（CI の `fmt` job に追加）で検査する——ファイルコピーそのものではなく、コピーの一致検証を構造的な保証にする。
4. **`plecto-host` に `pub const FILTER_WIT: &str = include_str!("wit/world.wit")` を追加し、`plecto-control` 経由で re-export する。** `plecto-server` は既存の設計判断（「`plecto-control` の re-export 経由で `plecto-host` の型を使い、直接の production 依存は持たない」——`crates/control/src/lib.rs` の既存コメントに明文化済み）を維持したまま、`new_filter.rs` の WIT テキスト embed を `plecto_control::FILTER_WIT` の参照に置き換える。これにより [[000072]] が確立した「host の bindgen が読む WIT と `new-filter` がコピーする WIT は同一ファイルである」という不変則を、2重 vendoring による drift check ではなく単一コピーの再利用として維持する。
5. **guest テンプレート（`examples/filters/filter-template/{Cargo.toml, src/lib.rs}`）を `crates/server/templates/filter-template/` へ vendoring する。** ネストした `Cargo.toml` はそのままだと `cargo package` がネスト manifest とみなして無条件除外する（`include` パターンを明示しても除外される——実地検証で確認）ため、`Cargo.toml.template` にリネームして回避する。正文との一致は (3) と同じ `check_wit_vendoring.py` で検査する。
6. **内部 path 依存を `[workspace.dependencies]` に集約し、`path` と `version` の両方を持たせる。** `crates/control` / `crates/server` は `{ workspace = true }` で参照する。`version` は `workspace.package.version` とリリース時に手動でロックステップ更新する（自動連動の仕組みは無い——cargo に workspace 間でその連動を取る機構が無いため。コメントで明記）。
7. **`publish` はワークスペース全体でまだ `false` のまま。** 本 ADR が承認するのは「公開できる状態に構造を直す」ことまで。`cargo publish --dry-run -p plecto-host -p plecto-control -p plecto-server`（一時的に `publish = true` へ切り替えてテストし、直後に `false` へ戻した）で3クレートともパッケージ化・検証ビルド・アップロード直前まで成功することを確認済みだが、実際の `publish = true` への恒久切り替えと初回 `cargo publish` の実行は、maintainer の別途の明示指示を要する独立したステップとする。

検討した代替案と却下理由:

- **[[000047]] の却下を維持し、M3 GA の技術的完了を待つ**——却下。crates.io の名前は待つ間にも第三者に取られうる恒久的リスクであり、`plecto` 系4名の予約価値は API 安定化の進捗と独立している。
- **バイナリのみ公開し、ライブラリ3クレートは skip する**——却下。`plecto`（バイナリ）は `plecto-control` → `plecto-host` の path 依存を持つため、cargo は依存クレートを先に（または同時に）公開することを要求する。部分公開はできない。
- **`plecto/wit/` を crates/host 側へ移し、そちらを正文にする**——却下。`CLAUDE.md` が「契約の正文は `wit/`」と明記しており、[[000064]] の wkg/OCI 配布・[[000072]] の self-vendoring 双方がこの前提に依存している。正文の位置を変えるのは本 ADR の scope 外の大きな変更であり、vendoring + drift check の方が既存規約と整合する。
- **`new_filter.rs` に host と別の WIT コピーを直接 vendoring する（2重 vendoring + 相互 diff）**——却下。[[000072]] の「同一ファイル」不変則をわざわざ「2つのコピーが一致すること」という弱い保証に後退させる。`plecto-control` 経由の re-export で単一ソースのまま解決できるので、そちらを採用した。
- **guest テンプレートの nested `Cargo.toml` 問題を `include` の工夫だけで解決する**——実地検証で `include` 指定があってもネスト manifest は除外されると判明したため却下。ファイル名変更（`.template` サフィックス）が唯一確認できた回避策。

本決定は [[000047]] の却下判断を撤回し、[[000072]] の self-vendoring 不変則をクレート境界を越えて延長する。`CLAUDE.md` の「セルフホスト性 × 運用の単純さ」——特に「契約の正文は `wit/`」の規約は変更せず、その正文をどう複数の配布先（wkg/OCI と crates.io）に矛盾なく届けるかという実装判断に留める。

## Consequences

### Pros
- crates.io の名前空間（`plecto` / `plecto-host` / `plecto-control` / `plecto-server`）を恒久的な squatting リスクから守れる状態に近づいた（実際の予約は初回 publish 時点）。
- WIT vendoring 構造は crates.io 公開のタイミングと無関係に、それ自体が正しい修正——`plecto/wit/` の正文性は変えず、CI drift check が唯一の追加コストで済む。
- `FILTER_WIT` の re-export 経由により、[[000072]] の「同一ファイル」不変則が3クレート構成でも壊れずに保たれた。
- `cargo publish --dry-run` の3クレート一括成功により、構造面の問題は初回 publish 前に潰し切れたことを確認済み。

### Cons / Tradeoffs
- `plecto-host` / `plecto-control` の内部型が crates.io 経由で外部消費者に触れうる状態になる——[[000047]] が懸念した「API 安定性の約束」はまだ技術的に果たされていない。0.x の pre-1.0 バージョニング方針（[[000047]]）がその緩衝材だが、それ以上の安定性保証はしていないことを README/docs.rs で誤読させない注意が要る。
- vendored コピー（WIT 4箇所 + guest テンプレート2ファイル）は正文と機械的に一致しているだけで、CI の `check_wit_vendoring.py` を通さないローカル手直しは drift を生む——CI green への依存が増える（既存の ADR 運用と同じ性質のコスト）。
- 内部 path 依存の `version` はリリース時に手動でロックステップ更新が必要（自動連動なし）。更新を忘れると `plecto-host` だけ新版が出て `plecto-control`/`plecto-server` が古い依存を指したまま、という半端な状態になりうる——次回リリース手順に明記する必要がある（本 ADR ではフラグのみ立てる）。
- `plecto-server` の `include` フィールドが whitelist 方式に切り替わったため、今後 `crates/server/` に新しいトップレベルディレクトリを足す変更は `include` の追随を忘れると `cargo package` から静かに漏れる。
- crates.io の publish は恒久——`plecto-host`/`plecto-control`/`plecto-server` v0.3.5 は既にレジストリに記録され、削除はできない（`cargo yank` は新規依存の解決対象から外すだけ）。
- **再検討条件**: (a) `plecto-host`/`plecto-control` の破壊的変更を外部消費者からの semver 違反報告として受け取るようになったら、公開の一時停止（`cargo yank`）や API 安定化の前倒しを検討する；(b) `plecto-server` → `plecto` へのパッケージ改名が決まったら、本 ADR の vendoring 構造（特に `documentation` フィールドの docs.rs URL）を追随させる；(c) 済——`publish = true` への切り替えと v0.3.5 の初回 `cargo publish` は maintainer 自身が 2026-07-13 に実行し、crates.io API で反映を確認済み。次の課題は Trusted Publishing への移行（`docs/plans/crates-io-publishing-guide-plecto.md` §6）とリリース手順への version-lockstep 明記。
