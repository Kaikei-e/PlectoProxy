# plecto バイナリを専用 bin クレートへ分離し、crates.io の素の plecto 名を実体で確保する

## Status
accepted（2026-07-14。実装済み・全テストグリーン（52 テストバイナリ / clippy all-features / fmt / vendoring drift check）。`cargo publish --dry-run` は 4 クレート（plecto-host / plecto-control / plecto-server / plecto）とも 0.3.6 で verify ビルドまで成功。実 publish と v0.3.6 タグはこの ADR の記録時点では未実施）

## Date
2026-07-14

## Affected Components
- crates/plecto（新設） — `plecto` バイナリと operator CLI 一式の新しい持ち場。`cargo install plecto` の一等導線
- crates/server — 純ライブラリ化（`[[bin]]` 撤去・CLI 専用依存 5 つを削減）。公開 API は不変
- CI / release / Dockerfile — バイナリ成果物のビルド対象を `-p plecto` へ切り替え
- scripts/check_wit_vendoring.py — テンプレート vendored 先の追随
- CHANGELOG — 0.3.6 として Added（plecto クレート）/ Changed（plecto-server 純ライブラリ化 + migration note）を記録

## Context

[[000090]] で `plecto-host` / `plecto-control` / `plecto-server` v0.3.5 を crates.io に初回公開した（2026-07-13）。その直後の外部フィードバックが、残った最優先の穴を指摘した: **素の `plecto` 名が crates.io 上で未確保のまま**である。

- squatting リスクの議論は実は `plecto-server` ではなく裸の名前にこそ当てはまる（同名の第三者企業が存在し、名前空間は先着順・恒久占有）。
- `cargo install plecto` は source-install の DX 名でもある。v0.3.5 時点では `plecto-server` が `[[bin]] name = "plecto"` を抱えており、`cargo install plecto-server` でバイナリ自体は入るが、名前の確保の論点は残っていた。
- [[000090]] は再検討条件 (b) として「`plecto-server` → `plecto` へのパッケージ改名が決まったら追随」を掲げていた。本 ADR はその分岐を「改名」ではない第三の形で解決する。

構造面の前提: CLI モジュール群（cli / dev_cmd / dev_key / new_filter）は `main.rs` 配下の private モジュールで、依存は `plecto_control` の公開 API・`plecto_server::serve_with_shutdown`・`libc`（dev の self-SIGHUP）に閉じていた——つまりライブラリの公開面を広げずにそのまま持ち出せる形をしていた。

## Decision

1. **新クレート `plecto`（`crates/plecto`）を作り、`[[bin]] plecto` と CLI モジュール一式・guest テンプレート vendoring・バイナリ駆動 E2E（tests/cli.rs / binary_signals.rs / listen.rs）を `crates/server` から移す。** CLI はこのクレートの private モジュールのまま——`plecto-server` の公開 API に昇格させない（bp-rust の最小 pub surface 規律。ライブラリの公開面が CLI 都合で汚れない）。エコシステムの標準形（runtime ライブラリ + 薄い cli クレート）に一致する。
2. **`plecto-server` は純ライブラリになる。** `[[bin]]` と CLI 専用依存（serde_json / tracing-subscriber / notify-debouncer-mini / toml_edit / wit-component）を外す。ライブラリ API・examples（`cargo run -p plecto-server --example <name>`）・bench-harness feature は不変。
3. **capability feature（outbound-http / outbound-tcp / fat-guest / capabilities）は `plecto` クレートにも定義し、`plecto-server` への転送のみとする**（[[000079]] の profile 連鎖を一段延長。`--version` の profile 自己申告 `cfg!` はバイナリ側クレートの feature を見る必要があるため、転送定義は必須）。
4. **バイナリ成果物のビルド・テスト配線を `-p plecto` へ切り替える**: release.yml（cargo auditable build）、Dockerfile、ci.yml（capabilities check / cli test / release parity）。examples・bench の `-p plecto-server` 参照は正しいまま据え置く。
5. **v0.3.6 として 4 クレート（host / control / server / plecto）を揃えて publish し、タグ v0.3.6 を切る**（実施は maintainer 判断）。`plecto` の初回バージョンは 0.3.6 で開始し、workspace 共有バージョンとの一貫性を優先する。

検討した代替案と却下理由:

- **`plecto-server` を `plecto` に改名する**（[[000090]] 再検討条件 (b) の想定形）——却下。crates.io の `plecto-server` 0.3.5 は恒久記録済みで、改名は「取り残される旧名クレート + 新名への実質再出発」を生む。bin クレート追加なら既存 3 クレートの名前と履歴が全て生きる。
- **CLI を `plecto_server` の公開 API に昇格させ、`plecto` は 1 行の thin wrapper にする**——却下。ライブラリの公開面に CLI 都合の型・関数が生え、最小 pub surface 規律に反する。CLI モジュールは自己完結しており、bin クレートへ移す方が結合が素直。
- **`plecto` を「README だけの空クレート」で先に取る**——却下。crates.io ポリシー上 squatting は歓迎されず、[[000090]] の判断メモでも「実体を持たせる形が無難」としていた。実体（バイナリ本体）を移せる目処が立っている以上、空予約を挟む理由がない。
- **`plecto-server` 0.3.6 にも `[[bin]]` を残し、両クレートから同名バイナリを出す**——却下。`cargo install` が衝突し、どちらが正か恒久に曖昧になる。過渡期の重複は「crates.io 上の plecto-server 0.3.5（bin 付き・yank しない）」として一世代だけ残るのを許容し、0.3.6 以降は一意にする。

バージョニングの注記（P9: 隠さない）: `plecto-server` から `[[bin]]` を外すのは、`cargo install plecto-server` 利用者にとっては patch bump（0.3.5→0.3.6）の「常に安全」原則の例外になりうる。公開から本変更まで 1 日・実利用者ゼロと見込める時点で、CHANGELOG に migration note（`cargo install plecto` へ）を明記した上で patch に載せる判断をした。ライブラリ API に破壊はない。

## Consequences

### Pros
- crates.io の最重要文字列 `plecto` を、空予約ではなく実体（バイナリ + operator CLI）で確保できる。`cargo install plecto` がそのまま成立する。
- `plecto-server` の依存が 5 つ減り（CLI 専用系）、ライブラリ消費者の supply-chain 表面が縮む。`.crate` も bin 分だけ軽くなる（65→55 ファイル）。
- bin / lib の責務分離がエコシステム標準形に揃い、[[000090]] の再検討条件 (b)（改名問題）が消滅する。
- `plecto` クレートの `.crate` は 16 ファイル / 53KiB と薄く、docs.rs / lib.rs 上の顔として description に「何であるか」を一文で示せる。

### Cons / Tradeoffs
- crates.io 上の `plecto-server` 0.3.5 は `plecto` バイナリを含んだまま恒久に残る（yank はしない——依存としては無害）。`cargo install plecto-server@0.3.5` と `cargo install plecto` が同名バイナリを入れる過渡期の重複は一世代限り。
- クレートが 4 つになり、リリース時の version-lockstep（[[000090]] Cons で指摘済み）の対象が 1 つ増えた。
- `crates/plecto` の `include` も whitelist 方式のため、新トップレベルディレクトリ追加時の追随忘れリスクを共有する（[[000090]] Cons の継承）。
- **再検討条件**: (a) `plecto` バイナリの CLI が肥大して bin クレートの responsibilities が曖昧になったら、サブコマンド単位の分割を検討する；(b) 0.3.6 の publish / タグ実施後、README の install 導線（`cargo install plecto` + crates.io バッジ）と Trusted Publishing 移行（[[000090]] 再検討条件の残件）を別途進める。
