# RLIMIT_NOFILE を soft→hard へ起動時に引き上げ、per-source-IP 同時接続数上限で inbound admission control を完成させる

## Status
accepted（2026-07-15 に設計を grill（[[000027]] との整合・per-IP 上限値・IPv6 丸め方針・ADR 要否を確認）→ 同日実装着地。`cargo fmt --all -- --check` / `cargo clippy --all-targets --all-features -- -D warnings` / `cargo test --all` 全て green）

## Date
2026-07-15

## Affected Components
- fast path（`plecto-server`）— `raise_nofile_limit()`（`lib.rs`）と `conn_limit::PerIpConnLimit`（`conn_limit.rs`、新設）を追加。TCP accept loop（`listener.rs`）と QUIC accept loop（`h3/endpoint.rs`）双方に per-IP admission チェックを組み込み

## Context

2026-07 の本番運用 hardening レビュー（内部の精査文書、外部非公開）が、[[000027]] が固めた inbound リソース上限に確定事実として2つの残存ギャップを指摘した:

1. **`RLIMIT_NOFILE` を起動時に引き上げるコードが無い**: 多くのディストリの既定 soft limit（1024）では、[[000027]] が導入した `MAX_CONNECTIONS`（10,000）のはるか手前で `accept()` が EMFILE を返す。accept ループは「transient error, continue」として扱う設計（同一障害クラスの `fd exhaustion` をクラッシュさせず継続する意図的な選択）だが、これは EMFILE の場合には「新規接続の受付をサイレントに停止する」という別の障害モードに転化する。
2. **接続元 IP 単位の同時接続数上限が無い**: [[000027]] が導入したのはグローバル `Semaphore` 1 本のみで、総量は縛るがその分布は縛らない。単一送信元が接続スロットを占有し、他クライアントを事実上締め出せる（インターネット対向 L7 プロキシとして最も指摘されやすい欠落）。

一次情報での裏取り（`getrlimit(2)` / `setrlimit(2)` man page、nginx `worker_rlimit_nofile` ドキュメント、Envoy `connection_limit` filter ドキュメント）:

- **soft limit を hard limit まで引き上げる操作は unprivileged process に無条件に許可される**（POSIX、`CAP_SYS_RESOURCE` 不要）。hard limit 自体を引き上げる操作にのみ `CAP_SYS_RESOURCE` が要る。nginx はこれを worker プロセスで hard/soft 双方に同一値をセットできるが、これは master が root 特権を保持したまま worker を fork してから setuid する特権分離アーキテクチャに支えられている。
- Envoy の `connection_limit` network filter は **filter chain 単位**の上限であり、per-source-IP の同時接続数上限をネイティブには持たない（per-IP はレート制限記述子経由が一般的）——Plecto の [[000033]] ネイティブ rate limiter と今回の per-IP 接続数上限は、この点で Envoy より一段具体的な admission control を fast path に持つ。
- IPv6 の per-source 制限は、単一ホストが /64 全体を保有し得るため /64 などのプレフィックス単位に丸めるのが公知の実務（[[000033]] が導入した `plecto_control::ratelimit` の `client-ip` バケットは既にこの丸めを採用済み）。

Tenet/Fork: 「安全 × ポータビリティ × セルフホスト性 × 運用の単純さ」の優先順位（`CLAUDE.md` コア原則）、および P4 deny-by-default / least-privilege（`CAP_SYS_RESOURCE` を要求しないという自己制限の判断根拠）。[[000027]] が具体化した「inbound リソース上限」を、OS レベルの fd 天井と per-source 分布の両輪で完成させる、直系の続きとして書く。

## Decision

### 1. RLIMIT_NOFILE: soft→hard のみを引き上げ、hard 自体は上げない

起動時（実 serve パスのみ、後述）に `getrlimit`/`setrlimit`（`libc` 直呼び、`cap_malloc_arenas` と同型）で soft limit を hard limit まで引き上げる。引き上げ後も soft limit が `MAX_CONNECTIONS * 4`（1 接続あたり client + upstream の 2 fd、admin/DNS/health/TLS resumption 等のヘッドルームで倍）を下回るなら `tracing::warn!` で運用者に外部（systemd `LimitNOFILE=` / `docker run --ulimit nofile=` / コンテナランタイムの ulimit 設定）での引き上げを促す。失敗しても起動は継続する（fail soft、ログのみ — RLIMIT_NOFILE が想定より低いことは起動を止めるべき異常ではなく、高負荷時に初めて顕在化する運用上の劣化モードだから）。

検討した代替案:
- **hard limit 自体も操作者が指定した値まで引き上げる（nginx 方式）** — 却下。Plecto は master-worker 特権分離アーキテクチャを持たない単一バイナリで、`CAP_SYS_RESOURCE` を要求すると自己ホスト型プロキシとしての最小権限原則・運用の単純さに反する。fd 天井そのものが不足する環境では、コンテナ/systemd/OS の責務として外部で解決してもらう。
- **`rlimit` crate 等の新規依存を追加** — 却下。既存の `libc = "0.2"`（`cap_malloc_arenas` の `mallopt` で既に直呼び済み）で必要十分。新規サプライチェーン面を増やさない（`deny.toml` / cargo-deny のコストに見合わない）。

呼び出し位置は `main.rs` の**実際に serve するパスのみ**（`TcpListener::bind` 直前）。`schema` / `validate` / `conformance` / `new-filter` / `dev` / `--version` の各ユーティリティサブコマンドでは呼ばない。実装時、`tracing_subscriber` 初期化直後という一見自然な位置に置いたところ、`plecto schema` が stdout に出す素の JSON へログ行が混入し `cargo test --all` の `schema_emits_a_draft07_json_schema_describing_the_manifest` を壊した——気づいて呼び出し位置を実 serve パスへ移し直した。この経緯を残すのは、同種のプロセス起動時フックを今後追加する際に同じ罠を踏まないため。

### 2. per-source-IP 同時接続数上限

新設 `conn_limit::PerIpConnLimit`: [[000033]] の `client-ip` バケットと同型の「固定 65536 スロットの hash テーブル（CWE-770 対策、メモリ O(1)）」を、`AtomicU32` カウンタ + RAII guard（`PerIpConnGuard`、drop 時に `fetch_sub`）で実装。デフォルト上限は 256（`MAX_CONNECTIONS` の約 2.6%）——攻撃者は少なくとも約 40 の異なる送信元（IPv6 はハッシュ衝突する /64 群）が必要になる一方、正当な単一クライアント（大規模 NAT/corporate gateway を含む）の実測より十分大きい。IPv6 は [[000033]] と同じく /64 に丸める——単一ホストによる /64 内ローテーションでの上限回避を防ぐためで、同一 /64 の複数正当ユーザーがスロットを分け合う副作用は許容する（丸めない場合の回避リスクの方が実害が大きいと判断）。

`TCP` 側は PROXY v2 で解決された peer で判定する。生の `accept()` peer で判定すると、PROXY v2 trust 構成時（L4 LB 配下）は全接続が LB 自身の IP に収束し、実質的に LB 配下のトラフィック全体を 256 接続に制限してしまう重大なリグレッションになるため、判定は PROXY v2 解決後・TLS ハンドシェイク前に置く。`QUIC/h3` 側は PROXY v2 の対象外（[[000057]] が明記済み）なので `Incoming::remote_address()` をそのまま使う。

検討した代替案:
- **`HashMap<IpAddr, AtomicU32>` の動的マップ** — 却下。CWE-770（多数/spoofed の送信元による無制限成長）。[[000033]] が既に固定ハッシュテーブルで解決済みの問題を再導入することになる。
- **`plecto_control::ratelimit` の非公開ハッシュヘルパーを `pub` にして共有** — 却下。`plecto-control` は crates.io 公開済み（[[000090]] / [[000091]]）で新規 `pub` API は semver コミットになる。関心も異なる（`ratelimit.rs` はマニフェスト駆動のルート単位ポリシー、本 ADR はルーティング前のトランスポート層グローバル admission control）。~20 行のハッシュヘルパーを複製する方が、公開クレートへの恒久的な新規契約より安い。
- **マニフェストで上限値・テーブルサイズを設定可能にする** — 却下（今回のスコープ外）。`MAX_CONNECTIONS` / `MAX_CONCURRENT_STREAMS` と同格の「ハードコード定数 v1、follow-up」慣習に揃えた。

## Consequences

### Pros
- 既定の soft limit（1024 等）による EMFILE 起因のサイレントな受付停止を、ほぼ全ての環境で解消する（hard limit は通常 soft よりはるかに大きい）。
- 単一送信元による接続プール専有（DoS）を、TCP/QUIC 両方の accept 経路で構造的に防止する。
- 新規依存クレート・新規 crates.io 公開 API のいずれも追加しない。
- [[000033]] と脅威モデル（CWE-770 の固定ハッシュテーブル設計、IPv6 /64 丸め）が一貫し、運用者が2つの独立した IPv6 ポリシーを覚える必要がない。

### Cons / Tradeoffs
- hard limit 自体が低いままの環境（一部の最小構成コンテナ等）では、Plecto は自動で解決せず警告ログのみを出す——意図的な選択だが、運用者が外部で対応しない限り根本原因は残る。
- per-IP 256 という値は既存定数とのバランスに基づく判断であり、実測（本番トラフィックでの分布計測）に基づくものではない。
- IPv6 /64 丸めの副作用（同一 /64 の複数正当ユーザーが同一スロットを分け合う）は残存する既知のトレードオフ。
- 上限値・テーブルサイズはマニフェスト化されておらず、環境ごとのチューニング手段がない（[[000027]] 以来の既存の「定数 v1」負債と同格、解消は別途）。
- 再検討条件: 本番運用で per-IP 256 が正当なクライアント（大規模 NAT/corporate gateway）を誤って締め出す事例が観測されたら値を見直す。あるいは per-IP/per-テーブルサイズのマニフェスト化が要望として顕在化したら、[[000027]] 以来の「定数 v1」路線全体の見直しと合わせて検討する。
