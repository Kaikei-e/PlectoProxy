# redb backend の書き込みを flat-combining group commit で 1 トランザクションに束ねる

## Status
accepted

## Date
2026-07-16

## Affected Components
- host runtime (`plecto-host`) — `backend.rs::RedbBackend` の書き込み経路を per-op write txn から combiner バッチへ変更。`KvBackend` seam（[[000011]]）・fail-closed 挙動・`set`/`delete` の即時永続保証は不変
- bench — `benches/kv_backend.rs` 新設（高並列 contended-write の criterion ベンチ。従来ベンチは低負荷点のみだった）

## Context

ホスト状態バックエンド（[[000004]] の redb）は、`host-kv.set` / `host-counter` / `host-ratelimit`
の各書き込みごとに `begin_write()` → 単独 txn → `commit()` を実行していた。redb は厳密な
single-writer（`begin_write` は先行ライターが居る間ブロック、公式 design.md）なので、
**全フィルタ・全ルートの書き込みがプロセス全体で 1 本のライターロックに直列化**され、
op 数 N に対して N 回の begin/commit コストを直列に支払う。2026-07-15 の戦略精査で
「高並列時に trusted インスタンスプールより先に天井になる」実害級所見（§3.2-3）として確定した。

counter/ratelimit の fsync は既に `Durability::None`（1024 op ごと Immediate）で償却済み
（[[000005]] のエフェメラル判定）。残っていたコストは **op ごとの txn 開始・コミット処理
そのものの直列化**である。

同条件 A/B 実測（criterion、1 キーへの increment、この開発機）:

| ケース | per-op txn（旧） | group commit（新） | 差 |
|---|---|---|---|
| 8-thread 競合 | ~48.9 µs/op | ~19.6 µs/op | **-60%** |
| 1-thread 非競合 | ~51.4 µs/op | ~50.0 µs/op | 同等 |

関係する既定決定: [[000004]]（redb 採用と単一 writer txn での原子性）を修正（amends）し、
[[000011]]（`KvBackend` seam — バックエンド実装の変更は callers に波及しない）を実証する形で
seam の内側に閉じた。

## Decision

`RedbBackend` の書き込みに、古典的 **group commit** の償却思想（DeWitt et al., SIGMOD'84 —
複数トランザクションで durable I/O を共有する）を適用し、その**形**は **flat combining**
（Hendler, Incze, Shavit, Tzafrir, SPAA 2010）の caller 選出型を採る。本実装は複数の論理 op を
1 本の redb write txn に束ねる（DeWitt の pre-commit + 共有 log fsync そのものではない）:

1. caller は op を MPSC キューに投入し、combiner 選出ロック（`parking_lot::Mutex`、非 poisoning）を
   `try_lock` で競る。
2. 勝者（combiner）がキューを drain し、最大 `MAX_BATCH_OPS = 64` 件を 1 つの write txn 内で順に
   read-modify-write して commit、各 caller へ reply channel で結果を返す。drain が空になるまで
   ラウンドを繰り返してからロックを解放する（上限 `MAX_COMBINE_ROUNDS = 16`）。ラウンド数と
   バッチ件数の積で徴用の仕事量を有界化する。storage エラー時はそのラウンドで combining を打ち切る。
3. バッチ形成は **self-clocking（タイマーなし）**: 前の commit 実行中に到着した op が次のバッチになる。
   低負荷ではバッチ = 1（レイテンシ現状同等）、高負荷では自動的に大バッチ化する。
4. **waiter は combiner ロックではなく自分の reply channel で待つ**。reply 送信で全 waiter が並列に
   起床する。ロック待ちにすると解放後の直列ハンドオフ（1 人ずつ起きて lock → 受領 → 解放）で
   バッチ化の利得が消えることを A/B で実証済み（本 ADR の実装過程で観測: ロック待ち版は 8-thread で
   改善が有意差なしまで縮退）。
5. durability はバッチ内最強に引き上げ: `set`/`delete` を含むバッチは `Immediate`（従来の保証を維持、
   同乗したホット op はタダで durable 化）、ホット op のみのバッチは従来どおり `None` + 1024 **op**
   ごと `Immediate`（カウントは commit 数ではなく op 数のままなので、バッチ化でファイル成長・
   クラッシュ損失窓の上限は伸びない）。flush カウンタ（`non_durable_run` / `forced_flushes`）の更新は
   **成功した `commit` の後だけ**行う — 失敗した Immediate がカウンタをクリアして窓を伸ばさない
   （main の `set`/`delete` 後リセットと同じ順序）。
6. 読み取り（`get` / `increment(_, 0)`）は従来どおり caller スレッドの MVCC read txn 直行
   （combiner を経由しない）。

検討した代替案と却下理由:

- **専用ライタースレッド常駐型 combiner**: 実装は単純で 8-thread 競合 -73% を実測したが、
  非競合時に op ごと caller→writer→caller の 2 回のスレッド起床（park/unpark）が乗り、
  1-thread per-op レイテンシが実測で明確に悪化（+15〜20 µs/op）。低負荷に税を課すため却下。
  caller 選出型は非競合時にスレッド切替ゼロ（旧実装と同経路 + チャネル/ロック操作のみ）。
- **redb Database のシャーディング（キーのハッシュで N DB に分割）**: ライターは N 本になるが
  op ごとの begin/commit コストは残り、状態ファイルが N 個になって運用の単純さ（設計原則）と緊張。却下。
- **write-behind キャッシュ（メモリ正 + 周期フラッシュ）**: ホット経路は最速になるが、`set` の
  即時永続（`Durability::Immediate`）の意味論が壊れ、クラッシュ損失窓が全 op に広がる。却下。
- **計測のみで様子見**: 所見は「可能性」止まりだったが、同時に高並列ベンチの不在も指摘されており、
  ベンチ新設と修正を一体で行った（ベンチだけ足して直さない、は二度手間）。

[[000004]] の再検討条件（「許容できないほどホットなカウンタが出たらホスト側ネイティブ lock-free
カウンタで WIT 越えを回避」）は本決定と独立に生きている——本 ADR は WIT 越え後のバックエンド側の
直列化を解消したもので、WIT 越えコスト自体には触れていない。

## Consequences

### Pros
- 8-thread 競合書き込みで per-op コスト -60%（A/B 実測）。バッチサイズは負荷に自動追従し、
  チューニングパラメータを増やさない（運用の単純さ）。
- 非競合時のレイテンシは旧実装と同等（combiner がインライン実行、スレッド切替なし）。
- `KvBackend` trait・`set`/`delete` の即時永続・fail-closed 挙動・既存テストのセマンティクスは全て不変。
  [[000011]] の seam が意図どおり機能した実例。
- 常駐スレッドなし。combiner が panic してもロックは poison せず（parking_lot）、drain 済み job の
  reply channel が切断されるので waiter は fail-closed に解決する（データプレーン panic 禁止の
  Tenet と整合）。
- 高並列 contended-write ベンチ（`kv_backend.rs`）が CI の bench workflow に載り、戦略精査の
  「ベンチは低負荷点のみ」も解消。

### Cons / Tradeoffs
- バッチ内の 1 op が storage エラーを起こすと txn ごと全 op が fail-closed になる（従来は単独失敗）。
  storage エラーは disk full 等の壊滅的状況であり、fail-closed の既定姿勢と整合すると判断。
- combiner に選出された caller（= tokio worker 上のリクエスト）は最大
  `MAX_COMBINE_ROUNDS × MAX_BATCH_OPS` 件分の commit 時間を他人のために消費する（徴用）。
  ラウンド上限・バッチ件数上限・タイムアウト再選出（100 µs）で有界化した。
- 選出レースの境界（combiner の最終 drain と解放の間に enqueue）では最大 100 µs の再選出待ちが乗る。
  稀（実測ベンチでは観測されない水準）。
- 再検討条件: (a) wasmtime の host call async 化（[[000021]] の後続）で `KvBackend` が async 化する
  とき、combiner 待ちを `.await` 化するか blocking pool に置くかを再設計する。(b) 実ワークロードで
  redb 書き込みが再び天井になったら、[[000004]] の再検討条件（ホスト・ネイティブ lock-free カウンタ）
  を先に検討する。
