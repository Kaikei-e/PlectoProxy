# plecto package と validate --resolve で CI パッケージングパイプラインを CLI の一等経路にする

## Status
accepted（2026-07-20 に grill セッションで設計を確定 → 同日 TDD（RED→GREEN）で実装着地。
`cargo test --all` 54 スイート / `cargo clippy --all-targets --all-features -- -D warnings` /
`cargo fmt --all -- --check` すべて green）

## Date
2026-07-20

## Affected Components
- crates/plecto — `package` / `validate --resolve` サブコマンド新設 + E2E（tests/package.rs 5 本）
- crates/host — `PemSigner` 切り出し・`TrustPolicy::verify_artifact` 切り出し（`Host::load` は委譲）

## Context

ドッグフーディング Phase 2 のフィールドレポート（docs/servey/draft_0004、§3.1 / §3.5）が、
配布系の構造的ギャップを特定した。ローダが要求する「署名済みローカル OCI image-layout」を作る
公式経路が (a) `plecto dev`（watch / dev 鍵 / SIGHUP に結合したインナーループ、ADR 000065）と
(b) `plecto_control::oci::write_layout`（`doc(hidden)` の Rust API、examples 専用）しか無く、
CI から「この component をこの鍵で署名して layout にして digest をくれ」だけを行う手段が
存在しなかった。その結果、利用者側 CI が oci.rs を読みながら layout 生成を約 60 行のシェルで
再実装しており、これは digest-pin 哲学が防ごうとしているドリフトそのものである。

前提の食い違いも確認した: `write_layout` の doc comment は「production artifacts come from
`wkg` (out-of-band)」と述べるが、ローダ（`crates/control/src/artifact.rs`）は現状ローカル
image-layout 専用で、remote fetcher は将来形にとどまる。リモート配布（Phase 4 相当）が届くまで
ローカル layout が事実上の production 経路である以上、その生成手段の不在は想定の隙間であり、
本 ADR は ADR 000065 の Filter Dev Kit を production/CI 側へ延長して修正する（amends）。

併せて §3.5: `plecto validate` は「artifact を要しない全 fail-closed 静的検査」（ADR 000046 の
`nginx -t` 型）であり、署名検証・digest pin 照合は起動/リロード時に初めて走る。CI が「この
manifest + この layout は本番でロードできるか」を事前に知る手段が、実際にプロセスを起動する
以外に無かった。

外部慣習の裏付け（Web 調査、2026-07-20 取得）: OCI Image Layout / image manifest の構造は
opencontainers/image-spec が正典で、現行 layout（config `application/vnd.wasm.config.v0+json`、
layer `application/wasm`）は CNCF の Wasm OCI Artifact 慣習に一致済み。署名は base64(DER
`Ecdsa-Sig-Value`, RFC 3279) over SHA-256 が sign-blob 系の互換形式で、既存の sigstore crate
経由の実装がそのまま該当する。SBOM の binding は in-toto Statement v1 の subject digest 意味論
に一致。「機械出力は stdout・診断は stderr」は POSIX の標準ストリーム意味論に基づく CLI 慣習で、
`DIGEST=$(plecto package …)` の合成を成立させる。

## Decision

**`plecto package` を新設し、`plecto validate --resolve` と対にして「package → digest pin →
resolve 検証」の CI パイプラインを CLI の一等経路として閉じる。**

1. **`plecto package <component.wasm> --key <pkcs8.pem> --out <layout-dir> [--sbom
   <statement.json>]`** — ビルド済み component を入力に、(a) `plecto conformance` と同一の
   汎用 conformance バッテリでゲートし（非適合は署名・書き込みに一切進まない。`plecto dev` の
   リロードゲートと同じ規律）、(b) 運用鍵（PKCS8 PEM、cosign `sign-blob` スキーム）で component
   と SBOM に署名し、(c) `write_layout` で署名済みローカル OCI image-layout を書き、(d) pin
   すべき image-manifest digest **のみ**を stdout へ印字する。`plecto dev` と異なり manifest に
   触れず、watch せず、鍵を生成しない。SBOM 既定は `bound_sbom` の最小 in-toto statement、
   `--sbom` で供給者の statement に差し替え可能（subject binding は供給者責任のまま、ローダと
   `--resolve` が検証する）。
2. **`plecto validate --resolve <manifest.toml>`** — 静的検査（従来の `validate`）に加え、各
   `[[filter]]` の layout を実際に解決（digest pin 照合）し、ローダと同一の provenance ゲート
   （信頼鍵による component / SBOM 署名検証 + SBOM↔component binding）を wasmtime なし・状態
   無変更で実走する。`init()` の挙動と契約バージョン検出は対象外（それらは compile /
   instantiate を要するため、実ロード時の責務のまま）と明記する。
3. **切り出しはコード共有のために最小限**: 署名コアを `PemSigner`（中立名）として切り出し
   `DevSigner` はそれを包む dev 専用皮に（公開 API 不変・additive）。ADR 000006 の provenance
   ゲート 4 検査（SBOM 存在 / component 署名 / SBOM 署名 / binding）を
   `TrustPolicy::verify_artifact` に集約し、`Host::load` と `--resolve` が**同一関数**を通る
   （検証経路の二重実装を作らない — ADR 000006 P5 の「dev と本番で検証コードは同一」と同型の
   規律）。

検討した代替案と却下理由:

- **(A) `wkg` / 汎用 OCI ツールに委ねる（現状維持）** — ローダがローカル layout 専用である現実
  と噛み合わず、署名 layer / SBOM layer の束ね方（ADR 000006/000007 固有）は汎用ツールの管轄外。
  利用者 CI の野良実装が既に発生しており、現状維持はドリフトの放置。却下。
- **(B) `write_layout` の `doc(hidden)` を外し Rust API として公開** — embedder には既に届いて
  いる（examples 6 本が使用）。CI の主体はシェルであり、Rust API の公開は「CI から使えない」
  問題を解かない。CLI を公の顔にし、`write_layout` は内部のまま留める（DECREE §2: pub 面は
  最小・意図的に）。却下。
- **(C) KMS / 事前計算済み署名の受け入れを初回に含める** — 署名対象が component と SBOM の
  2 つあり、事前計算方式は「SBOM バイト列を先に確定させる emit モード」まで要求して初回
  スライスが肥大化する。署名部を `PemSigner` という関数境界に切ってあるため、後続で呼び出し
  点の差し替えのみで足せる。見送り（再検討条件へ）。
- **(D) `validate --resolve` で eager instantiate まで行う** — trusted フィルタの `init()` trap
  まで事前検出できる利点はあるが、wasmtime engine 構築 + `[state]` の redb open という副作用が
  「validate は何も変異しない」という ADR 000046 の契約を壊す。artifact 検証までに留める。却下。

本判断はコア原則の「安全 ×  運用の単純さ」を具体化する: 署名・digest pin・fail-closed の既存
セキュリティモデル（ADR 000006/000007）を一切緩めず、それを CI から使える形にしただけである。

## Consequences

### Pros
- 利用者 CI の layout 手書き実装（フィールドレポートの ~60 行シェル）が 1 コマンドに置き換わり、
  layout 仕様変更時の黙った破壊が消える。
- `plecto conformance` → `plecto package` → `plecto validate --resolve` → digest pin が、全て
  文書化された CLI パイプラインとして閉じる（writing-a-filter.md §5）。
- 検証経路の単一化（`TrustPolicy::verify_artifact`）により、CI 検証と実ロードの判定が構造的に
  一致する — 「validate が通ったのにロードで落ちる」不一致の芽を artifact 層について排除。
- stdout = digest のみ、という契約により `DIGEST=$(plecto package …)` がそのまま成立する。

### Cons / Tradeoffs
- `--resolve` は `init()` trap と契約バージョン非対応を検出しない — 完全な「ロード可否」では
  なく artifact 層の検証である旨をドキュメントで明示し続ける必要がある。
- 秘密鍵ファイルを CLI 引数のパスで受ける形は KMS/HSM 運用と直交しない。**再検討条件**: (a)
  KMS / CI secrets に鍵が住む利用者が実際に現れたら、事前計算済み署名の受け入れ（`--component-signature`
  / `--sbom-signature` + SBOM emit モード）を `PemSigner` 境界の差し替えとして追加する。(b)
  remote fetcher（wkg 経路）が実装されローカル layout が主経路でなくなったら、`package` の
  出力形式（レジストリ push との関係）を再評価する。
- `PemSigner` の error 型は `DevKeyError` のまま（公開 API 互換のため）— 名前と実態のズレは
  次の breaking release で `SignerError` 等への改名を検討する。
