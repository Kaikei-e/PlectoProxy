# filter.config_files でシークレットのファイル間接参照を manifest 層に導入する

## Status
accepted（2026-07-20 に grill セッションで形状（別テーブル vs `file:` prefix）を確定 → 同日
TDD（RED→GREEN）で実装着地。`cargo test --all` 54 スイート / clippy / fmt すべて green）

## Date
2026-07-20

## Affected Components
- control/manifest — `[filter.config_files]` のパース + `resolved_config`（マージ・trim・上限・fail-closed）
- control — validate / build_active への配線（リロード毎の再読込 = SIGHUP でローテーション追従）
- wit — 無変更
- docs/writing-a-filter.md — 節追加

## Context

ADR 000066 の `[filter.config]` はインラインの string→string のみで、ファイル参照・env 展開を
持たない。一方、コンテナ運用のシークレットはファイルで届くのが標準形である（Docker
Compose/Swarm の `/run/secrets/<name>`、Kubernetes の Secret volume、systemd の
`LoadCredential`）。ドッグフーディング Phase 2（docs/servey/draft_0004 §3.2）では session-auth
フィルタの HMAC 鍵を渡すために、テンプレート manifest に Docker secret を注入して完成 manifest
を volume にレンダリングする one-shot ジョブが必要になり、(a) シークレットが manifest ファイル
として実体化する、(b) ローテーションが「render 再実行 + SIGHUP」の二段になる、(c) manifest が
「静的な単一ソース」でなく生成物になる、という歪みが確認された。checked-in で ro-mount される
manifest は、まさにシークレットをインライン化してはならないファイルである。

外部慣習の裏付け（Web 調査、2026-07-20 取得）: シークレットのファイル渡しは環境変数渡しより
推奨される（環境変数は IPC 経由の露出・プロセスツリー継承・ログ混入があり、OWASP Secrets
Management Cheat Sheet / systemd ドキュメントが明文で非推奨）。`X` / `X_file` の兄弟キー対は
監視系サーバの config で広く確立した慣習で、「両方設定はエラー（相互排他）」も同慣習の定型。
欠落時に起動失敗（fail-closed）は systemd `LoadCredential`（絶対パス源）・K8s の非 optional
Secret volume と同じ姿勢であり、クレデンシャルの合計 1 MB 上限という前例も systemd にある。
**trim 挙動は標準化されていない**（末尾改行のみ除去 / raw bytes / 両端 trim の三様が併存）ため、
選んだ姿勢を自前で明文化する必要がある。K8s の Secret mount はシンボリックリンク農場
（`..data` の atomic swap）で回転するため、読み手はシンボリックリンク追従が必須である。

## Decision

**`[filter.config_files]` という別テーブルを導入する。キー空間は `[filter.config]` と同一、
値はパス（絶対、または manifest 相対）で、ロード/リロード毎にファイル内容を読み、マージ済み
マップを既存の `host-config::get` から供給する。WIT 契約は無変更。**

意味論（すべて fail-closed、テストで固定済み）:

1. **解決はロード/リロード時**（`build_active`）— SIGHUP がローテーション済みシークレット
   ファイルを自然に拾う。`validate_manifest` も同じ解決を実行し、artifact 不要の fail-closed
   ファイル検査群（trust 鍵・TLS 証明書・upstream CA）に加わる — CI / pre-SIGHUP で欠落が
   発覚し、最初のリクエストでは発覚しない。
2. **同一キーの二重宣言はエラー**（`config` と `config_files` は相互排他 — `X`/`X_file` 慣習の
   定型。暗黙の優先順位を作らない）。
3. **内容は UTF-8 必須・両端 trim・1 MiB 上限**。trim は「`echo` が付ける末尾改行」事故の吸収
   で、標準が無い以上 docs で明文化する（意図的な前後空白を持つシークレットはこの機構に
   乗れない、と明記）。非 UTF-8 と上限超過は `InvalidFilterConfig`、欠落・読取不能は `IoAt`。
4. **シンボリックリンクは追従する**。orchestrator の Secret mount は symlink swap で回転する
   ため追従が必須。manifest は operator 所有の trusted 入力であり、供給チェーン由来の artifact
   blob（symlink 拒否のまま、CWE-59）とは信頼モデルが異なる — この線引きを明示する。
5. **単一 open の有界読み**（上限 +1 バイトの `take`）で stat→read の race を作らない。

検討した代替案と却下理由:

- **(A) `file:` scheme prefix を既存 `[filter.config]` の値に導入** — ADR 000066 の核心
  「host は値を一切解釈しない純パススルー」に正面から反する（host が値文字列を検査して scheme
  判定することになる）。さらに `file:` で始まる既存の正当な値を黙って壊す互換性問題を生む。
  別テーブル方式なら解釈されるのは**セクション名**であって値ではなく、`[filter.ratelimit]` 等の
  既存の型付きセクションと同じ建て付けで 000066 の stance を保てる。却下。
- **(B) 独立したホスト capability（secret-file 能力）として WIT に追加** — ADR 000081 再検討
  条件 (c) が予約する方向だが、フィルタ側 API が増え、契約の minor bump を要し、「フィルタは
  キーで設定を読む」という既存の単純なモデルを二重化する。フィルタから見て設定とシークレットを
  区別する必要は現状なく、manifest 層の解決で足りる。mTLS 鍵・STEK などホスト自身の秘密が
  同じ形を要求し始めたら改めて評価する（再検討条件へ）。却下。
- **(C) env 展開（`${VAR}` / `_FILE` env 変数）** — 環境変数経由はシークレット運搬手段として
  それ自体が非推奨（上記 Context）。manifest の「静的な単一ソース」性も、プロセス環境という
  外部状態への依存で弱まる。却下。

本判断は [[000066]] を amends する: 「host は値を解釈しない」は**値について**維持され、
`config_files` はセクション名による構造の宣言である。フィルタ側の必須キー検証が `init()` 責務
である点、「宣言必須」設定を持つフィルタは `isolation = "trusted"` で運用すべき点も 000066 の
まま変わらない。

## Consequences

### Pros
- moka-1 型の「render ジョブ + volume 実体化 + 二段ローテーション」が不要になり、manifest が
  静的な単一ソースに戻る。シークレットは manifest に一切現れない。
- ローテーションが「ファイル差し替え + SIGHUP」の一段に縮む。
- validate 段で欠落・衝突が検出され、fail-closed ファイル検査の既存ファミリと一貫する。
- WIT 無変更のため、既存フィルタ・多言語ゲスト・契約バージョニングに影響ゼロ。

### Cons / Tradeoffs
- 両端 trim のため、前後空白が意味を持つシークレットは表現できない（docs に明記済み）。バイナリ
  シークレットも不可（`host-config::get` が `string` である以上の制約ではない）。
- ローテーションの反映は SIGHUP 契機であり、ファイル変更の自動検知はしない（watch を持ち込む
  ことは validate/reload の単純さと引き換えにしない）。
- **再検討条件**: (a) ホスト自身の秘密（mTLS クライアント鍵、STEK 等）が同じファイル間接参照を
  要求し始めたら、ADR 000081 再検討条件 (c) と併せて共通のホスト能力へのくくり出しを評価する。
  (b) バイナリシークレットや非 trim の要求が実利用者から出たら、`host-config` の `list<u8>`
  変種（契約 minor bump）を評価する。
