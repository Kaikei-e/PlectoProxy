# wasmtime 47.0.1 へロックステップ更新し、default-on 化された GC / exception-handling proposal をエンジンで明示無効化する

## Status
accepted

## Date
2026-07-21

## Affected Components
- host runtime (crates/host) — `engine.rs` / `streaming.rs` の両 `Config` に `wasm_gc(false)` / `wasm_exceptions(false)` を追加
- workspace manifest (plecto/Cargo.toml) — `wasmtime` / `wasmtime-wasi` / `wasmtime-wasi-http` を 46.0.1 → 47.0.1 にロックステップ bump
- componentize 経路 (crates/plecto, crates/host build.rs) — `wit-component` を 0.252 → 0.254 に bump

## Context

wasmtime 47.0.0（2026-07-20）がリリースされた（47.0.1 は `wasmtime-cli` の crates.io 公開修正のみで、
ライブラリ実質差分は 47.0.0）。[[000021]] で乗った wasmtime 46 系（M3 Stage 1 ホスト async 化）の
後継 major であり、リリースノート精査（一次ソース: GitHub releases）で Plecto に関係する差分は
以下に絞られた（調査日 2026-07-21、いずれも確定事実）:

- **GC proposal と exception-handling proposal がデフォルト有効化**（#13594 / #13603）。
  現行の `build_engine()` は両者を明示設定していなかったため、無変更で bump すると
  untrusted フィルタに対して両 proposal の wasm 機能面が**暗黙に開く**。
- `wasi-common` / wasi-threads の削除（#13558）— Plecto は `wasmtime-wasi` /
  `wasmtime-wasi-http` のみ使用しており無影響。
- MSRV 1.94.0 — 現行ツールチェーン 1.97.1 で充足。
- component-model-async に破壊的変更なし（`Accessor::poll_ready_for_concurrent_call` の追加のみ）。

デフォルト有効化は Tenet「deny-by-default capability」（[[000006]] のセキュリティモデル）と
緊張関係にある: フィルタが触れる能力はホストが明示的に貸したものに限る、という原則は WIT 契約の
host-API 面だけでなく、エンジンが guest に公開する wasm 機能面にも適用されるべきである。

## Decision

1. **`wasmtime` / `wasmtime-wasi` / `wasmtime-wasi-http` を 47.0.1 へロックステップ bump** する
   （部分 bump による二重 wasmtime スタックの禁止は workspace manifest の既存方針）。
   併せて componentize 経路の `wit-component` を 0.254 へ更新する（[[000010]] の
   `ComponentEncoder` レシピは無変更で通る）。
2. **GC / exception-handling proposal を両エンジン（`engine.rs` の sync host、`streaming.rs` の
   streaming host）で明示無効化**する。フィルタゲストは wasm32-unknown-unknown（header-only）で
   両機能を使わず、ホストが貸すと決めていない機能面を untrusted コードに開かない。
   これは 46 系と完全同挙動でもあり、bump の挙動差分をゼロに保つ。

検討した代替案:

- **upstream デフォルトに追従（両 proposal 有効のまま）** — 却下。将来の GC 言語ゲスト
  （Kotlin/Dart 等）受け入れは楽になるが、現時点で貸す判断をしていない機能面を暗黙に開くのは
  deny-by-default に反する。貸すなら専用 ADR で明示的に決める。
- **無効化に加え、GC/EH 命令を含む component のロード拒否 conformance テストを追加** — 今回は
  見送り。wasmtime 自身が feature-off 時の validation reject を保証しており、ホスト側の
  二重テストは bump のスコープを超える。必要になれば polyglot conformance に追加する。

具体化する Tenet: 「deny-by-default capability」を、host-API 面に加えて**エンジンの wasm 機能面**
にも適用すると明文化した（upstream がデフォルトを動かしても Plecto の露出面は変わらない）。

## Consequences

### Pros
- untrusted フィルタへの露出面が 46 系と完全に同一のまま 47 系へ乗れる（挙動差分ゼロの bump）。
- upstream のデフォルト変更に対して受動的でなくなる: 機能面の開放は常に明示的な設定と ADR を通る。
- `cargo test --all`（54 スイート）/ `cargo clippy --all-targets --all-features -- -D warnings` /
  `cargo fmt --check` すべて green を確認済み。

### Cons / Tradeoffs
- GC 言語ゲスト（Kotlin/Dart/Scheme 等の GC proposal 依存ツールチェーン）はこのままでは動かない。
  **再検討条件**: GC 言語ゲストの需要が出たとき、`wasm_gc` を貸す判断を専用 ADR で行う。
- exception-handling ABI を使う guest ツールチェーンが現れた場合も同様に明示判断が要る。
- wasmtime major を追うたびに「新たに default-on 化された proposal はないか」をリリースノートで
  確認する運用負荷が残る（bump 手順の一部とする）。
